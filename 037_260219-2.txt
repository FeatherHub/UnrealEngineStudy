1 선언부 I 껍질
UINTERFACE()
URogueInteractionInterface: public UInterface
  UObject 시스템을 위한 메타데이터 허브
  리플렉션 시스템에게 “이런 인터페이스가 존재한다”고 알려주는 타입 표지(tag)
  “이 오브젝트는 이 인터페이스를 구현한다”를 UClass 메타데이터로 기록

2 선언부 II
// Macro Annotation 없음
class ACTIONROGUELIKE_API IRogueInteractionInterface // extends 없음
{
	GENERATED_BODY() // 인터페이스 메타데이터

public:
	UFUNCTION(BlueprintNativeEvent) // C++ & BP
  UFUNCTION(BlueprintImplementableEvent) // BP Only
	void Interact();
};

C++이 쓰는 몸통
액터가 실제로 구현해야 할 가상 함수들을 정의

3 구현부
UCLASS()
class ACTIONROGUELIKE_API ARogueItemChest : public AActor, public IRogueInteractionInterface
{
    GENERATED_BODY()

public:
    // 규약: _Implementation 접미사를 붙여서 구현
    virtual void Interact_Implementation() override;
};

---

왜 직접 호출(OtherActor->Interact())하지 않는가?

만약 블루프린트에서만 인터페이스를 구현했다면, 
C++ 레벨에서 실제 IMyInterface* 포인터가 nullptr일 수 있다.
(C++의 I-클래스 포인터로 캐스팅했을 때 nullptr이 반환)

그래서 Execute_Interact 중간자 레이어를 통해 내부적으로 ProcessEvent 경로로 빠질 수 있도록 기계장치를 셋업
그 결과 C++로 구현되었든 블루프린트로 구현되었든 상관없이 정확한 구현 지점을 찾아가도록 설계

if (BP_오버라이드_있음) {  // 런타임 분기
    Use_Channel_2();  // 느린 채널
} else {
    Use_Channel_1();  // 빠른 채널
}

Interface vs Component
Component
  Tick, Replication, Serialization, Editor Visibility 등이 기본적으로 따라옴 (필요 없을 때도)
  별도의 UObject 인스턴스 생성
  순수한 행동 계약에 쓰면 과도한 무게
  모든 액터가 똑같은 UHealthComponent 로직을 공유할 때 유리

Interface
  클래스에 붙은 **'표식(Tag)'** 
    Type 상속 - 고유한 형태를 획득 (형태Type은 구멍이자 열쇠)
    Unrelated 클래스(AActor, UDataAsset, UUserWidget) 사이에도 동일한 계약을 부여할 수 있음
  **'가상 함수 테이블(vtable)의 확장'
    한 Actor가 10개 인터페이스를 동시에 구현해도 메모리·성능 영향 미미
  각자 자기만의 방식(Implementation)으로 반응할 자유
    "데미지를 입는다"는 행위는 같지만, '바위'가 부서지는 방식과 '몬스터'가 아파하는 방식이 완전히 다를 때 유리

UHT

.h 파일을 파싱해서 리플렉션 메타데이터 생성
생성물
  *.generated.h (StaticClass, DECLARE_xxx 등)
  *.generated.cpp (Z_Construct_ 함수들)

Annotation Macro 와 UHT 가 협동하는 방법
  GENERATED_BODY, UCLASS, UINTERFACE, UPROPERTY, UFUNCTION
 
---
UClass
  하나의 UClass 인스턴스가 C++ 클래스 하나를 완전히 설명
  내부에 PropertyLink (연결 리스트)와 FunctionMap을 가지고 모든 멤버를 관리

UFunction
  함수 하나를 표현
  파라미터도 FProperty로 관리
  호출 메커니즘
    C++ 경로 Func->Invoke(this, Stack, Result)
    ProcessEvent이 우선순위 판단해서 BP가 있으면 BP 실행, 없으면 C++ 실행

FProperty (UE5+) / UProperty (legacy)
  변수 하나를 표현
  FIntProperty, FObjectProperty, FStructProperty 등 수십 개 서브클래스
  Offset, Size, Flags, ContainerPtrToValuePtr 등 실제 메모리 접근 정보 보유
---

class MyClass {
    int value;
};
MyClass obj;

obj는 자신이 MyClass인지 알까? (No)
obj는 자신이 value라는 멤버를 가졌는지 알까? (No)
obj는 자신을 파일에 저장할 수 있을까? (No)

Why? 타입 정보가 컴파일 이후 사라짐 (런타임에 존재하지 않음)

class UMyClass : public UObject {
    GENERATED_BODY()
    
    UPROPERTY()
    int32 Value;
};
UMyClass* obj = NewObject<UMyClass>();
obj->GetClass()->GetName();
obj->GetClass()->FindPropertyByName("Value");  
FArchive& Ar << obj;

---

Reflection이 없으면 엔진은 “이 객체가 어떤 UObject를 참조하고 있는지” 알 수 없다.
GC가 도는 순간, 엔진은 해당 UClass의 PropertyLink(또는 FField 체인)을 순회

TObjectPtr<T> (UE5)
  Reflection + GC write barrier를 동시에 제공
  UPROPERTY로 선언된 TObjectPtr은 GC 도중에도 즉시 reachable로 마킹

UE4
UClass → PropertyLink
UStruct → Field
→ UProperty와 UFunction이 완전히 다른 계층 구조를 가짐

UE5
모든 “필드”(프로퍼티, 함수, enum 등)를 FField라는 공통 베이스로 통합
Why?
  UProperty와 UField 별도 할당 → 메모리 단편화
  VS 연속 메모리 블록에 배치
