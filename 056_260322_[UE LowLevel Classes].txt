FEngineLoop Implements the main engine loop
게임 프레임 갱신의 소유자는 UGameEngine::Tick()
현재 시뮬레이션 우주의 갱신 책임자는 UWorld::Tick()

상위 루프가 UWorld를 매 프레임 방문한다
Engine이 정한 프레임 루프 안에서 tick entry를 받아요

“Engine이 Owner/Outer 트리를 재귀적으로 DFS 순회하는 방식”
  우선순위 개념 결여

등록된 tick function들을, 월드가 프레임 단계별 스케줄에 따라 실행하는 방식

Actor/Component ticking을 tick group으로 설명하고
각 tick group이 그 그룹에 속한 actor와 component의 tick을 모두 끝낸 뒤 다음 그룹으로 넘어간다
tick prerequisite를 통해 같은 그룹 안에서도 선후관계를 지정할 수 있다

FEngineLoop
→ UGameEngine::Tick()
→ 현재 UWorld::Tick()
→ 그 월드에 등록된 tickable 대상들(actors, components, world subsystems 등)을 
  tick group 순서와 prerequisite dependency에 따라 실행
→ 물리/카메라/스트리밍 같은 프레임 단계들을 사이에 끼워 넣음

phase + dependency 기반의 frame scheduler예요

tick group은 주로 물리 시뮬레이션과의 상대적 시점을 정하기 위해 존재하고, 
그룹은 순차적으로 진행돼요
TG_PrePhysics, TG_DuringPhysics, TG_PostPhysics 같은 그룹

“지금은 물리 전에 돌아야 하는 대상들”
→ “지금은 물리와 병행 가능한 대상들”
→ “지금은 물리 결과를 본 뒤 돌아야 하는 대상들”
→ “그 뒤 타이머/카메라/스트리밍”

의미 있는 프레임 단계들로 잘려 있다

int32 GuardedMain(const TCHAR* CmdLine)
{
    // ...초기화...
    
    while (!GIsRequestingExit)
    {
        EngineTick();  
    }
    
    // ...해제...
}

void EngineTick()
{
    // 1. 플랫폼 이벤트 수집
    FPlatformMisc::PumpMessages();      // OS 메시지 펌프
    
    // 2. 시간 계산
    GEngine->UpdateTimeAndHandleMaxTickRate();
    
    // 3. 엔진 틱
    GEngine->Tick(FApp::GetDeltaTime(), bIdleMode);
    
    // 4. 렌더 커맨드 플러시
    RHITick(FApp::GetDeltaTime());
}

void UEngine::Tick(float DeltaSeconds, bool bIdleMode)
{
    // 1. 모든 FWorldContext를 순회
    for (FWorldContext& Context : WorldList)
    {
        UWorld* World = Context.World();
        
        // 2. World에게 틱을 위임
        World->Tick(LEVELTICK_All, DeltaSeconds);
    }
    
    // 3. 스트리밍 관련 업데이트
    // 4. 가비지 컬렉션 조건 검사
    // 5. 통계 업데이트
}

void UEngine::Tick(float DeltaSeconds, bool bIdleMode)
{
    // 1. 모든 FWorldContext를 순회
    for (FWorldContext& Context : WorldList)
    {
        UWorld* World = Context.World();
        
        // 2. World에게 틱을 위임
        World->Tick(LEVELTICK_All, DeltaSeconds);
    }
    
    // 3. 스트리밍 관련 업데이트
    // 4. 가비지 컬렉션 조건 검사
    // 5. 통계 업데이트
}

void UWorld::Tick(ELevelTick TickType, float DeltaSeconds)
{
    // ── Phase 1: 입력 ──────────────────────────────
    // PlayerController들이 입력을 처리
    
    // ── Phase 2: 틱 그룹 순차 실행 ─────────────────
    RunTickGroup(TG_PrePhysics);
        // 물리 시뮬레이션 전 로직
        // 이동 의도 결정, 애니메이션 업데이트
    
    RunTickGroup(TG_StartPhysics);
        // Chaos 물리 시뮬레이션 시작 (비동기)
    
    RunTickGroup(TG_DuringPhysics);
        // 물리 시뮬레이션과 병렬 실행 가능한 것들
    
    RunTickGroup(TG_EndPhysics);
        // 물리 결과 수집 대기
    
    RunTickGroup(TG_PostPhysics);
        // 물리 결과를 읽는 것들 (IK, 래그돌 보정)
    
    RunTickGroup(TG_PostUpdateWork);
        // 모든 것이 끝난 후 마지막 정리
    
    // ── Phase 3: 충돌 이벤트 디스패치 ──────────────
    // Chaos가 수집한 충돌 정보를 Actor에게 전달
    // OnComponentHit, OnActorOverlapped 여기서 발화
    
    // ── Phase 4: 레벨 스트리밍 ─────────────────────
    // 로드/언로드 요청 처리
}

void RunTickGroup(ETickingGroup Group)
{
    // FTickTaskManager가 이 그룹에 등록된
    // 모든 FTickFunction을 실행
    
    TickTaskManager.RunTickGroup(Group, ...);
}

상주 스레드
프로세스
├── Game Thread (Main Thread)
│       앞서 다룬 것. GuardedMain 루프.
│
├── Render Thread
│       FRenderingThread::Run()
│       Game Thread가 생성한 렌더 커맨드를 소비하는 루프
│
├── RHI Thread (선택적)
│       Render Thread가 생성한 RHI 커맨드를 소비
│       GPU 제출 직전 단계
│
└── Audio Thread
        FAudioThread::Run()
        사운드 믹싱, 스패셜라이제이션 처리

아래 세 스레드는 독립 루프가 아니라 파이프라인이다.

Game Thread
    │
    │  FRenderCommandFence / 렌더 커맨드 큐
    ▼
Render Thread
    │
    │  FRHICommandList
    ▼
RHI Thread
    │
    │  GPU 드라이버 API (DX12, Vulkan, Metal)
    ▼
GPU (별도 하드웨어)

Game Thread와 Render Thread는 보통 1~2프레임 차이로 비동기 실행된다. 
Game Thread가 프레임 N+1을 계산하는 동안 Render Thread는 프레임 N을 렌더링하고 있다.

mid-tick preemption은 아니에요.
**“호출된 함수 본문을 중간에 잘라서 다음 프레임에 이어서 실행”**

레벨 스트리밍 중 actor registration 단계에 프레임당 최대 몇 ms를 쓰게 할지 정하는 값이에요
Level Streaming Hitching Guide

round-robin time slicing
“World A를 2ms 돌리고 끊고, World B를 2ms 돌리고, 다시 World A로”

상태 일관성
Tick 중간에 중단하면 World가 반쯤 업데이트된 상태가 된다
Actor A는 이번 프레임 위치로 이동했고, Actor B는 아직 이전 프레임 위치에 있다
물리, 충돌, 렌더링이 모두 이 불일치를 봐야 한다
처리가 복잡해진다.

Tick 중간 중단 시
    Actor A: 프레임 N+1 상태
    Actor B: 프레임 N 상태    ← 불일치
    물리 시뮬레이션: 어느 상태 기준으로 계산?

**Unreal의 Mass Entity + 멀티스레드 Processor**
각 Processor가 Time Budget을 인식하고
이번 프레임에 처리할 Entity 수를 조절한다

렌더링이 메인 스레드를 독점하면 사용자 입력, 스크롤, 애니메이션이 모두 멈춘다.

게임 World는 다르다.

프레임은 원자적이어야 한다
    물리가 절반만 진행된 World
    AI가 절반만 결정한 World
    충돌이 절반만 해결된 World
    → 이 상태를 렌더링하면 아티팩트
    → 이 상태에서 다음 로직이 읽으면 버그

React의 Virtual DOM은 중간 상태를 안전하게 보존할 수 있는 구조로 설계되었다. 
UE의 World 상태는 그렇지 않다. 
수백 개의 Actor가 서로의 상태를 읽으면서 업데이트하는 구조에서 중간 중단은 불일치를 만든다.

"Game Thread 바깥으로 작업을 꺼내라"는 방향으로 설계되었다.

지금 Unreal의 일반 gameplay 코드는 대체로 이런 계약을 전제해요.

Tick() 안에서 바로 위치를 바꿈
바로 damage 적용
바로 delegate broadcast
바로 actor spawn/destroy
바로 overlap/hit 반응
바로 component state 변경
바로 replicated state 수정

즉 계산과 커밋이 한 몸이에요.

읽기 단계: 오직 현재 확정 상태 N만 읽는다
계산 단계: 변경 의도만 기록한다
커밋 단계: 모든 의도를 합쳐서 N+1을 확정한다
이벤트 단계: 커밋 이후에만 dispatch한다

엔진을 transaction/commit 모델로 재해석하는 일이에요.

연속 신호 (Continuous)
    아날로그 스틱: -1.0 ~ +1.0 사이의 실수값
    마우스 이동: 매 프레임 델타값

이산 신호 (Discrete)
    버튼: Pressed / Released / 두 상태만

BindAxis의 콜백 시그니처:

매 프레임 들어오는 -1.0 ~ 1.0
방향과 세기가 하나의 float에 인코딩된다

BindAction의 콜백 시그니처
