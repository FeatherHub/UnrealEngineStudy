UE4 초기: AnimGraph와 AnimBlueprint의 도입

AnimBlueprint는 애니메이션 로직을 Blueprint 그래프로 표현

AnimBlueprint
  ├─ EventGraph   ← 상태 변수 계산 (IsMoving? Speed? IsFalling?)
  └─ AnimGraph    ← 그 변수들을 받아서 포즈를 생성

State Machine이 AnimGraph 안에 들어왔다
Blend Space가 도입됐다
AnimNotify가 공식화됐다
  애니메이션 타임라인의 특정 시점에 이벤트를 발생시키는 메커니즘

UE4 중기: 보조 시스템들

Skeleton과 SkeletalMesh의 분리가 정착했다
하나의 Skeleton 에셋을 여러 SkeletalMesh가 공유할 수 있데 되었다

IK(Inverse Kinematics) 노드들이 AnimGraph에 통합됐다
  Two-Bone IK로 팔다리 끝점을 월드 공간의 특정 위치에 붙이는 것이 가능해졌다
  계단 위에 발을 정확히 붙이거나, 아이템을 손으로 집는 것

Montage 시스템
  애니메이션 클립을 State Machine 밖에서 재생하고 제어하는 메커니즘
  공격, 상호작용, 피격 반응 같이 "일회성으로 발동하는" 애니메이션들이 Montage로 분류됐다

State Machine           Montage
지금 어떤 상태인가      지금 이 동작을 실행하라
지속적, 선언적          일회성, 명령적

Physics Asset과 Ragdoll이 정교해졌다
  사망 시 물리 기반 래그돌로 전환
  애니메이션과 물리를 블렌딩하는 Physical Animation 기능이 추가

  Control Rig는 리깅 로직 자체를 UE 안에서 Blueprint 스타일 그래프로 정의할 수 있게

DCC 시대:

UE5

Sequencer + Control Rig + AnimBlueprint의 통합이 더 깊어졌다
  Sequencer에서 Control Rig를 직접 키프레이밍할 수 있게 됐고, 
  이것이 인엔진 애니메이션 제작 파이프라인의 실질적 완성이다

Motion Warping이 공식화됐다
  State Machine과 Montage는 애니메이션 클립의 루트 모션이 고정되어 있어서, 
  문을 여는 동작이 항상 같은 거리에서 실행되어야 했다. 
  Motion Warping은 루트 모션을 런타임에 변형해서 
  캐릭터가 실제 문 위치에 정확하게 맞춰 동작하도록 한다.

Motion Matching이 도입됐다
  State Machine은 "지금 이 상태이면 이 애니메이션을 재생하라"는 규칙 기반 시스템이다. 
  상태가 늘어날수록 State Machine이 폭발적으로 복잡해진다.
  대량의 애니메이션 데이터를 데이터베이스로 두고, 
  현재 캐릭터의 운동 상태(위치, 속도, 예측 궤적)와 가장 유사한 포즈를 매 프레임 검색한다

State Machine:           Motion Matching:
규칙 → 애니메이션        현재 상태 → DB 검색 → 가장 유사한 포즈
경우의 수를 모두 정의    데이터가 경우의 수를 스스로 커버
           
**Full Body IK (FBIK)**와 IK Rig/IK Retargeter가 도입됐다
  서로 다른 체형의 스켈레톤 사이에 애니메이션을 자동으로 이식하는 도구다

뼈의 움직임에 따라 메시 버텍스를 변형
skinning

메시의 각 버텍스는 하나 이상의 뼈에 대한 skin weight
이 버텍스는 상완 뼈에 0.7, 전완 뼈에 0.3의 영향을 받는다

이 연산이 GPU에서 실행된다
스키닝은 버텍스 셰이더에서 수행

모든 상위 층의 일은 결국 본 행렬 배열을 생산하는 것으로 수렴
오디오에서 모든 상위 층의 일이 "최종 PCM 버퍼를 채우는 것"으로 수렴했던 것과 같아

---

스키닝이 작동하려면 "뼈가 몇 개이고, 어떻게 연결되어 있고, 기본 자세에서 각 뼈가 어디에 있는가"가 정의되어야 해.

이것이 USkeleton 에셋 안의 FReferenceSkeleton이야

FReferenceSkeleton의 구조

FMeshBoneInfo 배열
각 원소가 뼈 하나의 메타데이터(FName Name, int32 ParentIndex)를 담아

FTransform 배열
각 원소가 해당 뼈의 레퍼런스 포즈에서의 로컬 변환(부모에 대한 상대적 위치, 회전, 스케일)이야. 


이 두 배열의 인덱스가 일치해서, BoneIndex = 3이면 
FMeshBoneInfo[3]에서 이름과 부모를 읽고, FTransform[3]에서 레퍼런스 로컬 변환을 읽어

---

USkeleton이 "뼈 이름과 계층의 정규적 정의"로서, 
메시 USkeletalMesh와 애니메이션 UAnimSequence 사이의 
compatibility contract 제공

버텍스 위치가 에셋에 고정
런타임에 메시 형태가 변하지 않아

메시의 형태(shape)가 정적
문이 회전해서 열릴 수 있지만, 문 자체의 메시 형태(버텍스 배치)는 변하지 않아

내부에 뼈도 없고, 스킨 웨이트도 없고, 애니메이션 연결도 없어

GPU에 버텍스 버퍼를 한 번 올리면 매 프레임 변경할 필요 없이 그대로 드로우 콜만 날리면 돼
인스턴싱(같은 메시를 수천 개 찍기)도 쉬워

FStaticMeshRenderData
  LOD별 버텍스/인덱스 버퍼
  섹션(머티리얼 슬롯)
  콜리전 데이터

컴포넌트는 UStaticMeshComponent

아티스트가 수동으로 만드는 경우

LOD 0은 50,000 삼각형, LOD 1은 10,000 삼각형, LOD 2는 2,000 삼각형 

아티스트가 각 LOD에서 어떤 디테일을 보존하고 어떤 걸 버릴지 직접 판단


엔진이 에디터 타임에 자동 생성하는 경우

UStaticMesh 에디터에서 "Generate LOD" 버튼
엔진이 LOD 0의 메시를 입력으로 받아서 폴리곤 축소(polygon reduction) 알고리즘을 돌려 하위 LOD를 만들어

USkeletalMesh에서도 같은 기능이 있고, 추가로 뼈 제거(bone removal)도 지원

게임 상태에서 드로우 콜까지 많은 결정이 있고

첫째, 카메라에 안 보이는 건 안 그려야 해
  visibility determination
  프러스텀 컬링
  오클루전 컬링
  거리 컬링

둘째, 보이는 것들 중에서 그리는 순서를 정해야 해
  불투명한 오브젝트는 앞에서 뒤로 그리는 게 효율적
  반투명 오브젝트는 뒤에서 앞으로 그려야 정확한 블렌딩이 돼
  이 **정렬(sorting)**이 렌더 스레드의 일

셋째, 같은 머티리얼/셰이더를 쓰는 오브젝트들을 묶어야 해
  GPU의 상태 변경(셰이더 교체, 텍스처 바인딩 교체)이 비싸
  셰이더 A를 쓰는 오브젝트 50개를 연속으로 그리고, 그 다음 셰이더 B를 쓰는 30개를 연속으로 그린다

넷째, 각 오브젝트에 대해 "무엇을 어떻게 그릴지"를 결정
  이 메시의 어떤 LOD를 쓸지, 
  어떤 셰이더 변형(permutation)을 쓸지(라이트맵이 있는가, 스킨닝인가, 인스턴싱인가), 
  어떤 텍스처 밉맵 레벨을 쓸지
