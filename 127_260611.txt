물리스레드가 시점 0,3,6,9 에 돌았고 지금은 시점 10이라고 하자.
시점 10 에서의 physics simulated Actor 의 Transform 은 
물리스레드 시점 3과 6의 결과를 보간한 값을 사용하고 있다

게임스레드가 보는 건 최신 물리 상태(9)가 아니라 6만큼 과거(4)입니다. 
multiplier가 2인 이유는 안전 마진입니다
1이면 렌더 시간이 최신 결과 직전에 붙어서, 물리스레드가 한 스텝이라도 늦으면 보간할 "다음 결과"가 없어 끊기기 때문에, 
항상 완료가 보장된 두 결과 사이에 머물도록 한 스텝 더 뒤로 물러나 있는 겁니다

---

게임스레드는 렌더스레드보다 1프레임 이상 앞서지 못한다 (r.OneFrameThreadLag, FFrameEndSync)

---

스왑체인의 "최대 프레임 레이턴시"

CPU(렌더/RHI 스레드)는 GPU보다 앞서 달리면서 커맨드를 계속 제출할 수 있는데, 
무한정 앞서가게 두면 GPU 큐에 미실행 프레임이 쌓이고 
입력→화면 지연이 계속 늘어납니다

GPU가 아직 소화 못 한 프레임이 한도만큼 쌓여 있으면, 다음 Present() 호출이 블록됩니다

RHI 스레드가 멈추면 렌더스레드가 밀리고, 
렌더스레드가 밀리면 게임스레드도 (1프레임 lag 규칙 때문에) 멈춥니다

이게 PIE에서 깨졌습니다.
멀티플레이어 PIE는 한 프로세스에 게임 세션 N개인데, 
엔진은 하나뿐이니 세션 상태가 인스턴스끼리 섞였죠. 
그래서 4.4에 UGameInstance가 발명됩니다

---

① 에디터·커맨드릿처럼 GameInstance가 0개인 구성에서도 필요한가?
② 그 상태를 변경하는 동사는 누구의 코드인가(엔진 기계 vs 게임 코드)?
③ 게임 개발자에게 노출할 공개 표면인가, 엔진의 내부 장부인가?
④ 상태가 정말 세션에 귀속되는가, 아니면 키로 유도 가능한 무상태 서비스인가?

---

UGameInstance는 게임마다 서브클래스하고 BP에 노출되는 공개 객체입니다. 
전송 계층의 내부 장부를 거기 두면 엔진의 기계장치가 게임 코드의 손 닿는 곳에 놓입니다

---

GameViewportSubsystem이 엔진급인 이유: 세션 고유의 상태가 없습니다
AddWidget의 장부는 위젯을 키로 하고, 대상 뷰포트는 위젯→월드→뷰포트로 매번 유도됩니다. 
이런 무상태성 서비스를 GameInstance마다 N개 복제하면 얻는 게 없습니다


ReplaySubsystem이 인스턴스급인 이유
"지금 이 판을 녹화 중인가"는 PIE 인스턴스 A는 녹화하고 B는 안 할 수 있어야 하는, 
인스턴스 별 상태입니다

Mobility

Static Light
= 런타임에 움직이지도, 바뀌지도 않는다고 엔진에게 약속함
- Lightmass 빌드 때 lightmap에 baked됨
- dynamic shadow를 직접 만들지 않음

Stationary Light
= 런타임에 위치는 안 바뀌지만, 일부 속성은 바뀔 수 있다고 엔진에게 약속함
- static geometry에는 baked lighting 기여 가능
- movable object에는 dynamic shadow 가능


Static Mesh Actor / Static
- 런타임에 이동/회전/스케일 변경하지 않을 오브젝트
- precomputed lightmap/shadow에 참여 가능

Static Mesh Actor / Stationary
- Movable Light와 함께 쓸 때 cached shadow map을 재사용할 수 있음

The scene stores renderer state that is independent of any view or frame,

FNamedTaskThread : public FTaskThreadBase
virtual void ProcessTasksUntilQuit(int32 QueueIndex) override
{
  do
  {
    ProcessTasksNamedThread(QueueIndex, bAllowStall);
  } while (...); 
}

uint64 ProcessTasksNamedThread(int32 QueueIndex, bool bAllowStall)
{
    Task->Execute(NewTasks, ENamedThreads::Type(ThreadId | (QueueIndex << ENamedThreads::QueueIndexShift)), true);
}

bool TryExecuteTask()
{
    ExecuteTask();
}

virtual void FTaskBase::ExecuteTask() = 0;

TGraphTask<TTask>::ExecuteTask
virtual void ExecuteTask() override final
{
  TTask* TaskObject = TaskStorage.GetTypedPtr();
  TaskObject->DoTask(TaskObject->GetDesiredThread(), GraphEventRef);
}
