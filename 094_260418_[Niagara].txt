Interpolated Spawn Mode

문제:  고속으로 이동하는 Emitter(예: 총알, 레이저) 에서 파티클을 spawn 하면 
frame 단위로 뭉쳐서 스폰되어 점선처럼 보인다

해결: 한 frame 내에 N 개를 spawn 할 때 각 파티클에 sub-frame time offset 을 부여하고, 
이전 프레임과 현재 프레임의 Emitter transform 을 보간한 위치에서 각 파티클을 탄생시킨다

---

Sim Target (CPUSim / GPUCompute Sim)

CPU 모드에서 script 들은 Niagara VM bytecode 로 컴파일되어 
Game Thread(혹은 worker thread) 에서 실행

GPU 모드에서는 같은 module stack 이 HLSL compute shader 로 transpile 되어 GPU 에서 실행된다
  CPU Event 통신 불가능: GPU sim 은 CPU 로 readback 하지 않으므로 CPU 쪽 로직에서 파티클 상태를 알 수 없다
  Allocation Mode: GPU 는 정적 버퍼 할당
  GPU 에서는 wavefront 스케줄링 비결정성 때문에 Determinism 보장이 약해짐
  일부 Data Interface 는 CPU 에서만 작동

Requires Persistent IDs

Niagara 의 기본 자료구조는 SoA 이고 파티클이 죽으면 배열의 마지막 파티클로 덮어쓰는 방식(swap-and-pop) 으로 compaction 한다. 
이 과정에서 파티클의 배열 index 는 의미를 잃는다. 
Persistent ID 는 이와 별개로 유지되며 "파티클의 동일성(identity)" 을 배열 위치와 분리하는 장치다.


Ribbon renderer(이전 파티클과 현재 파티클을 순서대로 연결해야 함)나 
particle-to-particle event reference 가 필요할 때 사용

Parameter Namespace

PARTICLES.*     — 파티클별 속성 (SoA 배열 원소)
MODULE.*        — 한 module scope 내 지역변수
EMITTER.*       — Emitter 단위 변수 (1 인스턴스당 1개)

FScene 

태어나는 스레드는 Game Thread
그러나 "태어난 후의 삶"은 Render Thread 

UWorld::InitWorld() → UWorld::CreateScene() 
  GetRendererModule().AllocateScene(...) 를 호출
  FScene 객체를 힙에 할당

"World 가 태어날 때마다 FScene 이 하나 함께 태어난다"
한 UWorld 당 하나의 FScene, 1:1 관계


소유자는 UWorld 이다
관리(상태 조작)하는 스레드는 Render Thread 이다.

FScene 내부의 Primitives 배열에 항목을 추가/제거하는 것, Octree 에 삽입/제거하는 것, Light interaction 을 갱신하는 것

Game Thread 는 변경을 직접 수행하지 않고 ENQUEUE_RENDER_COMMAND 매크로로 "이런 변경을 해달라" 는 람다를 
Render Thread 의 command queue 에 투입한다
Render Thread 가 자기 tick 에서 그 람다를 실행하며 비로소 FScene 이 변한다

이 구조의 핵심은 "소유권(lifetime)" 과 "상태 수정권(mutation)" 이 서로 다른 스레드에 분리되어 있다는 점

deferred renderer 는 한 frame 에 여러 pass 를 순차 실행하며

techniques

의존관계 정의의 두 위치

의존자 쪽 선언(bottom-up): A 가 "나는 B 가 필요하다" 고 말함
관리자 쪽 선언(top-down): System 이 "A 는 B 뒤에 와야 한다" 고 말함


의존 관계가 System 에 선언되어 있다면, Emitter 를 다른 System 으로 옮길 때 개발자가 새 System 에서 의존 관계를 다시 셋업 해야 한다.
이는 Emitter 의 재사용을 저해한다

Emitter 가 자기 의존을 스스로 들고 다니면, 어느 System 에 drop-in 하더라도 
"이 Emitter 를 쓰려면 그 전에 X 라는 이름의 Emitter 가 있어야 한다" 는 요구가 자동으로 따라온다
