Mesh Shader (DX12 Ultimate SM6.5, Vulkan VK_EXT_mesh_shader; NVIDIA Turing+, AMD RDNA2+)
"GPU-driven rendering 으로의 패러다임 전환

IA/VS/HS/DS/GS 전체를
compute 스타일의 두 스테이지로 대체

Meshlet / Cluster data
        |
Optional Amplification Shader / Task Shader
        |     <- 어떤 meshlet을 처리할지, 몇 개의 mesh shader group을 띄울지 결정
        |
Mesh Shader
        |     <- 작은 meshlet 단위로 정점과 primitive를 직접 출력
        |
Rasterizer
        |
Pixel Shader

---

Mesh Shader 는 사실상 rasterizer 에 직접 출력하는 compute shader 입니다
하나의 threadgroup 이 협력해서 meshlet(정점 ≤ 256, 프리미티브 ≤ 256 의 작은 메시 조각) 하나를 만들어 
정점 배열 + 인덱스 배열을 on-chip 메모리에 직접 써서 rasterizer 로 넘깁니다

정점 fetch 는 고정 기능이 아니라 셰이더 코드가 버퍼에서 직접 읽습니다 (push model → pull model)

Amplification Shader 는 그 앞단에서 "Mesh Shader threadgroup 을 몇 개 띄울지"를 결정합니다
여기서 meshlet 단위 frustum culling, backface cone culling, occlusion culling, LOD 선택을 수행하고 
살아남은 meshlet 만 발사합니다.

---

Input Assembler가 고정적으로 vertex/index를 읽어주는 구조에서 벗어난다
compute shader처럼 thread group 단위로 실행되고, 그 group이 직접 버퍼를 읽고, 
필요한 정점과 삼각형 목록을 만들어 rasterizer로 넘깁니다

보이지 않는 geometry를 더 일찍, 더 큰 단위로 버리기 위해서

---

예전 방식에서는 정점 셰이더가 정점을 다 처리한 뒤에야 
primitive 단위의 여러 판단이 가능해지는 경우가 많았습니다.

반면 Mesh Shader에서는 보통 모델을 작은 cluster, 즉 meshlet으로 미리 쪼개둡니다. 
그러면 GPU는 meshlet 하나를 보고 
“이 묶음 전체가 화면 밖인가?”, “너무 작아서 낮은 LOD를 써도 되는가?”, “삼각형 일부를 버릴 수 있는가?” 
같은 판단을 rasterization 전에 할 수 있습니다.

---

기존 사고방식
  정점 -> 정점 -> 정점 -> 삼각형 조립 -> 나중에 버림

Mesh Shader 사고방식
  meshlet 하나를 먼저 판단
    보이지 않음 -> 통째로 버림
    보임        -> 필요한 정점/삼각형만 만들어 보냄

---

차이는 실행 단위

Vertex Shader: 입력 정점 하나를 출력 정점 하나로 변환

Mesh Shader: thread group이 작은 meshlet을 협력해서 처리하고, 출력 vertex와 primitive list를 직접 만든다

---

스레드당 정점 1 개라는 모델의 폐기

VS 는 정점 하나를 받아 정점 하나를 내는 1:1 함수였지만, 
Mesh Shader 는 threadgroup 이 groupshared memory 와 협력 연산으로 
지오메트리 조각 전체를 생성하는 N:M 모델

컬링이 파이프라인 안에서 메모리 왕복 없이 이루어지고,
per-primitive attribute 출력, 절차적 지오메트리 생성, 커스텀 정점 압축 해제 같은 것이 자연스러워집니다

Nanite 는 메시를 cluster(≈ 128 triangle) 로 쪼개 GPU 에서 계층 컬링하는데, 
작은 삼각형은 compute shader rasterizer 로, 
큰 삼각형은 하드웨어 rasterizer 경로로 보냅니다

큰 삼각형은 하드웨어 rasterizer 의 전용 유닛(fine raster, depth, attribute interpolation)이 압도적

TBDR 
Tile-Based Deferred Rendering

모바일 GPU의 아키텍처

모바일에서 DRAM 대역폭이 비싼 자원이라, 
데스크톱처럼 삼각형을 바로 DRAM 프레임버퍼에 그리는(IMR, Immediate Mode) 대신 
화면을 타일(예: 32×32 픽셀)로 쪼갭니다

1단계(binning pass): 모든 지오메트리의 위치만 먼저 변환해서
"이 삼각형은 어느 타일에 걸치는가"를 타일별 리스트에 기록

2단계: 타일 하나씩 on-chip 메모리 안에서 rasterize·셰이딩·블렌딩을 전부 끝내고 
최종 결과만 DRAM 에 한 번 씁니다

binning 은 픽셀 작업 전에 전체 지오메트리 위치를 알아야 하는데, 
mesh shader 는 지오메트리를 파이프라인 안에서 동적으로 생성

그 출력을 binning 하려면 결국 메모리에 받아 적어야 해서 
"메모리 왕복 제거"라는 mesh shader 의 핵심 이득이 사라집니다

Meshlet

엔진/빌드 파이프라인이 오프라인에서 만듭니다
DCC 툴은 관여하지 않고, GPU 가 자동 변환해 주지도 않습니다

임포트나 쿠킹 시점에 도구(meshoptimizer 라이브러리, DirectXMesh, Unreal 은 임포트 시 Nanite 빌드)가 
인덱스 버퍼를 분석해서 "정점 ≤ 64, 삼각형 ≤ 126" 같은 제약으로 클러스터를 자르고, 
각 클러스터의 정점 목록·삼각형 목록·바운딩·normal cone(backface culling 용)을 별도 데이터 구조로 저장

DX12/Vulkan 의 PSO (Pipeline State Object)

DX11 시절의 "슬롯에 바인딩" 모델과 달리
셰이더 조합 + rasterizer state + blend state + vertex layout 을 통째로 컴파일해 둔 불변 객체

Mesh shader 가 추가한 건 PSO 의 두 번째 종류

전통형: InputLayout + VS (+ HS/DS/GS) + PS
메시형: (AS) + MS + PS — InputLayout 없음, 정점 fetch 는 MS 코드가 직접

어느 쪽을 쓸지는 draw 단위 결정 
같은 프레임, 같은 render pass 안에서 두 종류가 공존

전통형은 DrawIndexedInstanced, 메시형은 DispatchMesh(x, y, z)

Nanite 의 하드웨어 rasterization 경로가 바로 이 메시형 PSO

---

Nanite 는 컬링을 통과한 cluster 들을 두 개의 리스트로 나눠 담고, 
SW 리스트는 compute PSO + Dispatch 로, 
HW 리스트는 메시형 그래픽스 PSO + draw 로 별도 제출

두 경로 모두 같은 visibility buffer (UAV) 에 64-bit atomic 으로 쓰기 때문에, 
depth 비교가 atomic InterlockedMax 안에서 일어나 결과가 자연스럽게 합쳐집니다

meshlet 하나에 배정되는 건 스레드 1 개가 아니라 threadgroup 1 개

meshlet 이 정점 ≤ 128 로 잘려 있는 이유가 바로 이것인데,
threadgroup 크기를 128 로 잡으면 스레드 1 개 = 정점 1 개가 됩니다

mesh shader 가 곧 정점 처리 단계이고, 출력은 rasterizer 로 직행합니다

---

GS 는 스레드 1 개가 프리미티브 1 개를 받아 가변 개수의 출력을 순차 생산했고, 
출력량 상한을 모르니 하드웨어가 최악 케이스만큼 버퍼를 예약하거나 메모리로 흘려야 했으며, 
출력 순서 보장까지 얹혀 직렬화가 생겼습니다

---

Mesh shader 는 정반대 설계

출력 상한을 컴파일 타임에 선언 (정점 ≤ 256, 프리미티브 ≤ 256)
on-chip 공간 정적 할당 가능, threadgroup 전체가 협력해서 출력 배열을 병렬로 채움, 
증폭이 필요하면 별도 스테이지(Amplification Shader)가 threadgroup 개수를 늘리는 방식으로 처리

"한 스레드가 많이 만든다"를 "많은 스레드가 하나씩 만든다"로 뒤집은 것

---

SW 경로는 compute PSO 인데, compute 파이프라인에는 그래픽스 스테이지라는 개념 자체가 없습니다
그냥 스레드들이 UAV 에 쓰는 프로그램

HW 경로조차 render target 에 출력하지 않습니다. 
PS 가 ROP 대신 SW 경로와 같은 visibility buffer UAV 에 InterlockedMax 로 씁니다

GS 는 현재 거의 버려진 스테이지입니다
Metal 은 아예 지원하지 않고, UE5 도 사실상 안 씁니다.
과거 용도였던 point sprite 확장·절차적 생성은 compute + instancing 으로

HS (Hull Shader) — patch(제어점 묶음) 단위로 실행
"얼마나 쪼갤까"를 결정
edge 별 tessellation factor 를 계산해서 출력 (예: 카메라 거리 기반 LOD).

Tessellator — 고정 기능. "쪼개기 자체"를 합니다
HS 가 준 factor 대로 patch 도메인을 균일 분할해서 새 점들의 (u, v) 좌표와 연결 토폴로지를 생성합니다

---

단조증가

값이 진행 방향으로 감소하지 않는다
x₁ ≤ x₂ ⇒ f(x₁) ≤ f(x₂)

DAG 를 leaf → root 로 올라갈수록(거칠어질수록) 
cluster 의 단순화 오차가 감소하지 않도록 빌드 시 강제

child error ≤ parent error

ParentError > Threshold ∧ ClusterError ≤ Threshold

---

DAG 를 왜 만드나

cluster 를 개별 단순화하면 이웃 cluster 와 만나는 경계 정점이 어긋나 틈이 생기므로 
경계는 잠그고(lock) 내부만 단순화해야 합니다

트리 구조로 매 레벨 같은 묶음을 유지하면 같은 경계가 영원히 잠겨서 
그 edge 들은 끝까지 단순화되지 못하고 조밀한 이음새로 남습니다.

---

cluster 여러 개를 group 으로 묶고 → group 경계만 잠근 채 내부 단순화 → 결과를 새로운 cluster 들로 다시 분할 
→ 다음 레벨에서는 다르게 group 핑. 

이러면 이번 레벨의 잠긴 경계가 다음 레벨에서는 group 내부가 되어 단순화

Nanite의 사고방식

"이 view에서 보이는 적절한 cluster들만 고른다"

LOD처럼 “mesh 전체가 LOD0, LOD1, LOD2 중 하나로 바뀐다”가 아니라, 
mesh 내부의 cluster들이 화면 조건에 따라 훨씬 세밀하게 선택

같은 object 안에서도 어떤 부분은 더 자세한 cluster, 어떤 부분은 더 거친 cluster가 선택될 수 있다

기존 LOD는 보통 mesh 전체가 LOD0, LOD1, LOD2로 바뀐다
실제 화면에서는 같은 mesh 안에서도 가까운 부분, 먼 부분, 실루엣에 중요한 부분, 가려진 부분의 필요 해상도가 다르다

Nanite는 object 전체가 아니라 cluster 단위로 선택한다.

Nanite는 단순한 자동 LOD가 아니다.

---

visible cluster index 12531
 -> VisibleClustersSWHW 배열에서 cluster record 찾기
 -> 이 cluster가 어떤 Nanite resource에 속하는지 찾기
 -> 어떤 instance / primitive인지 찾기
 -> cluster data에서 triangle 47의 세 정점 index를 찾기
 -> 세 정점의 position, normal, tangent, UV 등을 decode
 -> 픽셀 위치와 depth로 barycentric을 복원
 -> material input을 계산
 -> GBuffer에 base color / normal / roughness 등을 기록

Nanite도 rasterization을 한다. 
다만 rasterization의 출력이 바로 material shaded pixel이 아니라 visibility buffer

기존 Rasterization의 목적
 -> pixel shader를 실행할 fragment를 만든다

Nanite Rasterization의 목적
 -> 각 pixel에 보이는 cluster/triangle identity를 쓴다
