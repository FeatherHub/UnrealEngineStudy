Game Thread
Render Thread
RHI(Render Hardware Interface) Thread

Game Thread의 월드 상태와 Render Thread가 바라보는 월드 상태가 
다른 시점의 스냅샷

Game Thread가 Frame N을 시뮬레이션하는 동안, 
Render Thread는 Frame N-1의 데이터를 렌더링하고 있을 수 있습니다. 
이것은 네트워킹에서의 클라이언트-서버 상태 불일치와 구조적으로 동일한 문제이며, 
UE는 이를 Scene Proxy 패턴으로 해결합니다

---

UWorld → FScene: 게임 월드에서 렌더 월드로의 변환

Game Thread 쪽의 UWorld는 게임플레이 관점의 월드입니다
이 UWorld에 대응하는 렌더링 측 표현이 FScene입니다
UWorld의 시각적 측면만을 추출한 투영(projection)이라고 볼 수 있습니다
FScene은 오직 "이것을 어떻게 화면에 그릴 것인가"에 필요한 정보만 담고 있습니다.

---

UPrimitiveComponent → FPrimitiveSceneProxy: 프록시 패턴
Game Thread에서 시각적 표현을 가진 모든 컴포넌트(UStaticMeshComponent, USkeletalMeshComponent 등)는 
UPrimitiveComponent를 상속합니다. 
각 UPrimitiveComponent는 자신의 렌더링 측 분신인 FPrimitiveSceneProxy를 생성합니다.

이 프록시는 단순한 데이터 복사본이 아닙니다. 
Render Thread가 안전하게 접근할 수 있는 독립적인 객체로, 
Game Thread의 상태 변경이 명시적인 업데이트 명령을 통해서만 프록시에 반영됩니다. 
이것은 일종의 명령 기반 동기화(command-based synchronization) 패턴입니다. 
Game Thread는 "Transform이 바뀌었다"는 명령을 큐에 넣고, Render Thread가 이를 자신의 타이밍에 소비합니다.

---

FSceneRenderer: 프레임 렌더링의 오케스트레이터

FSceneRenderer(그리고 그 주요 구현체인 FDeferredShadingSceneRenderer)는 
한 프레임의 렌더링 전체를 조율하는 역할을 합니다

이것은 다음과 같은 질문에 순차적으로 답합니다.

카메라에 무엇이 보이는가(Visibility/Culling)
각 오브젝트를 어떤 순서로 그릴 것인가(Draw Policy/Sorting)
각 렌더 패스에서 무엇을 할 것인가(Render Pass Execution)
최종 이미지를 어떻게 합성할 것인가(Post Processing)

---

FMaterial → FMaterialRenderProxy: 머티리얼 시스템

UMaterial/UMaterialInstance는 Game Thread 측 에셋이고, 
실제 렌더링에 사용되는 것은 FMaterialRenderProxy입니다. 

머티리얼 에디터에서 노드 그래프로 구성한 것은 궁극적으로 HLSL 셰이더 코드로 컴파일되며, 
이 셰이더는 각 렌더 패스에서 해당 머티리얼이 적용된 메시를 그릴 때 GPU에 바인딩됩니다.

---

RHI (Render Hardware Interface): 플랫폼 추상화 레이어

RHI는 DirectX 11/12, Vulkan, Metal 등 플랫폼별 그래픽 API를 추상화하는 계층입니다
RHI가 그래픽 API에 대한 **가상 함수 테이블(vtable)**과 같다는 것입니다

FDynamicRHI 인터페이스를 통해 "텍스처 생성", "드로우 콜 발행" 같은 연산을 호출하면, 
런타임에 바인딩된 플랫폼별 구현체가 실제 API 호출로 변환합니다.

RHI Thread는 Render Thread가 생성한 명령 리스트를 받아 
실제 GPU 드라이버에 제출하는 역할을 하며, 
이로써 Render Thread까지도 드라이버 호출의 블로킹으로부터 해방됩니다

---

UE의 기본 렌더링 경로는 Deferred Shading
이것이 의미하는 바는, "이 픽셀에 어떤 머티리얼 속성이 있는가"를 먼저 기록한 뒤, 
나중에 "이 픽셀에 빛이 어떻게 영향을 미치는가"를 계산한다는 것입니다

Early Z / Depth Prepass
먼저 깊이 값만 렌더링하여 나중에 불필요한 픽셀 셰이딩을 방지합니다. 
이것은 "무엇이 앞에 있는가"라는 질문에만 집중하는 단계입니다.

Base Pass
각 오브젝트의 머티리얼 속성(색상, 노멀, 러프니스, 메탈릭 등)을 여러 장의 GBuffer 텍스처에 기록
이 시점에서는 라이팅을 전혀 고려하지 않습니다.
기하학적 복잡성과 조명 복잡성을 완전히 분리(decouple)하는 것입니다.

Lighting Pass
GBuffer에 기록된 정보를 기반으로, 씬의 각 광원이 각 픽셀에 미치는 영향을 계산합니다. 
여기에는 Shadow Map 생성(별도 패스), Screen Space Reflections, 
그리고 Lumen 같은 Global Illumination 시스템이 관여합니다.

Translucency 
반투명 오브젝트는 Deferred 파이프라인의 예외 케이스입니다
블렌딩이 필요하므로 Forward 방식으로 별도 처리되며, 
이것이 반투명 오브젝트가 성능적으로 비싼 이유 중 하나입니다

Post Processing
톤매핑, 블룸, DOF, 모션 블러, Anti-Aliasing(TSR/TAA) 등이 최종 이미지 위에 적용됩니다. 
이 단계는 이전 단계들이 생산한 여러 버퍼를 입력으로 받아 최종 화면을 합성합니다.

---

물리 시뮬레이션이나 애니메이션이 오브젝트의 Transform을 변경하면, 
이 변경은 Scene Proxy 업데이트 명령으로 변환되어 Render Thread에 전달됩니다

텍스처 스트리밍 시스템은 카메라 위치와 오브젝트의 화면 점유 크기를 기반으로 
텍스처 밉맵 레벨을 동적으로 조정합니다
렌더링 시스템이 "이 텍스처가 얼마나 크게 보이는가"라는 정보를 제공하면, 
에셋 시스템이 적절한 해상도의 텍스처를 메모리에 유지하는 협업이 이루어집니다.

APlayerCameraManager가 카메라 위치와 FOV를 결정하면 
이것이 FSceneView로 변환되어 렌더러에 전달됩니다.
HUD나 위젯 시스템(Slate/UMG)은 3D 렌더링 위에 오버레이되는 
별도의 2D 렌더링 경로를 사용합니다.

---

물리적 기반 GPU 하드웨어
대규모 병렬 SIMD 프로세서
수천 개의 작은 코어가 동일한 명령을 서로 다른 데이터에 동시에 실행합니다

GPU가 일을 하려면 세 가지가 필요합니다
처리할 데이터(버텍스, 텍스처 등)가 GPU 메모리에 있어야 하고,
어떻게 처리할지를 기술하는 셰이더 프로그램이 바인딩되어야 하며
파이프라인의 고정 기능 상태(블렌딩 모드, 뎁스 테스트 방식 등)가 설정되어야 합니다

이 세 가지를 묶어서 하나의 Draw Call을 발행하면, GPU가 실제로 픽셀을 생산합니다.

이 하드웨어 특성이 위의 모든 소프트웨어 레이어의 설계를 근본적으로 제약합니다.
GPU 자체의 연산 능력은 엄청나지만, **상태 전환(state change)**이 비쌉니다. 
셰이더를 교체하거나, 텍스처를 바꿔 바인딩하거나, 렌더 타겟을 전환하는 것은 
파이프라인을 비우고(flush) 다시 채우는 것을 의미합니다. 
UE의 상위 레이어들이 수행하는 정렬, 배칭, 머징의 궁극적 목적은 이 상태 전환을 최소화하는 것입니다.

---

Graphics API — 드라이버와의 계약

DirectX 12, Vulkan, Metal은 CPU에서 GPU에게 작업을 지시하는 프로토콜입니다

리소스 관리
GPU 메모리에 버퍼와 텍스처를 할당하고, CPU에서 GPU로 데이터를 전송합니다
D3D12/Vulkan 같은 현대 API에서는 메모리 힙 관리를 애플리케이션이 직접 해야 합니다
이것은 OS의 가상 메모리 관리를 애플리케이션 레벨에서 수행하는 것과 본질적으로 같은 문제입니다.

Pipeline State Object (PSO)
셰이더, 블렌딩 상태, 뎁스/스텐실 상태, 래스터라이저 상태 등을 하나의 불변 객체로 미리 컴파일합니다
D3D11 시절에는 이런 상태들을 개별적으로 설정했고, 드라이버가 내부적으로 조합을 캐싱했습니다
현대 API에서는 이 조합을 명시적으로 선언하게 함으로써, 
드라이버의 추측을 제거하고 검증 비용을 로드 타임으로 옮겼습니다

Command Buffer / Command List
GPU에 보낼 명령들을 버퍼에 기록한 뒤 일괄 제출합니다
이것이 중요한 이유는, 명령의 기록과 실행이 시간적으로 분리된다는 것입니다. 
CPU는 "미래에 GPU가 할 일"을 기술하고, GPU는 자신의 타이밍에 이를 실행합니다. 

이 비동기적 관계는 UE의 멀티스레드 렌더링 아키텍처의 하드웨어적 근거입니다.

동기화 프리미티브
Fence, Barrier, Semaphore 등은 CPU-GPU 간, 또는 GPU 내부의 작업 순서를 보장합니다.
특히 Resource Barrier(D3D12) / Pipeline Barrier(Vulkan)는 
"이 텍스처를 렌더 타겟으로 쓰던 것을 이제 셰이더에서 읽겠다"는 
전환을 명시적으로 선언하는 것으로, 
이전 API에서 드라이버가 암묵적으로 처리하던 것을 표면화한 것입니다.

---

RHI (Render Hardware Interface) — UE의 플랫폼 추상화

RHI는 위의 Graphics API들을 단일한 C++ 인터페이스 뒤로 숨기는 추상화 계층입니다
추상화의 수준 선택이라는 설계 문제가 여기 있습니다
추상화가 너무 높으면 플랫폼별 최적화를 활용할 수 없고, 너무 낮으면 추상화의 의미가 사라집니다.

UE의 RHI는 현대 API(D3D12/Vulkan)의 모델에 더 가깝게 설계되어 있습니다. 
명시적 리소스 상태 전환, Command List 기반 명령 기록, PSO 개념을 RHI 수준에서 노출합니다.
이는 구세대 API(D3D11, OpenGL)에서는 RHI 내부의 구현체가 이 명시적 모델을 암묵적 모델로 역변환해야 한다는 것을 의미합니다.

주요 RHI 추상화 객체들의 역할은 이렇습니다

FRHIResource
모든 GPU 리소스의 기반 클래스입니다. 
FRHITexture, FRHIBuffer, FRHIShader 등이 여기서 파생됩니다. 
이것은 UObject가 엔진 전체의 리소스 관리 기반을 제공하는 것과 평행한 패턴인데, 
RHI 리소스는 UObject 시스템 바깥에서 자체적인 참조 카운팅 수명 관리를 수행합니다. 

GPU 리소스의 수명은 게임플레이 객체의 수명과 다른 규칙을 따르기 때문입니다. 
GPU가 아직 해당 리소스를 참조하는 명령을 실행 중일 수 있으므로, 
CPU 측에서 참조를 해제해도 즉시 파괴할 수 없습니다

FRHICommandList 
Render Thread가 GPU 명령을 기록하는 인터페이스입니다
SetRenderTarget, DrawPrimitive, Dispatch 같은 호출이 여기에 기록되며, 
이 명령 리스트가 RHI Thread로 전달되어 실제 API 호출로 번역됩니다.

FGraphicsPipelineState 
셰이더 조합, 렌더 상태를 하나의 PSO로 캡슐화합니다
UE는 이 PSO를 캐싱하고 사전 컴파일하는 시스템을 갖추고 있으며, 
이것이 게임 첫 실행 시 "셰이더 컴파일" 과정의 실체입니다.

---

RDG (Render Dependency Graph) — 선언적 렌더 패스 관리

전통적 접근에서는 프로그래머가 
"먼저 이 패스를 실행하고, 리소스 배리어를 넣고, 다음 패스를 실행하고..."라고 
순서를 명시적으로 기술했습니다

RDG에서는 각 렌더 패스가 
"나는 이 리소스를 읽고, 저 리소스에 쓴다"고 의존성만 선언합니다. 
실제 실행 순서, 리소스 배리어 삽입, 임시 리소스의 할당과 해제는 
RDG가 그래프를 분석하여 자동으로 결정합니다

임시 렌더 타겟이 어느 패스까지 필요한지를 그래프가 알고 있으므로, 
더 이상 필요 없는 시점에 즉시 메모리를 회수하거나 다른 용도로 앨리어싱(aliasing)할 수 있습니다. 
이것은 컴파일러의 레지스터 할당 문제와 동일한 구조입니다. 
변수(리소스)의 live range를 분석하여 물리적 저장소(GPU 메모리)를 최적으로 재사용하는 것입니다.

리소스의 읽기/쓰기 의존성을 
그래프가 전역적으로 볼 수 있으므로, 
필요한 최소한의 배리어만 최적의 위치에 삽입합니다

최종 출력에 기여하지 않는 패스를 자동으로 제거할 수 있습니다. 
특정 기능이 비활성화되어 그 패스의 출력을 아무도 소비하지 않으면, 
해당 패스와 그 입력 생산 패스까지 연쇄적으로 제거됩니다.

makefile에서 의존성을 수동 관리하던 것을 
빌드 시스템이 의존성 그래프를 자동 분석하는 것으로 전환한 것과 같습니다. 
또는 수동 메모리 관리에서 
소유권 기반의 자동 수명 관리(RAII/스마트 포인터)로의 전환과도 유사합니다.

---

Mesh Draw Pipeline — 드로우 콜 생성의 기계

"이 메시를 이 머티리얼로 그려라"라는 높은 수준의 의도를 실제 GPU 명령으로 변환하는 곳입니다

FMeshBatch
Scene Proxy가 생성하는 것으로, 
"이 버텍스 버퍼와 인덱스 버퍼를, 이 머티리얼로, 이 Transform에서 그려라"라는 정보 묶음입니다

FMeshDrawCommand
FMeshBatch에서 생성되는, 
실제 드로우 콜에 1:1로 대응하는 구조체입니다.
PSO, 셰이더 바인딩, 버텍스 스트림 등 
GPU에 제출하기 직전의 모든 정보를 담고 있습니다. 

FMeshPassProcessor
각 렌더 패스(Base Pass, Depth Pass, Shadow Pass 등)마다 하나씩 존재하며, 
FMeshBatch를 받아서 해당 패스에 맞는 FMeshDrawCommand를 생성합니다. 
동일한 메시+머티리얼 조합이라도 
패스에 따라 다른 셰이더와 상태를 사용하므로, 
패스별로 다른 MeshDrawCommand가 필요합니다. 

그림자 패스에서는 깊이만 쓰면 되므로 
머티리얼의 복잡한 셰이더가 필요 없고, 
Base Pass에서는 전체 GBuffer 출력이 필요합니다.

이 레이어에서의 핵심 최적화는 **정렬(sorting)**입니다. 
FMeshDrawCommand들을 PSO, 머티리얼, 버텍스 버퍼 기준으로 정렬하면, 
연속된 드로우 콜 간의 상태 전환을 최소화할 수 있습니다. 
이것은 Layer 0에서 언급한 "상태 전환 비용"이라는 하드웨어 제약이 
소프트웨어 레이어의 설계를 결정하는 직접적인 예시입니다.

---

Scene Proxy와 Visibility — 월드의 렌더링 표현

이 레이어는 "무엇을 그릴 것인가"를 결정하는 곳입니다.

FPrimitiveSceneProxy는 앞서 설명한 Game Thread-Render Thread 경계의 다리 역할을 합니다.
각 프록시는 자신의 바운딩 박스, 렌더링 속성(캐스트 섀도, 수신 섀도, 라이팅 채널 등), 
그리고 FMeshBatch를 생성하는 능력을 갖고 있습니다.

Visibility 판정은 다단계로 이루어집니다. 
먼저 거리 기반 컬링(Distance Culling)이 가장 먼 오브젝트를 제거하고, 
절두체 컬링(Frustum Culling)이 카메라 시야 밖의 오브젝트를 제거하며, 
오클루전 컬링(Occlusion Culling)이 다른 오브젝트에 완전히 가려진 것을 제거합니다. 

UE5에서는 이 중 상당 부분이 GPU로 이동했습니다(GPU-driven rendering). 
Nanite의 경우 클러스터 단위의 GPU 오클루전 컬링을 수행하여, 
메시의 보이지 않는 부분까지 삼각형 수준에서 제거합니다

이 레이어에서 중요한 개념은 Relevance입니다. 
각 프리미티브가 각 뷰에 대해 
"이 패스에 참여해야 하는가"를 판정하는 과정으로, 
단순히 보이느냐/안 보이느냐를 넘어서 
"그림자를 드리우는가", "커스텀 뎁스에 쓰는가", "속도 버퍼에 기여하는가" 등 패스별 참여 여부를 결정합니다.

고수준 렌더링 시스템

Nanite
전통적 렌더링에서는 아티스트가 LOD를 수동 제작하고 엔진이 거리에 따라 전환했습니다. 
Nanite는 이 전체 과정을 자동화하면서 근본적으로 다른 접근을 취합니다. 
메시를 클러스터 계층 구조(DAG)로 분해하고, 
GPU에서 프레임마다 각 클러스터의 화면 기여도를 평가하여 적절한 LOD를 선택합니다. 
소프트웨어 래스터라이저까지 포함하여, 하드웨어 래스터라이저가 비효율적인 극소 삼각형을 자체 처리합니다. 
이것은 Mesh Draw Pipeline을 부분적으로 우회하는, 사실상 병렬적인 렌더링 경로입니다.

Lumen 
간접 조명 계산을 위한 하이브리드 시스템으로, 
화면 공간 추적(Screen Space Tracing)과 월드 공간 추적을 결합합니다. 
내부적으로 씬의 간소화된 표현(Surface Cache, Mesh Distance Fields, Voxel Lighting)을 유지하며,
이 표현들이 실제 씬 프록시와 동기화되어야 합니다. 
Lumen은 RDG의 선언적 패스로 구현되어, 여러 중간 버퍼를 생성하고 소비하는 복잡한 의존성 그래프를 형성합니다.

Virtual Shadow Maps
그림자의 해상도 문제를 가상 텍스처링으로 해결합니다. 
거대한 논리적 섀도우 맵을 작은 페이지들로 분할하고, 
카메라에서 보이는 영역의 페이지만 물리적으로 렌더링합니다. 
이것은 OS의 가상 메모리 시스템과 정확히 같은 원리로, 
논리적 주소 공간(거대한 섀도우 맵)을 
물리적 저장소(GPU 메모리의 페이지 풀)에 매핑하되, 
실제로 접근되는 페이지만 할당하는 것입니다

뇌가 이미 Deferred Renderer다
우리의 시각 피질(visual cortex)이 이미 UE와 놀랍도록 유사한 아키텍처를 사용하고 있다는 사실에서 출발합니다

망막에 도달한 빛은 먼저 V1(일차 시각 피질)에서 처리됩니다. 
V1이 하는 일은 방향(orientation), 공간 주파수(spatial frequency), 색상 대비를 추출하는 것입니다. 
여기서 핵심은, V1은 "이것이 무엇인가"를 전혀 모른다는 것입니다. 
단지 "이 위치에서 수직 경계선이 있다", "여기에 붉은/녹색 대비가 있다"는 저수준 속성만 기록합니다. 
이것은 Base Pass가 GBuffer에 노멀, 러프니스, 메탈릭을 기록하되 
최종 색상은 계산하지 않는 것과 구조적으로 동형(isomorphic)입니다

V1의 출력은 두 갈래 경로로 분기합니다. 
등쪽 경로(dorsal stream, "where pathway")는 공간적 위치와 움직임을, 
배쪽 경로(ventral stream, "what pathway")는 물체의 정체를 처리합니다. 
UE의 파이프라인에서 Velocity Buffer가 모션 블러와 TAA를 위해 별도로 생성되고, 
GBuffer의 머티리얼 속성이 물체의 시각적 정체성을 담당하는 것은 
이 이중 경로의 공학적 재현입니다.

시각 피질이 Deferred 방식을 쓰는 이유는 UE와 같습니다. 
뇌에 도달하는 시각 정보의 양은 초당 약 10^9 비트인데, 
의식적으로 처리할 수 있는 양은 초당 약 50비트에 불과합니다. 
이 엄청난 병목을 돌파하려면, 
먼저 저수준 속성을 병렬로 추출한 뒤(Base Pass), 
주의(attention)라는 메커니즘으로 어디에 연산 자원을 집중할지 결정하고(Visibility/Relevance), 
그 제한된 영역에서만 고수준 해석을 수행해야(Lighting/Shading) 합니다. 

무한한 입력을 유한한 연산으로 처리하는 문제에 대해, 
진화와 엔지니어링이 독립적으로 수렴한 해법인 셈입니다.

Nanite의 클러스터 기반 LOD 선택도 이 관점에서 새롭게 보입니다. 
인간의 망막에는 fovea(중심와)라는 작은 영역이 있는데, 
이곳에만 고밀도의 원추세포가 집중되어 있고 나머지 주변부는 해상도가 급격히 낮아집니다. 
우리가 세상을 선명하게 본다고 느끼는 것은, 
눈이 빠르게 움직이며(saccade) 관심 영역을 fovea로 가져가고, 뇌가 이를 합성하기 때문입니다. 
Nanite가 화면 기여도에 따라 클러스터별로 다른 해상도를 선택하는 것은, 
fovea-periphery 해상도 차등 전략의 일반화입니다.

---

렌더링 파이프라인의 구조 자체가 영화 후반 작업(post-production)의 공정 구조와 동형이라는 것입니다
영화 제작에서 촬영(production)은 원시 재료를 만드는 단계입니다. 카메라 앞에 배우와 세트를 두고 빛을 비추어 필름/센서에 기록합니다. 
후반 작업에서 합성(compositing), 색보정(color grading), 광학 효과(optical effects)를 겹겹이 쌓아 최종 이미지를 만듭니다.

이 후반 작업의 핵심 원칙이 **non-destructive editing(비파괴 편집)**입니다. 
원본 소스를 직접 수정하지 않고, 레이어와 패스를 분리하여 각각을 독립적으로 조정할 수 있게 합니다. 

GBuffer는 정확히 이 원칙의 구현입니다. 
Base Pass에서 각 픽셀의 "원본 소스"(노멀, 색상, 러프니스)를 별도의 버퍼에 기록해두면, 
Lighting Pass에서 다양한 광원을 적용하면서 기하학적 정보를 다시 렌더링할 필요가 없습니다. 

이것은 영화 후반 작업에서 Diffuse Pass, Specular Pass, Reflection Pass, Shadow Pass를 분리하여 렌더링하는 
"AOV(Arbitrary Output Variable) 렌더링"과 정확히 같은 구조입니다.

UE의 nDisplay 시스템과 LED 월(wall)을 이용한 In-Camera VFX는, 
렌더링 파이프라인이 영화 파이프라인을 흡수한 사례입니다. 

더 구조적인 유사성이 있습니다. 
영화에서 Dailies/Rushes란 그날 촬영분을 빠르게 확인용으로 보는 것인데, 
이때는 최종 색보정이나 VFX가 적용되지 않은 상태입니다. 
UE 에디터에서 "Unlit" 모드로 씬을 보는 것이 이에 해당합니다. 

이 두 세계에서 공유하는 통찰은, 
최종 이미지는 하나이지만 
그것을 구성하는 정보는 분해 가능해야 한다는 것, 
그래야 각 층위를 독립적으로 디버깅하고 조정할 수 있다는 것입니다.

---

수천 개의 셰이더 코어는 공장들이고, 
VRAM은 창고/물류 허브이며, 
버스(메모리 대역폭)는 도시의 도로망입니다. 
Draw Call은 이 도시에 도착하는 배송 트럭입니다

교통 공학에서 가장 치명적인 것은 **신호 전환(phase change)**입니다. 
교차로에서 동-서 방향 차선을 열었다가 남-북 방향으로 전환할 때, 
양쪽 모두 빨간불인 "전적색 시간(all-red clearance interval)"이 반드시 필요합니다. 
이 시간 동안 도로 용량은 0입니다. 
상태 전환이 많을수록 이 "죽은 시간"이 누적되어 전체 처리량이 떨어집니다.

GPU의 상태 전환(PSO 교체, 텍스처 바인딩 변경, 렌더 타겟 전환)이 비싼 이유가 정확히 이것입니다. 
파이프라인을 플러시하고 새로운 상태를 설정하는 동안 GPU 코어들은 유휴 상태가 됩니다.
이것은 교차로의 전적색 시간과 동형입니다.

교통 공학에서 이 문제를 해결하는 고전적 기법이 "Green Wave(녹색 파도)" 입니다. 
연속된 교차로의 신호를 동기화하여, 적정 속도로 달리면 계속 녹색 신호를 만나게 하는 것입니다. 
UE의 Mesh Draw Command 정렬이 정확히 Green Wave입니다. 
PSO가 같은 드로우 콜을 연속 배치하면, "상태 전환 없는 연속 처리"라는 녹색 파도가 만들어집니다.

**구도 스케치(compositional sketch)**
  캔버스에 물체의 위치와 깊이 관계만을 잡는 단계입니다
  공간적 관계만 확립하고, 색이나 질감은 아직 다루지 않습니다.
**밑칠(underpainting)**
**유약 층(glazing layers)**
  유화에서 유약 층을 여러 장 겹치면 빛이 층 사이에서 반사되어 깊이 있는 색감이 나오는 것처럼, 
  여러 광원의 기여가 누적되어 복잡한 조명 환경이 만들어집니다.
**바니시(varnish)와 액자**

유화 기법에서 이 계층적 접근이 발전한 역사적 이유가 
UE가 Deferred를 채택한 기술적 이유와 동일합니다. 

한 번에 모든 것을 칠하면(Forward의 alla prima 기법), 수정이 극도로 어렵습니다. 

빛의 방향을 바꾸고 싶으면 전체를 다시 칠해야 합니다. 
층을 분리하면, 밑칠을 유지한 채 유약 층만 조정할 수 있습니다. 

GBuffer를 유지한 채 광원만 추가/제거/수정할 수 있는 것과 같습니다

Lumen의 Global Illumination은 이 비유에서 빛의 간접 반사를 의식적으로 그리는 인상파 기법에 해당합니다. 
르네상스 이전의 화가들은 직접광만 표현했고, 
인상파에 이르러서야 "빛이 물체에 반사되어 다른 물체를 비추는" 간접광을 명시적으로 캔버스에 담기 시작했습니다. 

실시간 렌더링에서 직접광만 계산하던 것이 GI 시스템으로 진화한 것은, 
회화사에서 직접광 묘사에서 간접광 묘사로의 전환과 평행한 진화입니다.

---

수술에서의 **Triage(중증도 분류)**는 
렌더링의 Priority/Relevance 시스템과 같은 구조입니다. 

응급실에 환자 100명이 들어오면 전원을 동시에 치료할 수 없으므로, 
긴급도와 치료 효과에 따라 분류합니다. 

GPU에 드로우 콜 10,000개가 들어오면 전부를 동시에 처리할 수 없으므로, 
화면 기여도와 우선순위에 따라 어떤 것을 먼저, 어떤 해상도로 처리할지 결정합니다.

Nanite의 화면 기여도 기반 LOD 선택은 정확히 렌더링 트리아지입니다.

수술에서 가장 위험한 순간은 출혈이 발생하는 순간입니다. 
렌더링에서 이에 대응하는 것은 **프레임 예산 초과(frame budget overrun)**입니다. 
16.6ms(60fps) 안에 모든 렌더 패스가 완료되어야 하는데, 
예상치 못한 오버드로우나 셰이더 복잡도 폭발이 발생하면 프레임이 "출혈"합니다. 
수술에서 예방적 지혈 클램프를 미리 설치하듯, 
렌더링에서는 프레임 예산 모니터링, 동적 해상도 스케일링, LOD 강제 전환 같은 "예방적 클램프"를 사용합니다.

시각 신경과학은 "왜 이런 파이프라인이 필요한가"를 설명합니다. 
영화 제작은 "이 파이프라인이 어디서 왔는가"를 설명합니다. 
도시 계획은 "성능 제약을 어떻게 관리하는가"를 설명합니다. 
회화 기법은 "왜 층을 분리하는가"를 설명합니다. 
외과학은 "실패를 어떻게 방지하는가"를 설명합니다.

"전체를 한 번에 다루지 않고, 
관심사별로 분리된 패스를 순차적으로 적용하되, 
각 패스의 결과가 다음 패스의 입력이 되는 적층적(stratified) 처리"

데이터베이스 관점

UWorld는 원본 DB, FScene은 읽기 전용 물질화 뷰입니다

UWorld는 게임플레이를 위한 정규화된 원본 데이터베이스예요.
여기에는 물리, 입력, AI, 수명주기, 네트워크, 블루프린트 의미까지 뒤섞여 있습니다.

그런데 렌더러는 그런 정보 전체가 필요 없어요. 렌더러가 원하는 것은 오직
“어떤 기하를, 어떤 머티리얼로, 어떤 뷰에서, 어떤 패스에 참가시켜야 하는가”뿐입니다.

그래서 FScene은 UWorld를 그대로 복제한 것이 아니라, 
시각화에 필요한 열만 남긴 projection이고, 
더 정확히는 렌더링 전용 materialized view예요. 

FPrimitiveSceneProxy는 각 UPrimitiveComponent에서 추출된 비정규화된 읽기 모델이고요

Game Thread는 원본을 계속 갱신해요.
Render Thread는 약간 뒤처진 read replica예요.
프록시 업데이트는 락 공유가 아니라 변경 이벤트 전달이에요.
“지금 막 바뀐 Transform”이 즉시 렌더에 보이지 않을 수 있는 이유도 replica lag로 이해할 수 있어요.

왜 UE가 UObject를 그대로 Render Thread에서 읽게 하지 않는지가 명확해진다는 점이에요.
그건 “원본 OLTP 데이터베이스를 고빈도 분석 질의가 직접 두드리게 하지 않는다”와 같은 이유예요

컴파일러 백엔드 관점
렌더러는 월드를 GPU 명령으로 lowering 하는 컴파일러입니다

UPrimitiveComponent   = 고수준 소스 언어의 의미 있는 객체
FPrimitiveSceneProxy  = 렌더 친화적인 중간표현(IR)
FMeshBatch            = 아직 추상적이지만 드로우 가능성이 있는 lower IR
FMeshDrawCommand      = 거의 기계어 직전의 low-level IR
FRHICommandList       = 타깃 백엔드에 제출될 명령 버퍼
D3D12/Vulkan/Metal    = 실제 타깃 ISA에 해당하는 플랫폼별 백엔드

즉 “오브젝트를 그린다”가 아니라 “오브젝트를 특정 패스의 목표 기계에 맞는 명령 시퀀스로 낮춘다”예요

---

렌더러는 “현재 진실”을 원하지 않아요.
렌더러가 원하는 것은 동일 프레임 내내 흔들리지 않는 일관된 진실이에요.

즉 최신성보다 일관성이 중요합니다.

Game Thread가 물리, 애니메이션, 입력, AI 때문에 계속 움직이는 동안
Render Thread까지 같은 메모리를 동시에 직접 읽으면, 최신성이 아니라 불안정성만 커져요.

그래서 UE는 “원본 UObject 그래프를 직접 읽지 말고, 렌더 전용 복제 표현을 만들고, 명령으로만 갱신하라”는 원칙을 택합니다.

이건 단순한 프록시 패턴이 아니라
event-sourced replica 설계에 더 가까워요.

원본 상태는 Game Thread에 있음
변경은 이벤트 형태로 전달됨
Render Thread는 자신만의 replica를 유지함
replica는 약간 낡아도 되지만, 프레임 내부에서는 self-consistent해야 함

가우스의 놀라운 정리(Theorema Egregium)에 의해, 
곡률이 다른 표면 사이의 등거리 매핑은 존재하지 않습니다. 
즉, 어떤 지도 투영법을 선택하든 면적, 각도, 거리, 방향 중 무언가는 반드시 왜곡됩니다.
메르카토르 도법은 각도를 보존하지만 면적을 왜곡합니다. 
등적 도법은 면적을 보존하지만 형태를 왜곡합니다

원근 투영(perspective projection)은 시각적 사실감을 보존하지만 화면 가장자리에서 왜곡이 발생합니다
직교 투영(orthographic projection)은 측정 가능성을 보존하지만 깊이감을 잃습니다
VR에서의 렌즈 보정(lens distortion correction)은 
지도학에서 투영 왜곡을 역보정하는 것과 정확히 같은 연산입니다

항해용 해도는 수심과 암초를 강조하고, 지형도는 등고선과 식생을, 정치 지도는 국경을 강조합니다

GBuffer의 개별 채널이 정확히 이것입니다. 
같은 씬에 대해 Normal Map은 "표면 방향의 지도"이고, 
Roughness Buffer는 "표면 거칠기의 지도"이며, 
Metallic Buffer는 "금속성의 지도"입니다. 

하나의 픽셀에 기하학적 정보(노멀), 
물질적 정보(러프니스, 메탈릭), 
색채 정보(알베도), 
그리고 나중에 조명 정보까지 
"여러 잠재적 요소가 하나의 점으로 압축"됩니다. 
화면의 각 픽셀은 정점, 텍스처, 머티리얼, 광원이라는 여러 원천에서 온 정보가 응축된 결과물입니다.

하나의 세계에 대한 목적별 다중 투영.
Deferred Shading의 본질은 "하나의 지형을 여러 주제도(thematic map)로 동시에 제작하고, 
나중에 이 주제도들을 합성하여 최종 표현을 만드는 것"입니다.

**타일 피라미드(tile pyramid)**
전 세계를 여러 줌 레벨의 256x256 픽셀 타일로 분할하고, 현재 뷰포트에 필요한 타일만 로드합니다
줌 아웃하면 넓은 영역의 저해상도 타일을, 줌 인하면 좁은 영역의 고해상도 타일을 가져옵니다

---

전통적 LOD가 "축척별로 미리 만들어진 여러 버전의 지도"라면
(1:10,000, 1:50,000, 1:250,000 지도를 미리 제작해두고 전환), 
Nanite는 **연속적 축척 지도(continuous-scale map)**입니다. 

구글 맵에서 줌 레벨을 연속적으로 조절할 때, 
각 줌 레벨마다 별도의 지도가 존재하는 것이 아니라 
벡터 타일 기반으로 현재 뷰포트에 맞는 디테일이 동적으로 생성되는 것처럼, 

Nanite는 화면 기여도에 따라 
클러스터 단위로 연속적인 디테일 수준을 선택합니다. 

LOD 전환의 팝핑은 지도학에서 "축척 전환 시 시각적 불연속"에 해당하는데, 
Nanite는 이 불연속을 제거합니다

스레드 아키텍처

Frame N에서 Game Thread가 Tick을 돌리면서 액터의 Transform, 애니메이션 포즈, 머티리얼 파라미터 등을 갱신합니다. 
이 갱신이 끝나면 Game Thread는 FScene에 대한 업데이트 명령을 Render Thread의 명령 큐에 enqueue합니다. 
Game Thread는 이 제출 후 즉시 Frame N+1의 시뮬레이션을 시작할 수 있습니다.

Render Thread는 자신의 루프에서 이 명령들을 순차적으로 소비합니다. 
먼저 Scene Proxy 업데이트(Transform 갱신, 추가/제거)를 처리한 뒤, 
FSceneRenderer를 생성하여 실제 렌더링을 시작합니다. 

렌더링 과정에서 생성된 RHI 명령은 FRHICommandList에 기록되고, 
이것이 RHI Thread로 넘어가 실제 Graphics API 호출로 변환됩니다.

Game Thread가 너무 빨리 달려서 Render Thread보다 2프레임 이상 앞서가면, 
입력 지연(input latency)이 커지고 메모리 사용량이 증가합니다. 

이를 방지하기 위해 FFrameEndSync라는 메커니즘이 
Game Thread를 Render Thread가 특정 프레임을 완료할 때까지 대기시킵니다. 
OneFrameThreadLag가 이 동작을 제어하며, 기본값은 1(한 프레임 지연 허용)입니다.

Scene Proxy의 생성과 갱신 메커니즘

UPrimitiveComponent가 월드에 등록될 때(RegisterComponent → CreateRenderState_Concurrent), 
CreateSceneProxy() 가상 함수가 호출되어 
FPrimitiveSceneProxy 파생 객체가 힙에 생성됩니다. 
이 객체는 FScene::AddPrimitive()를 통해 FScene에 등록됩니다.

AddPrimitive 내부에서 일어나는 일이 상당합니다. 
프리미티브에 고유한 PrimitiveSceneInfo가 생성되고, 
여기에 Scene 내 인덱스(PersistentIndex)가 부여됩니다. 
이 인덱스는 FScene의 여러 병렬 배열(Primitives, PrimitiveTransforms, PrimitiveBounds, PrimitiveVisibilityIds 등)에서 
해당 프리미티브의 데이터를 빠르게 조회하는 데 사용됩니다. 
이 Structure of Arrays(SoA) 레이아웃은 캐시 효율을 극대화하기 위한 것입니다. 
프러스텀 컬링할 때 바운딩 박스만 연속적으로 순회해야 하는데, 
바운딩 박스가 별도의 연속 배열에 있으면 캐시 라인 활용률이 극적으로 올라갑니다

---

Transform 갱신 경로
오브젝트가 움직이면 UPrimitiveComponent::SendRenderTransform_Concurrent()가 호출됩니다.
이 함수는 새 Transform을 Scene Proxy에 직접 쓰지 않고, Render Thread 명령으로 제출합니다.
Render Thread 쪽에서 FScene::UpdatePrimitiveTransform()이 실행되면, 프록시의 로컬 Transform 캐시가 갱신되고, 
FScene의 PrimitiveTransforms 배열과 PrimitiveBounds 배열이 업데이트됩니다.

여기서 중요한 최적화가 있습니다. 
UE는 "이 프레임에 Transform이 변경된 프리미티브" 목록을 별도로 유지합니다. 
Visibility 계산 시 이 목록을 사용하여, 변경된 프리미티브의 오클루전 쿼리 결과를 무효화하거나, 
GPU Scene Buffer의 해당 엔트리만 선택적으로 업데이트합니다.

UE5에서 Scene Proxy 데이터의 상당 부분이 GPU 메모리에도 미러링됩니다
FGPUScene이 관리하는 구조화된 버퍼(StructuredBuffer)에 
각 프리미티브의 Transform, 바운딩 박스, 커스텀 데이터가 저장되며, 
GPU-driven 컬링과 Nanite가 이 버퍼를 직접 읽습니다. 
프리미티브가 변경되면 "dirty 리스트"에 추가되고, 
프레임 시작 시 변경분만 GPU에 업로드됩니다. 
이것이 전체 씬을 매 프레임 GPU에 재전송하는 대신, 
델타 업데이트만 수행하는 방식입니다.

Visibility 시스템

카메라 정보는 FSceneView로 캡슐화됩니다. 
여기에는 View Matrix, Projection Matrix, 뷰포트 크기, Near/Far Clip Plane, FOV가 포함됩니다. 

FViewInfo는 FSceneView를 확장하여, 
렌더링 과정에서 채워지는 추가 데이터(Visibility 결과, 동적 메시 목록, 오클루전 쿼리 결과 등)를 담습니다.

멀티플레이어 분할 화면이나 씬 캡처(SceneCapture2D)가 있으면 
한 프레임에 여러 FViewInfo가 존재합니다. 
각 뷰마다 독립적인 Visibility 판정이 수행되므로, 뷰 수가 렌더링 비용에 직접적으로 곱해집니다

FScene::Primitives 배열의 바운딩 스피어/바운딩 박스를 
뷰의 절두체 평면 6개와 교차 테스트합니다. 
이 연산은 SIMD로 벡터화되어 있으며, 한 번에 4개의 프리미티브를 동시에 테스트합니다. 
결과는 비트 배열(FSceneBitArray)에 기록되어, 이후 단계에서 비트 연산으로 빠르게 필터링할 수 있습니다.

바운딩 스피어 테스트가 먼저 수행되는 이유는 연산이 더 단순하기 때문입니다. 
스피어 중심과 절두체 평면의 거리만 계산하면 되므로, 
점-평면 거리 계산 6회로 끝납니다. 
이 테스트를 통과한 프리미티브만 더 정밀한 AABB(Axis-Aligned Bounding Box) 테스트를 받습니다
