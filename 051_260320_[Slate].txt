Slate는 UE의 실제 UI 렌더링 엔진
완전한 retained-mode UI 시스템
레이아웃 패스, 렌더링, 입력 처리를 모두 담당

그런데 Slate는 UObject와 무관하다
SWidget은 TSharedPtr로 관리되고, 
리플렉션 시스템이 모르며, GC가 추적하지 않는다. 
Blueprint는 SWidget을 건드릴 수 없다

여기서 UMG의 존재 이유가 결정된다
Blueprint와 Designer가 UI를 다루려면,
Slate 위에 UObject-compatible mirror가 있어야 한다. 
UMG는 그 mirror다

---

UWidget ↔ SWidget
UWidget은 UObject 계층에 산다
Blueprint에서 참조 가능하다

SWidget은 Slate 계층에 산다
TSharedPtr로 관리되고, 
실제 레이아웃과 렌더링을 수행한다.

---

class UWidget : public UVisual {
protected:
    TSharedPtr<SWidget> MyWidget; 
    virtual TSharedRef<SWidget> RebuildWidget(); // 서브클래스가 SWidget 생성
    
public:
    TSharedRef<SWidget> TakeWidget(); // SWidget 요청, 없으면 생성
};

UWidget이 SWidget을 소유하는 게 아니라, 
UWidget이 SWidget의 생성 레시피를 보유한다. 
실제 존재는 Slate 쪽이고, UMG 쪽은 그 존재를 기술하는 명세다.

---

UWidgetTree: UObject 측 그래프
class UWidgetTree : public UObject {
    UWidget* RootWidget; // 트리의 루트
    // 모든 자식 UWidget들을 UObject로 보유 → GC가 추적 가능
};

UMG 트리는 에디터에서도 살아있고, GC 대상이고, Serialize 가능하다. 
Slate 트리는 게임 실행 중에만 존재하고, 뷰포트가 닫히면 소멸한다.

Slate의 레이아웃 모델에서 "자식을 어떻게 배치할 것인가"는 부모가 결정하지 않는다. 
Slot이 결정한다.

UPanelWidget(Canvas, Overlay, HorizontalBox 등)은 
자식을 UPanelSlot을 통해 보유한다. 
Slot은 부모 패널 타입에 따라 다른 속성을 가진다. 
Canvas의 Slot은 Position과 Size를 가지고, 
HorizontalBox의 Slot은 Fill과 Padding을 가진다.

레이아웃 규칙은 부모 패널의 종류에 따라 달라지므로, 
규칙을 자식에 넣을 수도, 부모에 넣을 수도 없다. 
관계에 넣어야 한다. 
Slot은 그 관계를 객체로 승격시킨 것이다.

---

Invalidation: 변경이 전파되는 방식
레이아웃 관련 변경(크기, 가시성)은 부모까지 전파되어야 한다
페인트 관련 변경(색상, 텍스처)은 자신만 다시 그리면 된다.
이 둘을 구분하지 않으면 매 프레임 전체 레이아웃을 재계산해야 한다

---

좌표 모델: 절대 vs 흐름

절대 배치 (Canvas Panel)
Canvas Panel Slot은 Position, Size, Anchors, Alignment를 가진다
자식들은 서로를 모르고, 부모 공간 안에서 독립적으로 위치한다
자식이 추가되거나 크기가 변해도 다른 자식이 반응하지 않는다

흐름 배치 (Horizontal Box, Vertical Box, Grid Panel)
각 자식의 Slot은 "나는 얼마나 차지할 것인가"만 기술하고, 실제 위치는 Slate가 런타임에 계산한다.


---

Anchor 시스템: Canvas Panel의 핵심

Anchor가 Position의 기준점을 결정한다
부모 패널 공간에서 0~1로 표현되는 상대적 점이다
Anchor를 (0.5, 0)으로 설정하면 Position은 부모의 수평 중앙 상단을 기준으로 계산된다
(1, 1)로 설정하면 부모의 오른쪽 하단이 기준이다.

Min과 Max가 같은 점이면 고정 크기 위젯이 한 점에 고정된다.
Min과 Max가 다른 점이면 위젯이 부모의 영역 비율을 차지한다
부모 크기에 반응한다

Alignment(Pivot)는 별도의 개념이다

---

Size Rule: Fill / Auto / Fixed
흐름 배치 컨테이너(HBox, VBox, Grid)에서 Slot의 크기를

Fill(1)과 Fill(2) 두 자식이 있으면, 남은 공간을 1:2로 나눈다

이 세 규칙이 Slate의 레이아웃 패스에서 작동하는 순서
Auto 자식들이 먼저 크기를 요청하고,
Fixed 자식들이 고정 크기를 확보한 후, 
남은 공간을 Fill 자식들이 가중치에 따라 나눈다.

---

패널 유형마다 Slot이 제어할 수 있는 레이아웃 차원이 다르다.

Given a set of visual elements and a finite space, 
how do you determine where each element goes and how large it is?

Size negotiation: How does an element determine its size?
Position assignment: Given sizes, how do elements get placed?

bottom-up, where elements declare what space they need based on their content, 
and top-down, where containers dictate how much space is available

This tension between intrinsic and extrinsic sizing

constraint model
layout is fundamentally about satisfying constraints

Each element has minimum and maximum bounds
layout engine resolves these constraints to find valid positions and sizes.

Axes of independence let us decompose layout problems
width and height decisions can often be made separately
HorizontalBox only controls horizontal distribution and defers vertical layout to other mechanisms

content pushing outward or the container pulling inward

a protocol for negotiating size between parent and children
who initiates
how many rounds of communication happen
what information gets passed back and forth
who has the final say

---

A widget's size depends on its neighbors, which depend on their neighbors.
Space is inherently non-local - everything relates to everything. 
But computation must be local
How do you make a non-local problem tractable for local reasoning?

The constraint vs. imperative distinction
I can describe space as constraints (relative positioning, proportional sizing) 
or as a sequence of operations (place this, then that). 
These represent fundamentally different ways of thinking about space.

The reference frame problem
Position only makes sense relative to something else

whether space is the medium that holds objects, 
or whether space itself is just the web of relationships between them.

---

composability challenge is how to take local spatial rules and combine them into something globally coherent
each widget only understands its immediate context

Constraint propagation through a two-pass system works  
most layout constraints form a tree structure without cycles
but if you introduce circular dependencies, you need iterative solving or to forbid them entirely


---

assign a rectangle to every element
rectangles constrain each other
constraint propagate through a tree of elements

top-down constraints
bottom-up constraints

translating local specifications into globally consistent spatial assignments

two passes
gather and then distribute

every spatial element is simultaneously a requester (needing space) 
and a responder (fitting into available space)

2세대: 제약 기반, 흐름 기반. 역문제의 일부를 시스템이 흡수한다.

3세대: 예시 기반 추론.

시스템이 시선 이동 연구, 게슈탈트 원리, 문화적 읽기 패턴으로부터 기하를 도출한다.

Delegate 

Store, Subject, Observable, EventEmitter는 보통 무언가를 관리하거나 방출하는 쪽
Observer, Listener, Subscriber, Handler는 받거나 반응하는 쪽

Delegate는 그 둘 사이의 “관계 자체”를 하나의 값으로 만든 것
Delegate는 상태도 아니고, 이벤트도 아니고, 구독자도 아니에요

“나중에 누구를 어떤 형태로 호출할 것인가”를 캡슐화한 연결 형식

---

보통 함수 호출은 이렇게 생겨요.
내가 지금 저 함수를 안다
그래서 지금 저 함수를 직접 호출한다

그런데 delegate는 이것을 바꿔요.
지금은 누구를 호출할지 모르지만,
이 시그니처를 만족하는 누군가를 나중에 호출하겠다

즉 delegate는 미래의 호출 가능성을 현재의 값으로 가져와요.

코드가 더 이상 구체적 클래스나 구체적 함수 이름에 직접 묶이지 않고,
**“어떤 모양의 협력만 있으면 된다”**는 수준으로 추상화되기 때문이에요

delegate를 object-oriented, type-safe한 함수 포인터와 비슷하다

---

delegate는 행위의 소유권을 분리해요.
원래는 호출자가 “무엇을 할지”까지 알고 있어야 했는데,
delegate를 쓰면 호출자는 언제 호출할지와 어떤 형식으로 호출할지만 알고, 
실제로 무엇을 할지는 바깥에 위임해요

Store / Subject / EventEmitter
            │
            │  "이 일이 일어나면..."
            ▼
      Delegate / Signal / Callable Contract
            │
            │  "이 시그니처에 맞는 것을 호출"
            ▼
Observer / Listener / Subscriber / Handler

---

Delegate가 Store처럼 권위 있는 상태 저장소도 아니고, Publisher처럼 메시지 유통망의 생산자도 아니라는 점이에요
Delegate는 연결의 최소 단위예요.

Event나 Observer Pattern은 그 위에 쌓이는 더 큰 사회적 구조이고
delegate는 그 사회를 가능하게 하는 호출의 미시적 메커니즘에 가까워요.

---

개념의 혈통

1단계: 함수 포인터 / 콜백

2단계: 함수 포인터는 너무 거칠다
객체의 인스턴스 메서드, 타입 안전성, 고수준 언어의 안전한 계약
더 풍부하게 캡슐화

3단계: late binding
이제 “호출할 함수 그 자체”를 코드에 박아넣지 않고, 외부에서 주입할 수 있어요.

4단계: multicast
이 지점에서 delegate는 observer/event 세계와 직접 닿아요

개념적으로 보면 delegate의 기원은
원시적 callback + 타입 안전성 + 객체세계와 의 연결 + late binding + 1:N 호출 

---

void Shooter::Fire()
{
    sound->PlayFireSound();  // 컴파일 타임에 고정
}

DECLARE_DELEGATE(FOnFire)

void Shooter::Fire()
{
    OnFire.Execute();  // 슬롯에 무엇이 들어 있든 그것을 호출
}

---

일반 세계:  행위 = 코드에 박힌 것 (고정)
Delegate:  행위 = 변수에 담긴 것 (교체 가능)

---

인터페이스 상속 없이도 협력이 가능해진다. 
Delegate의 계약은 시그니처다. 
반환 타입과 매개변수 타입이 맞으면, 
그것이 어떤 클래스의 메서드인지는 호출자가 알 필요 없다.

---

다리는 양쪽을 잇지만,
경첩은 양쪽을 잇는 동시에 각자의 경계를 유지한 채 운동 가능성만 허용해요.

호출자는 수신자의 구체 타입 전체를 껴안지 않아도 되고
수신자는 호출자의 내부 흐름 전체를 알지 않아도 돼요
둘은 오직 “이 시점에, 이 모양으로, 이 호출이 가능하다”라는 약속만 공유해요

DECLARE_DELEGATE(FOnFire, int, Coefficient)

class Shooter {
    FOnFire OnFire;  // ← 자유 변수

Shooter는 완전한 클래스다. 컴파일된다. 인스턴스화된다
이 클래스는 행동 공간에 자유 변수를 가진 열린 시스템이다

Delegate를 가진 클래스는 고차 함수(higher-order function)다.

3 + 5는 닫힌 표현식
외부에서 아무것도 주입받지 않아도 완전하다

x² + 3x는 열린 표현식
닫히려면 그것이 채워져야 한다.

---

OOP reinforced the data/code split by tying methods to classes
making them non-values you can't store or pass around

Delegate is where that boundary starts to crack.

obj.Method()

코드와 데이터의 구분은 언제 평가되는가의 문제다.
데이터는 즉시 존재한다. 코드는 호출될 때 실행된다.

Delegate는 이 두 시제 사이에 세 번째 시제를 만든다.
지금 존재하지만, 아직 실행되지 않은 계산.

Delegate는 이것들의 OOP 방언이다.

=>
결정과 미결정의 분배 방식

인과의 사슬 안에 남겨 둔, 타입화된 빈칸

세계 = 이미 결정된 부분 + 아직 외부가 채울 부분
Delegate = 그 "채울 부분"을 형식으로 보존한 것

Event가 사건의 이름이라면,
Delegate는 사건 이후 세계가 진행될 통로의 형식이에요.

Event = 일어난 것
Delegate = 일어난 뒤 일어날 것

그래서 Event는 시제적이고,
Delegate는 구조적이에요.

Event는 과거형이고,
Delegate는 미래형이에요.

Valency (결합가)

동사 "주다"는 결합가 3을 가진다. 주는 자, 받는 자, 주어지는 것
세 자리가 비어 있다
세 자리가 채워지면 문장이 완성된다

모든 자리가 채워지면 그것은 닫힌 실체가 된다.

어떤 구조든 두 상태 중 하나다.
닫힌 구조: 모든 자리가 채워져 있다.
열린 구조: 하나 이상의 자리가 비어 있다
자리의 모양이 그것이 무엇인지를 이미 부분적으로 정의한다.

Valency는 열린 구조가 자신의 열림을 표현하는 방식이다. 
그냥 열린 것이 아니다. 몇 개의 자리가, 어떤 모양으로, 어디에 있는지를 명시한다.

perased apeiron
apeironized peras

닫기 연산이란 자리에 모양에 맞는 것을 대입하여, 열린 구조를 닫힌 구조로 변환하는 것이다.
자리가 세 개인 구조에서 하나만 채우면 자리가 두 개인 구조가 된다. 더 단순한 Valency를 가진 새로운 열린 구조다.
이것이 **커링(currying)**의 본질이다

수학의 함수는 차수 n의 Valency 구조다. 
f(x, y, z)는 세 자리를 가진 열린 구조다. 
f(1, y, z)는 하나를 채운 차수 2의 구조다.
