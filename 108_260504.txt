전체 replication loop의 진입점
매 tick 호출

UNetDriver::ServerReplicateActors()
  │
  ├─ 1. 모든 Connection에 대해 Actor Relevancy 갱신
  │      IsNetRelevantFor() 호출 → 거리/팀 기준
  │
  ├─ 2. Actor Selection (Replication Graph 로 교체 가능)
  │
  ├─ 3. Priority 정렬
  │      각 Actor에 대해 AccumulatedPriority 계산
  │      Priority = BasePriority × TimeSinceLastReplicated
  │
  ├─ 4. Bandwidth budget 내에서 상위 N개 Actor 선택
  │
  └─ 5. 선택된 Actor마다 UActorChannel::ReplicateActor() 호출


Priority는 굶주림 방지(starvation prevention) 스케쥴링 장치다. 
오래 복제되지 못한 Actor일수록 가중치가 누적되어 bandwidth를 확보하게 된다.

기본 UIpNetDriver의 gameplay traffic은 보통 UDP 기반이다.

각 UDP 패킷에는 source IP:port가 포함돼 있고, 서버는 이것으로 클라이언트를 구별한다.

SeamlessTravel은 "두 World가 동시에 존재하는 상태"를 거친다

TravelStart
    CurrentWorld (옛)         -> TransitionWorld (전환)
                              + PendingWorld (새, 로딩 중)

TravelMid
    TransitionWorld (전환 활성)
    + PendingWorld (계속 로딩)
    NetDriver는 TransitionWorld에 attach됨

TravelEnd
    PendingWorld → CurrentWorld로 승격
    NetDriver가 새 CurrentWorld에 attach

Pre-IRIS 도시

이 도시의 시민은 Actor였다. 도시는 시민들을 일렬로 세우고 매 프레임 묻는다
"당신, 변했습니까?"

시민은 답한다.
"모르겠습니다."


도시는 시민의 어제 사진을 꺼낸다. Shadow Copy라는 이름의 사진이다. 
지금의 시민과 Shadow Copy의 시민을 바이트 단위로 대조한다. 
다른 곳을 발견하면 Changelist라는 명부에 기록한다.


시민은 자신이 변했는지 모르고, 시스템도 미리 알지 못한다. 
진실은 비교를 통해서만 태어난다


한 시민에 한 명의 FObjectReplicator. 
FObjectReplicator가 시민을 관찰하고, 거울을 들고, 명부를 적고, 편지를 부쳤다. 

한 시민이 100명의 친구에게 편지를 보내려면 100명의 FObjectReplicator가 필요했다.
10,000명의 시민이 100명의 친구를 가지면 100만 명의 FObjectReplicator가. 도시는 FObjectReplicator의 무게로 가라앉았다

옛 도시에서 Actor는 자신이 무엇인지(Identity)와 자신이 어떤 상태인지(State)와 자신이 어떻게 직렬화되는지(Schema)를

NetObject        — 정체성

Protocol         — 타입의 복제 계약
NetSerializer    — 단일 필드의 변환 매뉴얼


Fragment는 인스턴스당 하나. 시민이 태어나면 생기고 죽으면 사라진다. 개체의 시간에 산다
State Buffer는 매 프레임 갱신된다. 순간의 시간에 산다.

SoA

Scope 게시판:    [1, 1, 0, 1, 0, 0, 1, ...]
Priority 게시판: [0.8, 0.0, 1.2, 0.3, ...]


CPU는 게시판을 한 번에 읽는다. 32명의 dirty 정보를 uint32 한 번 읽기로 처리한다. 캐시는 행복하다.
이것이 ECS의 정신이다. 개체가 데이터를 소유하지 않는다. 데이터가 자신의 배열에 산다.
