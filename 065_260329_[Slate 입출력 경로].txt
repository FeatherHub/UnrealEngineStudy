Phase 0: Before Any Key Is Pressed — The Tick Structure

The input flow doesn't begin with the keypress
It begins with the engine's tick loop FEngineLoop::Tick() 
It drives the entire frame

within it FSlateApplication::Tick() is called
This calls PlatformApplication->PumpMessages()
the engine asks the OS for pending input

Input processing is therefore poll-driven within a frame-driven loop, 
not interrupt-driven from the engine's perspective

later in the tick, UWorld::Tick()
which reaches APlayerController::TickPlayerInput()
accumulated input state gets dispatched to gameplay bindings

Phase 1: PumpMessages — Draining the OS Queue

FWindowsApplication::PumpMessages() calls the Win32 PeekMessage()
For each WM_KEYDOWN message, the registered WndProc callback fires.

If you pressed three keys between frames, 
all three OnKeyDown calls happen sequentially within this one pump

MessageHandler is FSlateApplication

Phase 2: FSlateApplication::OnKeyDown — The Routing Decision

Slate decides who gets the input.

determine the current focus state
maintains a FSlateUser per local user
  tracks which widget currently holds keyboard focus via a FWidgetPath.

input routing logic

Path A — UI Consumption
If a Slate widget has focus
the input event is routed down the widget path
Each widget in the path 
gets a chance to handle the event, and returns FReply::Handled().
If the input event is consumed, it does not reach gameplay layer.
This is why typing in a chat box doesn't make your character jump.

Path B — No UI Consumption
If no widget claims the event (or if the game viewport has focus)
the input falls through to the gameplay path
**Slate calls into FViewportClient::InputKey(),
which usually means UGameViewportClient::InputKey().**

Slate is not just a UI framework, but also the Input Arbiter

Phase 3: Viewport → PlayerController → PlayerInput

UGameViewportClient::InputKey() resolves which local player should receive this input
and gets the APlayerController
and calls APlayerController::InputKey().

The PlayerController delegates immediately to its owned UPlayerInput::InputKey()

the key event is not immediately dispatched to bindings.
Instead, UPlayerInput updates its internal FKeyState map
For the pressed key, it sets flags like bDown = true, EventAccumulator[IE_Pressed]++

The keypress is recorded as state, but no gameplay delegate fires yet
The reason for this buffering is frame coherence

all input for the frame should be processed 
together at a consistent point in the tick, 
not scattered across whenever PumpMessages happens to fire.

Phase 4: TickPlayerInput

Later UWorld::Tick(),
every PlayerController gets its TickPlayerInput() called
and accumulated input state becomes gameplay events.

UPlayerInput::ProcessInputStack() receives the InputComponent stack from the PlayerController

1. Axis Evaluation

For each entry in AxisMappings[], it checks whether the bound FKey is currently held
If so, it produces a float value (1.0 for digital keys) 
scaled by the mapping's Scale factor. 
Multiple keys mapped to the same axis name are summed

2. Action Evaluation

For each entry in ActionMappings[], 
it checks the EventAccumulator for IE_Pressed or IE_Released
If the "Jump" action is mapped to EKeys::SpaceBar and the accumulator shows a press event
"Jump Pressed" action event is generated.

3. Stack Walk

the generated action and axis events are walked against the InputComponent stack
For each InputComponent in the stack, 
it checks InputComponent's FInputActionBinding and FInputAxisBinding arrays for matching names.

When a match is found — say FInputActionBinding for "Jump" with IE_Pressed
the bound delegate fires, say AMyCharacter::Jump()

If that binding has bConsumeInput = true, the event is marked consumed

---

the static layers represent **ownership and abstraction boundaries**
the dynamic flow reveals **two temporal phases** that cut across those boundaries

The entire input system splits into an **ingestion half** and a **dispatch half**, 
separated by `UPlayerInput`'s `FKeyState` map 
acting as a frame-coherence buffer. 

This buffer is the single most important structural element 
for understanding the dynamic flow

---

Ingestion half
FSlateApplication::Tick()
early in the frame (which is FEngineLoop::Tick)

The path is a chain of **progressive abstraction**

each layer strips away one platform-specific detail
and hands up a more abstract representation

1 OS message queue          WM_KEYDOWN + VK_SPACE
2 FWindowsApplication       strips Win32 envelope, calls handler interface
3 FSlateApplication         routing decision: UI or gameplay?
4 UGameViewportClient       resolves local player
5 APlayerController         delegates to owned UPlayerInput
6 UPlayerInput::InputKey()  writes to FKeyState buffer

the traversal is a **write path**
nothing is *consumed*, only transformed 

**FInputKeyManager**
A global singleton that is not owned by any element
it holds maps that translates `VK_SPACE` into `EKeys::SpaceBar`

---

Dispatch half
UWorld::Tick()
later in the frame (which is FEngineLoop::Tick)

the PlayerController **actively reads** the buffer and **walks the InputComponent stack**:

APlayerController::TickPlayerInput()
  → UPlayerInput::ProcessInputStack()
      reads FKeyState buffer (the accumulated state)
      evaluates AxisMappings[]  → float values
      evaluates ActionMappings[] → discrete events
      walks InputComponent stack top-to-bottom
        → UInputComponent[n]::check bindings
          → delegate fires → AMyCharacter::Jump()

UPlayerInput, a reader of the buffer
UInputComponent is passive, it's a table being queried, not an agent
APlayerController is the orchestrator driving the whole dispatch

Why the Buffer Exists
if Jump() fired during PumpMessages()
The gameplay function might destroy an actor, 
which could modify the InputComponent stack. 
But PumpMessages is still iterating through OS messages

The buffer cleanly separates the gather invariant
(the InputComponent stack won't change while I'm recording key states)
from the dispatch invariant
(the key state map won't change while I'm walking the stack)

the EventAccumulator
a transient per-frame counter that gets cleared after dispatch
for discrete events like Actions 

different lifetimes
different temporal semantics
Axes are continuous queries
which persists across frames

Stage 0: The Electrical Event → OS Structures

The keyboard controller sends a scan code
a hardware-level identifier for the physical key
via the USB/PS2 interrupt

The Windows kernel driver translates this into a MSG structure 
and posts it into the thread message queue which is
associated with the thread that created the HWND

MSG {
    HWND   hwnd      →  the game window handle
    UINT   message   →  WM_KEYDOWN
    WPARAM wParam    →  VK_SPACE  (virtual key code, platform-defined integer)
    LPARAM lParam    →  [scan code, repeat count, extended flag, etc. packed into 32 bits]
    DWORD  time      →  OS timestamp
    POINT  pt        →  cursor position at time of event
}

엔터(Enter) 키나 숫자 패드의 엔터 키는 둘 다 VK_RETURN을 반환하지만, 
스캔 코드를 확인하면 두 키를 명확히 구분할 수 있습니다.

keyboard controller
sends a scan code, hardware identifier for the physical key
through a USB interrupt

kernel driver translates this into a MSG structure
{
    hwnd    
    message →  WM_KEYDOWN
    wParam  →  VK_SPACE (0x20)
    lParam  →  [scan code | repeat count | extended bit | previous key state | ...]
}

kernel posts this MSG into the game thread's message queue (hwnd)
(it will be found at the moment the game thread calls PumpMessage)

FWindowsApplication::PumpMessages()

{
    TranslateMessage(&msg);   // augments WM_CHAR for character input
    DispatchMessage(&msg);    // invokes associated callback
}

If three keys were pressed between frames, 
this loop runs three iterations

Inside WindowsApplication::AppWndProc, the MSG structure is decomposed
The packed lParam bitfield is split into its semantic components:

case WM_KEYDOWN:
    uint32 KeyCode    = wParam;                // VK_SPACE
    uint32 CharCode   = ...;                   // from TranslateMessage
    bool   bIsRepeat  = (lParam >> 30) & 1;    // previous key state bit

does two things: **translate** and **route**.

1 **Translation**
calls `FInputKeyManager::Get().GetKeyFromCodes(KeyCode, CharCode)`
which returns `EKeys::SpaceBar`

FSlateApplication then constructs FKeyEvent
{
    Key            →  EKeys::SpaceBar       (the UE-level key identity)
    ModifierKeys   →  {Shift=false, Ctrl=false, Alt=false, ...}
    CharacterCode  →  0x20 (' ')
    KeyCode        →  VK_SPACE              (preserved but now secondary)
    bIsRepeat      →  false
    UserIndex      →  0
}

The **second transformation**

lifted from platform vocabulary (`VK_SPACE`) 
to UE vocabulary (`EKeys::SpaceBar`)

The `FKeyEvent` also enriches the data

synthesizing context that wasn't in the original `WM_KEYDOWN` message

checks the focus state for this user via `FSlateUser` and its `FWidgetPath`

If a **UI widget** has keyboard focus
the event is routed through `FEventRouter::Route()`

The event goes to the **focused widget first** (the leaf), 
and if that widget returns `FReply::Unhandled()`, 
it **bubbles upward** through parent widgets. 
The first widget to return `FReply::Handled()` terminates propagation.

If **no widget consumes** the event
typically because the game viewport's `SViewport` has focus
Slate falls through to the gameplay path

FWindowsApplication creates and manages the HWND
FSlateApplication manages a tree of widgets that subdivide the HWND's client area into regions
the game world's rendered output occupies one of those regions (the one belonging to SViewport)

The renderer (the scene rendering pipeline — FSceneRenderer, the deferred shading passes, etc.)
renders the 3D world into a render target texture that FSceneViewport owns

This happens during the rendering phase of the frame, driven by the render thread
The result is a GPU-side texture containing the rendered scene

UGameViewportClient doesn't participate in this
it's a gameplay-level object concerned with input routing and HUD setup, not pixel production.

Then, during Slate's draw pass
Slate walks its widget tree and composites everything into the final frame
When it reaches SViewport
that widget says "draw my region 
using the render target texture that my owned FSceneViewport produced."

Slate treats the 3D scene's output as just another rectangular element to composite, 
no different in principle from a button texture or a text glyph

Other widgets (HUD elements, menus, debug overlays) are drawn in their own regions, 
potentially overlapping the viewport.

The final composited result 
Slate widgets and the viewport's rendered scene composited together
is what goes to the swapchain
which presents it onto the HWND's client area

So the pixel flow is
the renderer pushes pixels into a texture
and Slate pulls that texture during compositing
The HWND is the final destination

FEngineLoop::Init()
UGameEngine::Init()

creates the UGameViewportClient instance
At this point, it exists but is not connected to any viewport or any Slate widget

calls UGameEngine::CreateGameWindow()
creates an SGameLayerManager 
a composite Slate widget that owns an SViewport
The SViewport widget, upon initialization, constructs its owned FSceneViewport

Then the engine connects the two sides: 
FSceneViewport receives a pointer to UGameViewportClient as its FViewportClient,
and UGameViewportClient receives a reference back to the FSceneViewport.

FEngineLoop::PreInit()
  └─ FSlateApplication::Create()              ← empty Slate singleton

FEngineLoop::Init()
  └─ UGameEngine::Init()
       ├─ creates UGameViewportClient          ← gameplay handler, unattached
       └─ creates the game viewport widget
            ├─ SGameLayerManager               ← composite Slate widget
            │    └─ SViewport                  ← the Slate widget for the game view
            │         └─ creates FSceneViewport ← engine viewport, owned by SViewport
            └─ wires them together:
                 FSceneViewport.ViewportClient = UGameViewportClient
                 UGameViewportClient.Viewport  = FSceneViewport

The SGameLayerManager (containing SViewport) is 
then added to the Slate window's widget tree

During FEngineLoop::Tick(), 
rendering proceeds in two distinct stages

Stage 1: Scene rendering

UGameEngine::RedrawViewports()
UGameViewportClient::Draw().
it constructs FSceneView objects 
(camera position, projection matrix, visibility settings, post-process parameters) 
and hands them to FSceneViewport, which enqueues the actual scene rendering work 
to the render thread.

The render thread executes the deferred shading pipeline (FSceneRenderer), 
writing the 3D world into a render target texture that FSceneViewport owns
By the end of this stage, the texture exists on the GPU, containing the rendered scene.

Stage 2: Slate compositing

Slate's draw pass
FSlateApplication::DrawWindows() 
walks the Slate widget tree

When it reaches SViewport, the widget says: 
"for my rectangular region, sample the render target texture 
that my owned FSceneViewport holds"

Slate treats this texture as just another element to composite
