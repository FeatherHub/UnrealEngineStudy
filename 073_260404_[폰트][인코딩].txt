매체의 유한성
해상도가 무한하지 않음

글자의 형태는 연속적인 수학적 곡선이지만,
그것을 표시하는 매체는 유한한 격자(pixel grid)

1세대
비트맵 
처음부터 픽셀 격자 위의 on/off 패턴으로
각 글리프(glyph)는 특정 크기(예: 12px)에서의 비트맵이며
다른 크기가 필요하면 별도의 비트맵을 수작업으로 만든다
크기마다 별도의 데이터가 필요하고, 확대하면 계단 현상(aliasing)이 즉각 드러나며, 
인쇄와 화면이 동일한 원본을 공유할 수 없다

2세대

Adobe의 PostScript
글자를 픽셀 패턴이 아니라 **수학적 곡선의 윤곽(outline)**으로 정의
3차 Bézier 곡선(cubic Bézier)의 집합
제어점(control point) 4개로 정의되는 매개변수 곡선
B(t)= (1−t)^3 P0 ​+ 3(1−t)^2 t P1 ​ +3(1−t) t^2 P2 ​ + t^3P3
해상도와 무관한 순수 수학적 객체

하나의 아웃라인 정의로부터 어떤 크기, 어떤 해상도의 렌더링이든 도출할 수 있다

TrueType 
Apple과 Microsoft가 Adobe의 PostScript 라이센스 비용을 피하기 위해 만듦
**2차 Bézier(quadratic Bézier)**를 사용
수학적으로는 표현력이 약간 떨어지지만(3차 곡선 하나를 2차로 표현하려면 여러 세그먼트가 필요), 
연산이 더 빠르고 힌팅(hinting) 명령어를 위한 완전한 바이트코드 인터프리터를 내장한다

OpenType 
Microsoft와 Adobe가 합작하여 TrueType과 PostScript(CFF, Compact Font Format) 아웃라인을 모두 담을 수 있는 통합 컨테이너를 만듦
기술적 혁신이라기보다는 정치적 통합
이후 모든 폰트 기술의 표준 프레임워크

아웃라인 폰트가 존재한다는 것은, 렌더링할 때마다 연속 곡선 → 이산 픽셀 변환이 필요하다는 뜻이다
이 변환이 **래스터라이제이션(rasterization)**이며, 폰트 렌더링 품질의 핵심이다
가장 단순한 래스터라이저는 각 픽셀의 중심점이 글리프 윤곽 내부인지 외부인지를 판정하여 on/off를 결정한다
이것은 1비트 래스터라이제이션이며, 비트맵 폰트와 동일한 계단 현상을 만든다

3세대: **그레이스케일 안티앨리어싱(grayscale AA)**
샘플링 해상도가 신호의 주파수를 따라잡지 못할 때, **저역통과 필터(low-pass filter)**를 적용하여 앨리어싱을 줄인다
렌더링에서 이것은 픽셀 값을 on/off의 이진이 아니라 0~255의 그레이스케일로 표현하는 것이다.
한 픽셀이 글리프 윤곽에 의해 부분적으로만 덮이는 경우(coverage), 그 덮임 비율에 비례하는 밝기값을 할당한다
50% 덮인 픽셀은 128/255의 회색이 된다. 인간의 시각 체계가 이 회색 경계를 매끄러운 곡선으로 해석하는 것이다

macOS의 Quartz 렌더러가 오랫동안 이 접근의 대표적 구현이었다
Apple의 철학은 "글자의 수학적 형태에 최대한 충실하라"였고, 
그 결과 저해상도에서 글자가 약간 흐릿하게(blurry) 보이는 트레이드오프를 감수했다.

4세대: 힌팅
안티앨리어싱만으로는 저해상도(72~96 DPI의 화면)에서 충분한 가독성이 확보되지 않는다
여기서 등장하는 것이 힌팅(hinting), 또는 TrueType의 용어로 **인스트럭션(instruction)**이다.

래스터라이제이션이 수학적 곡선을 픽셀 격자에 "떨어뜨릴" 때, 
곡선의 특정 지점들을 **픽셀 경계에 의도적으로 정렬(snap)**시킨다. 
예컨대 'H'의 두 수직 획(stem)이 수학적으로는 4.3픽셀과 4.7픽셀 너비지만, 
힌팅이 둘 다 5픽셀로 맞추면 일관된 시각적 무게감을 얻는다

TrueType의 힌팅 시스템은 사실상 완전한 프로그래밍 언어다. 
스택 기반 바이트코드 인터프리터가 글리프의 제어점을 픽셀 경계에 맞춰 이동시키는 명령어들을 실행한다. 
전문 힌팅 작업은 각 글리프에 수백 줄의 인스트럭션을 수작업으로 작성하는 것을 의미했고, 
고품질 힌팅은 단일 폰트 패밀리에 수개월이 걸리는 장인적 작업이었다.

PostScript/CFF의 힌팅은 이보다 추상적이다
글리프별 세밀한 제어 대신 stem hint(획의 위치와 너비를 선언)와 같은 높은 수준의 정보를 제공하고
래스터라이저가 이를 해석하여 정렬을 수행한다. 제어의 정밀도는 떨어지지만 제작 비용이 훨씬 낮다.

정확성(fidelity to the outline)과 가독성(legibility on the grid)은 
저해상도에서 상충

5세대: 서브픽셀 렌더링

LCD 패널의 각 "픽셀"은 실제로 R, G, B 세 개의 독립적인 서브픽셀로 구성되어 있으며, 
이들은 수평으로 나란히 배열되어 있다. 
따라서 수평 방향의 실효 해상도는 공칭 해상도 Nominal Resolution의 3배다

표준적인 LCD 구조에서 픽셀은 수직으로는 길쭉한 세 개의 서브픽셀이 수평으로 붙어 있는 형태입니다.
R, G, B 각각의 밝기를 조절하여 1/3 픽셀 단위($0.333...$ px)로 경계선을 표현합니다.
실효 해상도(Effective Resolution)
이 덕분에 수직 획(Vertical Stem)의 위치를 훨씬 미세하게 조정할 수 있으며, 
곡선의 기울기를 표현할 때 계단 현상(Aliasing)이 눈에 띄게 줄어듭니다

인간의 시각 체계는 색상(chrominance) 해상도가 밝기(luminance) 해상도보다 훨씬 낮기 때문에, 
경계에 나타나는 미세한 색 번짐(color fringing)은 거의 지각되지 않는다

비등방성(Anisotropy)이란
방향에 따라 물리적 성질이나 측정값이 달라지는 특성
인간의 눈은 수평과 수직 방향의 정보를 처리하는 방식과 감도가 다릅니다

휘도와 색상의 분리
우리 눈의 망막은 색상 정보보다 휘도(밝기) 정보에 훨씬 민감합니다

수평 해상도의 우위
LCD의 서브픽셀 구조 덕분에 수평 방향으로는 공칭 해상도의 3배에 달하는 휘도 변화를 줄 수 있습니다

Windows는 ClearType과 강력한 힌팅의 결합으로 저해상도에서의 가독성을 극대화하는 전략을 취했고, 
macOS는 힌팅을 최소화하고 아웃라인 충실도를 우선하는 전략을 유지했다

6세대 고해상도의 도래
Retina 디스플레이(2010)와 그에 뒤따른 고DPI 디스플레이의 보편화

220+ DPI에서는 픽셀이 충분히 작아서 1비트 래스터라이제이션조차 인간의 눈에 매끄럽게 보인다
나이퀴스트 조건이 드디어 충족되기 시작한 것이다

픽셀 격자에 맞춘 의도적 왜곡은 격자가 충분히 미세할 때 오히려 해가 된다
아웃라인의 수학적 형태를 불필요하게 훼손하기 때문이다

Windows도 점진적으로 힌팅 의존도를 줄이고 있다
(DirectWrite의 자연스러운 레이아웃 모드 등).

FreeType / HarfBuzz 생태계
FreeType이 래스터라이제이션을 담당하고, 
HarfBuzz가 셰이핑(shaping) — 즉 유니코드 문자열을 글리프 시퀀스로 변환하는 과정
(리가처, 커닝, 아랍어/데바나가리 같은 복잡한 문자 체계의 재배열 포함)을 담당한다

Signed Distance Field (SDF) 렌더링
Valve가 2007년 논문으로 대중화

Variable Fonts (OpenType 1.8, 2016~)
하나의 폰트 파일 안에 연속적인 디자인 축(weight, width, slant 등)을 정의
개별 스타일마다 별도의 파일을 두지 않고 보간(interpolation)으로 임의의 중간 스타일을 생성
이산적 스타일 집합을 연속적 매개변수 공간으로 대체

인간의 망막에는 세 종류의 원추세포(cone cell)
장파장(Red, ~564nm), 중파장(Green, ~534nm), 단파장(Blue, ~420nm)에 피크 감도를 갖는다

물리적으로 빛은 파장별 에너지 분포(spectral power distribution)라는 고차원 신호이지만, 
세 종류의 센서를 통과하는 순간 3차원 값으로 압축된다.

RGB는 세 원추세포의 반응에 가장 가깝게 대응하는 좌표계다
하지만 인간은 색을 "빨강 0.7 + 초록 0.3 + 파랑 0.1"로 경험하지 않는다
대신 "저건 따뜻한 주황색인데 좀 탁하고 어둡다"고 경험한다

이 경험적 구조를 포착하는 것이 Hue-Saturation-Lightness(또는 Value, Brightness) 계열이다.

Hue (색상)
"형태, 외관, 색"을 의미

HSL/HSV 모델에서 hue는 0°~360°의 각도로 표현된다
빨강에서 주황, 노랑, 초록, 파랑, 보라를 거쳐 다시 빨강으로 돌아온다

Newton이 1704년 Opticks에서 프리즘 스펙트럼을 원형으로 배열한 **색상환(color wheel)**을 제안
자홍(magenta)처럼 스펙트럼에 존재하지 않는 hue도 있다(자홍은 장파장+단파장의 조합이며, 단일 파장으로 존재하지 않는다)
HSL과 HSV에서 "S"가 의미하는 바가 서로 다르다

Saturation (채도)
"가득 채우다." 염료나 안료가 직물에 얼마나 깊이 스며들었는가
색이 얼마나 순수한가 대 얼마나 회색에 가까운가의 축이다
순수한 빨강은 고채도이고, 같은 hue의 분홍이나 탁한 갈색은 저채도다

물리적으로는 스펙트럼 분포의 "뾰족함"에 대응한다
단일 파장에 집중된 빛은 고채도이고, 모든 파장이 균등한 빛(백색광)은 무채색이다

Luminance, Lightness, Brightness, Value — 밝기의 네 가지 이름

Luminance (휘도)
물리학/광학 전통
단위는 cd/m²(칸델라 퍼 제곱미터)
표면에서 특정 방향으로 방출되는 빛의 세기에 대한 물리적 측정값
CIE 1931 표색계에서 Y 값

Brightness (밝기)
순수한 지각적/심리적 용어
"이 표면이 얼마나 밝게 보이는가"에 대한 주관적 경험
물리적 luminance와 비선형적 관계
luminance를 두 배로 해도 brightness는 두 배로 느껴지지 않는다

Lightness (명도)
밝기를 상대적으로 정규화한 지각적 양
"이 표면이 같은 조건에서 조명된 흰색 기준면과 비교하여 얼마나 밝은가."
CIE L*a*b* 색 공간의 L*

Value
Munsell 체계(1905)
"이상적 검정(0)에서 이상적 흰색(10)까지의 지각적으로 균등한 척도."

감마(Gamma)
CRT 모니터는 물리적으로도 비선형 응답을 가졌는데(입력 전압의 약 2.2승에 비례하는 발광), 
이것이 우연히도 인간 지각의 비선형성과 대략적으로 역관계를 이루었다
이 행복한 사고가 **감마 보정(gamma correction)**의 기원이다.

**감마(γ)**라는 이름은 CRT의 전압-발광 전달 함수가 
거듭제곱 함수 L ∝ V^γ로 근사되며, 그 지수가 관례적으로 γ로 표기된 것에서 온다

sRGB 표준(1996)은 이 비선형 인코딩을 공식화하여, 
디지털 이미지의 픽셀 값이 물리적 luminance가 아니라 
지각적으로 균등한 공간에 대략 비례하도록 인코딩하는 것을 표준으로 삼았다. 

이 때문에 sRGB에서 값 128은 luminance의 50%가 아니라 약 21.4%에 대응한다. 
값 공간의 절반 이상이 어두운 영역의 미세한 차이를 인코딩하는 데 할당되는 것이며, 
이것은 인간의 눈이 어두운 톤의 차이에 더 민감하기 때문에 합리적이다.

이 비선형 인코딩을 무시하고 sRGB 값에 직접 선형 연산(보간, 블렌딩, 필터링)을 수행하면 물리적으로 부정확한 결과가 나온다. 
이것이 "선형 색 공간에서 연산하고 비선형 공간으로 출력하라"는 원칙의 근거이며, 
렌더링 파이프라인에서 감마-aware 처리가 필수적인 이유다.

Picture Element
1965년 JPL의 Frederic Billingsley가 디지털 이미지의 최소 단위를 지칭하기 위해 사용한 것이 문서화된 최초의 용례
그 자신은 이미 유통되던 용어를 가져왔다고 진술
2차원 이미지 격자의 하나의 표본(sample)
본질적으로 **점 표본(point sample)**
위치는 있지만 면적은 없다
"픽셀은 작은 정사각형이 아니다"(Alvy Ray Smith, 1995)라는 유명한 논문 제목이 이 점을 강조한다

픽셀은 공간의 한 점에서 취한 측정값이며, 
화면에 정사각형으로 표시되는 것은 **재구성 필터(reconstruction filter)**의 결과이지 
픽셀 자체의 속성이 아니다.

Texture Element
텍스처 맵의 한 샘플
텍스처가 3D 표면에 매핑될 때, 텍셀과 최종 화면 픽셀은 1:1 대응이 아니다
확대(magnification), 축소(minification), 원근 왜곡에 따라 
하나의 픽셀이 여러 텍셀에 걸치거나 하나의 텍셀이 여러 픽셀에 걸칠 수 있다.

Volume Element
3차원 격자의 한 샘플
의료 영상(CT, MRI)에서 기원
Minecraft 스타일의 복셀 지형이 대표적 사례

이산 표본 단위의 확장

타이포그래피

Character 
유니코드에서 **코드 포인트(code point)**로 표현되는 추상적 실체
어떤 시각적 형태와도 아직 결합되지 않은 순수한 의미론적 정체성

Glyph 
시각적 형태의 단위
하나의 character가 맥락에 따라 여러 glyph로 실현될 수 있다
아랍어의 각 글자는 단어 내 위치(초성/중성/종성/독립)에 따라 다른 glyph를 갖는다
여러 character의 조합이 하나의 glyph로 합쳐질 수도 있다
**리가처(ligature)** (f + i → ﬁ)
이 매핑을 수행하는 것이 앞서 언급한 셰이핑(shaping) 과정
γλυφή "조각, 새김." 돌이나 나무에 새긴 기호, 물리적 형태라는 의미가 핵심

Typeface (서체)
Garamond, Helvetica, 나눔고딕
일관된 글리프 스타일
디자인 개념이며, 물리적/디지털 구현과 독립적

Font 
Typeface의 특정 구현체
활판 인쇄 시대에 font는
"Garamond 12pt Roman"처럼 특정 서체의 특정 크기, 특정 스타일의 
**물리적 활자 세트(a set of metal type pieces)**를 의미
어원은 fonte(주조)

디지털 시대에 font는 "하나의 파일(예: NotoSans-Regular.ttf)"로 의미가 이동했으며, 
이 파일이 아웃라인 데이터, 힌팅 명령, 셰이핑 테이블, 메트릭스 등을 포함하는 완전한 기술적 패키지다. 
Variable Fonts는 하나의 파일이 연속적 스타일 공간을 포함

공간의 미세 조정
활판 인쇄에서 글자 사이, 줄 사이의 공간을 제어하는 것은 가독성의 핵심

Kerning
특정 글자 쌍 사이의 간격 조정
"AV"나 "To" 같은 조합

'f'의 상단 세리프가 옆 활자의 영역으로 침범하는 물리적 돌출
이것을 허용하기 위해 활자 사이를 조정하는 행위가 kerning

OpenType의 kern 테이블이나 GPOS 테이블이 이 쌍별 조정값을 저장한다.

"래스터 이미지"는 이 주사 방식으로 표시되는 격자 기반(grid-based) 이미지를 의미하게 되었다.

수학적 벡터에서 직접 온 용어

Alpha

각 픽셀에 RGB와 더불어 네 번째 채널 **A(alpha)**를 부여하여 **불투명도(opacity)**를 표현

선형 보간 공식 C = α C_foreground + (1-α) C_background에서 보간 계수가 관례적으로 α로 표기

Porter-Duff 합성 연산자(over, under, in, out, atop 등)는

이것이 모든 현대 그래픽스 합성의 수학적 기초다


유니코드 이전
ASCII(1963)
각 언어권, 각 운영체제, 각 벤더가 나머지 비트 공간(8비트 확장의 128~255 영역, 또는 멀티바이트 체계)을 자기 방식으로 채웠다
그 결과가 코드 페이지(code page)의 바벨탑이다
ISO 8859-1(서유럽), ISO 8859-5(키릴), Shift_JIS(일본), EUC-KR(한국), Big5(대만), GB2312(중국 대륙)
같은 바이트 시퀀스가 코드 페이지에 따라 완전히 다른 문자를 의미했다

바이트 0xC0은 ISO 8859-1에서 À이지만, ISO 8859-5에서는 Р(러시아어)이고, 
Shift_JIS에서는 멀티바이트 시퀀스의 선행 바이트다

텍스트 자체에는 자신이 어떤 인코딩인지에 대한 정보가 없었다
인코딩은 외부 메타데이터(파일 헤더, 시스템 설정, 또는 사용자의 추측)에 의존했으며, 이것이 틀리면 문자가 깨진다

각 인코딩 체계가 바이트 값 → 문자의 직접 매핑(direct mapping)을 정의하고 있었으며, 이 매핑이 전역적으로 합의되지 않았다
"이 바이트가 무슨 문자인가?"라는 질문에 대한 답이 맥락에 의존하는 것이다.

이것은 근본적으로 주소 공간의 분열 문제

유니코드(1987년 구상, 1991년 1.0 출판)
이전의 모든 인코딩은 하나의 층에서 두 가지 일을 동시에 했다
(1) "어떤 문자가 존재하는가"를 정의하고,
(2) "그 문자를 어떤 바이트 시퀀스로 표현하는가"를 정의했다

층 1 — 추상 문자 집합(Abstract Character Repertoire)과 코드 포인트 할당
"인류가 사용하는 모든 문자에 고유한 번호를 부여한다."
현재 약 15만 자가 할당되어 있고, 
최대 1,114,112개의 코드 포인트를 수용할 수 있다
(U+0000 ~ U+10FFFF).
총 21비트를 사용
10FFFF = 1,114,111

층 2 — 인코딩 형식(Encoding Form)
"코드 포인트를 어떤 바이트 시퀀스로 변환하는가."
UTF-8, UTF-16, UTF-32
UTF-16의 서로게이트 방식(10 bits x 2)으로 표현 가능한 최대 범위가 
17개의 평면(Plane), 즉 U+10FFFF까지였기 때문에 
유니코드의 전체 크기가 이 지점에서 확정되었습니다.

코드 페이지 시대에는 "문자의 정체성"과 "바이트 표현"이 용해되어 분리 불가능했다. 
유니코드는 이 둘 사이에 **명시적 경계(membrane)**를 설치하여, 
"이 문자는 무엇인가"라는 질문과 "이 문자를 어떻게 저장하는가"라는 질문이 독립적으로 답변 가능하게 만든 것이다.

코드 포인트 공간의 구조

1,114,112개의 코드 포인트 공간은 
17개의 **평면(plane)**으로 분할되며, 
각 평면은 65,536(= 2¹⁶)개의 코드 포인트를 담는다.

Plane 0: BMP (Basic Multilingual Plane, U+0000 ~ U+FFFF) 
라틴, 키릴, 그리스, 아랍, 히브리, 데바나가리, 한글, CJK 통합 한자
유니코드 초기에는 "모든 문자가 16비트에 들어갈 것"이라고 낙관했으며, BMP가 전부였다

Plane 1: SMP (Supplementary Multilingual Plane) 
고대 문자(이집트 상형문자, 선형 B, 고딕 문자 등), 이모지, 음악 기호, 수학 기호 확장 등

Plane 2: SIP (Supplementary Ideographic Plane) 
CJK 한자의 확장.

Plane 3 ~ 13: 대부분 미할당. 미래의 확장을 위한 예비 공간.

Plane 14: 특수 목적(태그 문자 등).

Plane 15~16: 사적 사용 영역(Private Use Area). 
유니코드 컨소시엄이 공식 할당하지 않으며, 당사자 간 합의로 사용.

16비트의 함정: UCS-2에서 UTF-16으로

유니코드 1.0(1991)은 "65,536자면 충분하다"고 가정했다
이 가정 하에 각 문자를 16비트 고정 폭으로 인코딩하는 UCS-2가 설계
(Universal Coded Character Set 2-byte)
Java(1995)와 Windows NT(1993)가 내부 문자열 표현으로 UCS-2를 채택
곧 CJK 한자의 통합 작업이 진행되면서 65,536자가 부족하다는 것이 명확해졌다.

해법이 **서로게이트 쌍(surrogate pair)**이다

BMP의 U+D800 ~ U+DFFF 영역(2,048개 코드 포인트)을 문자 할당에서 영구히 제외하고, 
이 영역의 코드를 두 개 조합하여 
BMP 바깥의 코드 포인트(U+10000 ~ U+10FFFF)를 표현하는 탈출구를 만든 것이다.
BMP 문자는 여전히 2바이트이고, BMP 밖의 문자는 4바이트(서로게이트 쌍)가 된다

초기 설계 가정("16비트면 충분하다")의 파괴가 낳은 영구적 상흔이며, 수많은 버그의 원천
String.length()가 반환하는 값은 문자 수가 아니라 UTF-16 코드 유닛 수
Java의 char는 16비트이므로 이모지(U+1F600 등)를 하나의 char로 담지 못한다
Java와 Windows의 내부 문자열은 UTF-16이며, "한 문자 = 한 코드 유닛"이 더 이상 보장되지 않는다

UTF-8
바이트 스트림의 어느 지점에서든 문자 경계를 찾을 수 있어야 한다(자기 동기화(self-synchronizing))

결과 설계

U+000000 ~ U+00007F는 1바이트: 0xxxxxxx. 선행 비트 0이 "이것은 1바이트 문자"를 신호한다.
U+000080 ~ U+0007FF는 2바이트: 110xxxxx 10xxxxxx.
U+000800 ~ U+00FFFF는 3바이트: 1110xxxx 10xxxxxx 10xxxxxx.
U+010000 ~ U+10FFFF는 4바이트: 11110xxx 10xxxxxx 10xxxxxx 10xxxxxx.

**선행 바이트(leading byte)**의 상위 비트 패턴이 후속 바이트의 개수를 인코딩
**후속 바이트(continuation byte)**는 항상 10으로 시작
10으로 시작하면 중간이므로 뒤로(또는 앞으로) 스캔하면 된다.

UTF-8은 가변 폭이므로 "n번째 문자에 오프셋 기반 즉시 접근(O(1) random access)"이 불가능하다
문자열의 n번째 문자를 찾으려면 처음부터 순차적으로 디코딩해야 한다(O(n))
UTF-32(모든 코드 포인트를 4바이트 고정 폭으로)는 O(1) 접근을 제공하지만 메모리 소비가 4배다. 

실무에서는 랜덤 접근이 필요한 경우가 생각보다 드물고(대부분의 텍스트 처리는 순차적 스캔이다), 
UTF-8의 공간 효율성과 ASCII 호환성이 유리하여 기본 선택이 되었다.

한자는 중국, 일본, 한국, 베트남에서 각각 약간씩 다른 자형(字形)으로 사용된다
예컨대 "골(骨)"의 일본 자형과 중국 자형은 획의 세부가 다르다.
이것을 하나의 코드 포인트로 통합할 것인가, 별도의 코드 포인트로 분리할 것인가?

유니코드의 결정은 **통합(unification)**이었다
"같은 기원(etymological origin)을 가진 한자는 자형 차이가 있더라도 하나의 코드 포인트로 합친다." 
자형의 차이는 폰트(glyph)의 영역에서 처리한다는 논리다. 
이 결정은 코드 포인트 공간을 절약했지만, 일본 사용자들을 중심으로 강한 반발을 불러일으켰다
자국어 텍스트가 중국어 폰트로 렌더링되는 현상이 발생할 수 있기 때문이다. 
이 논쟁은 유니코드 역사에서 가장 정치적으로 민감한 쟁점 중 하나이며, 
"추상 문자의 정체성"이 순수하게 기술적으로 결정될 수 없음을 보여주는 사례다.

선조합형과 조합형: 한글의 경우
한글 음절은 자모(초성 + 중성 + 종성)의 조합으로 구성되며, 유니코드는 두 가지 표현을 모두 제공한다.

선조합형(precomposed)
완성된 음절 "가", "나", "다" 각각에 코드 포인트를 부여. U+AC00(가) ~ U+D7A3(힣)
총 11,172자. 가능한 모든 현대 한글 음절을 커버한다.

조합형(decomposed)
자모(ᄀ, ᅡ, ᆨ 등)에 개별 코드 포인트를 부여하고, 
이들을 조합하여 음절을 표현한다. "가" = ᄀ(U+1100) + ᅡ(U+1161).

이 이중 표현은 정규화(normalization) 문제를 낳는다
선조합형 "가"(U+AC00)와 조합형 "가"(U+1100 U+1161)는 시각적으로 동일하지만 바이트 시퀀스가 다르다. 
문자열 비교 시 이 둘을 동일하게 취급해야 하는가? 
유니코드는 **정규화 형식(Normalization Form)**을 정의하여 이 문제에 답한다

동일성의 형식적 정의
"같은 문자"를 여러 코드 포인트 시퀀스로 표현할 수 있다는 사실은, 
"두 문자열이 같은가?"라는 질문을 비자명(non-trivial)하게 만든다

NFC (Canonical Decomposition, followed by Canonical Composition)
먼저 정준 분해한 뒤 다시 정준 합성. 가능한 한 선조합형으로 표현한다
웹과 대부분의 현대 시스템

NFD (Canonical Decomposition)
정준 분해만 수행
macOS의 파일 시스템(HFS+)
다른 OS와의 파일명 호환에서 문제를 일으킨 것은 유명

NFKC / NFKD
호환 분해(compatibility decomposition)를 포함. 
"ﬁ"(리가처, U+FB01)를 "fi"(두 개의 개별 문자)로 분해하는 등, 
의미론적으로는 같지만 형태적으로 구별되는 변이형을 통합한다

정규화의 존재는 유니코드의 "하나의 문자에 하나의 코드 포인트"라는 이상이 실현 불가능함을 인정한 것이다. 
역사적 호환성, 언어학적 복잡성, 정치적 타협이 중복 표현을 불가피하게 만들었고, 
정규화는 이 중복을 사후적으로 관리하는 메커니즘이다.

**결합 문자(combining character)**는 선행 기저 문자(base character)에 붙어서 하나의 시각적 단위를 형성하는 코드 포인트다. 
예컨대 U+0301(Combining Acute Accent, ◌́)은 앞의 "e"(U+0065)에 결합하여 "é"를 만든다. 
이론적으로 결합 문자는 무한히 쌓을 수 있다
하나의 기저 문자에 수십 개의 결합 부호를 붙이는 것이 유니코드 명세상 유효하며, 
이것이 "Zalgo text"(결합 부호를 극단적으로 쌓아 텍스트를 뒤덮는 장난)의 원리다.

이 구조는 "사용자가 하나의 문자로 인식하는 것"이 하나의 코드 포인트가 아닐 수 있음을 의미한다. 
"é"는 하나의 코드 포인트(U+00E9, 선조합형)일 수도 있고 두 개의 코드 포인트(U+0065 + U+0301, 조합형)일 수도 있다.

서기소 클러스터(grapheme cluster)
"사용자가 하나의 문자로 인식하는 최소 단위"이며, 하나 이상의 코드 포인트로 구성된다
텍스트 편집기에서 커서가 한 번 이동할 때 건너뛰는 단위, 백스페이스가 한 번에 지우는 단위

유니코드의 각 코드 포인트에는 번호뿐 아니라 풍부한 속성(property) 집합이 부여된다

이모지에는 피부색 수정자(skin tone modifier), 성별 수정자, ZWJ 시퀀스(Zero Width Joiner를 이용한 문자 결합)가 도입되어, 
하나의 "문자"가 수십 개의 코드 포인트로 구성되는 극단적 사례를 만들었다
단순한 텍스트 파일에서는 11바이트에서 많게는 수십 바이트(UTF-8 기준)를 차지
만약 프로그래밍 언어에서 string.length를 호출하면, 사용자의 눈에는 1글자이지만 
엔진은 7개(또는 UTF-16 서로게이트 페어 포함 시 그 이상)의 길이를 반환하는 불일치가 발생합니다.

1991: Unicode 1.0 출판. 7,161자. BMP만 존재.
1993: Unicode 1.1. CJK 통합 한자 20,902자 추가. 이 단일 추가가 BMP의 1/3을 차지.
1996: Unicode 2.0. 서로게이트 영역 예약으로 16비트 한계를 넘을 준비 완료.
1996~1999: UTF-8의 점진적 확산. HTML 4.0(1997)이 UTF-8을 권장.
2000: Unicode 3.0. 보조 평면(Plane 1~) 최초 사용.
2003: Unicode 4.0. 수학 기호, 음악 기호, 고대 문자 대거 추가.
2010: Unicode 6.0. 이모지 최초 공식 편입(일본 통신사들의 기존 이모지 집합을 표준화).
2015: Unicode 8.0. 피부색 수정자 도입.
2025 현재: Unicode 16.0 기준, 154,998개 문자. 168개 문자 체계(script).

1837~1844: Morse Code
가변 길이 인코딩
다른 언어를 위한 확장은 각국이 독자적으로 수행하여 호환되지 않는 변이형들이 난립
Huffman 코딩(1952)의 직관적 선행자

1870년대: Baudot Code
5비트 고정 길이 코드
SHIFT 메커니즘을 도입
모드를 전환하는 특수 코드를 보내면 이후의 동일한 코드가 다른 문자를 의미한다
코드 페이지 전환의 원형
"같은 바이트 시퀀스가 상태에 따라 다른 의미를 갖는다"는 문제의 최초 등장
전송 중 SHIFT 문자가 손실되면 이후의 모든 텍스트가 깨지는 문제

1890년대~1920년대: Hollerith 천공 카드
80열 카드의 각 열에 하나의 문자를 인코딩. 여기서도 확장 문제가 발생
IBM이 이후 수십 년간 독자적 천공 패턴을 정의하면서 여러 비호환 변이형이 생겼다.

1963: ASCII의 탄생
ASA(현 ANSI)가 표준화
대문자/소문자의 비트 패턴 관계: A = 0x41(100 0001), a = 0x61(110 0001). 
비트 5(0x20)를 뒤집으면 대소문자가 전환된다.

하드웨어 수준에서 대소문자 변환을 단일 비트 연산으로 만들기 위한 의도적 설계이며, 
현대까지도 char ^ 0x20이라는 트릭으로 살아 있다

숫자의 인코딩: '0' = 0x30(011 0000) ~ '9' = 0x39(011 1001). 
하위 4비트가 숫자의 실제 값과 일치한다. 

제어 문자 0x00~0x1F
NUL, BEL(터미널 벨 울리기), BS(백스페이스), LF(줄바꿈), CR(캐리지 리턴)
문자가 아니라 장치 제어 명령
관심사 혼합의 시작
CR+LF(Windows) 대 LF(Unix) 대 CR(구 Mac OS)의 줄바꿈 혼란이 여기서 기원

1963년의 동시대적 맥락: EBCDIC
IBM이 System/360을 위해 만든 8비트 코드. 
ASCII와 완전히 비호환
알파벳이 연속 배치되지 않아서(A~I, J~R, S~Z가 불연속)
for (c = 'A'; c <= 'Z'; c++)가 작동하지 않는다.
ASCII와의 변환 문제가 메인프레임↔PC 데이터 교환의 만성적 골칫거리였다
"표준화되지 않은 인코딩이 어떤 고통을 낳는가"의 첫 번째 대규모 사례

1967~1974: ISO 646과 국가별 변이형
ISO가 ASCII를 국제 표준으로 채택하면서, 
12개의 코드 포인트(0x23, 0x24, 0x40, 0x5B~0x5E, 0x60, 0x7B~0x7E)를
**국가별 대체 가능(national replacement)**으로 지정

결과는 혼돈이었다. 프랑스 변이형에서 # 위치에 £이, 독일 변이형에서 [ 위치에 Ä이 들어갔다. 
C 프로그램의 소스 코드가 국가별 터미널에서 다르게 보이는 문제가 발생했다
a[i]가 독일 터미널에서 aÄiÜ로 표시되는 상황
이것이 C89에 **트라이그래프(trigraph)**가 도입된 직접적 원인이다
??(를 [의 대안으로 허용한 것. 
트라이그래프는 C/C++ 역사에서 가장 혐오받는 기능 중 하나가 되었고 C++17에서 드디어 제거되었다

1981~1992: 8비트 확장의 시대, 코드 페이지의 난립

1981
IBM PC가 출시되면서 8비트 확장(128~255)을 사용하는 Code Page 437이 등장
상위 128자에 서유럽 악센트 문자, 상자 그리기(box drawing) 문자, 일부 그리스 문자, 수학 기호를 배치
공식 국제 표준이 아니라 IBM의 독자적 할당이었고, 곧 지역별 변이형이 쏟아져 나왔다
CP850(다국어 라틴), CP852(중/동유럽), CP855(키릴), CP866(러시아)

1985~1988: ISO 8859 시리즈
ISO가 8비트 인코딩을 표준화하려는 시도
ISO 8859-1(Latin-1, 서유럽), 8859-2(중유럽), 8859-5(키릴), 8859-6(아랍), 8859-7(그리스), 8859-8(히브리), 8859-9(터키) 등
각각이 0x00~0x7F는 ASCII와 동일하고, 0xA0~0xFF에 해당 언어권 문자를 배치

하나의 문서에 하나의 파트만 사용 가능
프랑스어와 러시아어가 혼재하는 텍스트를 ISO 8859로 표현하는 것은 불가능
각 파트가 상위 128자를 독점하기 때문

1978~1990: 동아시아의 멀티바이트 인코딩

JIS X 0208 (1978, 일본)
6,879자의 일본어 문자 집합 정의
이 집합을 바이트로 인코딩하는 방식이 여럿 난립했다
Shift_JIS (1982, Microsoft와 ASCII Corporation 공동), 
EUC-JP (Unix 계열), 
ISO-2022-JP (이메일용, 이스케이프 시퀀스로 인코딩 전환)

같은 문자 집합에 세 가지 비호환 바이트 표현이 존재

KS C 5601 (1987, 한국)
한글 2,350자와 한자 4,888자
2,350자는 현대 한글 음절 11,172자의 약 20%에 불과하며, "뷁", "쓩" 같은 음절을 표현할 수 없었다
완성형은 기존 빈도 기반으로 선별된 음절만 포함했고, 조합형은 자모를 조합하여 모든 음절을 표현할 수 있었다
한국 정부는 완성형(KS C 5601)을 공식 표준으로 밀었고, 많은 사용자들은 조합형을 선호했다.

GB2312 (1980, 중국 대륙) 간체자 6,763
Big5 (1984, 대만): 번체자 13,060

1987~1991: 유니코드의 구상과 탄생
1987: Joe Becker의 메모
Xerox의 Joe Becker가 "Unicode 88"이라는 내부 문서를 작성
"세계의 모든 문자에 고유한 16비트 코드를 부여하자."
Becker는 한자를 대략 14,000자로 추산했는데, 
이것은 중/일/한의 한자를 개별적으로 세지 않고 **통합(unify)**하겠다는 전제 하의 숫자였다

1988~1989: Apple과 Xerox의 합류
Mark Davis(Apple), Lee Collins(Apple)가 Becker의 구상에 합류
Apple은 Macintosh의 국제화 문제를 절실히 겪고 있었고 — Mac OS의 각 언어 버전이 서로 다른 인코딩을 사용했다
보편적 문자 집합이라는 비전에 강하게 공감했다.

1989: Unicode Inc. 설립 준비, ISO 10646과의 충돌
같은 시기에 ISO/IEC JTC1에서도 보편적 문자 집합 표준인 ISO 10646을 추진하고 있었다
그러나 ISO 10646의 초기 안은 4바이트(32비트) 인코딩으로, 
128개의 "그룹" × 256개의 "면" × 256개의 "행" × 256개의 "셀"이라는 거대한 계층 구조를 제안했다

1991년 1월: Unicode 1.0 출판
7,161자. BMP만 존재하며, UCS-2(16비트 고정 폭)가 유일한 인코딩. 한자는 아직 포함되지 않았다.

1991년: ISO와의 합의
유니코드 컨소시엄과 ISO가 "문자 목록(repertoire)을 동기화한다"는 합의에 도달
그러나 두 표준은 별개로 유지되며, ISO 10646이 코드 포인트 할당을 담당하고 
유니코드가 속성, 알고리즘, 규칙을 담당하는 사실상의 역할 분담이 형성되었다.

1992~1996: 결정적 확장과 결정적 실수
1992: CJK 통합 한자(URO, Unified Repertoire and Ordering)
20,902자의 한자가 U+4E00 ~ U+9FFF에 배치
중국(GB), 일본(JIS), 한국(KS), 대만(CNS)의 한자 집합을 하나로 통합하는 작업

일본의 반발이 가장 거셌다 — "우리의 한자가 중국 한자에 흡수되었다"는 인식
통합된 코드 포인트에 일본어 폰트를 적용할지 중국어 폰트를 적용할지가 텍스트에 명시되지 않으므로, 
시스템의 기본 폰트 설정에 따라 일본 사용자의 화면에 중국식 자형이 나타나는 현상이 발생했다
이 문제는 30년이 지난 현재까지도 완전히 해결되지 않았으며, 
HTML의 lang 속성으로 언어를 명시하여 폰트 선택을 유도하는 것이 현재의 권장 대응이다.

1992: Ken Thompson과 Rob Pike, UTF-8 설계
저녁 식사 중에 플레이스매트(종이 식탁보) 위에 UTF-8의 기본 구조를 스케치했다.
다음 날 Thompson이 Plan 9의 C 라이브러리에 구현을 완료
설계 요구사항은 명확했다

Shift_JIS와의 대비
Shift_JIS에서는 멀티바이트 문자의 두 번째 바이트가 0x5C(ASCII \)의 값을 가질 수 있다
C 컴파일러가 문자열 리터럴 안에서 이것을 이스케이프 문자로 해석하여 컴파일이 깨지는 문제가 일본의 C 프로그래머들을 수십 년간 괴롭혔다
UTF-8은 이런 종류의 문제를 구조적으로 불가능하게 만든 것이다

1993: Unicode 1.1 — 한글의 첫 번째 할당
Unicode 1.1은 한글 음절을 U+3400 ~ U+4DFF 영역에 배치했다.
현대 한글 음절 11,172자 전체를 커버하지 못했다
한국 커뮤니티의 불만이 폭발했다

1996: Unicode 2.0 — 한글 대이동(The Great Hangul Vowel Movement)
Unicode 2.0에서 한글 음절 블록이 완전히 재배치되었다
기존 U+3400 영역에서 U+AC00 ~ U+D7A3으로 이동하고, 11,172자 전체(현대 한글 음절 전부)가 포함되었다.
이후 유니코드는 아무리 과거의 결정이 부적절하더라도 코드 포인트를 재할당하지 않는다.

U+AC00을 기준으로, 각 음절의 코드 포인트가 초성(19개) × 중성(21개) × 종성(28개, 종성 없음 포함)의 수학적 공식으로 계산된다
코드 = 0xAC00 + (초성 인덱스 × 21 × 28) + (중성 인덱스 × 28) + 종성 인덱스
이것은 조합형의 논리를 선조합형의 형태 안에 내장한 것이며, 
프로그래밍적으로 자모 분해/합성이 산술 연산만으로 가능하게 만든 설계다.

1996: Unicode 2.0 — 서로게이트 
16비트의 한계가 명확해지면서 U+D800 ~ U+DFFF를 서로게이트 영역으로 예약
이 결정이 유니코드의 전체 크기를 확정한다
상위 서로게이트(U+D800 ~ U+DBFF, 1,024개) × 하위 서로게이트(U+DC00 ~ U+DFFF, 1,024개) = 1,048,576개의 추가 코드 포인트. 
BMP의 65,536자를 합하면 총 1,114,112자 — 이것이 U+10FFFF의 유래다.

1993~1996: Java와 Windows NT의 UCS-2 채택, 그리고 그 대가
Java (1995)
char 타입을 16비트로 정의. "하나의 char = 하나의 유니코드 문자"라는 설계 의도.
그러나 서로게이트 쌍이 도입되면서 이 등식이 깨졌다
Java 5(2004)에서야 Character.codePointAt(), String.codePointCount() 같은 서로게이트-aware API가 추가되었지만, 
기존 API(charAt(), length())의 의미론은 변경할 수 없었다

Windows NT (1993)
내부 문자열을 16비트 wchar_t로 처리. Win32 API의 W 변이형(예: CreateFileW)이 유니코드 경로
NT 3.1이 출시된 1993년에는 UCS-2가 충분했지만, 
이후 서로게이트 쌍이 필요해지면서 Windows의 유니코드 지원은 실질적으로 UTF-16으로 전환되었다
BMP 밖의 문자(이모지 포함)에서 깨지는 소프트웨어는 2020년대에도 여전히 존재한다.

1996~2003: UTF-8의 부상, 웹의 인코딩 혼란

1996: RFC 2044 — UTF-8의 공식 표준화
UTF-8이 IETF에 의해 RFC로 공식화
최대 길이가 6바이트에서 4바이트로 제한되었다(U+10FFFF까지만 필요하므로).

1997: HTML 4.0
HTML 4.0이 문서 인코딩으로 UTF-8을 권장
그러나 실제 웹은 여전히 인코딩의 무법지대
대부분의 웹 서버와 브라우저가 인코딩을 명시하지 않았고, 브라우저는 **인코딩 자동 감지(auto-detection)**에 의존
자동 감지는 바이트 패턴의 통계적 분석에 기반하며, 본질적으로 추측이다 — 짧은 텍스트에서는 틀리는 경우가 빈번

2000년대 초반: mojibake의 전성기
웹 게시판, 이메일, 데이터베이스에서 문자 깨짐이 일상이었던 시대

이메일: MIME 헤더에 charset=이 명시되지 않거나 잘못 명시된 이메일이 난무
한글 이메일이 일본어 인코딩으로 해석되어 의미 불명의 일본어 문자로 표시되는 일이 흔했다.

MySQL의 latin1 기본값
기본 문자 집합이 latin1(ISO 8859-1)
많은 웹 애플리케이션이 UTF-8 데이터를 latin1 컬럼에 바이트 그대로 저장했고
(이른바 "double encoding")
이후 인코딩을 바로잡으려 할 때 데이터 변환이 필요한 "UTF-8 저장된 것처럼 보이는 latin1" 데이터가 
수백만 개의 데이터베이스에 남아 있게 되었다. 
이 문제는 2020년대에도 레거시 시스템에서 발견된다.

Windows의 "ANSI" 인코딩
Windows가 로컬 코드 페이지를 "ANSI"라고 부른 것은 역사적 오명이다
실제로 ANSI(미국국가표준협회)가 정의한 적 없는 인코딩을 "ANSI"라고 부름으로써, 
코드 페이지 1252(서유럽 Windows)가 마치 보편적 표준인 것 같은 혼동을 만들었다
"Notepad에서 ANSI로 저장"이 시스템 로캘에 따라 다른 인코딩을 생성하는 현실은 이 오명의 귀결

2003~2010: UTF-8의 승리, 새로운 문제의 출현

2003: Unicode 4.0
49,194자. 수학 기호, 음악 기호, 고대 문자(Linear B, Cypriot 등)가 SMP(Plane 1)에 배치. 
BMP 밖의 세계가 실질적으로 채워지기 시작한 시점.

2006: RFC 4790 — 문자열 비교의 복잡성 인정

"collation"(정렬 순서)이 로캘에 의존한다는 사실의 표준화
같은 유니코드 문자열이라도 독일어에서는 ä를 ae로 취급하여 정렬하고, 스웨덴어에서는 ä를 z 뒤에 놓는다
"문자열 비교"가 보편적 연산이 아니라 문화적 맥락에 의존하는 연산임이 공식적으로 인정
ICU(International Components for Unicode) 라이브러리가 이 복잡성을 관리하는 사실상의 표준 구현

2007~2008: UTF-8이 웹의 지배적 인코딩으로

2008: Unicode 5.1 — 보안 문제의 부상

호모글리프(homoglyph) 공격
시각적으로 동일하거나 유사한 문자를 이용한 피싱

양방향 텍스트(Bidi) 악용
파일 확장자를 시각적으로 위장
report[U+202E]fdp.exe가 화면에 reportexe.pdf로 표시

2010~2016: 이모지 혁명

2010: Unicode 6.0 — 이모지의 공식 편입

일본의 이동통신사(NTT DoCoMo, au, SoftBank)가 1990년대 후반부터 독자적으로 사용하던 이모지 집합을 유니코드에 공식 편입
그런데 이제 유니코드 컨소시엄이 "어떤 이모지가 존재할 수 있는가"를 결정하는 기관이 되었다
인류의 시각적 어휘를 관장하는 표준 기구라는, 원래 의도하지 않았던 역할.

2015: Unicode 8.0 — 피부색 수정자

Fitzpatrick 스케일에 기반한 5단계 피부색 수정자(U+1F3FB ~ U+1F3FF) 도입
기저 이모지 뒤에 수정자를 붙이면 해당 피부색으로 렌더링된다

2016~현재: ZWJ 시퀀스의 폭발적 증가
Zero Width Joiner(U+200D)를 이용하여 여러 이모지를 하나의 "문자"로 결합
실제로 렌더링이 지원되는 조합은 플랫폼마다 다르다.
지원되지 않는 조합은 구성 요소가 개별적으로 표시되는 "fallback" 동작을 하며, 
이것이 플랫폼 간 이모지 표시의 불일치를 낳는다
"내 폰에서 보낸 이모지가 상대방 폰에서 다르게 보이는" 경험이 일상이 되었다

2017~현재: 성숙기의 과제들

2019: Windows Notepad, UTF-8을 기본값으로
코드 페이지 시대의 마지막 상징적 잔재 중 하나가 제거된 사건

2021: Trojan Source 공격 공개
유니코드의 표현력이 보안 표면(attack surface)이 되는 문제는 완전한 해결이 불가능하며, 지속적 관리의 대상
GitHub, Rust, Go 등이 Bidi 제어 문자에 대한 경고/차단 메커니즘을 도입

2022~: 유니코드와 AI
대규모 언어 모델의 토크나이저가 유니코드를 어떻게 처리하는가가 새로운 문제로 부상

BPE(Byte Pair Encoding) 토크나이저가 UTF-8 바이트 시퀀스를 기반으로 작동하며, 
비라틴 문자가 더 많은 토큰으로 분해되어 동일한 의미의 텍스트가 
언어에 따라 다른 비용(토큰 수)을 부과하는 불평등이 발생

한국어 한 음절이 UTF-8에서 3바이트이고, 토크나이저에 따라 1~3 토큰을 소비하는 반면, 영어 단어는 보통 1 토큰이다.

낙관적 가정의 파괴와 그 상흔
"7비트면 충분하다"(ASCII)가 8비트 확장을 낳았고, 
"8비트면 충분하다"가 멀티바이트를 낳았고, 
"16비트면 충분하다"(UCS-2)가 서로게이트 쌍을 낳았다.

UTF-8만이 가변 길이를 처음부터 설계에 내장하여 이 패턴에서 벗어났다

층 분리(layer separation)의 점진적 발견
코드 페이지 시대에는 문자 정체성과 바이트 표현이 용해되어 있었다
유니코드가 이 둘을 분리했다.
유니코드 내부에서도 코드 포인트, 서기소 클러스터, 글리프라는 층위가 점진적으로 명확해졌다

~3300 BCE: 쐐기문자(Cuneiform)

쐐기문자 이전의 수메르 원시 문자는 곡선이 있는 그림 문자(pictograph)였지만, 
점토+갈대라는 매체 조합이 직선적 쐐기 형태를 강제했다

~1300 BCE ~ 600 CE: 필사 도구의 분기

파피루스 + 갈대 펜(reed pen): 이집트, 그리스, 로마
갈대의 절단면이 만드는 **넓적한 끝(broad nib)**이 획의 굵기 변화(thick-thin contrast)를 자연스럽게 생성한다
이 물리적 현상이 서양 타이포그래피에서 **세리프체(serif typeface)**의 굵기 대비(contrast)의 기원이다 
타이포그래피의 미학적 전통이 도구의 물리학에서 출발한 것이다.

모필(毛筆) + 먹 + 종이/비단: 동아시아
붓의 유연한 끝이 필압에 따라 획의 굵기를 연속적으로 변화시킨다
서양의 broad nib이 각도에 의한 기계적 굵기 변화를 만든다면, 동양의 붓은 필압에 의한 역학적 굵기 변화를 만든다
이 차이가 서양 타이포그래피와 동아시아 서예의 미학적 분기의 물질적 기반이다

7세기~15세기: 중세 필사본과 서체의 분화
중세 유럽의 수도원 필사실(scriptorium)에서 다양한 필사 서체가 발전한다.

운셜체(Uncial, 4~8세기): 둥근 대문자체. 라틴어 성경 필사에 널리 사용.

카롤링 소문자체(Carolingian Minuscule, 8세기)
샤를마뉴 대제의 궁정 학자 Alcuin of York이 주도한 서체 개혁
유럽 전역에서 통일된 필사 서체를 사용하게 한 최초의 "표준화" 시도
오늘날 우리가 사용하는 소문자(minuscule) 형태의 직접적 조상이다

블랙레터(Blackletter/Gothic, 12세기~)
카롤링 소문자체가 변형되어 각진 형태로 진화
좁은 글자 폭이 양피지를 절약하고 한 페이지에 더 많은 텍스트를 넣을 수 있게 했다
경제적 압력이 서체 형태를 변형시킨 사례
구텐베르크가 처음 인쇄한 성경이 바로 이 블랙레터체다

1040년경: 필승(畢昇)의 교자(膠泥) 활자
중국 송나라. 점토로 만든 개별 활자를 조합하여 인쇄. 세계 최초의 가동 활자(movable type)
그러나 한자의 엄청난 수(수만 자)로 인해 실용적 확산이 제한되었다

1230년경: 고려의 금속 활자
세계 최초의 금속 활자 인쇄 실시
현존 최고의 금속활자 인쇄물은 직지심체요절(1377)
한자의 방대한 문자 수와 유교적 필사 전통의 존중으로 인해, 
활자 인쇄가 사회를 변혁하는 수준의 확산에는 이르지 못했다.

1450년경: 구텐베르크(Johannes Gutenberg)
기술적 혁신의 핵심은 활자 자체보다 활자 주조 시스템에 있다
강철 **펀치(punch)**에 글자를 새기고, 
이것을 구리 **매트릭스(matrix)**에 찍어 틀을 만들고, 
이 틀에 납-주석-안티몬 합금을 부어 **활자(type)**를 대량 주조한다

구텐베르크는 필사본의 시각적 품질을 충실히 재현하려 했으며, 
리가처(ligature), 약어 기호(abbreviation), **문맥별 글리프 변이(contextual alternates)**를 위해 
약 290개의 서로 다른 활자를 주조했다. 
26자 알파벳을 인쇄하는 데 290개의 글리프가 필요했다는 사실은, 
character와 glyph의 관계가 디지털 시대의 발명이 아니라 
인쇄술의 태초부터 존재한 현실이었음을 보여준다.

1470년대: Nicolas Jenson의 로만체
베네치아에서 활동한 프랑스인 펀치 커터 Nicolas Jenson이 
카롤링 소문자체와 로마 비문 대문자체를 결합한 **로만체(Roman type)**를 제작. 
블랙레터의 각진 형태를 버리고 둥글고 열린 형태를 채택. 
이것이 오늘날 "본문용 세리프체"의 원형이며, 
400년 뒤의 Times New Roman까지 이어지는 계보의 시작이다

1495~1500: Aldus Manutius와 이탤릭체의 발명
베네치아의 출판인 Aldus Manutius가 펀치 커터 Francesco Griffo에게 의뢰하여 **이탤릭체(Italic)**를 제작. 
원래 목적은 미학적이 아니라 경제적이었다
기울어지고 좁은 이탤릭체로 한 페이지에 더 많은 텍스트를 넣어 소형 판본(pocket edition)을 만들기 위해서. 
블랙레터가 양피지 절약을 위해 좁아진 것과 동일한 경제적 동기

1530년대: Claude Garamond
파리의 펀치 커터
최초로 자신의 이름으로 활자를 판매한 독립 활자 디자이너
이전까지 활자 디자인은 인쇄소에 종속된 익명의 기술

1720년대: William Caslon
영국 활자 디자인의 확립
영국 식민지 전역에 퍼져 미국 독립선언서(1776)도 Caslon체로 인쇄되었다. 
"의심스러우면 Caslon을 써라(When in doubt, use Caslon)"라는 격언이 생길 정도로 범용적 서체가 되었다

1757: John Baskerville
인쇄 품질에 집착하여 더 매끄러운 종이(wove paper)와 개선된 잉크를 개발했다. 
활자 디자인이 종이와 잉크라는 매체의 물성과 공동 진화한다

1784: Firmin Didot (프랑스)
개선된 인쇄 기술(더 매끄러운 종이, 더 정밀한 프레스)이 가능케 한 형태다
직선에 가까운 헤어라인 세리프, 극단적인 굵기 대비(가는 획은 극히 가늘고 굵은 획은 극히 굵다), 완전히 수직인 축
거친 종이와 원시적 프레스로는 헤어라인 세리프가 인쇄에서 사라져 버린다

Didot는 동시에 **포인트 시스템(point system)**을 체계화했다
활자 크기를 물리적 단위로 정량화한 것이며, 
오늘날의 72pt = 1인치(DTP 포인트)의 먼 조상이다
이전에는 크기를 "Cicero", "Pica", "Brevier" 같은 이름으로 불렀고 국가/인쇄소마다 실제 치수가 달랐다.

1816: William Caslon IV — 최초의 산세리프체
Caslon의 후손이 세리프 없는 활자를 발표. 
당시에는 "기괴한(grotesque)" 것으로 받아들여졌으며, 실제로 산세리프의 초기 분류명이 Grotesque다

1880년대~1890년대: 기계식 조판의 등장
Linotype (1886)
키보드 입력으로 한 줄(line)의 활자를 한 번에 주조하는 기계
신문 조판의 속도를 혁명적으로 높였다
Linotype은 한 줄을 통째로 주조하므로 개별 글자의 수정이 불가능
오타 하나를 고치려면 그 줄 전체를 다시 주조해야 했다

Monotype (1887)
개별 활자를 기계적으로 주조. 수정이 용이하여 서적 조판에 선호. 
Linotype이 신문을, Monotype이 서적을 지배하는 분업이 형성되었다.

글자의 크기는 매트릭스의 물리적 크기에 묶여 있었고, 
하나의 기계에 장착 가능한 매트릭스 수가 곧 사용 가능한 글리프 수의 상한이었다.

III. 20세기: 과도기

1927: Futura
Bauhaus의 미학적 이상 — 기하학적 순수성, 장식의 제거 — 을 서체로 구현한 기하학적 산세리프
원, 삼각형, 직사각형의 순수 기하학 형태로 글자를 구성하려는 시도
순수 기하학만으로는 시각적으로 균형 잡힌 글자를 만들 수 없었고(순수한 원형의 'o'는 사각형의 'n' 옆에서 작아 보인다)
광학적 보정(optical compensation)을 적용

1928: Gill Sans (Eric Gill)
Humanist Sans-Serif
Futura의 기하학적 순수성과 달리, 로만체 필기의 인간적 비례를 산세리프 형태에 유지
런던 지하철, BBC 등 영국 기관의 정체성에 사용
"영국의 Helvetica" 역할

1932: Times New Roman

The Times 신문을 위해 Monotype이 의뢰하여 설계
신문 인쇄의 구체적 제약 — 좁은 열(column), 거친 신문 용지, 빠른 인쇄 속도 — 에 최적화

1957: Helvetica
**네오그로테스크 산세리프(Neo-Grotesque Sans-Serif)**의 정점
중립성(neutrality)을 설계 목표로 삼은 서체
글자 자체가 메시지에 개성을 부여하지 않고 투명한 그릇이 되는 것
2007년 다큐멘터리 영화 Helvetica가 이 서체의 문화적 의미를 탐구

1960년대: Phototypesetting의 시대

글자의 마스터가 **필름 네거티브(film negative)**에 기록되고, 
빛을 필름 네거티브를 통해 감광지(photosensitive paper)에 투사하여 조판한다
렌즈를 교체하면 크기 변경이 가능했으므로, 
하나의 마스터에서 여러 크기를 생성하는 것이 최초로 실현되었다.

금속 활자 시대에 숙련된 펀치 커터는 각 크기마다 미묘하게 다른 디자인을 만들었다
작은 크기에서는 획을 약간 더 굵게, 글자 간격을 약간 더 넓게, x-height를 약간 더 크게.

Phototypesetting는 하나의 마스터를 기계적으로 확대/축소하므로 이 조정이 사라졌고, 
작은 크기의 텍스트가 이전보다 가독성이 떨어지는 결과를 낳았다

이 문제는 디지털 폰트에서도 수십 년간 지속되다가, 
Variable Fonts의 opsz(optical size) 축으로 2016년에야 체계적으로 재도입된다

금속 활자에서는 활자 몸체(body)의 물리적 폭이 최소 간격을 결정했지만, 사진 식자에서는 글자를 겹칠 수도 있었다
이것은 1960~70년대의 tight letterspacing 유행을 낳았고, 
Herb Lubalin 같은 디자이너가 글자를 극단적으로 밀착시키는 표현주의적 타이포그래피를 가능케 했다.

1974~1978: Donald Knuth의 METAFONT

글자를 윤곽(outline)이 아니라 **펜의 궤적(stroke)**으로 정의한다
수학적 매개변수(획의 굵기, 세리프의 길이, 곡률 등)를 변수로 두고, 
이 변수를 조정하면 하나의 프로그램에서 다양한 스타일의 글자를 생성할 수 있다.

"사용자(타이포그래피 디자이너)의 인지적 모델에 부합하지 않는 추상화는 아무리 강력해도 채택되지 않는다"

1976: Ikarus 시스템
글자의 윤곽을 디지타이저(digitizer)로 입력
수학적 곡선으로 보간(interpolation)하는 시스템
기존의 금속 활자나 사진 식자 마스터를 디지털로 변환하는 도구
"아날로그 → 디지털 변환"이라는 폰트 산업의 전환


1982~1984: Adobe PostScript
John Warnock과 Charles Geschke가 Xerox PARC를 떠나 Adobe Systems 설립
device-independent page description language. 
페이지의 모든 요소(텍스트, 그래픽, 이미지)를 수학적으로 기술하고, 
출력 장치(프린터, 식자기, 화면)가 자신의 해상도에 맞게 래스터라이즈
PostScript의 폰트 
3차 Bézier 곡선으로 글리프 윤곽을 정의. 
힌팅 메커니즘을 포함하지만, Adobe가 힌팅 명세를 **영업 비밀(trade secret)**로 유지
이 폐쇄성이 이후 Apple과 Microsoft의 반발, 그리고 TrueType의 탄생을 촉발하는 직접적 원인이 된다.

1985: Apple LaserWriter + Aldus PageMaker
Desktop Publishing Revolution
전에는 전문 식자 업체에 의뢰해야 했던 출판물 제작이 개인 컴퓨터에서 가능해진 것이다.

1989: Adobe Type 1 명세 공개
Apple과 Microsoft가 TrueType 개발을 발표하자
Adobe가 방어적으로 Type 1 명세를 공개
영업 비밀로 유지하던 힌팅 명세를 포함하여 전체 스펙을 발표

1991: TrueType
Apple의 동기: Adobe에 대한 의존도 감소. PostScript 라이선스 비용을 지불하지 않고도 고품질 폰트 렌더링을 확보하려는 전략적 결정
Microsoft는 대가로 Apple에 TrueColor 기술을 제공

Apple/Microsoft의 TrueType 대 Adobe의 Type 1
사용자와 출판 업계는 두 형식을 모두 지원해야 하는 부담을 짊어졌다

Apple은 TrueType을 GX Typography(후에 AAT, Apple Advanced Typography)로 확장하여 
고급 타이포그래피 기능(문맥별 대체, 리가처 자동화 등)을 도입했고, 이것은 후에 OpenType의 기능 영감이 된다.

1994: Multiple Master Fonts (Adobe)
하나의 폰트 파일 안에 여러 "마스터" 디자인을 두고 그 사이를 보간하여 연속적인 변이를 생성하는 기술
예컨대 Light 마스터와 Bold 마스터 사이에서 Medium, Semi-Bold 등 임의의 중간 무게를 생성.

실패한 이유들
프린터 드라이버와의 호환성 문제
소프트웨어 지원이 미흡했다 - UI가 불편했다
인스턴스를 생성하는 워크플로우가 직관적이지 않았다

"기술적으로 올바르지만 생태계가 준비되지 않은" 기술이 실패하는 패턴의 전형적 사례
22년 뒤 Variable Fonts가 거의 동일한 개념을 더 나은 생태계 조건에서 재시도한다.

1996: OpenType 발표 (Microsoft + Adobe)
TrueType 아웃라인(.ttf)과 PostScript/CFF 아웃라인(.otf)을 모두 하나의 컨테이너 형식에 수용

OpenType Layout 테이블
GSUB(Glyph Substitution, 글리프 대체)
GPOS(Glyph Positioning, 글리프 위치 조정). 
이 두 테이블이 문맥별 리가처, 커닝, 스타일 대체(stylistic alternates), 
분수 표기, 작은 대문자(small caps) 등 "스마트 폰트" 기능의 기반

OpenType의 cmap 테이블이 유니코드 코드 포인트를 글리프 인덱스에 매핑한다
이것은 유니코드와 폰트 기술의 합류점이며, 
이전의 인코딩별 폰트 파일(같은 서체의 "Western" 버전과 "Central European" 버전이 따로 존재하던 상황)을 
하나의 파일로 통합하는 것을 가능케 했다

OpenType의 발표 시점(1996)과 실질적 채택 사이에는 상당한 시차가 있었다
운영체제와 애플리케이션의 OpenType Layout 지원이 본격화된 것은 2000년대 중반이며, 웹에서의 지원은 더 늦었다.

1996~2004: 화면용 폰트의 문제와 ClearType
Matthew Carter가 Microsoft를 위해 Verdana와 Georgia를 설계
화면 렌더링을 일차적 설계 목표로 만들어진 서체다

인쇄물의 미학이 아니라 72~96 DPI 화면에서의 가독성이 모든 설계 결정의 기준

유사 형태 간의 명확한 구분(I/l/1, O/0), 정교한 수작업 힌팅. 
Verdana의 힌팅은 단일 폰트 사상 가장 정교한 것 중 하나로 간주되며, 
각 글리프에 수백 줄의 TrueType 바이트코드가 들어 있다.

1998: Microsoft Research, ClearType 기술을 개발
서브픽셀 렌더링의 최초 상업적 구현.
ClearType이 기본 활성화된 것은 Windows Vista(2006)부터

2000년대 초반: "웹 안전 폰트(Web-Safe Fonts)"의 시대
웹에서 사용 가능한 폰트는 사용자의 컴퓨터에 설치된 폰트로 제한되었다
이 제약 때문에 웹 디자인은 사실상 Arial, Verdana, Georgia, Times New Roman, Courier New 등 
소수의 "웹 안전 폰트"에 묶여 있었다

이 시기에 sIFR (Scalable Inman Flash Replacement, 2004) 같은 해킹적 기법이 등장
Flash 플러그인을 이용하여 커스텀 폰트로 텍스트를 렌더링하는 방식.
접근성, 검색 엔진 최적화, 성능 모두에서 문제가 있었지만, 웹 디자이너들의 절박함을 보여주는 사례

2006: DirectWrite (Microsoft)
Windows Vista / 7에서 도입된 텍스트 렌더링
글자 위치를 픽셀 경계에 강제 정렬(snap)하지 않고 서브픽셀 위치에 둠
힌팅 퇴각의 시작이며, 아웃라인 충실도(macOS의 전통적 접근)에 한 발짝 다가선 것이다.

DirectWrite로 전환한 애플리케이션(Firefox 4, 2011 등)에서 "텍스트가 흐릿해 보인다"는 불만이 폭주
강한 힌팅에 익숙한 Windows 사용자들이 자연스러운 간격의 텍스트를 흐릿하게 느낀 것이다
사용자 기대치는 기술적 우열과 독립적이며, 익숙함이 품질 판단을 지배한다

2009~2010: @font-face — 웹 폰트 혁명

CSS3에서 표준화
웹 페이지가 서버에서 폰트 파일을 다운로드하여 사용할 수 있게 함
CSS2(1998)에 이미 명세되어 있었지만, 브라우저 지원 부족과 폰트 라이선스 문제로 10년간 사장되어 있었다.

WOFF (Web Open Font Format, 2009)
OpenType/TrueType을 웹 전송에 최적화한 래핑 형식
핵심적으로 WOFF는 기술적 혁신이라기보다 라이선스 타협
폰트 파운드리들은 원본 OpenType 파일이 웹에 직접 노출되는 것을 거부했고, 
WOFF는 "웹 전용"이라는 구분을 형식적으로 제공하여 라이선스 우려를 완화했다.

Google Fonts (2010)
오픈소스 폰트를 무료로 호스팅하는 서비스 출시
웹 타이포그래피의 다양성을 폭발적으로 증가
2025년 현재 1,700개 이상의 폰트 패밀리를 호스팅

2010: Apple Retina Display — 힌팅의 퇴장

iPhone 4의 Retina Display(326 DPI)
폰트 렌더링 패러다임을 변경
Apple은 2018년 macOS Mojave에서 서브픽셀 안티앨리어싱을 완전히 폐기했다

2012: WOFF 2.0 / Brotli 압축
WOFF 2.0은 Brotli 압축 알고리즘을 사용하여 WOFF 1.0 대비 약 30% 추가 압축
웹 폰트의 로딩 속도 개선에 기여

2013: Noto — "No more Tofu" 프로젝트
Google과 Monotype의 협업
유니코드의 모든 문자를 커버하는 폰트 패밀리를 만드는 프로젝트
100개 이상의 문자 체계(script)를 지원하는 역사상 가장 거대한 단일 폰트 프로젝트다

2016: OpenType 1.8 — Variable Fonts
Microsoft, Google, Apple, Adobe의 공동 발표
하나의 폰트 파일 안에 연속적인 **디자인 축(design axis)**을 정의. 
표준 축: wght(weight), wdth(width), ital(italic), slnt(slant), opsz(optical size). 
커스텀 축도 정의 가능.

(1) OpenType이라는 보편적 표준 안에 통합되었다.
(2) CSS font-variation-settings로 웹에서 직접 제어 가능. 
(3) 운영체제, 브라우저, 디자인 도구의 지원이 동시에 진행되어 생태계 준비도가 높았다

사진 식자 시대에 잃어버린 **광학적 크기 조정(optical sizing)**을 디지털에서 복원한 것
활판 인쇄 시대의 펀치 커터가 수작업으로 하던 것을 매개변수 보간으로 자동화.

2018~현재: 컬러 폰트(Color Fonts)
글리프에 색상 정보를 포함하는 기술. 네 가지 비호환 형식이 경쟁

COLR/CPAL (Microsoft)
글리프를 여러 색상 레이어로 분해. 
COLR v1(2021)에서 그라디언트, 변환, 합성 모드 추가.

SVG (Mozilla/Adobe)
글리프를 SVG로 정의. 가장 표현력이 높지만 파일 크기가 크고 렌더링 성능이 낮을 수 있다.

sbix (Apple)
비트맵 이미지 내장.
Apple의 이모지 폰트(Apple Color Emoji)가 이 형식.

물리적 객체
크기, 스타일, 매체가 불가분하게 결합된 금속 덩어리
디지털 전환은 이 결합을 하나씩 풀어냈다

아웃라인이 크기를 분리했고,
OpenType이 인코딩을 분리했고, 
@font-face가 설치를 분리했고, 
Variable Fonts가 스타일 변이를 분리했다

입력: 유니코드 코드 포인트의 시퀀스 + 폰트 지정 + 크기 + 색상 + 대상 표면
출력: 대상 표면의 특정 픽셀들에 특정 색상값이 기록된 상태

텍스트 렌더링 파이프라인

① Text Analysis
유니코드 문자열에서 스크립트(Latin, Hangul, Arabic...) 식별, 
양방향(Bidi) 레벨 결정, 줄바꿈 가능 지점(line break opportunity) 판별.

② 폰트 선택
지정된 폰트가 해당 코드 포인트의 글리프를 가지고 있는가?
없으면 대체 폰트(fallback font)를 어떻게 찾는가?

③ Shaping
코드 포인트 시퀀스를 글리프 시퀀스로 변환.
리가처 적용(f + i → ﬁ), 아랍어 문맥별 형태 선택, 한글 자모 합성, 데바나가리 재배열 등
OpenType Layout 테이블(GSUB, GPOS)이 이 단계의 규칙을 제공한다

④ 레이아웃
각 글리프의 위치 결정. 전진 폭(advance width), 커닝, 줄바꿈, 정렬(alignment), 행간(line spacing)
여러 줄의 텍스트라면 줄바꿈 알고리즘(Knuth-Plass 등)

⑤ 래스터라이제이션
각 글리프의 아웃라인을 특정 크기, 특정 해상도의 픽셀 격자 위에서 비트맵(또는 알파 맵)으로 변환. 
힌팅 적용, 안티앨리어싱, 서브픽셀 렌더링이 이 단계

⑥ 합성/렌더링
래스터라이즈된 글리프 비트맵을 대상 표면에 지정된 색상으로 그리기. 
alpha 블렌딩(Porter-Duff "over" 연산)으로 배경과 합성. 
GPU 가속 환경에서는 텍스처에 글리프 비트맵을 캐시(glyph atlas/texture atlas)하고 셰이더로 그린다.

프레임 A: "텍스트는 그래픽의 특수한 경우다"
텍스트를 선, 원, 다각형과 동일한 2D 그래픽 프리미티브로 취급한다
텍스트와 그래픽이 같은 좌표 공간에서 자유롭게 조합된다
텍스트에 임의의 변환(회전, 스케일, 투영)을 적용할 수 있다.

텍스트의 언어적 구조(단어, 문단, 줄바꿈, 선택, 편집)를 표현하지 않는다
Canvas에 텍스트를 그린 후에는 "3번째 단어를 선택하라"는 작업이 극히 어렵다
한번 래스터라이즈되면 텍스트는 그냥 픽셀이다.

PostScript, PDF, SVG, HTML5 Canvas

프레임 B: "텍스트는 문서의 구성 요소다"

텍스트를 구조화된 문서 모델의 요소로 취급한다

문단, 줄, 단어, 글자의 계층 구조가 있고, 
스타일(폰트, 크기, 색상)이 범위(range/span)에 적용되며, 
선택(selection), 커서 이동, 편집이 가능하다

검색, 선택, 복사, 접근성(스크린 리더) 등이 자연스럽게 지원된다.
렌더링의 세밀한 제어가 어렵다. "이 글자를 정확히 (142.7, 88.3) 좌표에 놓겠다"는 요구가 문서 모델의 추상화와 충돌한다.

프레임 C: "텍스트는 실시간 렌더링의 오브젝트다"
게임 엔진과 GPU 가속 UI에서의 프레임
텍스트는 3D 공간의 오브젝트이거나 GPU 텍스처 위의 쿼드(quad)다.
매 프레임 수만 글자를 60fps 이상으로 그려야 한다.
SDF(Signed Distance Field) 렌더링, 글리프 아틀라스(glyph atlas), 인스턴스드 렌더링(instanced rendering)이 이 프레임의 핵심 기법이다.

셰이핑과 레이아웃의 복잡성을 CPU 측에서 선행 처리해야 하며, 
복잡한 문자 체계(아랍어, 데바나가리 등)의 지원이 불완전한 경우가 많다.

폰트 파일은 단순한 "글리프 이미지의 모음"이 아니라, 
여러 소비자(consumer)가 서로 다른 목적으로 접근하는 구조화된 데이터베이스다

셰이핑 엔진이 소비하는 데이터

cmap: 유니코드 코드 포인트 → 글리프 인덱스(glyph index) 매핑
없으면 .notdef 글리프(빈 사각형, "tofu")가 반환되고, 시스템은 대체 폰트를 탐색한다.

GSUB (Glyph Substitution)
글리프-to-글리프 대체 규칙. 리가처(f + i → ﬁ)
스타일 대체(stylistic alternates), 작은 대문자 등

GPOS (Glyph Positioning)
글리프 위치 미세 조정
커닝(특정 글리프 쌍의 간격 조정), 마크 위치(결합 문자의 정확한 배치 좌표), 이중 마크 부착 등
GSUB이 "어떤 글리프를" 결정한다면, GPOS는 "어디에" 결정한다.

일반 애플리케이션 프로그래머가 이 테이블을 직접 읽을 일은 거의 없다
셰이핑 엔진이 이 복잡성을 흡수
"코드 포인트 시퀀스를 넣으면 위치 지정된 글리프 시퀀스가 나온다"는 인터페이스를 상위 층에 제공

래스터라이저가 소비하는 데이터

glyf + loca (TrueType)
글리프의 아웃라인 데이터
2차 Bézier 곡선의 제어점 좌표와 온커브/오프커브 플래그

CFF / CFF2 (PostScript/OpenType)
3차 Bézier 곡선으로 정의된 아웃라인

fpgm + prep + cvt (TrueType 힌팅)
이 바이트코드를 실행하는 것이 TrueType 인터프리터이며
역사적으로 Apple의 특허 문제로 기본 비활성화되어 있다가 2010년대에 특허 만료 후 활성화되었다

레이아웃 엔진이 소비하는 데이터

hmtx / vmtx (Horizontal/Vertical Metrics)
각 글리프의 전진 폭(advance width)과 좌측 여백(left side bearing)
텍스트의 수평 진행에서 다음 글리프의 시작 위치를 결정하는 기본 데이터
수직 조판(세로쓰기)에서는 vmtx가 사용된다.

hhea / vhea / OS/2
폰트 전체의 메트릭.
ascender(기선 위 최대 높이), descender(기선 아래 최대 깊이), line gap(행간). 
이 세 값이 행 높이(line height)를 결정한다
Windows는 OS/2를, macOS는 hhea를 우선한다
"같은 폰트, 같은 크기인데 Windows와 macOS에서 행 높이가 다르다"는 흔한 문제의 원인

kern
글리프 쌍별 커닝 값
레거시 테이블
현대 폰트는 GPOS의 PairPos Lookup으로 커닝을 정의하는 것이 권장
일부 렌더링 엔진(특히 오래된 것)이 kern 테이블만 읽고 GPOS를 무시하므로, 
실무에서는 양쪽 모두에 커닝 데이터를 넣는 경우가 많다.

폰트 선택/분류 시스템이 소비하는 데이터
name: 폰트의 이름, 저작권, 라이선스 정보 등 문자열 메타데이터
폰트 패밀리 이름(Family Name), 서브패밀리(Regular, Bold, Italic), 전체 이름(Full Name) 등.
CSS의 font-family가 매칭하는 대상

OS/2
무게(weight), 너비(width), 스타일 플래그(italic, bold) 등 분류 속성
CSS의 font-weight: 700이 매칭하는 대상

fvar 
변이 축(variation axis)의 정의
CSS의 font-variation-settings가 제어하는 대상

최저 수준: 폰트 파싱 라이브러리
폰트 파일을 열고 테이블을 직접 읽는 수준.

FreeType
래스터라이제이션에 특화. 글리프 아웃라인 추출, 비트맵 렌더링, 힌팅 실행을 담당
FreeType은 셰이핑을 하지 않는다
래스터라이제이션과 셰이핑은 독립적 관심사

FT_New_Face() — 폰트 파일로부터 Face 객체 생성
FT_Set_Char_Size() — 렌더링 크기 설정
FT_Get_Char_Index() — 유니코드 코드 포인트 → 글리프 인덱스
FT_Load_Glyph() — 글리프 아웃라인 로드 (힌팅 적용 옵션 포함)
FT_Render_Glyph() — 아웃라인을 비트맵으로 래스터라이즈
FT_Glyph_Get_Bitmap() — 래스터라이즈된 비트맵 접근

이 인터페이스로 텍스트를 렌더링하려면 
호출자가 직접 문자열을 순회하고, 각 코드 포인트마다 글리프를 로드하고, advance width를 누적하여 위치를 계산하고, 
비트맵을 대상 표면에 복사해야 한다

HarfBuzz
OpenType Layout 테이블(GSUB, GPOS)을 해석하여 코드 포인트 시퀀스를 올바른 글리프 시퀀스(+ 위치 오프셋)로 변환한다

hb_buffer_add_utf8() — 텍스트 입력
hb_buffer_set_direction() — 쓰기 방향(LTR, RTL, TTB)
hb_buffer_set_script() — 스크립트(Latin, Arabic, Hangul...)
hb_buffer_set_language() — 언어(셰이핑 규칙이 언어에 따라 달라질 수 있음)
hb_shape() — 핵심 함수. 셰이핑 실행
hb_buffer_get_glyph_infos() — 결과 글리프 인덱스 배열
hb_buffer_get_glyph_positions() — 결과 위치(x_advance, y_advance, x_offset, y_offset) 배열

HarfBuzz는 레이아웃(줄바꿈, 문단 정렬 등)을 하지 않는다
하나의 "런(run)" — 동일한 폰트, 동일한 방향, 동일한 스크립트의 연속 텍스트 구간 — 에 대해 셰이핑만 수행한다
여러 폰트/스크립트가 혼재하는 텍스트를 처리하려면 
상위 레이아웃 엔진이 문자열을 런으로 분할하고 각 런에 대해 HarfBuzz를 호출해야 한다.

레이아웃 수준

ICU (International Components for Unicode)
줄바꿈(line breaking), 단어 경계(word boundary), 양방향 알고리즘(BiDi), 정규화, 정렬(collation), 로캘별 규칙 등
레이아웃 엔진이 필요로 하는 언어적 분석을 제공

Pango (Linux/GNOME)
HarfBuzz와 FreeType 위에 구축된 레이아웃 엔진
다중 폰트/다중 스크립트 텍스트의 런 분할, 줄바꿈, 양방향 처리를 담당

각 플랫폼의 레이아웃 엔진은 이름과 구조가 다르지만 동일한 문제를 해결한다
유니코드 문자열을 입력받아, 폰트 선택 → 런 분할 → 셰이핑 → 위치 계산 → 줄바꿈까지를 처리한 후, 
"이 글리프를 이 좌표에 그려라"는 명령 시퀀스를 출력한다

OS 텍스트 API
"문자열과 좌표를 주면 화면에 글자가 나타난다"는 인터페이스를 제공한다.

Windows — GDI(Graphics Device Interface TextOut() / ExtTextOut()
Device Context(HDC)에 폰트를 선택하고
문자열과 좌표를 전달하면 렌더링. 글리프 캐싱, 힌팅, 서브픽셀 렌더링을 내부적으로 처리
글리프 위치가 정수 픽셀로 양자화(snap)되며
OpenType Layout 기능(리가처 등)의 지원이 제한적

Windows — DirectWrite
서브픽셀 글리프 위치, OpenType 기능 전체 지원, 자연스러운 글자 간격
IDWriteTextFormat → IDWriteTextLayout → ID2D1RenderTarget::DrawTextLayout()
TextLayout 객체가 텍스트 분석, 셰이핑, 레이아웃을 모두 수행하고, 
렌더링은 Direct2D가 담당. 
DirectWrite는 셰이핑 엔진을 내장하고 있으며, HarfBuzz를 사용하지 않는다.

macOS / iOS — Core Text
Apple의 저수준 텍스트 API
CTFont → CTTypesetter → CTFramesetter → CTFrame → CTLine → CTRun의 계층 구조
Core Text의 설계 철학은 **불변 객체(immutable object)**의 체인
멀티스레드 환경에서의 안전성과 캐싱 효율을 위한 설계

macOS / iOS — TextKit (AppKit/UIKit 수준)
Core Text 위에 구축된 고수준 프레임워크
NSTextStorage (텍스트 데이터 + 속성) → NSLayoutManager (레이아웃 계산) → NSTextContainer (레이아웃 영역 정의) 의 MVC 패턴
TextKit 2(2021~)는 이 구조를 비선형 레이아웃으로 확장

Android 
Canvas API(Canvas.drawText())가 최종 렌더링
내부적으로 Minikin(레이아웃/폰트 선택) + HarfBuzz(셰이핑) + FreeType(래스터라이제이션)의 조합
Skia 그래픽 라이브러리가 래스터라이즈된 글리프의 GPU 렌더링을 담당.
고수준에서는 TextView/EditText 위젯이 전체 파이프라인을 감춘다.

Chrome은 HarfBuzz + FreeType/Skia, 
Safari는 Core Text, 
Firefox는 HarfBuzz + FreeType/Skia(또는 플랫폼 API)

게임 엔진에서의 텍스트 렌더링은 
OS API와 근본적으로 다른 제약 하에서 작동한다.

각 글자마다 FreeType을 호출하여 래스터라이즈하는 것은 성능적으로 불가능하다.
해법은 **사전 래스터라이제이션(pre-rasterization)**과 GPU 배치 렌더링이다.

Glyph Atlas
필요한 글리프들을 미리 래스터라이즈하여 하나의 큰 텍스처("아틀라스")에 패킹한다
런타임에는 각 글자를 이 텍스처의 해당 영역을 참조하는 텍스처드 쿼드(textured quad)로 그린다
수천 글자를 하나의 드로 콜(draw call)로 처리할 수 있다

Signed Distance Field
아틀라스에 비트맵 대신 거리 필드를 저장
GPU 셰이더가 임계값 비교로 선명한 윤곽을 생성
확대/축소/회전에도 품질이 유지
외곽선(outline), 그림자(drop shadow), 글로우(glow) 효과를 셰이더에서 저렴하게 추가할 수 있다
