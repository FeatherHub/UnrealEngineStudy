why does PlayerController exist as a distinct entity from both the Pawn and the player's "self"?

persistence problem
A player's identity must survive across Pawn deaths
level transitions, possession changes
The Pawn is mortal; the player is not
the persistent projection of player identity into the game world

In networked games, the server needs a stable anchor for a connected client's intent
PlayerController persists on the server for the lifetime of a connection 
while Pawns are ephemeral

input-to-world translation boundary
sits at the threshold between raw human input and world commands
PlayerController manages the camera through PlayerCameraManager, 
handling perception that exists outside the physical actor hierarchy.

The architecture splits identity into three layers
APlayerState holds statistical identity (score, name, ping) and replicates everywhere
APlayerController manages volitional identity (commands, input, camera, HUD) server-side per connection
APawn provides the physical body that can be swapped or destroyed

The crucial insight is that PlayerController embodies agency without substance
it has will and perception but no physical form

The network topology reinforces this separation
PlayerController only exists on the owning client and server, 
creating a private channel for the player

PlayerState replicates to all clients as public identity

Pawn exists everywhere as physical presence

When the controller possesses a pawn through the Possess mechanic, 
it's claiming a body as its instrument while remaining distinct

The HUD is presentation, owned by the controller but disconnected from physics and game logic

All of this reinforces that the PlayerController sits at the boundary between human intent and the game world
between persistent identity and mortal physics, 
between private information and public knowledge.

---

subject-object distinction

the Pawn is the player as object
something physical in the world with position and physics. 

The PlayerController is the player as subject—the perceiver, the intender, the commander

└─────────────────────┘

따라서 실행 순서를 명시적인 선으로 표현해야 했다. 
그것이 Exec Pin이다.

FEngineLoop Implements the main engine loop
게임 프레임 갱신의 소유자는 UGameEngine::Tick()
현재 시뮬레이션 우주의 갱신 책임자는 UWorld::Tick()

상위 루프가 UWorld를 매 프레임 방문한다
Engine이 정한 프레임 루프 안에서 tick entry를 받아요

“Engine이 Owner/Outer 트리를 재귀적으로 DFS 순회하는 방식”
  우선순위 개념 결여

등록된 tick function들을, 월드가 프레임 단계별 스케줄에 따라 실행하는 방식

Actor/Component ticking을 tick group으로 설명하고
각 tick group이 그 그룹에 속한 actor와 component의 tick을 모두 끝낸 뒤 다음 그룹으로 넘어간다
tick prerequisite를 통해 같은 그룹 안에서도 선후관계를 지정할 수 있다

FEngineLoop
→ UGameEngine::Tick()
→ 현재 UWorld::Tick()
→ 그 월드에 등록된 tickable 대상들(actors, components, world subsystems 등)을 
  tick group 순서와 prerequisite dependency에 따라 실행
→ 물리/카메라/스트리밍 같은 프레임 단계들을 사이에 끼워 넣음

phase + dependency 기반의 frame scheduler예요

tick group은 주로 물리 시뮬레이션과의 상대적 시점을 정하기 위해 존재하고, 
그룹은 순차적으로 진행돼요
TG_PrePhysics, TG_DuringPhysics, TG_PostPhysics 같은 그룹

“지금은 물리 전에 돌아야 하는 대상들”
→ “지금은 물리와 병행 가능한 대상들”
→ “지금은 물리 결과를 본 뒤 돌아야 하는 대상들”
→ “그 뒤 타이머/카메라/스트리밍”

의미 있는 프레임 단계들로 잘려 있다

void EngineTick()
{
    // 1. 플랫폼 이벤트 수집
    FPlatformMisc::PumpMessages();      // OS 메시지 펌프
    
    // 2. 시간 계산
    GEngine->UpdateTimeAndHandleMaxTickRate();
    
    // 3. 엔진 틱
    GEngine->Tick(FApp::GetDeltaTime(), bIdleMode);
    
    // 4. 렌더 커맨드 플러시
    RHITick(FApp::GetDeltaTime());
}

void UEngine::Tick(float DeltaSeconds, bool bIdleMode)
{
    // 1. 모든 FWorldContext를 순회
    for (FWorldContext& Context : WorldList)
    {
        UWorld* World = Context.World();
        
        // 2. World에게 틱을 위임
        World->Tick(LEVELTICK_All, DeltaSeconds);
    }
    
    // 3. 스트리밍 관련 업데이트
    // 4. 가비지 컬렉션 조건 검사
    // 5. 통계 업데이트
}

void UWorld::Tick(ELevelTick TickType, float DeltaSeconds)
{
    // ── Phase 1: 입력 ──────────────────────────────
    // PlayerController들이 입력을 처리
    
    // ── Phase 2: 틱 그룹 순차 실행 ─────────────────
    RunTickGroup(TG_PrePhysics);
        // 물리 시뮬레이션 전 로직
        // 이동 의도 결정, 애니메이션 업데이트
    
    RunTickGroup(TG_StartPhysics);
        // Chaos 물리 시뮬레이션 시작 (비동기)
    
    RunTickGroup(TG_DuringPhysics);
        // 물리 시뮬레이션과 병렬 실행 가능한 것들
    
    RunTickGroup(TG_EndPhysics);
        // 물리 결과 수집 대기
    
    RunTickGroup(TG_PostPhysics);
        // 물리 결과를 읽는 것들 (IK, 래그돌 보정)
    
    RunTickGroup(TG_PostUpdateWork);
        // 모든 것이 끝난 후 마지막 정리
    
    // ── Phase 3: 충돌 이벤트 디스패치 ──────────────
    // Chaos가 수집한 충돌 정보를 Actor에게 전달
    // OnComponentHit, OnActorOverlapped 여기서 발화
    
    // ── Phase 4: 레벨 스트리밍 ─────────────────────
    // 로드/언로드 요청 처리
}

즉 계산과 커밋이 한 몸이에요.

읽기 단계: 오직 현재 확정 상태 N만 읽는다
계산 단계: 변경 의도만 기록한다
커밋 단계: 모든 의도를 합쳐서 N+1을 확정한다
이벤트 단계: 커밋 이후에만 dispatch한다

엔진 전체를 transaction/commit 모델로 재해석하는 일이에요.

---

연속 신호 (Continuous)
    아날로그 스틱: -1.0 ~ +1.0 사이의 실수값
    마우스 이동: 매 프레임 델타값

이산 신호 (Discrete)
    버튼: Pressed / Released / 두 상태만

BindAxis의 콜백 시그니처:
void MoveForward(float Value)
매 프레임 들어오는 -1.0 ~ 1.0
방향과 세기가 하나의 float에 인코딩된다

BindAction의 콜백 시그니처
void PrimaryAttack()
IE_Pressed와 IE_Released를 따로 바인딩하는 것이 이것을 드러낸다
눌림과 뗌이 별개의 사건이다

Enhanced Input은 이 이분법을 해체했다.

UPROPERTY(EditDefaultsOnly)
UInputAction* Input_Move;    // Value Type: Axis2D (FVector2D)

UPROPERTY(EditDefaultsOnly)
UInputAction* Input_Sprint;  // Value Type: Digital (bool)

// 콜백 시그니처도 통일됨
void Move(const FInputActionValue& Value)
{
    FVector2D Input = Value.Get<FVector2D>();
}

void Sprint(const FInputActionValue& Value)
{
    bool bPressed = Value.Get<bool>();
}
