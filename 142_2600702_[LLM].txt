
Tokenization

모델은 텍스트를 직접 다루지 않습니다. 
첫 단계는 문자열을 정수 ID 배열로 바꾸는 것

"The renderer draws" → [464, 31837, 13979] 같은 식

토큰 ≠ 단어

대부분의 모델은 BPE(Byte Pair Encoding)를 씁니다
원리는 압축 알고리즘과 같아요
학습 코퍼스에서 가장 자주 붙어 나오는 바이트 쌍을 반복적으로 병합해서 어휘 사전을 만듭니다.

흔한 단어 "the" → 토큰 1개
드문 단어 "FViewInfo" → "F", "View", "Info" 처럼 여러 조각

GPT-2 기준 어휘 크기는 50,257개. 
이게 이후 모든 단계에서 "선택지의 개수"가 됩니다

TArray<int32> Tokenizer::Encode(const FString& Text); // "hello world" → {31373, 995}

Embedding

이제 정수 ID를 벡터로 바꿉니다
모델 안에는 거대한 2차원 행렬이 하나 있습니다

E ∈ ℝ^(50257 × 768)  (GPT-2 small 기준)

토큰 ID가 464면 E의 464번째 행(768차원 벡터)을 꺼내옵니다

FVector768 EmbeddingLayer::Lookup(int32 TokenId) { return E[TokenId]; }

---

이 행렬의 값들이 학습으로 채워진다는 것
학습이 끝나면 의미가 비슷한 토큰들의 행 벡터가 기하적으로 가까워집니다

E["king"] − E["man"] + E["woman"] ≈ E["queen"]

의미의 방향성이 벡터 공간의 방향으로 인코딩

위치 정보

어텐션 연산 자체는 순서를 모르기 때문에
("A hits B"와 "B hits A"가 구분 안 됨), 
각 위치마다 위치 벡터를 더해줍니다

x_i = E[token_i] + P[i]

이제 시퀀스 전체가 벡터의 배열이 됐습니다. 문장이 10토큰이면 10 × 768 행렬 X 하나

---

"The renderer draws"
  → [464, 31837, 13979]        (토큰화: 문자열 → 정수)
  → X ∈ ℝ^(3 × 768)            (임베딩: 정수 → 벡터, + 위치)

이 단계까지는 문맥이 전혀 반영 안 된다는 것

토큰화

토큰: ["I", " like", " Unreal", " Engine"]
ID:   [40, 588, 47382, 9146]

모델 입장에서 최초 입력은 의미가 있는 문장이 아니라 숫자 ID 목록

각 토큰 ID는 벡터로 바뀐다

토큰 ID 9146 -> [0.12, -0.44, 1.03, ...]

이 벡터를 embedding이라고 해

임베딩은 입력 토큰의 출발점이고, 
실제 추론 능력은 Transformer 내부의 수많은 weight에 분산되어 있다

Transformer 블록

입력 벡터들
 -> Transformer Block 1
 -> Transformer Block 2
 -> Transformer Block 3
 -> ...
 -> Transformer Block N
 -> 출력 벡터들

각 블록 안에는 크게 두 가지가 있다.

Self-Attention
MLP / Feed Forward Network

Self-Attention = 문맥 안에서 어떤 토큰을 참고할지 결정
MLP = 각 토큰 표현을 더 복잡한 패턴으로 변환

---

Self-Attention

"The character picked up the weapon because it was powerful."

여기서 it은 무엇을 가리킬까? 아마 weapon이지. 이런 관계를 모델은 문맥 속 토큰들 사이의 관련도로 계산한다

Self-Attention은 각 토큰이 다른 토큰들을 얼마나 참고할지 계산한다

"it" -> "weapon"을 강하게 참고
"powerful" -> "weapon"을 강하게 참고
"character" -> "picked up"과 관계

내부적으로는 각 토큰에서 Query, Key, Value라는 벡터를 만든다.

Query = 내가 지금 찾고 싶은 정보
Key   = 내가 어떤 정보인지 나타내는 표지
Value = 실제로 전달할 내용

각 토큰이 질문한다: "내 의미를 정하려면 문장 안의 누구를 봐야 하지?"
다른 토큰들은 답한다: "나는 이런 정보를 가지고 있어."
관련도가 높은 토큰의 정보가 더 많이 섞인다.

---

Multi-Head Attention

실제 모델은 attention을 한 번만 하지 않는다
여러 개의 attention head를 동시에 쓴다

어떤 head는 주어-동사 관계를 본다.
어떤 head는 대명사가 가리키는 대상을 본다.
어떤 head는 코드의 괄호나 들여쓰기 관계를 본다.
어떤 head는 앞에서 정의한 개념을 뒤에서 다시 참조한다.

Multi-Head Attention은 하나의 문장을 여러 관점에서 동시에 읽는 구조

Head 1: 문법 관계
Head 2: 참조 관계
Head 3: 코드 구조
Head 4: 의미적 연결

---

MLP / Feed Forward

MLP는 그 정보를 바탕으로 각 토큰의 표현을 더 복잡하게 변환한다

Attention: 주변 토큰에서 필요한 정보를 모은다.
MLP: 모은 정보를 바탕으로 패턴을 가공한다

"UFUNCTION(Server, Reliable)"

MLP는 그 조합이 “Unreal RPC 선언과 관련 있다”는 식의 더 추상적인 표현으로 바꿔갈 수 있다.

물론 모델 내부에 사람이 읽을 수 있는 RPC 개념 노드가 있는 건 아니야. 
하지만 벡터 공간 안에서 그런 패턴에 해당하는 방향과 조합이 형성된다고 볼 수 있다.

---

다음 토큰 점수, logits

Transformer 블록들을 모두 지나면, 
모델은 마지막 위치의 벡터를 가지고 다음 토큰 후보들의 점수를 계산한다

모델의 vocabulary에 토큰이 10만 개 있다면,
다음에 올 수 있는 10만 개 토큰 각각에 대해 점수를 만든다

현재 문맥: "Unreal Engine uses reflection through"

다음 토큰 후보 점수:
" UHT"        14.2
" macros"     12.7
" metadata"   11.9
" a"           8.1
" banana"     -3.4
...

이 점수를 logits라고 부른다. 
아직 확률은 아니고, 확률로 바꾸기 전의 원점수야.

그다음 softmax 같은 과정을 통해 확률 분포로 바뀐다.

" UHT"        46%
" macros"     25%
" metadata"   18%
" a"           5%
...

토큰 선택

이 확률 분포에서 실제 다음 토큰을 하나 고른다

temperature, top-p 같은 샘플링 설정

temperature 낮음 -> 보수적, 예측 가능한 답
temperature 높음 -> 다양하지만 불안정한 답

LLM 답변이 항상 똑같지 않은 이유는 
다음 토큰 선택 과정에 확률적 샘플링이 들어갈 수 있기 때문이다.

 -> 반복

Self-Attention

X ∈ ℝ^(3 × 768)
문맥을 모르는 벡터 3개가 있습니다

어텐션의 임무는 각 토큰 벡터에 다른 토큰들의 정보를 섞어 넣는 것입니다

---

각 토큰이 세 가지 벡터를 만듭니다. 학습된 행렬 3개를 곱해서요:

각 토큰이 세 가지 벡터를 만듭니다. 학습된 행렬 3개를 곱해서요:
Q = X·W_Q  (Query: "나는 뭘 찾고 있나")
K = X·W_K  (Key: "나는 뭘 제공할 수 있나")
V = X·V_V  (Value: "실제로 전달할 내용")

---

모든 키와의 유사도만큼 모든 값을 섞어오는 soft한 조회입니다

for (int i = 0; i < NumTokens; ++i)           // 각 토큰이
    for (int j = 0; j <= i; ++j)              // 자기 이전의 모든 토큰에게
        Score[i][j] = Dot(Q[i], K[j]);        // "너 나랑 관련 있어?" 물어봄

Attention(Q, K, V) = softmax(Q·Kᵀ / √d_k) · V

---

Q·Kᵀ — 모든 토큰 쌍의 내적
"The cat sat on the mat because it..." 
"it"의 Query와 "cat"의 Key가 내적이 크면, 모델이 it→cat 관계를 잡은 겁니다

/ √d_k — 스케일링
차원이 크면 내적 값이 커져서 softmax가 극단으로 치우치는 걸 방지

softmax 
각 행의 점수를 합이 1인 확률(가중치)로 변환.
"it"의 행이 [cat: 0.7, mat: 0.2, 나머지: 0.1] 같은 분포가 됨

· V
그 가중치로 Value 벡터들을 가중합

행렬곱 3번 + softmax.

Causal Masking

구현은 점수 행렬의 위쪽 삼각형을 −∞로 채워서 softmax 후 0이 되게 합니다

Multi-Head

실제로는 이 연산을 한 번이 아니라 병렬로 여러 번 합니다
GPT-2 small은 12헤드 — 768차원을 64차원 × 12개로 쪼개서 
각각 독립적으로 어텐션을 수행하고 결과를 이어붙입니다

헤드마다 다른 종류의 관계를 학습합니다
어떤 헤드는 대명사→선행사, 어떤 헤드는 동사→목적어, 어떤 헤드는 그냥 직전 토큰을 복사

MLP(2층 완전연결망, 768→3072→768)

수집한 정보를 각 토큰이 개별적으로 소화하는 단계
모델 파라미터의 약 2/3가 여기 있고
사실적 지식("파리는 프랑스의 수도")이 주로 MLP 가중치에 저장된다고 알려져 있습니다.

---

residual connection

원본을 유지한 채 수정분만 더합니다
이 덕분에 수십 층을 쌓아도 학습이 됩니다
각 층이 벡터를 통째로 교체하는 게 아니라 조금씩 다듬는 구조

---

이 블록을 GPT-2 small은 12번, GPT-3는 96번 쌓습니다
층을 지날수록 벡터가 점점 문맥화됩니다

초반 층의 "bank"는 아직 중의적이지만, 
후반 층에서는 문맥상 강둑인지 은행인지가 벡터에 반영돼 있습니다.

---

LLM은 파일 내용을 토큰으로 받아서, 
변수명, 함수명, 타입명, 주석, 주변 코드 패턴을 self-attention으로 연결한다

그러고 나서 “이 함수는 이런 역할을 한다”는 설명을 생성한다

하지만 이것은 컴파일러처럼 완전한 보장을 가진 분석은 아니야. 
그래서 중요한 경우에는 실제 정적 분석 도구, 빌드, 테스트, 검색 결과로 검증해야 한다

---

LLM 자체는 파일 시스템을 직접 탐색하지 못합니다.
에이전트 환경에서는 모델에게 사용 가능한 도구의 schema가 prompt로 주어지고,
모델은 현재 질문에 필요한 tool call을 구조화된 출력으로 생성합니다.
실제 파일 검색이나 읽기는 런타임이 수행하고, 
그 결과 텍스트가 다시 context에 들어갑니다.

함수를 찾을 때는 보통 문자열 검색, 심볼 인덱스, LSP, AST, embedding search 같은 retrieval 기법을 씁니다.
그렇게 가져온 코드 조각을 모델이 context로 받아 Transformer 계산을 통해 관계를 해석하고 답변을 생성합니다.

정확도가 필요한 경우에는 AST, type checker, call graph, 빌드, 테스트 같은 외부 검증 도구와 결합해야 합니다.

Autoregressive Decoding

N층을 통과한 뒤, 마지막 토큰 위치의 최종 벡터 h ∈ ℝ^768 하나를 꺼냅니다
이 벡터가 "지금까지의 문맥 전체"의 요약

---

로짓 → 확률 분포

logits = h · W_U  (W_U ∈ ℝ^(768 × 50257))

결과는 어휘 사전의 모든 토큰마다 점수 하나씩, 50,257개의 실수
임베딩의 역방향 연산이라 unembedding이라고 부릅니다

여기에 softmax를 씌우면

p = softmax(logits) — 합이 1인 확률 분포

즉 모델의 출력은 "다음 단어"가 아니라 50,257개 선택지 전체에 대한 확률 분포입니다
"The renderer draws" 다음에 → "the": 18%, "a": 12%, "triangles": 3%, ... 전부에 대해

---

샘플링: 분포에서 하나 뽑기

Greedy — 항상 최고 확률 토큰. 결정적이지만 반복 루프에 빠지기 쉽습니다("the the the...")

Temperature (T) — softmax 전에 logits를 T로 나눕니다: softmax(logits / T)

T < 1이면 분포가 뾰족해져서 안전하고 뻔한 선택, 
T > 1이면 평평해져서 다양하지만 헛소리 위험 증가. 
T → 0이 greedy와 같습니다

---

Top-p (nucleus)

확률 상위부터 누적합이 p(예: 0.9)가 될 때까지만 후보로 남기고 그 안에서 샘플링. 
꼬리의 쓰레기 토큰을 잘라냅니다

같은 질문에 매번 다른 답이 나오는 이유가 이 확률적 샘플링
모델이 "생각을 바꿔서"가 아니라, 같은 분포에서 다른 주사위가 나온 것.

오토리그레시브

토큰 하나를 뽑았으면, 그걸 입력 끝에 붙이고 전체 과정을 다시 돌립니다

while (!IsEndToken(Next))
{
    Logits = Model.Forward(Tokens);       // 전체 파이프라인 1회
    Next   = Sample(Softmax(Logits / T));
    Tokens.Add(Next);                     // 자기 출력이 자기 입력이 됨
}

1000토큰짜리 답변 = 이 루프 1000바퀴. 
100번째 토큰을 뽑을 때 모델이 보는 것은 프롬프트 + 자기가 이미 뱉은 99토큰이 전부입니다.

KV 캐시

위 루프를 그대로 구현하면 매 바퀴마다 이전 토큰들의 K, V를 전부 재계산하게 됩니다
그런데 인과 마스킹 때문에 앞 토큰들의 K, V는 절대 변하지 않죠. 그래서 캐싱합니다

새 토큰의 Q, K, V만 계산하고 이전 것은 재사용

Pretraining

인터넷 텍스트를 주고, 각 위치에서 다음 토큰을 맞혀라

"The cat sat on the ___" 에서 정답이 "mat"였다면, 모델이 뱉은 확률 분포에서 "mat"에 준 확률을 봅니다. 
손실 함수는 cross-entropy:
L = −log p(정답 토큰)

정답에 90%를 줬으면 손실이 작고(−log 0.9 ≈ 0.1), 1%를 줬으면 큽니다(−log 0.01 ≈ 4.6)

인과 마스킹 덕분에 한 문장의 모든 위치가 동시에 학습 문제가 됩니다
10토큰 문장이면 예측 문제 10개

backpropagation

손실을 줄이려면 각 파라미터를 어느 방향으로 움직여야 하는지, 
미분의 연쇄법칙으로 전체 네트워크를 거꾸로 타고 내려가며 계산합니다. 

모든 파라미터 θ에 대해 기울기 ∂L/∂θ를 구하고
θ ← θ − η·∂L/∂θ  (η는 학습률)

손실이 줄어드는 방향으로 아주 조금씩 이동
이걸 수조 토큰에 대해 반복합니다

개별 스텝은 단순한데, 스케일이 본체입니다
수천 개 GPU로 수개월, 비용은 수천만~수억 달러

---

"다음 토큰 예측"이라는 단순한 과제를 인터넷 스케일에서 잘하려면, 
문법·사실·추론·코드의 구조가 전부 압축돼서 가중치에 새겨질 수밖에 없다는 것. 

"파리는 프랑스의 수도"를 저장하는 별도의 지식 DB는 없습니다. 
"The capital of France is" 다음에 "Paris"의 확률을 높이는 방향으로 조정된 
수십억 개의 가중치가 있을 뿐. 

그래서 AI의 "지식"은 조회가 아니라 통계적 압축의 산물이고, 
이게 hallucination의 근본 원인이기도 합니다
압축이 손실 압축이라서, 그럴듯하지만 없는 사실도 매끄럽게 생성됩니다.

영어: house → 프랑스어: maison
영어: renderer → 프랑스어:

예시들이 하는 일은 가중치 안에 잠재된 수많은 능력 중 지금 어떤 것을 활성화할지 특정하는 것입니다.
p(다음 토큰 | 컨텍스트)에서 컨텍스트가 분포를 "번역 모드"의 영역으로 밀어 넣는 것.

Induction Head
Anthropic의 2022년 induction heads 연구

두 개의 어텐션 헤드가 연쇄된 회로입니다:

헤드 1 (previous-token head) 
각 토큰 위치에 "내 직전 토큰이 뭐였는지"를 복사해 둡니다
residual stream에 그 정보를 얹어두는 거죠
[A][B]가 있었다면, B 위치에 "내 앞은 A였다"가 기록됩니다.

헤드 2 (induction head)
현재 토큰이 [A]일 때, 컨텍스트를 거슬러 "앞이 A였다고 기록된 위치"를 Query·Key 매칭으로 찾아냅니다
그게 예전의 [B] 위치죠. 그리고 그 위치의 Value에서 [B]를 가져와 "다음은 B"로 예측을 밀어 올립니다.

즉 이 회로가 구현하는 알고리즘은
[A][B] ... [A] → [B]  ("전에 A 다음에 B가 왔으니, 이번 A 다음도 B")

fuzzy 버전으로 일반화되면 in-context learning 그 자체가 됩니다
확히 같은 토큰이 아니라 의미적으로 유사한 패턴에도 작동

예시들이 만든 패턴을 어텐션이 소프트 매칭으로 완성하는 것
"배운 것처럼 보이는" 행동의 상당 부분이 이 패턴 완성입니다.

사전학습 초반의 특정 시점에 induction head들이 상전이처럼 한꺼번에 형성되는데, 
정확히 그 시점에 모델의 in-context learning 능력이 급격히 뛰어오릅니다. 
손실 곡선에 눈에 보이는 꺾임(bump)이 생길 정도로

---

induction head가 완성하는 건 패턴이므로, 
예시들의 형식이 흔들리면 매칭할 패턴 자체가 흐려집니다. 
심지어 예시의 정답이 일부 틀려도 형식이 일관되면 성능이 꽤 유지된다는 연구도 있습니다.

softmax 가중치는 합이 1입니다. 
관련 없는 토큰이 많을수록 어텐션이 희석되고, 
긴 컨텍스트 중간의 정보가 무시되는 현상도 생깁니다. 
소스를 줄 때 통째로 붓는 것보다 관련 부분을 추리는 게 나은 이유

Chain-of-thought
단계별로 생각해봐
모델은 토큰당 고정된 연산량(N층 1회)만 씁니다. 
중간 추론을 토큰으로 뱉게 하면, 그 토큰들이 컨텍스트에 쌓여 다음 토큰의 조건이 됩니다

Interpretability

수천억 개의 가중치는 사람이 쓴 게 아니라 경사하강이 찾아낸 값입니다
우리는 학습 절차는 완전히 이해하지만, 그 절차가 만들어낸 결과물은 이해하지 못합니다
컴파일된 바이너리만 있고 소스가 없는 상황과 비슷한데, 
애초에 "소스"에 해당하는 것이 존재한 적이 없다는 점이죠

Interpretability는 이 바이너리를 리버스 엔지니어링하는 분야입니다

뉴런은 읽을 수 없다 — Superposition

순진한 기대: 뉴런 하나하나가 의미 하나씩 담당하겠지 ("고양이 뉴런", "포인터 역참조 뉴런")

실제로 열어보면 뉴런 하나가 학술 인용, 영어 대화, HTTP 요청, 한국어 텍스트에 동시에 반응합니다
이걸 polysemanticity라고 합니다

Anthropic의 2022년 "Toy Models of Superposition"

모델이 표현해야 할 개념(feature)의 수는 가진 차원 수보다 훨씬 많습니다
768차원에 수백만 개의 개념을 담아야 해요

그런데 고차원 공간의 기하학적 성질 하나가 이걸 가능하게 합니다
n차원에서 완전히 직교하는 방향은 n개뿐이지만, 
거의 직교하는(내적이 ≈ 0인) 방향은 지수적으로 많이 넣을 수 있습니다

그래서 모델은 개념들을 거의-직교 방향들로 겹쳐 담습니다
개념들이 희소하게(동시에 안) 등장하는 한, 약간의 간섭 노이즈를 감수하고 압축하는 게 이득이니까요

의미의 단위는 뉴런이 아니라 방향(direction)이고,
뉴런 하나에는 여러 방향이 겹쳐(superposition) 있다

Sparse Autoencoder (SAE)

그럼 겹쳐진 방향들을 풀 수 있나

2023년 "Towards Monosemanticity"의 접근

residual stream의 활성값을 훨씬 큰 희소 기저로 분해하는 보조 네트워크(SAE)를 학습시킵니다

"이 768차원 활성값을, 수백만 개 후보 방향 중 극소수만 켜진 조합으로 재구성하라"는 제약을 걸면, 
풀려 나온 방향들이 놀랍게도 하나씩 해석 가능해집니다.

2024년 "Scaling Monosemanticity"

실제 상용 모델(Claude 3 Sonnet)에 적용해 수백만 개의 feature를 추출했는데, 발견된 것들이 이렇습니다
금문교 feature(텍스트든 이미지든 반응), 코드의 보안 취약점 feature, 내적 갈등 feature, 아첨(sycophancy) feature

그리고 결정적으로, 이 방향들은 인과적입니다. 읽기만 되는 게 아니라 쓰기도 됩니다
금문교 feature 값을 인위적으로 크게 고정하자 모델이 모든 대화에서 자기가 금문교라고 주장하기 시작했어요
("Golden Gate Claude" 실험)

상관관계가 아니라 그 방향이 실제로 그 개념을 구현한다는 증거입니다.
디버거로 메모리 값을 직접 써서 동작을 바꾼 것과 같은 급의 증거

---

12 attention heads

한 Transformer layer 안의 self-attention을 하나의 관점으로만 하지 않고,
여러 관점으로 나눠서 병렬로 수행

Head 1: 직전 토큰 관계를 봄
Head 2: 주어-동사 관계를 봄
Head 3: 괄호/코드 구조를 봄
Head 4: 대명사 참조를 봄

예를 들어 d_model=768이고 head가 12개면 보통:

768차원 전체를
64차원짜리 head 12개로 나눠 attention 수행

"텍스트 ↔ 정수" 양방향 Map이 실제로 존재하나?

신경망 바깥에 토크나이저는 학습된 가중치가 아니라 평범한 결정적 코드 + 데이터 파일입니다
GPT-2라면 vocab.json(문자열 → ID 5만 개)과 merges.txt(병합 규칙)라는 파일이 있고, 열어서 읽을 수 있습니다

디코딩(정수 → 텍스트)은 문자 그대로 Map 조회입니다
인코딩(텍스트 → 정수)은 조회만으로는 안 되고, 텍스트를 어떻게 쪼갤지 병합 규칙을 순서대로 적용하는 알고리즘
"unhappiness"를 "un"+"happiness"로 쪼갤지 "unhap"+"piness"로 쪼갤지

인코더는 Map + 세그먼테이션 알고리즘, 디코더는 순수 Map입니다.

신경망 내부에는 텍스트라는 것이 아예 존재하지 않습니다. 정수 ID와 float 벡터뿐입니다

---

정수 ID로 인덱싱하는 대상의 실체
Matrix.GetRow(RowIndex)

실체는 float 2차원 배열 하나입니다
float EmbeddingTable[50257][768] // 학습으로 채워진 값들

Lookup: return EmbeddingTable[TokenId];

---

X = X + Attention(LayerNorm(X));   // 교체가 아니라 덧셈
X = X + MLP(LayerNorm(X));

각 층의 출력으로 입력을 대체하지 않고, 입력에 더합니다
역전파 때 기울기가 96층을 거슬러 올라가야 하는데, 곱셈 연쇄만 있으면 기울기가 지수적으로 소멸합니다
덧셈 경로가 있으면 기울기가 무손실로 통과하는 고속도로가 생겨서 깊은 망이 학습 가능해집니다

파이프라인/흐름의 관점

upstream
대량 데이터로 일반 표현을 학습

downstream
그 표현을 번역, 요약, 분류, QA 같은 구체 작업에 적용

위쪽에서 일반 능력이나 representation을 만들고, 
아래쪽에서 그것을 구체적인 과제에 흘려보낸다.

예전에는 모델이 어떤 task 하나만 잘하게 훈련되는 경우가 많았는데, 
점점 “먼저 범용 표현을 학습하고, 그것을 여러 작업에 전이하자”는 관점이 강해졌어
