(1) 메모리 확보
일반 힙 할당이 아니라 GUObjectAllocator가 해당 클래스 크기만큼 메모리를 가져옵니다. 
이 메모리는 아직 의미 있는 비트 패턴이 없습니다.

(2) C++ 생성자
메모리 위에서 돌면서 vtable 포인터를 심고, 
부모 클래스 서브오브젝트들을 (베이스 생성자 호출을 통해) 구성하고, 
멤버들의 생성자를 호출합니다. 
이 시점이 끝나야 그 메모리 영역은 "C++ 객체로서 유효" 합니다.

(3) archetype 으로부터의 프로퍼티 복사
UClass 의 메타데이터를 보고 archetype (대개 CDO, 또는 더 구체적인 archetype) 의 UPROPERTY 값들을 복사 대입(copy-assignment)으로
인스턴스에 부어 넣습니다. 
즉 생성자에서 잡힌 "C++ 디폴트" 위에 archetype 값들이 덧씌워집니다.

(4) (있다면) 직렬화 복원
디스크에서 로드된 객체라면 Serialize 와 PostLoad 가 추가로 돌아 저장된 값들이 다시 위에 덧씌워집니다

(5) PostInitProperties hook
위 단계들이 다 끝나고 호출되는 사용자 entry point. 
이제 "이 객체는 자기 프로퍼티에 들어 있는 모든 UObject 참조가 의미 있는 값을 가졌다" 가 보장됩니다


"UObject 인스턴스가 만들어진다" 는 사건이 사실 여러 종류이고, 
종류마다 위의 (1)~(5) 가 어떻게 조합되는지가 다르다

---

경로 A: CDO 생성


게임이 부팅될 때 (모듈이 로드될 때) UE 는 그 모듈에 정의된 모든 UClass 를 등록하면서 
각 UClass마다 하나의 Class Default Object 를 만듭니다

(1) GUObjectAllocator 가 메모리를 할당합니다. 

(2) C++ 컨스트럭터가 호출됩니다. 
FObjectInitializer 가 컨스트럭터의 인자로 전달되는데, 
CDO 인 경우 이 ObjectInitializer 는 "지금 만들고 있는 객체는 archetype 이 없다" 라는 사실을 알고 있습니다. 

따라서 (3) archetype 으로부터의 프로퍼티 복사가 — 적어도 이 클래스의 자기 프로퍼티에 대해서는 — 일어나지 않습니다. 
대신 부모 클래스의 CDO 를 복사한 위에 자기 컨스트럭터의 코드가 디폴트를 생성 합니다. 
즉 CDO 자체가 archetype 의 시작점이고, 그 시작점은 컨스트럭터 코드가 직접 만들어낸다. 


CDO 는 한 번 만들어지면 게임이 끝날 때까지 살아 있습니다. 
GC 의 루트로 등록되어 회수되지 않습니다. 
UClass 가 이 CDO 의 포인터를 들고 있어 GetDefaultObject<T>() 로 언제든 꺼낼 수 있습니다.

---

경로 B: NewObject, SpawnActor (런타임 생성)

(1) 메모리 할당.
(2) C++ 컨스트럭터 실행 - CDO 만들 때 돌았던 그 코드가 다시 돕니다. 
(3) FObjectInitializer 가 archetype 의 모든 UPROPERTY 값을 인스턴스에 복사 대입합니다. 
(4) 이 객체는 새로 만들어진 것이므로 직렬화 복원은 없습니다. 
(5) PostInitProperties 가 호출됩니다

---

경로 C: LoadObject (직렬화 복원)
uasset 파일에서 객체를 로드하는 경우


(4) Serialize 가 추가로 호출
디스크에서 읽은 값들이 덧씌워지고, 그 다음 PostLoad 가 불립니다. 
PostLoad 는 직렬화 의존성이 모두 해소된 시점의 후처리 hook 으로, NewObject 경로에는 없는 단계입니다


파일에는 차이만 적혀 있고, 베이스라인은 archetype 이 제공한다. 
이것은 .uasset 파일이 작은 이유이고, 
클래스 디폴트가 바뀌면 인스턴스의 (변경되지 않은) 값들이 자동으로 따라가는 이유이기도 합니다.

"archetype 이라는 데이터 베이스라인 위에 인스턴스별 차이를 얹는 합성 시스템" 

---

경로 D: DuplicateObject

PostInitProperties 가 불리고, 
PostDuplicate 라는 추가 hook 이 호출

---

경로 E: 재인스턴싱 (Hot Reload / Blueprint 컴파일)

UPROPERTY 가 아니면 재인스턴싱 때 자동으로 옮겨지지 않아 에디터에서 BP 컴파일 한 번에 상태가 사라집니다


(1) 새 UClass 를 만들고 새 CDO 를 생성
(2) 기존 인스턴스 각각에 대해 새 클래스 인스턴스를 만들고
(3) 옛 인스턴스의 프로퍼티 값들을 이름-매칭으로 새 인스턴스에 옮겨 적습니다 (CopyTaggedProperties)
(4) 옛 인스턴스에 대한 모든 UObject 참조를 새 인스턴스 포인터로 patch 합니다 (FArchiveReplaceObjectRef).

컴포넌트가 가진 두 종류의 "준비됨"

첫째: 등록(Registration) 완료
둘째: 사용자 초기화(InitializeComponent) 완료

---

액터의 RegisterAllComponents 가 호출
각 호출 안에서 그 컴포넌트의 OnRegister 와, 
SceneComponent 라면 CreateRenderState_Concurrent, OnCreatePhysicsState 등이 차례로 돕니다


그 다음에 AActor::PreInitializeComponents 가 호출
이 단계에서 사용자가 가질 수 있는 보장
이 액터의 모든 컴포넌트는 등록되어 있다. 즉 모든 컴포넌트가 첫째 readiness 에 도달


bWantsInitializeComponent 가 true 인 것들에 대해 InitializeComponent 를 호출


PostInitializeComponents 
둘째 readiness 가 모든 컴포넌트에 대해 보장된 시점입니다.


BeginPlay 
이 액터뿐 아니라 월드에 있는 다른 액터들도 자기 초기화를 마친 상태
