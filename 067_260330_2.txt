GetControlRotation()
캐릭터의 방향이 아니라 컨트롤러의 방향
플레이어가 카메라로 바라보고 있는 방향

Pitch = 0 
Pitch를 0으로 강제하는 것은 "수평면에 투영한다"는 의미다
캐릭터는 어디를 바라보든 수평으로만 이동한다

ControlRotation (원본)           ControlRotation (처리 후)
Pitch: 30° (위 바라봄)     →     Pitch: 0°
Yaw:   45° (북동쪽)        →     Yaw:  45°
Roll:  0°                  →     Roll: 0°

ControlRot.Vector()
→ Yaw 방향의 단위 전방 벡터
   "내가 바라보는 방향이 월드에서 어느 쪽인가"

ControlRot.RotateVector(FVector::RightVector)
→ 전방 벡터에 수직인 우측 벡터
   "내가 바라보는 방향 기준으로 오른쪽이 월드에서 어느 쪽인가"

FVector::RightVector는 엔진 기본 좌표계의 오른쪽(Y축)
이것을 컨트롤 회전으로 돌려야 비로소 현재 시점 기준의 오른쪽이 된다

플레이어가 북쪽을 바라볼 때의 오른쪽(동쪽)과 
동쪽을 바라볼 때의 오른쪽(남쪽)은 다르기 때문이다

SpringArm의 bUsePawnControlRotation = true
"카메라는 컨트롤러 회전을 따른다"는 선언

플레이어가 마우스를 오른쪽으로 돌림
    ↓
Controller::ControlRotation이 바뀜
    ↓
SpringArm이 그 회전을 따라감 → 카메라가 오른쪽으로 돌아감 (시각)
Move()에서 GetControlRotation()이 새 값을 가져옴 → 이동 기준도 바뀜 (이동)

  문을 여는 동작이 항상 같은 거리에서 실행되어야 했다. 
  Motion Warping은 루트 모션을 런타임에 변형해서 
  캐릭터가 실제 문 위치에 정확하게 맞춰 동작하도록 한다.

Motion Matching이 도입됐다
  State Machine은 "지금 이 상태이면 이 애니메이션을 재생하라"는 규칙 기반 시스템이다. 
  상태가 늘어날수록 State Machine이 폭발적으로 복잡해진다.
  대량의 애니메이션 데이터를 데이터베이스로 두고, 
  현재 캐릭터의 운동 상태(위치, 속도, 예측 궤적)와 가장 유사한 포즈를 매 프레임 검색한다

State Machine:           Motion Matching:
규칙 → 애니메이션        현재 상태 → DB 검색 → 가장 유사한 포즈
경우의 수를 모두 정의    데이터가 경우의 수를 스스로 커버
           
**Full Body IK (FBIK)**와 IK Rig/IK Retargeter가 도입됐다
  서로 다른 체형의 스켈레톤 사이에 애니메이션을 자동으로 이식하는 도구다

이 CDO의 해당 프로퍼티가 갱신
.uasset 파일은 이 CDO의 프로퍼티 값들을 직렬화해서 저장

UBlueprintGeneratedClass
부모 CDO와 다른 기본값 정보를 표현할 수 있다

Blueprint에서 이 값을 지정하는 것은
“CastEffect라는 슬롯에 기본값을 넣는다”는 뜻이고, 
그 기본값은 그 Blueprint가 생성한 클래스의 CDO 값으로 저장돼요

Class Defaults 탭이 바로 그 값을 보여주는 거예요

Blueprint는 C++ 클래스의 인스턴스가 아니라, 그 위에 새 클래스를 하나 더 만든다

보이는 형태는 포인터 같지만, 저장과 로드의 배후에는 Unreal의 package/object reference 체계가 있어요

C++ 쪽 표면:
  TObjectPtr<UNiagaraSystem>

에디터 쪽 표면:
  "Niagara System asset picker"

저장/로드 배후:
  serialized UObject reference
  + package/object path resolution
