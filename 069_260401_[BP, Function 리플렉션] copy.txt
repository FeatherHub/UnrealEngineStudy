the diamond inheritance problem with UObject

Multiple UObject inheritance is forbidden
systems assume single UObject inheritance chain

UE's GC and reflection systems
make assumptions about the exact memory layout of UObject subclasses
all depend on knowing exactly where fields are relative to the object pointer

Virtual inheritance breaks these assumptions
Virtual inheritance changes the memory layout of objects
the base class offset becomes dynamic (stored in the vtable)

vptr 
Stored at a fixed, known offset within the object (usually offset 0)
The compiler knows exactly where it is at compile time for each class
Field offsets (UPROPERTY offsets) remain compile-time constants relative to this

vbptr virtual base table pointer
The problem is specifically about the base class subobject offset
With virtual inheritance, casting from ARogueItemChest* to UObject* requires runtime indirection through the vbtable
The this pointer adjustment when converting between derived and virtual base is determined at runtime, not compile time

When you have AActor : virtual UObject and 
URogueInteractionInterface : virtual UObject, and 
ARogueItemChest : AActor, URogueInteractionInterface, 
the single UObject subobject's position relative to the ARogueItemChest "this" pointer 
can only be determined at runtime

The UPROPERTY offsets are stored relative to each subobject
not the most-derived object
so they should remain compile-time constants

The GC identifies an object as ARogueItemChest
and walks its UClass hierarchy to find all UPROPERTYs
(모든 멤버 집합 중 UPROPERTY 로 마크업된 멤버 부분 집합이 자동 관리의 대상으로 등록된다)

Across MSVC, Clang, and GCC, virtual inheritance produces different vtable structures

Virtual override는 어떤 코드가 실행되는가를 런타임에 결정하지만, 데이터의 위치는 건드리지 않아.
vptr은 객체의 맨 앞 고정 오프셋에 놓이고, 그 크기는 컴파일 타임에 알려진 상수야

Virtual inheritance는 다른 문제를 일으켜
virtual base subobject — 즉 UObject의 실체 데이터 덩어리 — 의 위치 자체가 런타임이 되어야 결정돼
컴파일러는 vbtable(virtual base table)을 도입하고, 
"derived pointer에서 virtual base까지의 오프셋"을 런타임에 vbtable을 통해 간접 참조해야 해

호출 시 C++ 구현과 Blueprint 구현 중 무엇을 탈지 결정하는 dispatch 레이어

Blueprint에서 함수 호출은 ProcessEvent + 바이트코드 실행

UFunction 메타데이터를 통해 런타임에 동적으로 dispatch돼.

C++에서 타입의 정체성은 vtable 슬롯으로 표현
다형성은 포인터 역참조와 함수 포인터 호출이야

C++의 타입 정보는 빌드 타임에 소멸한다
런타임에 "이 클래스에 어떤 프로퍼티가 있는지"를 동적으로 묻고 답할 방법이 없다
UClass, UFunction, FProperty로 이루어진 타입 시스템이 런타임 내내 살아있는 객체로 존재한다

Blueprint 노드 그래프 (UEdGraph)
        │
        ▼
  FKismetCompilerContext        ← Blueprint 전용 컴파일러
  ├─ 그래프 검증 (타입 체크, 핀 연결 검사)
  ├─ Statement 생성 (중간 표현)
  ├─ Bytecode 생성 → UFunction::Script (TArray<uint8>)
  └─ UBlueprintGeneratedClass 갱신

각 이벤트/함수에 대응하는 UFunction 객체들이고, 
그 안의 Script 필드에 Blueprint VM이 해석할 바이트코드가 들어간다

UBlueprintGeneratedClass 
이 Blueprint 타입의 런타임 타입 기술자

UBlueprintGeneratedClass ("BP_MyCharacter_C")
  │
  ├─ SuperClass: ACharacter (C++ 부모 클래스)
  ├─ Children UFunction들
  │     ├─ "EventBeginPlay" → Script: [0x0D, 0x19, ...바이트코드...]
  │     ├─ "EventTick"      → Script: [...]
  │     └─ "MyCustomFunc"   → Script: [...]
  ├─ FProperty들 (Blueprint에서 추가한 변수들)
  └─ ClassDefaultObject (CDO)

C++ 클래스의 UClass가 UHT가 생성한 정적 코드로 만들어지는 것과 달리
UBlueprintGeneratedClass는 Blueprint 컴파일러가 런타임에 동적으로 생성하고 갱신한다
이것이 Blueprint가 재빌드 없이 즉시 변경될 수 있는 이유다.

CDO와 레벨 에디터 즉시 반영: Reinstancing

UE의 모든 UClass는 자신의 **CDO(Class Default Object)**를 하나 가진다
에디터에서 프로퍼티 기본값을 편집하면 실제로는 CDO를 편집하는 것이고, 
레벨에 배치된 인스턴스들은 CDO로부터 프로퍼티를 상속받는다.

Blueprint를 컴파일하면 UBlueprintGeneratedClass가 바뀐다
이때 레벨에 이미 배치된 인스턴스들이 문제가 된다
그것들은 이전 버전의 클래스에 속해있기 때문이다
이를 처리하는 것이 FBlueprintCompileReinstancer다.

Blueprint 컴파일 완료
        │
        ▼
FBlueprintCompileReinstancer
  │
  ├─ 레벨 내 BP_MyCharacter 인스턴스 전체 수집
  ├─ 각 인스턴스의 현재 프로퍼티 값을 직렬화로 추출
  ├─ 인스턴스를 새 UBlueprintGeneratedClass로 재생성
  └─ 추출해둔 프로퍼티 값을 새 인스턴스에 역직렬화로 복원

레벨에 배치된 인스턴스들은 이 새 CDO를 기준으로 
"CDO와 다른 값"만 오버라이드로 들고 있는 구조이므로, 
CDO가 바뀌면 오버라이드하지 않은 프로퍼티는 자동으로 새 기본값을 따른다.

이미 컴파일된 바이트코드가 메모리에 있고, 
PIE는 그것을 그대로 사용한다.

[에디터 월드]                     [PIE 월드]
  UWorld (Editor)     복제 →      UWorld (PIE)
    │                               │
    ├─ AMyCharacter 인스턴스        ├─ AMyCharacter 복사본
    ├─ AFloor 인스턴스              ├─ AFloor 복사본
    └─ ...                          └─ ...
         ↑                              ↑
  같은 UBlueprintGeneratedClass 를 공유
  같은 UFunction::Script 바이트코드를 공유

그 인스턴스들이 ProcessEvent()를 통해 
클래스에 있는 같은 바이트코드를 실행한다.

PIE 실행 중 Blueprint를 수정하고 다시 컴파일하면 
에디터 월드의 클래스가 바뀌지만, 
PIE 월드는 이미 실행 중이라 영향을 받지 않는다
PIE를 재시작해야 새 코드가 반영되는 이유가 여기 있다.

int main()
OS 로더와 C 런타임 사이의 계약이다

Windows에서 .exe를 실행하면 
로더는 그 바이너리의 PE 헤더를 읽어서 
"진입점(entry point) 주소"를 찾는다

C 런타임 라이브러리(CRT)가 그 진입점에 앉아서 
스택과 힙을 초기화한 다음, 
최종적으로 main() 또는 WinMain()을 호출한다. 
링커가 이 계약을 알고 있기 때문에, 
main()이 없으면 "unresolved external symbol" 에러를 낸다

DLL은 다른 종류의 바이너리다
.dll은 실행파일이 아니다
OS가 직접 실행하는 것이 아니라, 
다른 프로세스에 의해 주소공간 안으로 매핑되는 바이너리다.

.exe → OS가 직접 실행 → 진입점(main) 필요
.dll → 다른 프로세스가 LoadLibrary()로 로드 → 진입점 불필요

OS 층위의 모듈: 단순히 .dll 파일 하나다. 
운영체제가 LoadLibrary()로 프로세스 주소공간에 매핑하는 바이너리 단위.

UE 층위의 모듈: IModuleInterface를 구현한 C++ 클래스 하나다
이것이 UE가 각 DLL에 부여하는 논리적 생명주기 관리 단위다.

OS 층위:  LoadLibraryW("UnrealEditor-MyGame.dll")
               │
               ▼
UE 층위:  FModuleManager가 "InitializeModule" 심볼을 찾아
          IModuleInterface* 를 얻어냄
               │
               ├─ StartupModule()   ← DLL이 주소공간에 들어왔을 때
               └─ ShutdownModule()  ← DLL이 언로드될 때

StartupModule()이 UE 모듈의 실질적 진입점이다. 
여기서 UClass 등록, 팩토리 등록, 서브시스템 초기화 같은 
"이 모듈이 UE 시스템에 자기 자신을 알리는" 작업들이 일어난다.

MyGame 모듈은 코드를 들여다보면 어딘가에 이런 구조가 있다
class FMyGameModule : public IModuleInterface
{
public:
    void StartupModule() override { /* 초기화 */ }
    void ShutdownModule() override { /* 정리 */ }
};

IMPLEMENT_MODULE(FMyGameModule, MyGame)
// ↑ 이 매크로가 "InitializeModule" 심볼을 DLL에 심어넣는다

내 프로젝트는 실행파일 전체가 아니라, 엔진이 불러오는 모듈 조각
UE C++ 프로젝트는 보통 “프로그램 전체”를 직접 만드는 게 아니라, 
“이미 존재하는 엔진 프로그램이 불러올 모듈”을 만드는 것

IDE가 빌드 과정에서 호출하는 도구

런타임 동적 링커(Dynamic Linker / Loader)
OS의 일부다
Windows에서는 ntdll.dll 안의 로더가 담당한다
LoadLibrary()가 호출되거나 프로세스가 시작될 때 OS 커널이 자동으로 개입한다

빌드 타임 링커는 "심볼의 주소를 확정"한
런타임 동적 링커는 "DLL이 메모리 어디에 올라갔는지를 확정"한다.

DLL은 빌드될 때 자기가 메모리의 몇 번지에 로드될지 모른다
그래서 DLL 안의 코드는 재배치 가능한(relocatable) 형태로 컴파일된다

런타임 로더가 DLL을 실제로 메모리에 올린 다음, 
"이 DLL은 0x7FF00000에 올라갔으니, 
내부의 모든 주소 참조를 그에 맞게 수정"하는 작업을 한다. 
이것을 **재배치(relocation)**라 한다

빌드 타임 링커 (link.exe):
  MyCharacter.obj + Engine.lib → UnrealEditor-MyGame.dll

런타임 동적 링커 (OS ntdll 로더):
  LoadLibrary("UnrealEditor-MyGame.dll") 호출 시
  "이 DLL을 0x7FF012340000에 올렸다"
  "TakeDamage의 실제 주소 = 0x7FF012340000 + 0x3A20 = 0x7FF01234_3A20"

IAT (Import Address Table) 
DLL 간 의존 관계

빌드 타임 링커가 IAT라는 포인터 테이블을 DLL 안에 만들어두고
런타임 로더가 `CoreUObject.dll`을 로드한 뒤 실제 주소를 그 테이블에 채워넣는다

UnrealEditor-MyGame.dll 내부:
  IAT[ProcessEvent] = ???        ← 빌드 타임: 모름

런타임 로더:
  CoreUObject.dll 로드
  ProcessEvent 실제 주소 = 0x7FF0_ABCD
  IAT[ProcessEvent] = 0x7FF0_ABCD  ← 채워넣음

빌드 타임 링커는 모든 .obj 파일을 훑어서 static 객체 생성자 포인터들을 .CRT$XCU라는 섹션에 모아둔다.
DLL이 로드될 때 CRT가 이 섹션을 순회하며 포인터들을 하나씩 호출한다

GUObjectArray에 들어간 순간 그 객체는 UE의 Garbage Collector 관할에 들어간다
GC는 이 배열을 주기적으로 순회하며 참조 추적을 수행한다

그 경우들
1. Blueprint VM
   노드 그래프 실행 중 TakeDamage 노드를 만남
   → 이름 "TakeDamage"로 UFunction 조회
   → ProcessEvent 호출
   → thunk가 FFrame에서 파라미터 복원
   → TakeDamage(50, 3.0f) 호출

2. 네트워크 RPC
   서버에서 클라이언트로 함수 호출을 직렬화해서 전송
   받은 쪽: 패킷에서 "TakeDamage", [50, 3.0f] 꺼냄
   → ProcessEvent 호출
   → thunk가 복원

3. 리플렉션 기반 동적 호출
   UFunction* Func = FindFunction("TakeDamage");
   ProcessEvent(Func, &Params);
   → thunk 경유

4. AddDyanmic
    함수 포인터가 아니라 이름 문자열 "OnOverlapBegin"을 저장
    Broadcast() 시 FindFunction("OnOverlapBegin") → ProcessEvent

vtable 자체:
  → 클래스당 하나. 바이너리의 .rdata 섹션에 존재.
  → 컴파일/링크 타임에 확정되어 바이너리에 구워짐.
  → 스택도 힙도 아님. 프로세스가 살아있는 동안 영원히 같은 주소.
  → 읽기 전용. 런타임에 수정 불가.

vptr (vtable을 가리키는 포인터):
  → 인스턴스마다 하나. 객체 메모리의 맨 앞에 박힘.
  → 객체가 스택에 있으면 vptr도 스택에.
  → 객체가 힙에 있으면 vptr도 힙에.
  → 생성자에서 초기화됨.

type_info 객체:
  → 클래스당 하나. 역시 .rdata 섹션.
  → vtable이 포인터로 참조.

MSVC, Clang, GCC가 각자 자기 방식으로 vtable을 만든다
컴파일러가 암묵적으로 생성해서 바이너리에 심어두는 것이다

static 멤버는 BSS/data 섹션에, vtable은 .rdata 섹션에 있다

class AMyCharacter : public APawn // 전체 크기 컴파일러가 계산
    { int health; };

컴파일러는 이 선언들을 보는 순간 `AMyCharacter`의 전체 크기와 각 멤버의 오프셋을
**컴파일 타임에 완전히 확정**한다

sizeof(AMyCharacter) = 20바이트
오프셋 0:  UObject::a
오프셋 4:  AActor::b
오프셋 8:  AActor::c
오프셋 12: APawn::d
오프셋 16: AMyCharacter::health

NewObject<AMyCharacter>()가 하는 일은

void* mem = FMemory::Malloc(sizeof(AMyCharacter));
// ↑ 컴파일러가 이미 계산한 전체 크기만큼 한 번에 할당
// UClass 체인을 순회하며 크기를 계산하는 것이 아님

new (mem) AMyCharacter();
// placement new로 생성자 호출



컴파일 타임:  컴파일러가 AMyCharacter 전체 레이아웃 확정
                → sizeof, 각 멤버 오프셋 전부 확정
                → 바이너리에 구워짐

런타임:       NewObject()가 sizeof(AMyCharacter) 크기 블록 한 번에 할당
              UClass 체인을 따라다니며 메모리를 조각조각 할당하는 일 없음

// AActor 안:
virtual void BeginPlay();           // C++ virtual. UFUNCTION 아님.

UFUNCTION(BlueprintImplementableEvent)
virtual void ReceiveBeginPlay();    // Blueprint용 별도 함수.

두 개가 **별도로 존재**한다. 이것이 핵심이다.

UWorld::InitializeActorsForPlay()
        │
        ▼
AActor::DispatchBeginPlay()
        │
        ▼
AActor::BeginPlay()          ← virtual 함수
        │
        ├─ vtable 디스패치
        │       │
        │       └─ AMyActor::BeginPlay()  ← 네 C++ 오버라이드 직접 실행
        │                                    Thunk 없음. ProcessEvent 없음.
        │
        └─ ReceiveBeginPlay() 호출
                │
                └─ ProcessEvent("ReceiveBeginPlay")
                        │
                        ├─ Blueprint 오버라이드 있으면 → VM 실행
                        └─ 없으면 → 아무것도 안 함

---

BlueprintNativeEvent 

호출자
  │
  ▼
TakeDamage()  ← UHT가 생성한 dispatcher
  │
  ├─ Blueprint 오버라이드 있으면 → ProcessEvent → VM
  └─ 없으면 → TakeDamage_Implementation() (C++)
             (C++ OR Blueprint, 둘 중 하나)

DispatchBeginPlay (Epic이 손으로 작성한 dispatcher)

엔진
  │
  ▼
DispatchBeginPlay()
  │
  ├─ BeginPlay() → vtable → C++ 오버라이드
  └─ ReceiveBeginPlay() → ProcessEvent → VM
             (C++ AND Blueprint, 둘 다)

엔진은 항상 내 코드를 직접 호출하지 않는다. 반드시 dispatcher를 통한다.

// 케이스 1: 부모가 UFUNCTION 없이 선언
class AActor {
    virtual void OnHit();  // UFUNCTION 없음
};

class ASMyActor : public AActor {
    UFUNCTION(BlueprintNativeEvent)
    virtual void OnHit();  // X
};

// 케이스 2: 부모가 UFUNCTION으로 선언
class AMyBase {
    UFUNCTION(BlueprintNativeEvent)
    virtual void OnHit();      // UFUNCTION으로 최초 선언
};

class AMyDerived : public AMyBase {
    UFUNCTION(BlueprintNativeEvent)
    virtual void OnHit() override;  // O
};

UFUNCTION 스펙 자체를 파생 클래스에서 변경하는 것을 UHT가 허용하지 않는다
최초 선언 시점에 확정된 계약을 파생 클래스가 바꿀 수 없다
