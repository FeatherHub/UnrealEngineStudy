모든 에셋은 UObject의 서브클래스
디스크 상에서는 UPackage라는 컨테이너 안에 직렬화(serialize)된 형태로 존재

UObject 시스템이 이미 제공하는 
리플렉션, 직렬화, 가비지 컬렉션이라는 세 가지 인프라를 
에셋을 위한 별도의 저장/로딩 메커니즘을 만들 필요가 없어집니다

UPROPERTY로 선언된 필드는 자동으로 직렬화되고, 
에셋 간의 참조는 GC가 추적하며, 
에디터에서의 프로퍼티 편집도 리플렉션이 처리합니다.

---

UPackage는 디스크의 .uasset 파일 하나에 대응하는 논리적 단위입니다
하나의 패키지 안에는 보통 
하나의 "주요 에셋"(Primary Asset)과 
그것이 의존하는 여러 "내부 객체"들이 함께 들어갑니다
스태틱 메시 에셋 하나의 패키지 안에는
UStaticMesh 객체 자체뿐 아니라 
UBodySetup(콜리전 데이터), UMaterial 슬롯 정보 등이 함께 존재
디스크 I/O의 효율성 때문

UE3에서는 하나의 패키지에 여러 주요 에셋
UE4에서 "하나의 패키지 = 하나의 주요 에셋" 원칙으로 전환
버전 관리 시스템과의 통합
바이너리 포맷인 .uasset은 텍스트처럼 머지할 수 없기 때문

"어떤 파일인가"에 대한 답

/Game/Characters/Hero/SK_Hero.SK_Hero
내부 서브오브젝트는 콜론(:)으로 더 깊이
"그 파일 안의 어떤 객체인가"에 대한 답

에셋을 "참조"할 수 있어야 하기 때문
그 텍스처 패키지가 메모리에 없어도 경로 문자열만으로 관계를 기록

하드 레퍼런스
일반적인 UObject 포인터 UPROPERTY로 선언
소유자 객체가 로딩될 때 참조 대상도 반드시 함께 로딩
"참조 체인 폭발"

소프트 레퍼런스(FSoftObjectPath, TSoftObjectPtr)
경로 문자열만 저장하고, 
실제 객체 로딩은 프로그래머가 명시적으로 요청할 때까지 일어나지 않습니다. 
이것은 에셋 시스템에서의 "lazy evaluation"입니다

게임의 초기 로딩 시간과 메모리 사용량이
이 참조 그래프의 구조에 의해 결정

의존성의 긴급도를 표현

가벼운 메타데이터 캐시를 구축합니다

Streaming과 비동기 로딩

에셋을 동기적으로(즉, 로딩이 끝날 때까지 게임을 멈추고) 로딩하면 
"히칭(hitching)"이라 불리는 프레임 드랍이 발생

FStreamableManager와 비동기 로딩 API
에셋 로딩 요청을 발행하고, 로딩이 완료되면 콜백을 받는 방식
텍스처의 MIP 스트리밍(저해상도부터 먼저 보여주고 고해상도를 점진적으로 로딩),

쿠킹(Cooking): 에디터 포맷에서 런타임 포맷으로
에디터에서 작업하는 .uasset 파일과 최종 빌드에서 사용하는 에셋은 같지 않습니다
"쿠킹" 과정에서 에디터 전용 데이터가 제거되고, 대상 플랫폼에 최적화된 포맷으로 변환됩니다
텍스처는 플랫폼별 압축 포맷으로 변환되고, 셰이더는 대상 GPU 아키텍처용으로 컴파일됩니다.

이 과정에서 .uasset은 .uexp(에셋 데이터)와 .ubulk(대용량 벌크 데이터) 파일로 분리될 수 있으며, 
또한 여러 에셋이 하나의 .pak 파일로 묶입니다. 
.pak 파일은 본질적으로 가상 파일 시스템으로, 
수천 개의 개별 파일 대신 소수의 큰 아카이브를 읽는 것이 
디스크 I/O에서 훨씬 효율적이기 때문입니다

바이트 스트림 — 디스크와 메모리의 경계

에셋은 디스크 위의 바이트 시퀀스입니다
메모리의 상태를 디스크에 기록하고 복원할 수 있어야 합니다

FArchive라는 추상 스트림 클래스를 이 층위의 기반
FArchive는 << 연산자를 오버로딩해서 읽기와 쓰기를 동일한 코드 경로로 처리
같은 함수가 FArchive의 방향(로딩인지 세이빙인지)에 따라 반대 동작을 수행
직렬화 코드를 한 번만 작성하면 저장과 로딩이 모두 해결된다는 것입니다. 
읽기/쓰기 로직이 분리되어 불일치가 발생하는 버그를 구조적으로 방지합니다.

FArchiveFileReaderGeneric은 디스크 파일에, 
FMemoryReader는 메모리 버퍼에, 
FNetBitReader는 네트워크 스트림에 
동일한 FArchive 직렬화 로직을 적용합니다

---

원시 바이트 스트림 위에, UObject의 직렬화 시스템이 놓입니다
UPROPERTY 메타데이터를 이용하면 
각 클래스마다 수동으로 직렬화 코드를 작성할 필요 없이, 
리플렉션 정보를 순회하면서 프로퍼티들을 자동으로 읽고 쓸 수 있습니다.

Tagged Property Serialization
버전 호환성
게임이 업데이트되면서 클래스에 프로퍼티가 추가되거나 제거되거나 타입이 바뀔 수 있습니다. 
이전 버전에서 저장된 에셋을 새 버전의 엔진이 로딩할 수 있어야 합니다. 
UE는 이를 위해 프로퍼티를 이름-값 쌍으로 태깅해서 저장합니다. 
로딩 시 현재 클래스에 해당 이름의 프로퍼티가 없으면 건너뛰고, 새로 추가된 프로퍼티는 기본값을 사용합니다. 

**“시간을 견디는 객체 저장 형식”**이라는 요구에 대한 답

UPackage — 물리적 컨테이너와 .uasset 포맷
하나의 에셋은 보통 여러 개의 하위 객체를 포함하고, 다른 패키지의 객체를 참조합니다
이 모든 것을 하나의 일관된 파일로 묶어야 합니다
이것이 UPackage의 역할입니다

.uasset 파일의 물리적 구조를 보면
파일 헤더에는 매직 넘버, 엔진 버전, 커스텀 버전 정보가 들어갑니다
그 다음 Name Table이 옵니다
이 패키지 안에서 사용되는 모든 FName 문자열의 목록으로, 
본문에서는 문자열 대신 이 테이블의 인덱스만 사용합니다

Import Table은 이 패키지가 외부에서 참조하는 객체들의 목록
Export Table은 이 패키지가 다른 패키지에 제공하는 객체들의 목록
이 두 테이블이 에셋 간 의존성 그래프의 물리적 실체

---

FLinkerLoad

UPackage 구조를 실제로 해석해서
UObject들을 메모리에 복원하는 주체

.uasset 파일을 열고, 헤더를 파싱하고, Name/Import/Export 테이블을 읽고, 
필요한 외부 패키지의 Linker를 재귀적으로 생성하고, 
최종적으로 Export 객체들을 역직렬화

컴파일러 링커가 여러 오브젝트 파일의 심볼 참조를 해결하듯, 
FLinkerLoad는 여러 패키지 간의 객체 참조를 해결합니다.

헤더와 테이블을 읽는 "Summary" 단계
외부 Import를 해결하는 "Preload" 단계
실제 객체 데이터를 역직렬화하는 "Load" 단계
로딩 후 초기화를 수행하는 "PostLoad" 단계

이 단계적 접근은 순환 참조 문제를 해결하기 위한 것
패키지 A가 B를 참조하고 B가 A를 참조할 때, 
모든 패키지의 "껍데기"(Export 엔트리)를 먼저 만들어두고 
실제 데이터는 나중에 채우는 방식으로 교착을 방지합니다.

경로 시스템과 에셋 레지스트리 — 주소 지정과 색인

"특정 에셋 하나를 어떻게 로딩하는가"라는 저수준 문제에서 
"수만 개의 에셋 전체를 어떻게 관리하는가"라는 중간 수준의 문제로 전환

경로 시스템은 디스크의 물리적 위치를 논리적 이름 공간으로 추상화
FPackageName 클래스가 이 매핑을 관리
마운트 포인트 시스템을 통해 임의의 물리 경로를 논리 경로에 연결할 수 있습니다. 
이것은 OS의 파일시스템 마운트와 같은 개념입니다

FAssetData와 IAssetRegistry는 
이 경로 공간 위에 구축된 
메타데이터 색인

Asset Registry
에셋의 메타데이터를 에셋 자체를 로딩하지 않고 파악
각 .uasset 파일의 헤더 부분에 
Asset Registry Tag라는 키-값 쌍이 포함되어 있고, 
Asset Registry는 이 태그만 빠르게 스캔해서 
인메모리 데이터베이스를 구축

에디터에서는 시작 시 Content 디렉토리를 비동기적으로 스캔하고, 
쿠킹된 빌드에서는 DevelopmentAssetRegistry.bin이라는 
사전 생성된 파일에서 한 번에 로딩합니다

참조 시스템 — 의존성 그래프

하드 레퍼런스(TObjectPtr, 직접 UObject 포인터)
이 에셋이 존재하려면 저 에셋이 반드시 있어야 한다
로딩이 전이적(transitive)으로 전파

소프트 레퍼런스(TSoftObjectPtr, FSoftObjectPath)
이 에셋이 저 에셋을 알고 있지만, 당장 필요하지는 않다
실제 로딩은 명시적 요청 시에만 발생

소프트 클래스 레퍼런스(TSoftClassPtr)
UClass 자체를 지연 로딩
로딩 비용이 높은 클래스
블루프린트 클래스를 하드 참조하면 
그 블루프린트가 참조하는 모든 에셋이 연쇄 로딩

프로젝트의 로딩 프로파일
Reference Viewer(에디터 도구)로 에셋의 의존성 그래프를 시각화하면
잘 설계된 프로젝트는 느슨하게 연결된 클러스터들로 보이고, 
문제가 있는 프로젝트는 모든 것이 모든 것에 연결된 거대한 덩어리로 보입니다

---

동기 로딩(StaticLoadObject, LoadObject)은 
호출 스레드를 블로킹하고 
에셋이 완전히 로딩될 때까지 기다립니다.

비동기 로딩(FStreamableManager, LoadPackageAsync)
로딩 요청을 발행하고 즉시 반환
실제 로딩은 비동기 로딩 스레드(AsyncLoadingThread)에서
완료 시 델리게이트 콜백이 호출
FStreamableHandle을 통해 로딩 진행 상태를 조회
여러 에셋의 로딩이 모두 완료될 때까지 기다리는 복합 요청

Event-Driven Loader(EDL)는 UE 4.20 이후의 비동기 로딩 백엔드
패키지 로딩의 각 단계를 이벤트로 모델링해서 
여러 패키지의 로딩을 인터리빙(interleaving)합니다. 
하나의 패키지가 I/O를 기다리는 동안 
다른 패키지의 직렬화 작업을 진행할 수 있는 것입니다.

Insight: React 18 아키텍처 

IO Dispatcher(FIoDispatcher)는 더 아래 층위에서 
디스크 I/O 요청을 최적화합니다. 
특히 .pak 파일이나 IoStore 컨테이너에서 여러 에셋을 로딩할 때, 
I/O 요청을 병합하고 순서를 재배치해서 디스크 시크를 최소화합니다.

---

레벨 스트리밍은 World를 여러 Sub-Level로 분할하고, 
플레이어의 위치나 게임 로직에 따라 Sub-Level을 동적으로 로딩/언로딩
World Partition(UE5)은 이것의 진화형
월드를 격자(grid)로 분할해서 스트리밍 단위를 자동화

오디오 스트리밍
긴 음악 파일을 청크 단위로 디코딩하면서 재생
전체 파일을 메모리에 올릴 필요가 없습니다.

스트리밍 시스템의 공통 패턴
에셋의 데이터를 "항상 필요한 핵심부"와 "상황에 따라 필요한 가변부"로 분리

스트리밍은 비동기 로딩의 특수한 적용
"전체를 한 번에"가 아닌 "필요한 부분을 점진적으로" 로딩
텍스처의 벌크 데이터가 .uasset에서 분리되어 .ubulk 파일에 저장되는 이유가 여기
헤더와 메타데이터는 항상 필요하지만, MIP 데이터는 선택적으로 로딩

Insight: 스트리밍은 읽거나 쓰는 방식뿐만 아니라 데이터 형식까지 시야에 넣어야 하는 개념이다.
즉 순전한 작용의 문제라기보다는 작용자-피작용자의 조화, 협력, 설계의 문제이다.

Asset Manager

Primary Asset은 독립적으로 로딩/해제의 단위가 되는 에셋

Asset Bundle은 Primary Asset에 연관된 에셋들의 명명된 집합
메뉴 화면에서는 UI 번들만 로딩하고, 
실제 게임플레이에 진입할 때 Gameplay 번들을 추가로 로딩하는 식의 
제어가 가능해집니다

Primary Asset Rules
각 Primary Asset의 쿠킹 및 청크 할당 정책을 정의합니다. 
어떤 에셋이 어떤 .pak 청크에 들어갈지, 
어떤 에셋이 "항상 로딩" 상태를 유지할지를 지정

Insight: 먼저 하나라는 단위를 정립한다. 그리고 여럿을 하나로 모으는 (묶는) 층위가 얹힌다.

---

쿠킹 파이프라인
개발 중에는 .uasset이 에디터 친화적 포맷으로 존재
쿠킹은 이것을 특정 대상 플랫폼에 최적화된 형태로 변환
대상 플랫폼에서 불필요한 데이터 제거, 
텍스처 포맷 변환(PC의 BC7, 모바일의 ASTC 등), 
셰이더 크로스 컴파일, 
에셋 의존성 분석을 통한 청크 분배, 
최종적으로 .pak 또는 IoStore 컨테이너로의 패키징

IoStore(.ucas/.utoc)는 UE5에서 도입된 새로운 컨테이너 포맷으로, 
.pak보다 더 효율적인 I/O를 제공합니다. 
컨테이너 레벨의 압축, 빠른 검색을 위한 목차 구조, 그리고 on-demand 로딩에 최적화된 레이아웃

Insight: 공간 내의 배치는 Potential 을 가진다.
목적에 맞는 공간 배치를 통해 효율적인 Work를 설계한다
Layout은 가장 효과적인 Operation을 암시하고 있고
수행하고자 하는 Operation에서 그것에 가장 알맞는 대상의 형태/배치를 유도한다

형태는 굳어진 작용이고, 작용은 풀려난 형태
---

LoadObject(TEXT("/Game/Meshes/SM_Rock.SM_Rock"))
이 한 줄이 트리거하는 호출 체인

Phase 0: 문자열 경로 해석
/Game/Meshes/SM_Rock이라는 논리 경로를 물리적 파일 위치로 변환
FPackageName::TryConvertLongPackageNameToFilename이 호출
마운트 포인트 테이블을 조회
/Game/이 프로젝트의 Content/ 디렉토리에 매핑
../Content/Meshes/SM_Rock.uasset이라는 물리 경로가 산출

분기
해당 패키지가 이미 메모리에 있는가
FindObject로 글로벌 UObject 해시 테이블을 조회
이 "캐시 히트" 경로가 지배적이어야 성능이 나옵니다

Phase 1 FLinkerLoad 생성
캐시 미스라면, 실제 로딩이 시작됩니다
GetPackageLinker가 호출되어 
해당 패키지에 대한 FLinkerLoad 인스턴스를 생성

FLinkerLoad는 파일을 열고 Summary(헤더)를 읽습니다
매직 넘버로 파일 유효성을 확인하고, 
엔진 버전과 커스텀 버전을 읽어서 호환성을 검증하고, 
Name Table, Import Table, Export Table의 오프셋과 크기를 파악

Phase 2: Import 해결 — 재귀적 Linker 생성
Import Table의 각 엔트리는 "이 패키지가 외부에서 필요로 하는 객체"를 기술
SM_Rock이 특정 머티리얼을 참조하고 있다면, 
그 머티리얼 패키지에 대한 FLinkerLoad가 생성
하드 레퍼런스의 전이적 전파가 실제로 실행되는 지점

순환 참조의 해결
FLinkerLoad는 Export 엔트리에 대한 "플레이스홀더" UObject를 먼저 생성합니다
데이터는 나중 Phase에서 채워집니다

트리
  파일을 열고
  위에서부터 읽고
  객체를 완성
  하는 선형 절차 

---

Phase 3: Export 객체 생성과 직렬화
Import가 모두 해결되면
이 패키지가 "제공하는" 객체들(Export Table)을 실제로 생성하고 데이터를 채웁니다.
각 Export 엔트리에 대해 StaticConstructObject가 호출되어 UObject 인스턴스가 메모리에 할당됩니다
이 시점에서 객체는 존재하지만 프로퍼티 데이터는 비어있습니다

그 다음 각 객체의 Serialize 함수가 호출됩니다. 

여기서 FArchive가 등장합니다
FLinkerLoad 자체가 FArchive의 서브클래스이므로, 
객체의 Serialize(FArchive& Ar) 호출에 자기 자신을 전달합니다

리플렉션 시스템이 UPROPERTY들을 순회하면서, 
Tagged Property Serialization 포맷으로 저장된 이름-값 쌍을 읽어 
프로퍼티에 할당합니다.

이 과정에서 UObject 포인터 타입의 프로퍼티를 만나면, 
직렬화된 값은 실제 포인터가 아니라 Import/Export 인덱스입니다
FLinkerLoad가 이 인덱스를 앞서 해결해둔 실제 UObject 포인터로 치환합니다. 
이것이 "링킹"이라는 이름의 핵심 동작입니다

---

Phase 4: PostLoad
이 단계가 별도로 존재하는 이유는, 
직렬화 시점에는 참조 대상 객체가 아직 불완전한 상태일 수 있기 때문입니다.

같은 로딩 배치에 포함된 모든 객체가 직렬화를 마친 상태이므로, 
참조 대상의 데이터에 안전하게 접근할 수 있습니다.

UStaticMesh의 렌더 데이터 빌드,
UTexture의 플랫폼별 리소스 생성, 
UMaterial의 셰이더 맵 바인딩 등
이것들은 직렬화된 "원시 데이터"를 런타임이 실제로 사용할 수 있는 "가공된 리소스"로 변환하는 과정

비동기 로딩

LoadPackageAsync가 호출
로딩 요청이 FAsyncLoadingThread의 요청 큐에
호출자는 FStreamableHandle(또는 패키지 로딩의 경우 델리게이트)을 받고 즉시 반환
실제 I/O와 직렬화는 별도 스레드에서 진행

Event-Driven Loader의 실행 모델
로딩의 각 단계를 독립적인 "이벤트 노드"로 모델링
패키지 하나의 로딩이 선형적 파이프라인이 아니라 의존성 그래프가 되는 것입니다.

패키지 A의 I/O 완료를 기다리는 동안 패키지 B의 직렬화를 진행하고, 
B의 Import 해결이 C의 로딩 완료에 의존하면 C를 우선 처리하는 식의 
스케줄링이 가능해집니다

CreateLinker → SerializePackageFileSummary → CreateImportsAndExports → SerializeExports → CallPostLoad
이 각 노드 사이에 제어가 양보(yield)될 수 있고, 다른 패키지의 노드가 끼어들 수 있습니다

---

I/O 층위: FIoDispatcher
"디스크에서 바이트를 어떤 순서로 읽을지"를 최적화

.pak 파일 안에 여러 에셋이 순차적으로 배치되어 있을 때, 
패키지 A와 C가 파일의 인접한 영역에 있고 B가 먼 영역에 있다면, 
A와 C의 I/O 요청을 병합해서 하나의 큰 읽기로 처리하는 것이 디스크 시크를 줄입니다.

IoStore(.ucas/.utoc)는 이 원리를 더 극단적으로 추구한 포맷입니다. 
.utoc(Table of Contents)가 각 청크의 오프셋과 압축 정보를 기록하고, 
.ucas(Container Archive Storage)가 실제 데이터를 담습니다. 
쿠킹 시점에 함께 로딩될 가능성이 높은 에셋들을 물리적으로 인접하게 배치하는 최적화

---

게임 스레드와 로딩 스레드 사이의 동기화

로딩 스레드에서 하는 일
파일 I/O, 패키지 헤더 파싱, Name/Import/Export 테이블 처리, 프로퍼티 직렬화. 
이것들은 기존 UObject의 상태를 변경하지 않으므로 스레드 안전합니다.

반드시 게임 스레드에서 해야 하는 일
새 UObject의 생성(StaticConstructObject), PostLoad 호출, 로딩 완료 콜백 실행. UObject 생성은 글로벌 UObject 배열과 해시 테이블을 수정하므로 게임 스레드의 독점 영역입니다.

이 경계 때문에 비동기 로딩에는 "플러싱" 포인트가 존재
로딩 스레드가 준비해둔 작업(객체 생성, PostLoad 등)을 
게임 스레드가 매 틱마다 일정량씩 처리하는 것입니다

ProcessAsyncLoading이 매 프레임 호출되어, 
로딩 스레드가 큐에 넣어둔 "게임 스레드 작업"을 
시간 예산 내에서 소화합니다
이 시간 예산을 초과하면 남은 작업은 다음 프레임으로 미뤄집니다.

프레임의 시간 예산을 초과하면, 게임 스레드가 그 프레임에서 멈추게 됩니다

---

소프트 레퍼런스의 실행 경로
하드 레퍼런스가 FLinkerLoad의 Import 해결 단계에서 자동으로 재귀 로딩을 트리거

TSoftObjectPtr의 직렬화는 FSoftObjectPath(본질적으로 FTopLevelAssetPath + 서브패스 문자열)만 읽고 씁니다. 
Import Table에 엔트리를 만들지 않으므로, 
FLinkerLoad는 이 참조의 존재 자체를 모릅니다. 
의존성 그래프에 간선이 추가되지 않는 것입니다

나중에 게임 코드에서 
TSoftObjectPtr::LoadSynchronous()나 FStreamableManager::RequestAsyncLoad()를 호출하면, 
그때서야 위에서 설명한 전체 로딩 파이프라인이 해당 에셋에 대해 시작됩니다. 
하지만 이 시점은 프로그래머가 제어합니다

"lazy evaluation"

---

Asset Manager 
정책이 메커니즘을 구동
FStreamableManager를 내부적으로 소유하면서 로딩 요청을 위임

게임 코드가 UAssetManager::LoadPrimaryAsset(FPrimaryAssetId)을 호출
Asset Manager는 먼저 Asset Registry에서 해당 Primary Asset의 에셋 번들 정보를 조회
번들에 포함된 에셋들의 FSoftObjectPath 목록을 구성
이것을 FStreamableManager::RequestAsyncLoad에 전달

FStreamableManager는 각 경로에 대해 LoadPackageAsync를 호출
EDL/FIoDispatcher를 통해 실제 I/O로 이어집니다

Primary Asset이 "하나"이고, 
Asset Bundle이 "여럿을 하나로 묶는" 것이며, 
Asset Manager가 이 묶음 단위로 하위 시스템에 로딩을 요청하는 것입니다

---

쿠킹은 런타임 로딩의 "거울 이미지"
로딩이 .uasset → 메모리 객체 방향
쿠킹은 메모리 객체 → 플랫폼 최적화된 .uasset/.uexp/.ubulk 방향

UCookOnTheFlyServer(또는 커맨드렛 모드의 쿠커)
먼저 Asset Registry를 사용해서 쿠킹 대상 에셋 목록을 수집
각 에셋을 에디터에서 로딩(위의 전체 로딩 파이프라인이 동작)
대상 플랫폼용 데이터를 생성
결과를 새로운 패키지 파일로 직렬화

이번에는 "세이빙 방향"의 FArchive이 등장

쿠커는 Asset Manager의 Primary Asset Rules를 참조해서, 
각 에셋을 어떤 .pak 청크에 배치될지를 결정합니다

여러 Primary Asset이 공유하는 에셋은 공통 청크로, 
특정 레벨에서만 쓰이는 에셋은 해당 레벨의 청크로 분배됩니다. 
이 결정이 런타임의 I/O 패턴을 결정짓습니다.

---

소유자와 피소유자 & 고수준에서 저수준 & Chain of Delegation
Asset Manager → FStreamableManager → LoadPackageAsync → EDL → FLinkerLoad → FArchive → FIoDispatcher

각 층위는 자기 아래 층위에 작업을 위임하면서, 자기 수준의 추상화를 추가

Asset Manager는 "정책"을, 
FStreamableManager는 "핸들과 콜백"을, 
EDL은 "이벤트 스케줄링"을, 
FLinkerLoad는 "패키지 구조 해석"을, 
FArchive는 "바이트 스트림 추상화"를 담당

펼쳐놓기 & Lookup-then-Execute 

거의 모든 단계에서 "이미 있는가?"를 먼저 확인

FindObject로 이미 로딩된 객체를 확인하고, 
Asset Registry에서 메타데이터를 조회하고, 
패키지의 Linker가 이미 존재하는지 확인합니다. 

이 패턴은 불필요한 중복 작업을 방지하는 캐싱 전략입니다

---

분할-후-합류(Split-then-Join) & 분기와 합류

비동기 로딩에서 요청이 여러 패키지로 분산되었다가,
모든 의존성이 해결된 후에 
PostLoad와 콜백으로 합류합니다

FStreamableHandle이 여러 로딩 요청의 "조인 포인트" 역할을 합니다.

---

두 가지 존재 양식
메모리 안의 살아있는 UObject
디스크 위의 직렬화된 바이트 시퀀스
같은 것의 두 표현

어떤 것이 자신이 생성된 프로세스보다 오래 살아남아야 하거나, 
자신이 존재하는 주소 공간 바깥으로 전달되어야 할 때, 
반드시 직렬화/역직렬화라는 Transport/Translate 비용이 발생한다

---

대칭성의 강제는 정합성의 강제
FArchive의 방향성 설계
같은 코드가 읽기/쓰기 양쪽으로 동작
저장 경로와 복원 경로의 로직이 분리되면
두 경로가 점진적으로 어긋나는 것은 필연입니다

---

"안다"는 것의 비용 구조
한 엔티티가 다른 엔티티를 "안다"는 사실 자체가 비용을 발생시킨다
"아는 방식"이 그 비용의 시점과 규모를 결정한다.

하드 레퍼런스는 "존재적 앎"입니다. A가 B를 하드 참조하면, B 없이는 A 자체가 성립하지 않습니다
하나의 논리적 존재의 실체는 이 전이로 인해 그래프적인 물리적 존재의 총체라는 것을 볼 수 있어야 한다.

소프트 레퍼런스는 "잠재적 앎"입니다

안다는 것은 결합된다는 것이다.
이 결합의 비용(결합이 가져오는 모든 실제적, 인지적 effect)을 볼 수 있어야 한다.

---

대상 자체를 활성화하지 않고도 
대상에 대해 추론할 수 있는 경량 표상(representation)이 있으면, 
탐색과 결정의 비용이 극적으로 낮아진다

도서관의 카드 카탈로그, 파일 시스템의 inode, 데이터베이스의 인덱스, 
사과를 먹어보지 않고도 사과가 빨갛다는 것을 아는 것, 
파일을 열지 않고도 파일 크기를 아는 것. 
메타데이터는 대상의 형식화이며, 
내용이 아닌 형식만으로 판단할 수 있는 결정은 
대상 자체를 끌어올 필요가 없습니다

.uasset 헤더에 Asset Registry Tag를 포함시키는 설계 결정
파일의 앞부분만 읽으면 그 파일의 정체를 파악할 수 있게
쿠킹된 빌드에서 DevelopmentAssetRegistry.bin을 사전 생성하는 것은
메타데이터의 메타데이터를 만드는 것입니다. 
원본 파일들을 스캔하는 비용조차 제거한 것이죠.

"비싼 작업 앞에는 반드시 싼 확인이 선행해야 한다"

---

여러 에셋을 하나의 패키지에 넣으면 I/O 효율은 좋지만 버전 관리 충돌이 증가합니다. 
하나씩 분리하면 충돌은 줄지만 I/O 요청이 파편화됩니다

UE4는 개발 시에는 하나씩 분리(작업 충돌 최소화)하고, 
배포 시에는 .pak으로 묶는(I/O 최적화) 방식으로, 
두 상충하는 요구를 생명주기의 다른 단계에 배치해서 해결했습니다.

같은 데이터라도 "누가 수정하는가"의 관점에서는 잘게 쪼개야 하고,
"누가 읽는가"의 관점에서는 크게 묶어야 합니다
묶음의 최적 단위는 고정값이 아니라 맥락에 의해 결정됩니다.

소스 코드가 파일 단위로 관리되지만 배포는 바이너리로 링크되는 것, 
마이크로서비스가 독립 배포되지만 요청 처리 시에는 오케스트레이션되는 것

분해와 재결합은 서로 다른 기준으로 일어나고, 
그 기준의 차이가 시스템의 유연성을 결정한다

순환의 해결: 불완전한 존재의 허용

상호 의존하는 엔티티들의 집합을 초기화하려면, 
"부분적으로 존재하는 상태"를 허용해야 한다.
모든 객체의 "껍데기"를 먼저 만들어두고 내용은 나중에 채우는 것
"아직 완전하지 않지만 참조 가능한" 중간 상태를 도입

데이터베이스의 deferred constraint checking(트랜잭션 끝까지 외래 키 검증을 미룸), 
TCP의 three-way handshake(완전한 연결이 성립하기 전의 반개방 상태),
사회적으로도 "잠정적 합의 → 세부 조율"이라는 협상 구조

직렬화(데이터 복원)와 초기화(데이터 가공)를 분리
"모든 데이터가 복원된 후에야 가능한 작업"의 안전한 수행을 보장하는 시점을 확보
이것은 "존재의 단계"를 명시적으로 구분하는 설계입니다. 
객체가 "할당됨 → 직렬화됨 → 초기화됨"이라는 단계를 거치며, 
각 단계에서 보장되는 불변량(invariant)이 다릅니다

동기 로딩은 "지금 전부 지불"합니다
지불 시점에 다른 모든 활동이 멈춥니다

비동기 로딩은 "지금 약속하고 나중에 지불"합니다
지불 완료 시점이 불확실해집니다

스트리밍은 "할부"입니다
전액을 한 번에 지불하지 않고 필요에 따라 분할 지불합니다

현재 프레임에서 60fps를 유지하는 것

c.f. React 18 Concurrent Mode

복잡도 일반 원리
N개의 독립 주체 사이에서 일관성을 유지해야 할 때, 
중앙화하는 비용은 O(N)이고 
분산 상태로 일관성을 유지하는 비용은 O(N²)이다
   N×N 통신
   single source of truth 문제

N개의 소스 포맷과 M개의 대상 포맷이 있을 때, 
직접 변환은 N×M개의 변환기가 필요하지만, 
공통 중간 표현을 두면 N+M개로 줄어듭니다
