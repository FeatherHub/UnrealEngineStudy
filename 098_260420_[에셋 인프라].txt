UAssetManager 
IAssetRegistry를 참조

UAssetManager의 핵심 기능: "Primary Asset" 이라는 의미 단위 를 관리하는 것
저수준에서 보면 에셋은 "패키지 안의 export" 일 뿐이다

화염검은 메시 에셋, 머티리얼 에셋, 사운드 에셋, 이펙트 에셋 등 여러 저수준 에셋으로 구성되지만, 
게임 로직에서는 하나의 "아이템" 이다.

FPrimaryAssetId ("Weapon:Sword_Fire") 는 이 게임적 의미 단위에 붙는 이름이고
UAssetManager는 이 이름을 통해 "화염검을 로드하라"는 요청을 "이 메시, 이 머티리얼, 이 사운드를 한꺼번에 로드하라" 로 번역


Asset Bundles 는 이 번역을 더 세분화한다
같은 화염검이라도 인벤토리 UI 에서는 아이콘 텍스처만 필요하고, 
실제 전투에서는 메시·머티리얼·이펙트·사운드 전부가 필요하다

"UI" 번들과 "Gameplay" 번들로 나누면, 인벤토리 화면에서는 UI 번들만 로드해서 메모리를 아끼고 
전투 진입 시 Gameplay 번들을 추가 로드할 수 있다

번들은 "같은 에셋이지만 맥락에 따라 필요한 부분이 다르다" 는 요구를 표현하는 장치다.

Export 안에 있는 것 

1 Export Table
ClassIndex — 이 객체의 UClass
SuperIndex — 부모 클래스 (Blueprint 서브클래스에서 의미가 있다)
ObjectName — FName
ObjectFlags — RF_Public, RF_Standalone
SerialSize — .uexp 안에서 이 export 의 바이트 크기
SerialOffset — .uexp 안에서 이 export 가 시작하는 바이트 위치


2 .uexp 안의 실제 바이트 스트림
SerialOffset 에서 SerialSize 만큼이 이 export 의 "본문"

2-1 프로퍼티 영역
UPROPERTY 값들의 직렬화

2-2 UObject::Serialize(FArchive&) 오버라이드에서 직접 기록하는 바이트들

2-3 벌크 데이터 참조

텍스처 픽셀, 메시 버텍스/인덱스 버퍼, 오디오 샘플 같은 대용량 데이터는 .uexp 안에 직접 들어가지 않는다. 
대신 .ubulk (또는 IoStore 의 BulkData 청크) 에 별도로 저장되고, 
.uexp 안에는 "이 벌크 데이터는 .ubulk 의 오프셋 X 에서 Y 바이트" 라는 참조 만 남는다

.uexp 의 프로퍼티 데이터는 항상 필요하지만 (객체를 구성하려면), 
벌크 데이터는 지연 로드하거나 스트리밍 단위로 다르게 취급하기 위해 분리했다
