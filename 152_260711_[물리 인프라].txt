Proportional-Derivative controller

제어 이론의 기본 장치
목표와 현재의 오차(error)에 비례해 출력을 내는 기계

F = K_p · e + K_d · ė

P(비례) 항은 "멀수록 세게 당겨라" — 물리적으로 스프링(강성 k)과 동형
D(미분) 항은 "빠르게 접근 중이면 브레이크를 걸어라" — 물리적으로 damper(감쇠 c)와 동형

P만 있으면 목표를 지나쳐 영원히 진동하고, D가 그 진동을 죽입니다

즉 PD 제어기 = 스프링–댐퍼

템플릿 구현이 안 보이는 번역단위는 암시적 인스턴스화를 못 함

add.h:    template<class T> T Add(T a, T b);       

add.cpp:  template<class T> T Add(T a, T b) { return a + b; }

main.cpp: #include "add.h"
          int x = Add<int>(1, 2);                    


C++ 는 cpp 마다 독립적으로 컴파일합니다(분리 컴파일 ). 

각 번역단위의 운명을 보면:
  main.cpp 컴파일: Add<int> 에 대한 수요가 여기서 발생했습니다. 
  찍어내려면 레시피 본문이 필요한데 — 선언밖에 안 보입니다. 
  컴파일러는 포기하고 "Add<int> 라는 심볼이 어딘가 있겠지"라는 외부 참조만 남깁니다.

  add.cpp 컴파일: 레시피 본문이 여기 있습니다. 
  그런데 add.cpp 안에는 Add<int> 를 쓰는 코드가 없습니다.
  수요 기반 생성이므로, 수요가 없으면 아무것도 안 찍습니다. 
  산출물: 기계어 0줄.

링크: main 은 Add<int> 를 찾고, 어디에도 없습니다. 링크 에러

기계어가 태어나려면 "본문"과 "수요"가 같은 번역단위 안에서 만나야 하는데, 
본문은 add.cpp 에, 수요는 main.cpp 에 있어서 만나지 못했다

---

명시적 인스턴스화

// add.cpp 맨 아래
template int Add<int>(int, int);   // "int 판을 지금 여기서 찍어라"

함수 호출이 아니라 "수요 발생" 자체를 문법으로 적은 것

이 순간부터 Add<int> 는 더 이상 템플릿이 아니라, cpp 에 구현이 있는 평범한 함수와 완전히 동일한 존재입니다
main.cpp 는 평범한 외부 함수를 부르듯 선언만 보고 호출하고, 링커가 연결합니다.
