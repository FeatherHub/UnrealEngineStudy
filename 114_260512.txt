런타임 데이터는 세 자리 중 하나에 있다.


(1) FNiagaraDataSet — per-emitter 파티클 attribute 저장소

두 개의 FNiagaraDataBuffer (current / destination) 를 보유
각 DataBuffer 는 SoA (Structure of Arrays) 레이아웃


(2) FNiagaraParameterStore — parameter map 의 런타임 표현
모든 namespaced parameter (User., System., Emitter. 등) 가 결국 이 store 의 한 자리


(3) UNiagaraDataInterface
Niagara 가 외부 세계의 어떤 것을 샘플하거나 쓰려 할 때는 항상 DataInterface 추상화를 거친다

UNiagaraScriptSourceBase 가 보유한 UNiagaraGraph (UEdGraph)
노드들은 NiagaraNode 의 자손


이 그래프가 컴파일되어 두 갈래로 갈라진다

CPU 경로: 그래프 → Niagara VM 바이트코드 → FNiagaraVMExecutableData
런타임에 FNiagaraScriptExecutionContext 가 이걸 들고 VM 으로 실행

컴파일 결과의 정체성은 FNiagaraVMExecutableDataId 라는 hash 가 결정


GPU 경로: 그래프 → FHlslNiagaraTranslator → HLSL 소스 → 표준 셰이더 컴파일러 → FNiagaraShader
NiagaraShader 모듈이 셰이더 측 컴파일/관리

2 System Update (매 tick)

  Emitter Spawn (emitter 첫 활성화 시 1회)
  Emitter Update (매 tick, 1회)
  Particle Spawn (이번 tick 에 새로 만들어질 N 개의 입자 각각)
  Particle Update (현존하는 모든 입자 각각)
  Sim Stages (옵션, 별개 thing — 아래에서 짧게)
  Event Handlers (옵션)

4 Render

Emitter Spawn / Update

"이번 tick 에 몇 개를 spawn 할지" 라는 결정이 어디에 살아야 하는가?
매 tick 1회 돌면서 "이번 frame 에 spawn rate × deltaTime 개를 spawn"
"burst 이벤트로 50 개 추가" 같은 결정


그 결과를 FNiagaraSpawnInfo 류의 출력으로 시스템에 알린다
시스템이 그 N 을 받아 DataSet 에 N 개의 새 slot 을 할당한 다음, ParticleSpawn 을 N 번 dispatch

---

emitter 가 자기가 spawn 한 입자들보다 오래 산다

자기 나이 (Emitter.Age) — 0 → 30 단조 증가.
자기 루프 상태 (Emitter.LoopAge, CurrentLoopIndex) — 사이클을 돌고 있는지, 몇 번째 루프인지.
자기 위상 — 예컨대 1 초마다 강하게 펄스하는 분수라면 sin(2π · LoopAge / LoopDuration) 같은 값.

DataSet 

입자만이 아니라 시스템 인스턴스 파라미터, event record, sim stage 의 iteration source 등 비-입자 상황에도 그대로 쓰인다

만약 "ParticleArray" 였다면 이 재사용이 부자연스러웠을 것이다. "DataSet" 은 입자성을 추상화에서 제거 한다



사용자 UI / 문서 / 입자 컨텍스트 → "Attribute"

백엔드 저장·메모리·전송 계층 → "Data"


Attribute 는 의미를 함의하고 ("이건 입자가 가진 위치속성 이다"), 
Data 는 의미를 비운다 ("이건 그냥 Float buffer 의 N 번째 column")

같은 DataSet/DataBuffer 메커니즘이 grid cell, event record, sim cache frame 같은 
입자가 아닌 row 들에도 그대로 적용된다
