게임플레이 프레임워크

이 질문들에 대한 표준 답안
"이런 것들은 이런 클래스로 표현하고, 이런 관계로 연결하라."

의미론적 어휘 체계
**"어떤 생각을 프레임워크의 어휘로 표현하는 것"**


세계가 있어야 한다
세계에 뭔가 놓여 있어야 한다 (풍경, 적, 아이템, 플레이어)
플레이어의 입력을 받아야 한다
플레이어가 화면으로 세계를 봐야 한다 (카메라)
게임에는 규칙이 있어야 한다 (승리/패배 조건, 스폰 규칙, 팀 구성)
여러 플레이어가 있을 수 있다 (로컬, 네트워크)
게임이 저장되고 로드될 수 있어야 한다

블루프린트의 이벤트 그래프 전체를 담은 하나의 거대한 UFunction

UberGraphFunction (단일 UFunction)
  ├── BeginPlay 진입점 → 오프셋 0
  ├── Tick 진입점 → 오프셋 342
  ├── OnDamaged 진입점 → 오프셋 891
  └── ... 

이벤트 그래프의 노드들이 지역 변수와 중간 상태를 공유할 수 있게 하기 위함

---

class UBlueprint : public UBlueprintCore

    // 편집 중인 그래프들
    TArray<UEdGraph*> UbergraphPages;       // 이벤트 그래프 페이지들
    TArray<UEdGraph*> FunctionGraphs;       // 사용자 정의 함수 그래프들
    MacroGraphs, DelegateSignatureGraphs

    // 변수 정의들
    TArray<FBPVariableDescription> NewVariables;

    // 컴포넌트 트리 (에디터 버전)
    USimpleConstructionScript* SimpleConstructionScript;

    // 컴파일 결과
    UClass* GeneratedClass;         // → UBlueprintGeneratedClass*
    UClass* SkeletonGeneratedClass; 

    // 부모 클래스
    TSubclassOf<UObject> ParentClass;

[단계 1] SkeletonGeneratedClass 생성 또는 갱신
  - 변수와 함수의 "서명"만 있는 빈 UClass 생성
  - 다른 블루프린트가 이 블루프린트를 참조할 수 있도록

[단계 2] UBlueprintGeneratedClass 생성 또는 갱신
  - 부모 클래스의 FProperty들 상속
  - FBPVariableDescription들을 FProperty로 변환하여 추가

[단계 3] 함수 그래프, 이벤트 그래프 컴파일

[단계 4] UBlueprintGeneratedClass의 CDO 구성

[단계 5] 후처리
  - 참조 갱신 (이 클래스를 참조하는 다른 블루프린트들)
  - 핫 리로드 (Reinstancing)

C++ 생성자는 CDO 생성 시에도 실행되고, 일반 인스턴스 생성 시에도 실행된다. 생성자는 매 인스턴스마다 불린다.

[1] 메모리 할당 (엔진이 담당)
    │
    ▼
[2] C++ 생성자 호출
    - 사용자 작성 생성자 본문 실행
    - 멤버 초기화, CreateDefaultSubobject 등
    │
    ▼
[3] CDO로부터 프로퍼티 값 덮어쓰기
    - 생성자가 채운 값 위에 CDO의 값을 복사
    │
    ▼
[4] PostInitProperties 등 생명주기 훅

GUObjectAllocator

GC 시스템, 레지스트리, 해시 테이블이 모두 GUObjectAllocator가 관리하는 객체만 보고 있다
UObject를 new 로 만들면 GUObjectAllocator에 연결이 안 된다

---

void (*ClassConstructor)(const FObjectInitializer&)

void AMyActor::__DefaultConstructor(const FObjectInitializer& X)
{
    new((EInternal*)X.GetObj()) AMyActor(X);
    // placement new로 이미 할당된 메모리에 대해 C++ 생성자 루틴 실행
    // vtable이 설정되고, 멤버가 초기화되고, 사용자 생성자 본문이 실행된다
    // placement new 덕분에 엔진이 메모리 관리를 하면서도
    // C++ 생성자 기능이 정상 작동한다
}

// 이후 바인딩
AMyActor::StaticClass()->ClassConstructor = &AMyActor::__DefaultConstructor;

생성자(코드 기반 구조화)와 CDO 복사(데이터 기반 값 설정)의 분리

C++ 객체의 구조적 유효성(vtable, 가상 함수 디스패치)은 생성자 없이는 불가능하다.

반면 "디자이너가 편집 가능한 기본값"은 코드에 하드코딩될 수 없다 - 재컴파일 없이 바뀌어야 하므로
