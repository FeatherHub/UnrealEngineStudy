x86인지 ARM인지, 메모리가 4GB인지 64GB인지, SSD인지 HDD인지
CPU의 명령어 집합, 메모리 계층 구조, 인터럽트 메커니즘
이것들은 주어진 조건이고, 이 조건 위에서 소프트웨어가 설계

건축에서 암반의 특성을 바꿀 수는 없지만, 컴퓨터에서는 하드웨어 자체를 재설계할 수 있습니다 (FPGA, 주문형 반도체). 
건축에서는 "자연"과 "건축물"의 경계가 절대적이지만, 
컴퓨터에서는 "하드웨어"와 "소프트웨어"의 경계가 설계 결정에 의해 이동할 수 있습니다. 
어떤 기능을 하드웨어에 넣을지 소프트웨어에 넣을지는 선택의 문제입니다.

건축에서 기초(foundation)는 건물의 하중을 지반에 전달하는 구조입니다. 
건물 자체는 지반 위에 직접 서는 것이 아니라 기초 위에 섭니다. 
기초는 지반의 불균일성을 흡수하고, 건물에게 균일하고 안정적인 지지면을 제공합니다.

운영체제 커널이 정확히 이 역할을 합니다
응용 프로그램은 하드웨어 위에 직접 실행되는 것이 아니라 커널 위에서 실행됩니다.
하드웨어는 종류마다 다른 인터페이스를 가집니다
이 NIC는 MMIO로 통신하고, 저 디스크는 DMA를 쓰고, 이 GPU는 고유한 레지스터 레이아웃을 가집니다
커널은 이 불균일성을 흡수하여, 응용 프로그램에게 균일한 추상화(파일, 소켓, 프레임버퍼)를 제공합니다

건물의 1층 바닥이 아래에 있는 기초의 복잡한 구조를 의식할 필요 없이 평평한 것처럼, 
응용 프로그램은 아래에 있는 디바이스 드라이버의 복잡성을 의식할 필요 없이 
read(), write() 같은 균일한 인터페이스를 사용합니다.

기초 공사의 또 다른 핵심 기능은 하중 분산입니다
한 기둥에 하중이 집중되면 지반이 불균등하게 침하하여 건물이 기울어집니다
기초는 하중을 넓은 면적에 분산시킵니다

커널의 스케줄러가 하는 일이 이것입니다.
여러 프로세스가 CPU라는 유한한 자원 위에서 실행될 때, 스케줄러가 시간을 분할하여 각 프로세스에 할당합니다.
한 프로세스가 CPU를 독점하면 다른 프로세스가 기아(starvation) 상태에 빠지는데, 
이것은 한 기둥에 하중이 집중되어 지반이 침하하는 것과 같은 구조의 문제입니다.

기초 공사의 품질은 건물의 수명을 결정하고, 커널의 안정성은 시스템의 가동 시간을 결정합니다.
커널에 버그가 있으면 시스템 전체가 크래시합니다

---

거주자가 실제로 건물을 "사용"하려면 
물이 들어오고 나가야 하고(배관), 
전기가 공급되어야 하고(전기 계통), 
공기가 순환해야 합니다(HVAC: Heating, Ventilation, and Air Conditioning)

이 시설물(facilities)들은 건물의 구조체와는 별개의 시스템이지만, 
건물의 벽과 바닥 안에 매립되어 함께 존재합니다.

키보드로 입력이 들어오고(배관의 급수), 
모니터로 출력이 나가고(배관의 배수), 
네트워크로 데이터가 오가고(공조의 환기), 
디스크에 데이터가 저장됩니다(전기의 축전)

배관 시스템에는 
급수관(외부에서 건물로)과 
배수관(건물에서 외부로)이 있고, 
중간에 밸브, 펌프, 수조, 정화 장치가 있습니다.

네트워크 I/O에도 
수신(RX)과 송신(TX) 경로가 있고, 
중간에 버퍼, DMA 엔진, 프로토콜 스택이 있습니다

---

전기는 건물의 다른 모든 시설(배관의 펌프, 공조의 팬, 엘리베이터, 조명)을 작동시키는 에너지원입니다.

건물의 전기 계통에는 배전반(분전반)이 있어서, 
외부에서 들어온 전기를 여러 회로로 분배합니다. 
각 회로에는 차단기(breaker)가 있어서, 
과전류가 발생하면 해당 회로만 차단합니다

컴퓨터의 인터럽트 컨트롤러(APIC)
여러 I/O 장치에서 발생하는 인터럽트를 받아서 CPU에 분배하고, 우선순위를 관리합니다

---

건물의 공조 시스템은 온도, 습도, 공기 질을 일정 범위 내로 유지합니다. 
거주자가 의식하지 못하는 사이에 끊임없이 작동하며, 문제가 생기면(너무 덥거나 추우면) 즉시 쾌적성이 무너집니다

CPU의 캐시 시스템
프로그래머가 의식하지 못하는 사이에 끊임없이 작동하며(L1/L2/L3 캐시가 자동으로 데이터를 옮김),
문제가 생기면(캐시 미스가 빈번하면) 즉시 성능이 무너집니다

둘 다 "보이지 않게 작동하면서 환경을 최적으로 유지하는 백그라운드 시스템"

---

건축에는 건축법, 소방법, 장애인접근성법 등 방대한 법적 요구조건이 있습니다

건물의 구조적 안전성(내진 설계, 하중 기준)

방화벽, 스프링클러, 비상구, 피난 계단.
  "한 구역의 재난이 다른 구역으로 전파되는 것을 물리적/논리적으로 차단하는 구조물"입니다.

"모든 사용자가 건물의 기능에 접근할 수 있어야 한다"는 요구 - 경사로, 점자 안내판, 넓은 문

도시계획에서 건폐율과 용적률 - 한 필지에 얼마나 큰 건물을 지을 수 있는지
  인접 건물의 일조권, 통풍, 교통 용량 등 공유 자원의 보호

CPU의 ISA(Instruction Set Architecture) 스펙은 명령어의 의미론, 메모리 모델, 인터럽트 동작을 규정합니다
API/ABI 호환성 요구 - 특정 컴파일러로 빌드된 프로그램만 작동하거나, 특정 버전의 라이브러리에서만 돌아간다면 이것은 접근성 위반입니다

건축은 사전 설계에 막대한 시간을 투자합니다. "한 번에 제대로" 해야 하기 때문입니다. 
건축에서 시공 후 설계를 변경하는 비용은 극도로 높습니다. 콘크리트를 타설한 후에 기둥의 위치를 바꾸는 것은 사실상 불가능합니다
소프트웨어는 반복적 수정(iterative development)이 가능하므로, "먼저 만들고 고쳐가기"가 실행 가능한 전략입니다.

데이터베이스 스키마, 네트워크 프로토콜, 공개 API — 이런 것들은 한번 배포되면 변경 비용이 급격히 높아집니다
수천 개의 클라이언트가 의존하는 API의 인터페이스를 바꾸는 것은, 
이미 다른 건물들과 연결된 배관의 규격을 바꾸는 것만큼 어렵습니다. 
이런 의미에서 API 설계는 소프트웨어에서 가장 "건축적인" 행위입니다.

건물의 존재 이유는 거주자입니다
건물은 거주자에게 공간, 보호, 시설을 제공합니다
거주자는 건물의 내부 구조(철근의 배치, 배관의 경로)를 알 필요 없이, 
문을 열고 수도꼭지를 틀고 스위치를 누르는 것만으로 건물의 기능을 사용합니다.

프로세스는 커널의 내부 구조(스케줄러 알고리즘, 페이지 테이블 관리)를 알 필요 없이, 
시스템 콜이라는 인터페이스를 통해 OS의 기능을 사용합니다

open()은 문을 여는 것이고, 
read()/write()는 수도꼭지를 트는 것이며, 
fork()는 방을 하나 더 만드는 것입니다.

건물에서 여러 세대가 거주할 때, 
각 세대는 자기 공간(전용 면적)을 가지면서 
복도, 엘리베이터, 주차장 같은 공용 공간을 공유합니다
한 세대의 활동이 다른 세대에 영향을 주지 않아야 합니다
방음, 방수, 사생활 보호. 주차 갈등, 층간 소음 

멀티프로세스 환경에서 각 프로세스가 자기 가상 주소 공간(전용 면적)을 가지면서
CPU 시간, 물리 메모리, 네트워크 대역폭(공용 자원)을 공유하고,
한 프로세스의 오류가 다른 프로세스에 영향을 주지 않아야 하는 것(프로세스 격리)과 같은 구조

컴퓨터에서 프로세스 간 경쟁 조건(race condition), 교착 상태(deadlock), 자원 기아(starvation)가 발생하는 이유도 동일합니다. 
공유 자원에 대한 접근 규칙(락, 세마포어, 스케줄링 정책)이 불완전하기 때문입니다

공유 환경에서 개별 사용자의 자원 사용을 제한하여 전체의 안정성을 보장하는 메커니즘

30년 전 건축 기준으로 지어진 건물에 현재 기준의 시설을 설치해야 하는 상황이 빈번합니다
원래 전화선만 고려했던 건물에 광케이블을 넣어야 하고, 에어컨이 없던 건물에 냉방 설비를 추가해야 합니다
벽 안의 배관을 교체해야 하는데, 도면이 유실된 경우도 있습니다.

20년 전에 설계된 시스템에 현재의 요구사항(클라우드 배포, REST API, 실시간 처리)을 추가해야 합니다
원래 단일 서버만 고려했던 아키텍처에 분산 처리를 넣어야 하고, 문서가 유실된 코드베이스를 수정해야 합니다

건축에서 시공 중 감리(construction inspection) 디버깅
건축에서 준공 검사(completion inspection) QA/테스트
건축에서 하자 보수(defect repair) 버그 수정

건축에서 최하위 계층은 재료(materials)입니다. 콘크리트, 철강, 목재, 유리. 
각 재료는 고유한 물리적 성질(인장 강도, 압축 강도, 열전도율, 내화성)을 가지며, 
이 성질이 상위 계층에서 무엇이 가능한지를 제약합니다.

컴퓨터에서 이 위치에 있는 것은 트랜지스터와 기본 논리 게이트입니다
AND, OR, NOT 게이트의 스위칭 속도, 전력 소비, 집적도가 상위 계층의 가능성을 제약합니다.

그 위에 구조 요소(structural elements)가 있습니다
기둥, 보, 슬래브, 벽. 재료들이 특정 형태로 조합되어, 
하중을 전달하고 공간을 구획하는 기능 단위를 형성합니다

컴퓨터에서 이에 대응하는 것은 ALU, 레지스터 파일, 캐시, 버스 같은 하드웨어 기능 블록입니다
논리 게이트들이 특정 구조로 조합되어, 연산을 수행하고 데이터를 저장하고 전달하는 기능 단위를 형성합니다

그 위에 시스템(systems)이 있습니다
구조 시스템, 배관 시스템, 전기 시스템, 공조 시스템
건물 전체에 걸쳐 하나의 기능을 담당합니다

컴퓨터에서 이 위치에 있는 것은 메모리 서브시스템, I/O 서브시스템, 인터럽트 시스템 같은 것들입니다

그 위에 공간(spaces)이 있습니다
거실, 주방, 사무실, 복도. 여러 시스템이 통합되어 사용자가 실제로 점유하고 활동하는 기능적 단위를 형성합니다

컴퓨터에서 이에 대응하는 것은 프로세스의 실행 환경입니다
가상 주소 공간, 파일 디스크립터 테이블, 시그널 핸들러 등이 통합되어 프로그램이 실제로 실행되는 "공간"을 형성합니다.

최상위에 건물 전체(building)가 있습니다
여러 공간이 조합되어 하나의 완결된 기능적 단위를 이룹니다
주거, 상업, 의료 등 건물의 "용도"가 여기서 정의됩니다
컴퓨터에서 이에 대응하는 것은 운영 중인 시스템 전체
OS 위에서 여러 프로세스가 협력하여 하나의 서비스(웹 서버, 데이터베이스, 게임)를 제공하는 것입니다.

거주자는 벽 안의 철근 배치를 모르고, 
프로세스는 페이지 테이블의 엔트리 구조를 모릅니다. 
이 "위 계층이 아래 계층의 내부를 모르면서도 기능을 사용할 수 있다"는 성질은 
두 분야에서 동일한 설계 원리(정보 은닉, 관심사의 분리)에 의해 달성됩니다

---

구조는 건물의 뼈대입니다
기둥, 보, 슬래브, 내력벽. 이것들은 중력이라는 근본적 힘에 대항하여 건물을 세우는 역할을 합니다
구조는 건물이 존재하는 데 필수적이며, 구조를 제거하면 건물이 물리적으로 무너집니다.

설비는 건물의 장기(organs)입니다
배관, 전기, 공조, 엘리베이터, 통신. 이것들은 건물이 "살아있게" — 즉 사용 가능하게 — 만드는 역할을 합니다.

구조에 대응하는 것은 CPU의 실행 파이프라인과 메모리 계층입니다
명령어 페치 → 디코드 → 실행 → 메모리 접근 → 라이트백이라는 파이프라인은 프로그램이 "서 있기" 위한 뼈대입니다
이것 없이는 어떤 소프트웨어도 실행될 수 없습니다
운영체제 수준에서 구조에 해당하는 것은 프로세스 스케줄링과 메모리 관리입니다.

설비에 대응하는 것은 I/O 서브시스템, 네트워크 스택, 파일 시스템, 디스플레이 시스템입니다
CPU와 메모리만 있으면 프로그램은 "실행"되지만, 외부 세계와 소통할 수 없습니다 

건물의 수명(50~100년) 동안 변하지 않지만 설비는 15~20년 주기로 교체됩니다
배관이 노후화되고, 전기 계통이 현행 기준에 맞지 않게 되고, 공조 시스템이 에너지 효율 요구를 충족하지 못하게 됩니다. 

ISA(명령어 집합 아키텍처)는 수십 년 동안 유지되지만(x86은 1978년부터)
I/O 인터페이스는 빠르게 변합니다
ISA를 사용하는 소프트웨어, SATA → NVMe, USB 1.0 → 4.0, VGA → HDMI → DisplayPort, Ethernet 10Mbps → 100Gbps

건물의 6층 모델
각 층이 서로 다른 속도로 변하고, 
빠르게 변하는 층이 느리게 변하는 층에 영향을 주지 않아야 한다는 원칙이 건축과 컴퓨터 시스템 모두에서 성립합니다
잘 설계된 건물은 배관을 교체할 때 구조를 건드리지 않아도 되고, 
잘 설계된 컴퓨터 시스템은 네트워크 드라이버를 교체할 때 커널 코어를 건드리지 않아도 됩니다.

건축 구조학의 핵심 개념 중 하나가 **하중 경로(load path)**입니다
지붕에 내리는 눈의 무게가 지붕 → 보 → 기둥 → 기초 → 지반으로 전달되는 연속적 경로

컴퓨터 시스템에서 데이터 흐름과 제어 흐름이 이 하중 경로와 동형입니다
사용자 입력이 키보드 → USB 컨트롤러 → 인터럽트 → 커널 드라이버 → 유저 프로세스로 전달되는 경로,
프로그램의 함수 호출이 main → processInput → parseCommand → executeAction으로 이어지는 경로

이 경로의 어느 지점에서든 실패가 발생하면 
그 위의 모든 것이 영향을 받습니다.

구조공학의 잉여도(redundancy) 개념
잘 설계된 구조물은 한 부재가 파괴되어도 다른 부재가 하중을 분담할 수 있는 잉여 경로를 가집니다
잘 설계된 소프트웨어 시스템도 한 컴포넌트가 실패해도 다른 컴포넌트가 기능을 인계할 수 있는 
잉여 경로(failover, 복제, 재시도 메커니즘)를 가집니다

단일 장애점(single point of failure)이라는 개념은 양쪽에서 동일하게 사용됩니다

벽과 바닥이 만나는 곳, 창문 프레임과 벽이 만나는 곳, 서로 다른 재료가 접하는 곳. 누수의 대부분은 접합부에서 발생합니다
각 재료의 열팽창률이 다르기 때문에, 온도 변화에 따라 접합부에 틈이 벌어지고, 그 틈으로 물이 침투합니다

소프트웨어에서 버그가 가장 많이 발생하는 곳도 인터페이스(interface) — 즉 모듈과 모듈이 만나는 곳입니다. 
함수와 함수 사이, 라이브러리와 응용 프로그램 사이, 서비스와 서비스 사이. 
한쪽이 NULL을 반환할 수 있는데 다른 쪽이 NULL을 처리하지 않거나, 
한쪽이 UTF-8을 보내는데 다른 쪽이 ASCII로 해석하거나, 
한쪽이 밀리초 단위인데 다른 쪽이 초 단위로 읽거나. 

각 모듈의 내부가 정확하더라도, 접합부의 가정 불일치(assumption mismatch)에서 오류가 발생합니다.

서로 다른 맥락에서 만들어진 것들이 만날 때, 각각이 암묵적으로 가정하고 있는 조건들이 충돌하는 것입니다

건축에서 콘크리트와 철재의 열팽창률 차이는 각 재료가 "자기 나름대로" 온도에 반응하는 것이고, 
소프트웨어에서 두 모듈의 인코딩 가정 차이는 각 모듈이 "자기 나름대로" 문자열을 해석하는 것입니다

평면도는 수평적 공간 배치
단면도는 수직적 구조
상세도는 접합부의 정밀한 구성
설비도는 배관/전기의 경로

하나의 건물을 완전히 기술하려면 이 모든 도면이 필요하고,
각 도면은 서로 정합(consistent)해야 합니다

소프트웨어에서 설계 의도를 전달하는 방법
UML 다이어그램, API 문서, 아키텍처 결정 기록(ADR), 코드 주석, 그리고 코드 자체

건축에서는 설계 의도가 도면이라는 별도의 형식적 문서에 기록되므로, 
"의도와 구현의 불일치"를 도면과 현장을 비교함으로써 검출할 수 있습니다

소프트웨어에서는 설계 의도가 종종 프로그래머의 머릿속에만 존재하고 별도로 기록되지 않으므로, 불일치의 검출이 더 어렵습니다
테스트 코드는 "의도"를 형식적으로 기록한 것이고, 테스트 실행은 "구현(코드)과 의도(테스트)를 비교하는" 행위입니다. 
테스트 코드는 건축의 도면에 가장 가까운 소프트웨어의 대응물입니다.
