0. Naive - Material이 곧 Shader
각 Material은 "색은 붉다, 금속성은 0.3이다, 법선은 이 텍스처를 쓴다" 같은 결정된 값들을 품고 있다

붉은 공 하나와 파란 공 하나가 있으면 Material도 두 개이다.

색만 다른 공 1000개를 화면에 띄우려면 1000개의 shader를 컴파일해야 하고,
draw call마다 pipeline state change가 일어난다


"붉은 공"과 "파란 공"은 계산의 형태는 같고 계산에 공급되는 값만 다르다
Lambert 조명 계산의 구조, 법선을 읽어오는 방식, 광원 루프의 전개는 완전히 동일하다
shape of computation

---

1. 첫 분리 — Material은 "계산 그래프 + parameter 스키마"가 된다

Scalar Parameter, Vector Parameter, Texture Parameter로 표시된 노드들은 명령어 안에 immediate로 박히지 않고 
constant buffer의 슬롯으로 번역된다

상수로 박힌 값은 shader compiler가 constant folding, 곱셈을 시프트로 치환하는 등 온갖 접기(folding)를 수행할 수 있지만,
parameter는 compiler 입장에서 "알 수 없는 값"이므로 이 접기가 봉쇄된다

---

Static Switch Parameter 범주

shader permutation을 만들어내는 축

---

2. 두 번째 분리 — Material Instance

Material이 "계산 + parameter schema"를 내놓았다면, 
그 schema를 채우는 binding이 Material Instance

그 자체로 새로운 계산 그래프를 정의하지 않는다
단지 "부모 Material이 선언한 parameter들에 대해 어떤 값을 덮어쓸 것인가"를 선언

Static Switch를 override하는 특수한 경우에만 새로운 shader permutation이 비용을 치르고 만들어진다.

---

3. 세 번째 분리

MaterialInstanceConstant(MIC)

에디터에서 만들어지고, .uasset 파일로 디스크에 저장되며, cook 타임에 최종 값으로 직렬화되어 패키지에 포함된다
MIC 객체는 메모리상 하나이고 shader도 하나이고 constant buffer 내용도 하나이다


MaterialInstanceDynamic(MID)

인스턴스별로 독립된, mutable한 parameter binding을 담기 위해 기술적으로 다른 객체
런타임에 생성되고, 디스크에 직렬화되지 않으며, 일반적으로 그것을 만든 Component·Actor가 독점적으로 소유한다.

---

Custom Primitive Data

MID가 "parameter 값을 material 객체에 저장한 후 'material 단위'로 uniform buffer를 만들어 GPU에 올린다"는 접근이라면, 
CPD는 "parameter 값을 CPD 객체에 저장한 후 primitive 데이터의 일부로 GPU에 올린다"는 접근이다


shader는 vertex 단계에서 PrimitiveId로 이 레코드를 lookup한다

"material uniform buffer의 특정 offset에서 읽어라"가 아니라 
"primitive의 custom data 배열에서 특정 index를 읽어라"로 HLSL이 생성된다.


100개의 primitive가 같은 UMaterial 또는 같은 MIC을 공유하고, 각자 자기 primitive에 다른 CPD 값만 찔러 넣으면 된다.
메모리에 MID 100개가 살 필요가 없다

---

CPD vs MID

Texture parameter는 못 바꾼다
(GPUScene의 flat float 배열 모델과 맞지 않다)

Static Switch도 못 건드린다


순수 Scalar/Vector 값 수준의 mutability

Resource Assets

프로그램 전체 수명 동안 VRAM에 상주

Texture
색상 텍스처(base color/albedo), 법선맵, 거칠기·금속성·AO를 담는 material texture, 
라이트맵, 섀도우맵, environment cubemap, BRDF LUT 

"2D/3D 격자에 담긴 sampler로 읽히는 데이터"
Mip chain이 함께 적재되어 원본 크기의 약 1.33배
BC1~BC7이나 ASTC 같은 압축 포맷으로 4~8배 축소되어 저장

텍스처 유닛이 압축 상태 그대로 디코딩해서 읽는다
CPU RAM처럼 압축 해제 후 올리는 것이 아니다

---

메시 버퍼(Vertex/Index Buffer)

Vertex buffer는 정점마다 위치·법선·탄젠트·UV·뼈 가중치 등을 담고, 
Index buffer는 "이 삼각형은 정점 3, 7, 12로 이루어진다"는 연결성을 담는다

최근에는 Meshlet 기반 파이프라인(Nanite, Mesh Shader)으로 구조가 복잡해졌다
정점이 cluster로 묶이고, cluster별 bounding sphere와 culling 메타데이터가 함께 상주한다

---

컴파일된 shader 바이트코드(SPIR-V, DXIL 등)와 Pipeline State Object(PSO)

PSO는 shader 뿐 아니라 blend state, depth state, rasterizer state를 묶은 GPU pipeline 구성의 완전한 기술이다

GPU-driven rendering이 성립하려면 "CPU가 매 draw call마다 뭘 그릴지 지시"하는 대신 
GPU가 자기 안의 데이터로부터 스스로 판단해야 한다. 
이를 위한 간접 참조 구조들이 VRAM에 상주한다


Bindless descriptor heap / tables
모든 텍스처와 버퍼를 하나의 거대한 descriptor array에 등록해두고, shader가 인덱스로 원하는 것을 고른다
씬의 텍스처가 수만 장이어도 바인딩 전환 없이 GPU가 직접 선택한다.


Indirect Draw/Dispatch buffer
GPU가 자기 계산 결과로 다음 draw의 파라미터를 쓰는 버퍼
Frustum/occlusion culling이 compute shader로 일어나고, 
그 결과가 이 버퍼에 "이번 프레임에 실제로 그릴 primitive들의 인덱스와 instance 수"로 쌓인다
