왜 지연 등록 이 필요한가?

C++ 정적 초기화는 다음 두 가지 어려움이 있다

(1) 순서가 부분적으로만 정의됨. translation unit 내에서는 선언 순서대로지만, TU 간에는 보장 없음 
(Static Initialization Order Fiasco).

(2) 요구 시스템이 아직 없을 수 있음
UObject 시스템 (GUObjectArray, FName 시스템, GC 시스템 등) 이 아직 init되지 않았을 수 있음.


해법: 정적 초기화는 "큐에 넣는 일"만 하고, 실제 객체 구성은 미룬다

---

시점 0:  로더가 .dll/.so를 메모리에 매핑 (.text/.rodata/.bss 매핑)

시점 1:  C 런타임 정적 초기화 (main 진입 전, 모듈마다 한 번씩)
  ─ Global static CTOR: 실행됨 (= "Eager static init")

시점 2:  main 진입 → UObjectBaseInit 호출 (GUObjectArray, FName 시스템 등 부팅)

시점 3:  Module load 완료 시점에 ProcessNewlyLoadedUObjects 호출

"BSS" (Block Started by Symbol)

프로그램이 실행될 때 "0"으로 초기화되는 전역 변수나 정적(static) 변수들이 저장되는 메모리 영역

"BSS" 영역에 할당된 변수들은 실행 파일(Binary) 내에 실제 데이터를 저장하지 않습니다. 
대신 실행 파일에는 "이만큼의 BSS 크기가 필요하다"라는 정보만 기록

프로그램이 메모리에 로드되어 실행을 시작하기 직전, 커널이나 런타임 환경에 의해 해당 영역이 모두 "0"으로 채워집니다

---

Text (Code): 읽기 전용 실행 코드가 저장되는 곳

Data (GVAR): 초기값이 있는 전역/정적 변수가 저장되는 곳

BSS: 초기값이 없는 전역/정적 변수가 저장되는 곳

Heap: 동적 할당되는 데이터가 저장되는 곳 (위쪽으로 확장)

Stack: 지역 변수와 함수 호출 정보가 저장되는 곳 (아래쪽으로 확장)

Rehash, Reserve, Compact


C++ 내부에서는 이런 위험을 “문맥과 규율”로 관리합니다

이 포인터를 오래 저장하지 마라.

하지만 Blueprint 에는 이런 빌림 규칙이 표현되지 않아요
Blueprint pin 은 이렇게 말하지 못합니다.

이 포인터는 AttributeMap 이 수정되기 전까지만 유효합니다.
이 포인터는 owner component 보다 오래 보관하면 안 됩니다.
이 포인터는 map rehash 후 dangling 될 수 있습니다.
이 포인터는 C++ stack/local/internal container address 입니다.


그래서 “빌린 내부 주소”를 Blueprint 세계로 내보내는 계약을 만들지 않습니다.

BP VM 의 답: "나는 C++ 함수의 stack frame 이 정리된 후 에 이 포인터를 받는다. 이 포인터가 가리키는 메모리가 아직 살아있는지, 5초 뒤에도 살아있을지 나는 알 수 없다. C++ 함수가 자기 멤버를 가리키는 포인터를 줬는지, 임시 객체를 가리키는 포인터를 줬는지 보이지 않는다.

"다른 머신에 포인터 주소를 보내면 의미가 없다. 가리키는 값을 보내면 그 머신의 사본이 만들어지고 원본과의 연결이 끊긴다

UObject* 를 .uasset 에 쓴다는 것은 — 그 객체의 path name (예: /Game/MyAsset.MyAsset:MyComponent) 을 쓴다는 의미다
로드 시 그 path 로 다시 lookup 한다


USTRUCT 는 항상 어떤 UObject 의 멤버, 또는 어떤 컨테이너의 원소, 또는 어떤 함수의 지역변수 로 존재한다
자기 path 를 가지지 않는다

---

UObject 는 자기 집 을 가진다. UObject 인스턴스는 항상 자기만의 메모리 블록에 거주한다 (GUObjectAllocator 가 할당). 
다른 UObject 의 멤버로 박혀있지 않는다. UObject 는 다른 UObject 를 참조 할 수는 있어도 내부에 흡수 하지 않는다
(즉, UObject 멤버는 항상 UObject* — 포인터로)

UObject 는 항상 독립된 entity 이고, 그 entity 의 위치 = 그 entity 의 정체성


USTRUCT 인스턴스는 항상 어딘가에 박혀 있다
UObject 의 멤버로, TArray 의 원소로, 다른 USTRUCT 의 멤버로, 함수의 지역변수로
USTRUCT 는 항상 호스트의 일부 로 존재한다
그 호스트가 사라지면 USTRUCT 도 사라진다 (호스트의 일부이므로). 호스트가 옮겨지면 USTRUCT 도 같이 옮겨진다

---

다만 기술적으로 struct bytes 를 별도 heap buffer 에 둘 수는 있다.
중요한 점은 그래도 GUObjectArray 에 등록된 UObject entity 는 아니라는 것이다.

---

FRogueAttribute* 를 들고 있을 때, 그 포인터는 호스트의 변동에 따라 언제든 무효화된다. 
호스트가 TArray 라면 — 그 TArray 가 reallocate 되는 순간 (push_back 으로 capacity 가 늘어남, remove 로 원소 이동) 
모든 원소의 포인터가 무효화된다

---

이건 BP VM 이 USTRUCT* 를 함수 반환값으로 받았을 때 마주하는 정확한 문제다. 
그 포인터가 다음 BP 노드 실행 시점까지 유효하다는 보장이 어디에서도 나오지 않는다

---

UObject 인스턴스 하나를 만들 때마다

GUObjectArray 에 entry 등록, 고유 path name 부여, FName 시스템에 이름 등록, 
GC reference walking 의 일부, NetGUID 시스템의 후보, package 와 outer chain 의 일부. 
이 모든 인프라가 하나의 객체를 둘러싼다. 



USTRUCT 의 raison d'être 는 UObject 의 모든 인프라 비용을 지불하지 않으면서도 reflection 의 혜택은 받는 것 이다
USTRUCT 인스턴스는 GUObjectArray 에 등록되지 않고, GC 가 직접 추적하지 않고, path 를 가지지 않고, NetGUID 를 가지지 않는다

전역 등기소 GUObjectArray 


NewObject<> 가 일어나면, 객체는 힙에 할당되고 동시에 GUObjectArray 라는 전역 배열에 entry 가 등록


어떤 UObject 가 살아있다 = 그 entry 가 GUObjectArray 에 valid 상태로 존재한다. 
어떤 UObject 가 destroy 됐다 = 그 entry 가 invalid mark 됐다.

---

(1) 포인터의 검증 가능성: 누가 UMyActor* 를 들고 있을 때, 
그 포인터가 진짜 살아있는 UObject 를 가리키는지 를 시스템이 검증할 수단 이 있다
dangling 검출이 가능한 시스템 이다


USTRUCT 는 그런 등기부가 없다. FRogueAttribute* 를 받으면 그것이 살아있는 USTRUCT 를 가리키는지 어디에서도 확인할 방법이 없다


(2) Weak reference 가능성
이게 가능한 이유는 destroy 라는 사건이 등기부에서 detectable 하기 때문이다.


USTRUCT 의 destroy 는 호스트의 변동 이다
TArray reallocation, 호스트 UObject destroy, 함수 return. 각각이 다른 종류의 사건이고, 어디에서도 통일된 detection point 가 없다.
포인터 invalidation 을 검출할 곳이 없다.

UFUNCTION 이 호출되는 그 순간 을 생각해보자.

그 변수는 어딘가의 UObject 의 멤버 에 살고 있고 (BP 그래프 자체가 UObject 다), 그 UObject 는 root 에서 도달 가능하다.


USTRUCT* 에는 이 보증이 없다. C++ 함수가 FRogueAttribute* 를 반환했다고 하자. 그 포인터는 — 어떤 호스트의 일부를 가리킨다.

FArrayProperty 
  Header 의 layout: 어떤 offset 에 Data, Num, Max 가 있는지
  Inner property: payload 의 원소 하나가 어떤 타입인지 (FIntProperty)


1 호스트 인스턴스 주소 + offset → header 위치
2 Header 에서 Num 을 읽음 (원소 개수)
3 Header 에서 Data 를 읽음 (payload 시작 주소)
4 payload 위에서 i = 0..Num-1 에 대해, Inner property 에게 각 원소의 직렬화 를 위임
