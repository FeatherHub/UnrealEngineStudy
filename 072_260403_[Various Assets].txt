Hierarchical Level of Detail Layer

멀리 있는 건물 수백 채
"건물 세 채"를 "하나의 합쳐진 메시"로 대체

여러 개의 액터/메시를 하나의 단순화된 대표 메시로 합치는 시스템

LOD(Level of Detail)가 단일 메시의 폴리곤 수를 거리에 따라 줄이는 것이라면
HLOD는 여러 메시를 하나로 병합하는 것

가까우면 개별 메시, 
중간 거리에서는 몇 개씩 묶은 클러스터,
아주 멀면 전체를 하나로 합친 초저해상도 메시

HLOD 정책
"어떤 액터들을 어떤 기준으로 묶을 것인가", 
"병합 방식은 메시 합치기인가 임포스터(billboard)인가",
"어떤 거리에서 전환할 것인가"

HLOD Layer를 할당하면, 빌드 시 자동으로 병합된 프록시 메시가 생성된다

Data Layer

하나의 레벨에 "낮 버전의 NPC 배치"와 "밤 버전의 NPC 배치"가 공존하거나, 
"메인 퀘스트 진행 전/후의 월드 상태"가 달라야 할 때, 
전통적으로는 서브레벨을 만들어서 로드/언로드했다. 
그러나 서브레벨은 물리적 파일 분리를 강제하므로, 같은 공간의 서로 다른 "상태"를 관리하기에 불편하다.

Data Layer는 하나의 레벨 안에서 액터들을 논리적 그룹으로 태깅하여, 런타임에 그룹 단위로 활성화/비활성화하는 시스템이다
Photoshop의 레이어와 가장 가까운 비유
같은 캔버스(레벨) 위에 여러 레이어가 겹쳐 있고, 레이어를 켜고 끄면 보이는 것이 달라진다.

에디터에서 Data Layer를 만들고, 
액터들을 해당 레이어에 할당한 뒤, 
런타임에 UDataLayerManager를 통해 레이어를 Loaded/Unloaded/Activated 상태로 전환한다. 
World Partition과 결합하면 스트리밍 단위로도 작동한다 — 특정 Data Layer에 속한 액터들만 선택적으로 스트리밍.

퀘스트 진행에 따라 마을의 모습이 바뀌는 것(파괴 전/후 레이어), 
멀티플레이어에서 팀별로 다른 오브젝트가 보이는 것, 
시간대에 따른 NPC 배치 전환

Brush 

Unreal Engine 1~3 시대에 레벨 지오메트리를 만드는 주된 방법이 
BSP(Binary Space Partitioning) Brush
"레벨은 솔리드(solid) 공간에서 빈 공간을 깎아내는 것"이라는 조각적(subtractive) 모델

Brush는 볼록 다면체(convex polyhedron)를 정의하는 볼륨
Additive Brush — 빈 공간에 솔리드를 추가한다
Subtractive Brush — 솔리드 공간에 구멍을 뚫는다

현재의 위상
UE5에서 BSP Brush는 사실상 레거시다
현대 워크플로우에서 레벨 지오메트리는 
스태틱 메시(3D 모델링 툴에서 만든 것)로 구성하며, 
프로토타이핑 단계에서도 BSP 대신 Modeling Tools(UE5의 내장 메시 에디터)를 사용하는 것이 권장된다

페인팅에서 브러시로 물감을 칠하듯, 공간에 "솔리드"를 칠한다는 비유
Landscape 모드에서 지형을 "페인팅"하는 것, Foliage 모드에서 식생을 "칠하는" 것도 같은 Brush 메타포의 확장이다.

PCG (Procedural Content Generation) Framework

오픈 월드에 나무 10만 그루, 바위 5만 개, 풀 100만 포기를 수작업으로 배치하는 것은 불가능하다. 
규칙 기반으로 자동 생성하되, 아티스트가 결과를 제어할 수 있어야 한다

PCG는 UE5.2에서 도입된 노드 기반 프로시저럴 콘텐츠 생성 프레임워크다
생성 규칙 그래프가 생성하는 것은 "로직"이나 "셰이더"가 아니라 액터/인스턴스 배치 데이터다.

"Landscape에서 경사도가 30도 이하인 영역을 찾아라" 
→ "그 영역에 포아송 디스크 샘플링으로 점을 분포시켜라" 
→ "각 점에 나무 메시를 배치하되, 높이에 따라 침엽수/활엽수를 선택하라" 
→ "도로 근처의 점은 제거하라" 
→ "최종 결과를 Instanced Static Mesh로 스폰하라."

파라미터를 바꾸면 결과가 즉시 재생성된다
선언적 노드 그래프가 도메인 특화 결과물로 컴파일된다.

Interchange Pipeline (Interchange Framework)

Interchange는 UE5에서 도입된 새로운 에셋 임포트/익스포트 프레임워크
기존 FBX 임포터를 대체하기 위해 설계

Pipeline 기반 아키텍처
임포트 과정이 "파일 파싱 → 중간 표현(Interchange Graph) 생성 → UE 에셋으로 변환"의 파이프라인으로 분리

중간 표현(Interchange Graph)
소스 파일(FBX, glTF, USD 등)의 내용을 포맷 독립적인 그래프 구조로 변환한 것

다중 포맷 지원
FBX뿐 아니라 glTF, USD, OBJ 등을 통합된 프레임워크로 처리

"이 프로젝트에서 FBX를 임포트할 때는 항상 이 규칙을 적용하라"를 팀 전체에서 공유 가능.

Camera Rig

시네마틱 카메라 워크 — 크레인 샷, 레일 샷, 스테디캠 — 는 
단순히 카메라 위치를 애니메이션하는 것보다 복잡한 
물리적 장비의 동작을 시뮬레이션해야 한다

크레인은 붐(boom arm)이 피벗을 중심으로 회전하고, 그 끝에 카메라가 달려 있으며, 카메라 자체도 팬/틸트가 가능하다

Camera Rig는 실제 촬영 장비의 기계적 구조를 시뮬레이션하는 액터다
Camera Rig Crane
크레인 피벗 위치, 붐 길이, 붐 상하 각도, 붐 회전 각도를 파라미터로 제어

Camera Rig Rail
레일(dolly) 샷 시뮬레이션. 스플라인으로 레일 경로를 정의하고, 카메라가 그 경로 위를 0~1 파라미터로 이동한다

Level Sequencer(시네마틱 에디터)와 함께 사용된다

게임 월드의 물리적 공간을 구성하는 요소들
지면(terrain), 수면(water), 식생(foliage), 구조물(geometry)
각각에 대해 전용 시스템을 제공

광활한 야외 지형 — 산, 계곡, 평원
스태틱 메시로 만들면 메모리와 렌더링 비용이 감당 불가능하다
수 킬로미터 규모의 지형을 효율적으로 표현

Landscape는 하이트맵(heightmap) 기반 지형 시스템
2D 그리드의 각 점에 높이값 하나를 저장하면 3D 지형이 된다
"지형이란 2D 영역 위에 정의된 스칼라 함수 h(x, y)"라는 수학적 모델

조각(sculpting) 메타포

Sculpt 브러시로 높이를 올리고/내리고,
Smooth 브러시로 매끄럽게 하고
Flatten 브러시로 평탄하게 만든다

BSP Brush가 "공간에 솔리드를 칠하는" 메타포였다면, 
Landscape 브러시는 "하이트맵에 높이값을 칠하는" 메타포다.

Landscape는 **컴포넌트(Component) → 섹션(Section) → 쿼드(Quad)**의 계층으로 분할된다
각 ULandscapeComponent가 하이트맵의 한 타일을 소유하며, 거리에 따라 LOD가 자동 적용된다
이 LOD 전환은 Nanite와 별도로 Landscape 전용 시스템이 처리한다
(UE 5.4부터 Landscape의 Nanite 지원이 실험적으로 도입 중).

Landscape Material: 레이어 페인팅
일반 메시 머티리얼과 다른 고유한 패턴
여러 레이어(잔디, 흙, 바위, 눈 등)를 정의하고, 
각 레이어의 가중치를 지형 위에 "페인팅"하여 혼합한다
이 가중치 맵이 Landscape Layer
에디터의 Paint 모드에서 브러시로 "여기는 잔디, 저기는 바위"를 칠한다

Landmass Plugin: 대규모 지형의 프로시저럴 생성

수십 킬로미터 규모의 오픈 월드 지형을 브러시로 수작업 조각하는 것
대규모 지형 특징은 프로시저럴하게 생성한 뒤 디테일만 수작업으로 다듬는 워크플로우가 필요하다.

Landmass는 Blueprint 기반의 프로시저럴 지형 편집 도구 모음이다
Landscape를 직접 조각하는 대신, Landscape Blueprint Brush라는 특수한 액터를 레벨에 배치하면 
이 액터가 자기 주변의 Landscape 높이와 레이어를 자동으로 수정한다.

예를 들어 "Mountain Brush"를 배치하면 그 위치에 산이 솟아오르고, 
"River Brush"를 스플라인으로 그리면 강줄기를 따라 지형이 파이고, 
"Erosion Brush"를 적용하면 침식 패턴이 생긴다

각 Brush는 Blueprint이므로, 파라미터(산의 높이, 경사 각도, 노이즈 강도 등)를 조절하면 결과가 실시간으로 변한다

Water System

시각적으로 반사/굴절/파도/거품이 필요하고, 
물리적으로 부력/수영 판정이 필요하며, 
지형과의 상호작용(해안선, 강바닥)이 필요하다.

이 모든 것을 수작업으로 조립하면 엄청난 노동이 소요된다.

body of water를 하나의 의미 있는 단위로 정의하고, 
렌더링/물리/지형 상호작용을 통합 처리하는 시스템이다

AWaterBodyOcean
무한히 펼쳐지는 바다. 레벨 전체를 덮는 수면이며, 섬/대륙은 이 바다에서 "솟아오른" 것으로 처리된다.

AWaterBodyLake 
스플라인으로 윤곽을 정의하는 호수. 닫힌 영역

AWaterBodyRiver 
스플라인을 따라 흐르는 강. 스플라인의 각 지점에서 폭, 깊이, 유속을 설정할 수 있다.

AWaterBodyCustom 
수영장, 분수 등 커스텀 형태

Water Body를 배치하면, 그 아래의 Landscape를 자동으로 파내어 강바닥/호수 바닥을 만든다. 
이것이 Landmass와의 통합 지점이다
Water Body가 내부적으로 Landscape Blueprint Brush처럼 작동하여 지형을 비파괴적으로 수정한다.

Water Mesh
수면의 렌더링을 담당하는 메시
카메라 근처는 고밀도 쿼드로, 먼 곳은 저밀도 쿼드로 수면을 생성하는 동적 테셀레이션을 수행한다
파도는 Gerstner Wave 함수로 정점을 변위시켜 생성

Water Zone
Water Body를 내부를 감지하여 "물 안에 있음" 상태를 판정
수영 물리, 수중 포스트 프로세스(블러, 색조 변환), 부력 시뮬레이션의 기반

물리 상호작용
UBuoyancyComponent가 부력을 시뮬레이션
물에 들어간 물체의 침수 깊이를 계산하여 위로 미는 힘을 적용하고, 
파도에 따라 흔들리는 효과를 낸다

Foliage System

나무, 풀, 관목 등을 하나하나 배치하면 시간이 걸리고
수만~수십만 개의 인스턴스를 개별 액터로 관리하면 성능이 붕괴한다.

브러시 페인팅 기반의 인스턴스드 배치 도구

설정된 메시(나무, 풀 등)가 브러시 영역에 확률적으로 뿌려진다

Instanced Static Mesh
Foliage로 배치된 메시들은 개별 액터가 아니라
UInstancedStaticMeshComponent (또는 UHierarchicalInstancedStaticMeshComponent, HISM)로 관리된다
동일한 메시의 수만 개 인스턴스가 하나의 컴포넌트에 속하며, GPU에 하나의 드로우 콜로 전체를 그린다. 
각 인스턴스의 차이(위치, 회전, 스케일)는 인스턴스 버퍼로 전달된다.
나무 1만 그루를 개별 AStaticMeshActor로 만들면 드로우 콜이 1만 개이지만, 
HISM으로 관리하면 LOD 그룹별로 수십 개의 드로우 콜로 줄어든다

HISM의 "Hierarchical"
카메라에서 보이지 않는 클러스터를 통째로 컬링(제거)할 수 있으므로, 
수십만 인스턴스 중 화면에 보이는 것만 효율적으로 렌더링한다.

수작업 페인팅이 Foliage이고, 규칙 기반 자동 생성이 PCG다
PCG 그래프의 최종 출력이 Foliage 인스턴스일 수 있으며, 
이 경우 PCG가 "어디에 무엇을 배치할지" 결정하고, 
Foliage/HISM 시스템이 "그것을 효율적으로 렌더링하는" 역할을 담당한다

Foliage와 Nanite/Lumen
Nanite가 지원되는 메시를 Foliage로 사용하면, 
각 나무/바위의 LOD를 Nanite가 자동 관리하므로 
아티스트가 LOD를 수작업으로 만들 필요가 없다

Quake/Unreal 초기에는 월드가 "솔리드로 꽉 찬 공간"에서 시작하여 
방과 복도를 Subtractive Brush로 깎아내는 것이 기본 워크플로우였다

현대 UE에서 BSP의 역할은 두 가지로 축소되었다
프로토타이핑 
그레이박스(graybox) 레벨 디자인에서 빠르게 공간 레이아웃을 잡을 때
이마저도 UE5의 Modeling Tools가 대체하는 추세

보이지 않는 볼륨
AVolume 파생 클래스들(BlockingVolume, TriggerVolume, AudioVolume, PostProcessVolume, NavModifierVolume 등)이
ABrush를 상속한다
이 볼륨들은 렌더링되지 않으며, "이 공간 영역에 이 규칙을 적용하라"는 공간적 규칙 정의 도구다.

바닥에 다각형 윤곽을 그리고 높이를 주어 돌출(extrude)

Revolve / Path Sweep
커브를 회전체로 만들거나 경로를 따라 밀어내어 복잡한 형태 생성

Boolean (CSG)
Constructive Solid Geometry
두 메시의 합집합(Union), 교집합(Intersection), 차집합(Difference)을 계산
BSP의 Additive/Subtractive를 현대적으로 재해석한 것
"이 큐브에서 저 실린더 모양을 빼라" = 창문 구멍 뚫기. 
BSP와 달리 결과가 스태틱 메시이므로 성능이 좋고, Nanite/Lumen과 호환된다

Mesh Merge — 여러 메시를 하나로 합쳐 드로우 콜을 줄인다.

Voxel Merge 
여러 메시를 복셀화한 후 리메시하여 유기적으로 합친다. 
바위 무더기나 동굴 벽처럼 "녹아붙은" 느낌.

PolyEdit — 정점, 엣지, 폴리곤을 직접 선택하여 이동/회전/스케일
PolyGroup Edit — 연속된 폴리곤 그룹을 선택하여 일괄 편집. 한 면 전체를 밀어넣거나 돌출.
TriEdit — 삼각형 단위 편집. 가장 정밀한 편집
Hole Fill — 메시의 구멍을 자동으로 메운다. Boolean 연산 후 열린 면을 정리할 때 유용.
Displace — 노이즈나 텍스처로 표면을 변위시킨다. 매끄러운 벽에 바위 질감을 주는 등.
Offset — 표면을 법선 방향으로 균일하게 밀어내거나 당긴다. 벽 두께를 주거나 인셋 효과

텍스처
셰이더가 좌표로 인덱싱하여 값을 읽는 데이터 배열
법선 맵, 노이즈 데이터, 룩업 테이블
"좌표 → 값" 매핑을 GPU가 고속으로 수행하는 자료 구조

Texture 2D Array
같은 해상도의 2D 텍스처 여러 장을 하나의 배열로 묶은 것
GPU에서 Texture2DArray로 바인딩
셰이더에서 (u, v, index)로 특정 "슬라이스"의 특정 텍셀을 읽는다

Landscape의 레이어 텍스처들(잔디, 흙, 바위)을 배열 하나에 묶어서 셰이더에 전달하면, 
레이어 인덱스로 텍스처를 선택할 수 있다. 
레이어 간 블렌딩을 단일 셰이더에서 깔끔하게 처리할 수 있다. 

또 다른 사례는 플립북 애니메이션 — 각 프레임을 배열의 슬라이스로 저장.

Texture Cube Array

큐브맵 여러 개를 하나의 배열로 묶은 것
큐브맵 자체가 6면(+X, -X, +Y, -Y, +Z, -Z)의 텍스처
Cube Array는 (방향벡터, index)로 특정 큐브맵의 특정 면의 특정 값을 읽는다

환경 반사 프로브(Reflection Capture)를 여러 개 배치하면 각각이 큐브맵을 갖는데, 
이것들을 하나의 Cube Array에 패킹하여 셰이더에서 가장 가까운 프로브의 큐브맵을 인덱스로 선택. 
런타임에 여러 반사 환경 사이를 블렌딩할 때 사용.

Volume Texture
Volume Texture는 (u, v, w) — 3차원 좌표로 복셀(voxel)을 읽는다
3D 공간에 채워진 데이터 필드

체적 안개(Volumetric Fog)의 밀도 분포를 3D 텍스처로 정의, 
체적 구름(Volumetric Cloud)의 형태 데이터, 
의료 시각화(CT 스캔 데이터), 
3D 노이즈 필드(머티리얼에서 월드 좌표로 샘플링하여 볼류메트릭 효과). 
머티리얼에서 World Position을 UV로 변환하여 Volume Texture를 샘플링하면, 
오브젝트의 내부에 3차원 패턴이 있는 것처럼 보이게 할 수 있다(대리석 무늬, 나무 결 등).

C++ 대응: UVolumeTexture, GPU에서 Texture3D 리소스.

Media Texture

비디오 스트림의 현재 프레임을 텍스처로 노출하는 것
UMediaPlayer가 재생하는 비디오의 현재 프레임이 매 틱 이 텍스처에 기록되며, 
머티리얼에서 이것을 샘플링하면 메시 표면에 동영상이 재생된다.

게임 내 TV 화면, 보안 카메라 모니터, 영화관 스크린, 광고판. 
라이브 웹캠 피드를 텍스처로 표시하는 것도 가능.

Runtime Virtual Texture (RVT)

런타임에 필요한 영역만 동적으로 렌더링하여 캐시하는 가상 텍스처

일반 텍스처가 전체 해상도를 메모리에 올려놓는 것과 달리, 
RVT는 카메라 근처의 영역만 고해상도로 렌더링하고, 먼 영역은 저해상도 또는 렌더링하지 않는다.

바위가 Landscape 위에 놓여 있을 때, 바위 밑면과 지면의 경계가 부자연스럽게 뚝 끊기는 문제가 있다. 
RVT를 사용하면: Landscape의 머티리얼이 RVT에 자기 색상/법선을 기록하고, 
바위의 머티리얼이 RVT에서 자기 아래의 지면 색상을 읽어서 밑면에 블렌딩한다. 
결과적으로 바위가 지면에 자연스럽게 "녹아드는" 효과.

극도로 큰 텍스처(수만 텍셀 해상도의 월드 머티리얼)를 가상화하여 메모리를 절약

렌더 타겟 계열: 렌더링 결과를 기록하는 것

미니맵

Data Asset

Data Table이 "행 단위의 정형 데이터"라면, 
Data Asset은 **"자유 형식의 구조화된 데이터 묶음"**

C++에서 UDataAsset을 상속하여 원하는 프로퍼티를 정의하면, 에디터에서 이 에셋을 만들어 값을 채울 수 있다.

캐릭터 능력치 프리셋, 레벨 설정 데이터, AI 행동 파라미터 세트

Data Table이 "같은 구조의 많은 행"에 적합하다면, 
Data Asset은 "구조가 다른 개별 설정 묶음"에 적합

UPrimaryDataAsset 

Asset Manager와 통합되어 
에셋의 비동기 로드, 스트리밍, 번들링을 선언적으로 관리할 수 있다

GetPrimaryAssetId()를 오버라이드하여 에셋을 고유 ID로 식별하고, Asset Manager가 이 ID로 로드/언로드를 관리.

Data Registry

여러 Data Table이나 Data Asset을 중앙에서 통합 관리하고 조회하는 레지스트리
게임이 커지면 Data Table이 수십 개가 되는데, "이 아이템 ID의 데이터를 찾아라"를 할 때 
어떤 테이블에 있는지 호출자가 알 필요 없이 레지스트리에 물으면 된다.

대규모 게임에서 아이템/스킬/퀘스트 데이터가 여러 테이블에 분산되어 있을 때 
단일 접근점 제공. 
비동기 로드와도 통합되어, 아직 로드되지 않은 테이블의 데이터를 요청하면 자동으로 로드를 트리거

가까워지면 실제 에셋으로 교체하는 스트리밍 전략에 사용.

Level Variant Sets

"이 차를 빨간색으로 보시겠어요? 파란색으로 보시겠어요? 가죽 시트는요, 천 시트는요?"
"이 거실에 오크 바닥을 깔면? 대리석을 깔면?"

각 조합마다 별도의 레벨을 만들거나, 
Blueprint에서 SetMaterial, SetVisibility, SetRelativeLocation을 모든 조합에 대해 하드코딩

차량 색상 5종 × 휠 3종 × 인테리어 4종 = 60개의 조합

Variant Set — 하나의 변주 축. 
"외장 색상"이라는 Variant Set 안에 "빨강", "파랑", "검정" 등의 Variant가 있다.

Variant — 하나의 선택지. 
"빨강"이라는 Variant 안에는 "Body 메시의 Material 슬롯 0을 M_CarPaint_Red로 변경", "반사 강도를 0.9로 변경" 같은 
프로퍼티 오버라이드 목록이 있다

Actor Binding
하나의 Variant가 여러 액터의 여러 프로퍼티를 동시에 변경할 수 있다
 "빨강"을 선택하면 차체 색상만 바뀌는 게 아니라 사이드미러, 범퍼, 스포일러의 색도 함께 바뀐다.

런타임에 UVariantSet::SetCurrentVariant()를 호출

캐릭터 커스터마이징(피부색, 의상 조합), 
건물 파괴 단계(온전/손상/폐허), 
시간대 변화(낮/밤 조명 프리셋)

Variant Sets는 주로 시각화/데모/시네마틱 용도에서 강점을 갖는다.

Variant Sets는 Sequencer와 통합된다
Level Sequence 안에서 특정 시점에 Variant를 전환하는 키프레임을 찍을 수 있다
자동차 광고 영상 제작에서 UE가 사용되는 핵심 워크플로우

Groom — 가닥 단위 헤어/퍼 시스템

전통적으로 머리카락은 **텍스처가 입혀진 면(hair cards/sheets)**으로 표현
반투명 폴리곤 여러 장을 겹쳐서 "머리카락처럼 보이게" 하는 것

영화/시네마틱에서는 수십만 가닥의 개별 머리카락을 시뮬레이션하고 렌더링하는 것이 표준이다(Pixar의 RenderMan, Weta의 시스템 등)
Groom은 이 영화급 헤어 품질을 UE의 실시간 렌더링으로 가져오는 시스템

가닥(strand) 기반 표현
각 머리카락이 제어점(control point)의 연쇄로 정의된 커브다
수만 개의 커브가 캐릭터 두피에 분포하며
각 커브는 수십 개의 제어점을 갖는다
렌더링 시 각 커브가 카메라를 향한 얇은 스트립으로 확장되어 화면에 그려진다.

물리 시뮬레이션
각 가닥이 스프링-댐퍼 체인으로 시뮬레이션되어, 캐릭터 이동 시 관성에 의해 자연스럽게 흔들린다. 
바람, 중력, 충돌(두피/어깨와의)이 적용된다

GroomBindingAsset
Groom은 독립적인 에셋이고, 캐릭터의 Skeletal Mesh도 독립적인 에셋이다. 
Binding이 "이 Groom의 뿌리 점들이 이 Skeletal Mesh의 어떤 표면 위치에 부착되는가"를 정의

LOD 전략

가까울 때는 개별 가닥, 중간 거리에서는 가닥 수를 줄이고(decimation), 먼 거리에서는 전통적 hair cards로 전환
전환 거리와 블렌딩을 세심하게 설정해야 한다.

영화 시네마틱(UE로 렌더링하는 버추얼 프로덕션)에서는 가닥 수를 최대화하고 시뮬레이션 정밀도를 높인다. 
실시간 게임에서는 가닥 수를 줄이고, 시뮬레이션을 단순화하며, LOD를 적극적으로 사용한다. 
Groom 에셋 자체는 동일하되 렌더링/시뮬레이션 설정으로 품질-성능 트레이드오프를 조절한다.

Data Table은 동질적(homogeneous) 컬렉션
모든 행이 같은 구조체 타입이다
SQL 테이블과 동형
"이름으로 행을 찾아 특정 열의 값을 읽는다"가 기본 연산

Data Asset은 이질적(heterogeneous) 단일체
각 Data Asset 서브클래스 인스턴스가 독자적 구조를 가질 수 있다

Asset Manager 통합
수백 개의 Data Asset(각 레벨 설정, 각 퀘스트 정의 등)이 있을 때, 
게임 시작 시 전부 메모리에 올리면 로드 시간과 메모리가 폭발한다.

UPrimaryDataAsset은 Asset Manager에 등록되어, 
"이 에셋은 레벨 3에 진입할 때 비동기로 로드하고, 레벨 3을 떠나면 언로드하라"는 정책을 선언적으로 정의할 수 있다

Data Registry — 데이터의 DNS
Data Registry에서 ItemID_Sword_001이라는 ID를 입력하면, 어떤 Data Table에 있는지 모르지만 결국 해당 행의 데이터를 얻는다.
레지스트리가 내부적으로 올바른 테이블에서 찾아준다. 테이블 구조가 바뀌어도 호출 코드는 변경 없음

비동기 로드 통합

Chooser Table

if (Speed > 300 && bIsCrouching) return Anim_CrouchRun;
else if (Speed > 300) return Anim_Run;
else if (Speed > 0 && bIsCrouching) return Anim_CrouchWalk;
else if (Speed > 0) return Anim_Walk;
else return Anim_Idle;

Chooser Table은 이것을 스프레드시트로 바꾼다
결정의 구조(코드)와 결정의 내용(데이터)을 분리

Proxy Asset

"이 에셋은 아직 디스크에 있고 메모리에 없다"는 경계
로드된 세계와 로드되지 않은 세계 사이의 경계

Proxy Asset은 원본 에셋의 축소판
10만 폴리곤의 건물 메시에 대해, Proxy는 500 폴리곤의 단순화된 메시 + 저해상도 텍스처를 가진다

Proxy Table은 "원본 에셋 ID → Proxy 에셋 ID"의 매핑을 정의한다
스트리밍 시스템이 이 테이블을 참조하여 "이 원본 대신 이 Proxy를 보여줘라"를 결정

HLOD는 여러 오브젝트를 하나로 병합하는 것이고, 
Proxy Asset은 개별 오브젝트를 단순화된 버전으로 대체하는 것

플레이어가 게임패드에서 키보드로 전환하면 실시간으로 아이콘이 바뀌어야 한다

이것을 하드코딩하면 각 위젯마다 if (Platform == Xbox) ShowIcon(A); else if ... 같은 분기가 필요하다
Common UI InputActionDataTable은 이 매핑을 데이터 테이블로 중앙 관리한다

테이블의 각 행이 하나의 논리적 액션(확인, 취소, 인벤토리 열기)이며, 
열이 플랫폼/입력 장치별 표시 정보(키보드 아이콘, Xbox 아이콘, PS 아이콘, Switch 아이콘, 표시 텍스트)

Common UI는 입력 장치 전환을 자동 감지한다

Interchange Import Test Plan

대규모 프로젝트에서 외부 DCC에서 수천 개의 에셋을 임포트한다
임포트 설정(스케일 팩터, 축 변환, 머티리얼 매핑 규칙)을 한 번 바꾸면 기존에 정상이던 에셋이 깨질 수 있다. 
"지난주까지 멀쩡했던 캐릭터의 팔이 오늘 뒤집혀 있다"는 상황이 
CI(Continuous Integration) 파이프라인에서 자동으로 감지되어야 한다.

아트 팀이 매일 수정된 에셋을 커밋하고, 엔지니어링 팀이 Interchange Pipeline 설정을 업데이트할 때, 
양쪽 변경이 서로를 깨뜨리지 않았는지를 야간 빌드에서 자동 검증한다.

Object Mixer

레벨의 모든 액터를 스프레드시트 행으로, 프로퍼티를 열로 보여주므로, 
필터링("PointLight만 보기" + "Intensity > 5000")하고 일괄 편집할 수 있다

Object Mixer Filter 에셋은 이 필터 설정("어떤 클래스, 어떤 프로퍼티 조건")을 저장하여 반복 사용할 수 있게 한다.
