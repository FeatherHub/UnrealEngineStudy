Scale 1.5
  논리 1칸 ≈ 물리 1.5×1.5칸
  이 경우 정확히 정수 픽셀로 딱 떨어지지 않음

1.5배, 1.25배 같은 비정수 스케일에서는 내부적으로 보간, 래스터화, 반올림이 일어나고 
레이아웃 snapping 같은 문제가 생길 수 있어요.


물리 픽셀 위치 ≈ 논리 좌표 × scale factor

---

비정수 스케일 문제

Scale 1.5에서 논리 좌표 (0, 0)과 (10, 0)은 물리 픽셀 0과 15에 대응된다. 
하지만 논리 좌표 (1, 0)은 물리 픽셀 1.5 — 존재하지 않는 위치다. 

두 가지 대응이 가능하다. 픽셀 1에 넣거나 픽셀 2에 넣거나. 
혹은 두 픽셀에 가중 평균으로 분배 (서브픽셀 정밀도 렌더링)

이 결정이 레이아웃 엔진마다 다르고, 이 때문에 경계 픽셀이 약간 다르게 렌더링되는 것이 레이아웃 Snapping 문제다.

DPI는 원래 인쇄 쪽 용어
Dots Per Inch
1인치 안에 점이 몇 개 들어가는가

디스플레이에서는 더 정확히는 PPI
Pixels Per Inch
1인치 안에 픽셀이 몇 개 들어가는가


가로 PPI = 가로 픽셀 수 ÷ 물리적 가로 길이 (인치) 
PPI = 대각선 픽셀 수 ÷ 물리적 대각선 길이 (인치)


A: 10인치에 대각선 1000픽셀
B: 10인치에 대각선 2000픽셀

A = 100 PPI
B = 200 PPI

---

대각선 2400 픽셀, 24인치 -> 2400 / 24 = 100 PPI 
대각선 2400 픽셀, 6인치  -> 2400 / 6  = 400 PPI

대각선 PPI 에서 손실되는 정보
  60 x 40 일수도 있고, 40 x 60 일수도 있고, 120 x 20 일수도 있다
  즉, 화면비 (aspect ratio) 정보가 손실된다

24인치, 80 PPI -> 대각선 픽셀 개수 1920
6인치,  80 PPI -> 대각선 픽셀 개수 480

버튼 크기 = 100px × 50px

낮은 PPI 화면에서는 이 버튼이 손가락으로 누르기 적당할 수 있어요
그런데 PPI가 2배 높은 화면에서는 물리 픽셀 하나가 훨씬 작아요
그러면 같은 100픽셀 버튼은 물리적으로 절반 크기처럼 보일 수 있어요.

물리 크기 = 픽셀 수 / PPI => 몇 인치


100 PPI 에서 100 px => 100 px / 100 ppi => 1 inch
200 PPI 에서 100 px => 100 px / 200 ppi => 0.5 inch

a logical pixel space in memory, a physical grid of light emitters, and the mapping between them
Logical Grid  →  Mapping  →  Physical Grid

---

In the most basic embedded system,

you'd write directly to the framebuffer as if it were the physical pixel grid itself
a perfect 1:1 mapping

that creates a fragility: the moment you change the display hardware, your code breaks because the pixel count changed

The logical/physical split emerges as a practical solution to the portability problem

Without it, your UI code is locked to specific pixel grid configuration


The logical pixel system is a practical way to encode "desired physical size in pixel units" relative to a reference PPI

to keep physical sizes consistent across devices, DPI scaling compensates for PPI differences
At its core: pixels are just a count with no inherent physical size, 
but the screen's pixel grid has physical dimensions, represented by PPi

OS DPI Scale

OS가 앱의 좌표계를 실제 출력 픽셀 쪽으로 몇 배로 매핑할 것인가?

Windows Scale 200%
CSS/앱 논리 100px
→ OS 200px


UE DPI Scale

UMG 위젯의 100이라는 크기를 현재 viewport 안에서 몇 픽셀로 그릴 것인가?

UMG Button Width = 100
UE DPI Scale = 0.67

Viewport 상 크기 ≈ 67 px

---

Device Content Scale = Physical Pixel / Viewport Pixel

Device Content Scale 1.0 = Viewport Pixel과 Physical Pixel이 1:1.


Viewport:             852 × 393   (iOS logical points)
Device Content Scale: 3.0

Physical Pixel = 852  × 3.0 = 2556
               = 393  × 3.0 = 1179

---

[Device Profile]
   │ Device Content Scale (= Mobile Content Scale Factor)
   │  "이 기기를 몇 해상도로 렌더할까"
   ▼
Content / Render Resolution  (= 1280×720)
   │ DPI Curve가 이 크기를 읽음
   ▼
DPI Scale (0.67)
   │
   ▼
UMG Slate Unit → Render Pixel 변환

---

Device screen: 2560 × 1440
Device Content Scale: 0.5

Preview/content size: 1280 × 720

그러면 UMG DPI Curve는 2560×1440이 아니라, preview/content size인 1280×720 쪽을 기준으로 평가될 수 있습니다.

Device Content Scale이 UMG 버튼을 직접 0.5배 하는 게 아니라,
DPI Curve가 바라볼 viewport/content size를 바꿀 수 있다.

Layout Transform
  위젯이 “레이아웃 세계에서” 어디에 있고,
  얼마만큼의 공간을 배정받는가를 정하는 변환.

Render Transform
  이미 배치가 끝난 위젯을 “그릴 때만”
  이동/회전/스케일/기울이기 하는 변환.


1. Desired Size 계산
   “이 위젯은 자연스럽게 어느 정도 크기를 원하나?”

2. Layout / Arrange
   부모 패널이 자식들에게 공간을 나눠줌
   여기서 Layout Transform이 중요함

3. Paint
   배정된 Geometry를 바탕으로 실제 Draw Element를 만듦
   여기서 Render Transform이 적용됨

4. Render
   Slate draw elements가 화면에 그려짐

---

Render Transform Scale을 2.0으로 하면 시각적으로는 대략 200 × 80로 보일 수 있습니다.

Layout상 크기:
  100 × 40

Render 결과:
  200 × 80

하지만 부모 패널과 형제 위젯은 여전히 이 버튼을 100 × 40짜리로 취급할 수 있습니다.

레이아웃 세계:
  [ Button A 100 ][ Button B ]

렌더 결과:
  [ Button A 200 ][ Button B ]
         겹칠 수 있음


반면 Layout 쪽 크기를 200 × 80으로 바꾸면 부모와 형제들도 그것을 반영합니다

레이아웃 세계:
  [ Button A 200 ][ Button B pushed ]
           겹치지 않음

---

Render Transform은 레이아웃을 다시 계산하지 않고도 시각적으로 흔들거나, 커지게 하거나, 회전시킬 수 있습니다.

OS는 앱에게 말했어요.

너희는 물리 모니터를 직접 신경 쓰지 마세요.
나에게 논리 좌표로 말하세요.

앱은 말했어요.

버튼은 100입니다.
창은 800입니다.
글자는 14입니다.

OS는 뒤에서 생각했어요.

이 모니터는 고밀도군요.
이 사용자는 200% 배율을 선택했군요.
이 앱은 DPI-aware인가요?
아니면 오래된 앱인가요?

DPI-aware 앱은 OS와 대화했어요.

OS: 이 모니터에서는 이렇게 스케일하세요.
App: 알겠습니다. 처음부터 크게 그리겠습니다.

DPI-unaware 앱은 말했어요.

나는 96 DPI 세계만 압니다.

그러면 OS는 한숨을 쉬고, 앱이 그린 화면을 통째로 늘렸어요.

그 결과는 때로 흐릿했어요.
  DPI를 아는 앱은 OS에게 배율을 통보받아 처음부터 그만큼 곱해 그리고, 
  DPI를 모르는 앱은 1로 그린 뒤 나머지를 OS에 떠넘긴다. 
  떠넘겨진 점은 흐리다. 정보가 늘지 않은 채 부풀려졌으므로.

연속(連續)

잴 수만 있고 셀 수는 없는 세계

자리와 자리 사이에 또 자리가 있었고, 그 사이에 또 있었다. 끝없이 나뉘어도 빈틈이 없었다
밝기는 한 점에서 옆 점으로 매끄럽게 흘렀고, 어디서 한 색이 끝나고 어디서 다음 색이 시작되는지 묻는 것은 무의미했다. 경계가 없었으므로

여기에는 개수라는 말 자체가 성립하지 않는다

---

누군가 격자를 들이댔다
연속의 들판 위에 그물을 던지듯, 일정한 자리마다 점을 찍어 그 자리의 빛을 한 숫자로 받아 적었다
자리와 자리 사이는 버렸다

그물코보다 가는 무늬는 잡히지 않고, 잡히지 않은 무늬는 거짓 무늬가 되어 돌아온다
너무 성긴 그물로 잔물결을 뜨면 엉뚱한 큰 물결이 떠오른다

---

해상도

"몇 개로 나누었는가"만을 안다.

이 층의 점은 위치의 상대적 관계만 있다
(0, 0) (0, 1)
(1, 0) (1, 1)

그 위치가 손톱 위인지 극장 벽인지는 모른다 (정해져 있지 않다, 무지)

이 층의 점은 크기가 없다
