Recast는 Mikko Mononen(핀란드 게임 프로그래머)이 2009년경 시작한 오픈소스 프로젝트
nav mesh generation/pathfinding을 깔끔한 라이브러리로 정리
Recast는 단일 라이브러리가 아니라 라이브러리 family


Recast — collision geometry → NavMesh 생성 (build-time)
Detour — 만들어진 NavMesh 위에서 path query (runtime)
DetourCrowd — 여러 agent가 동시에 움직일 때의 충돌 회피


Havok AI, Kynapse, PathEngine 같은 상용 미들웨어가 있었고 비싸고 라이선스 부담
Recast가 zlib license(거의 public domain)로 풀린 게 결정적
주요 게임 엔진이 Recast를 채택

---

"collision geometry → walkable polygon graph"라는 변환이 단순하지 않다

입력은 임의의 mesh들의 collection (벽, 바닥, 계단, 비탈, 가구)
출력은 캐릭터가 걸을 수 있는 표면을 나타내는 convex polygon graph.


Recast의 pipeline:

1 Voxelization — 입력 geometry를 3D voxel grid로 rasterize. solid voxel과 empty voxel로 분류.
2 Walkable surface 추출 — 위가 비어 있고 충분한 height clearance가 있고 경사가 너무 가파르지 않은 voxel만 walkable로 표시. 
3 Region 분할 — Walkable voxel들을 watershed algorithm 같은 걸로 region으로 묶음.
4 Contour 추출 — 각 region의 경계를 polygon으로 변환.
5 Convex polygon partitioning — 일반 polygon을 convex polygon들로 분해 (pathfinding에 convex가 편하니까).
6 Polygon mesh 생성 — 인접 polygon 간 연결 정보 포함된 graph

---

Detour

State Tree

AI에 한정된 시스템이 아니라 generic state-driven logic을 표현하는 framework로 설계

---

BT가 해결 못한 문제들

BT의 reactivity가 implicit하다

BT의 reactivity는 BB observer + Decorator의 abort cascade에서 온다
LowerPriority/Self/Both abort mode를 머릿속에서 시뮬레이션해야 행동이 예측된다

"체력 낮으면 도망"이라는 단순한 의도를 BT로 짤 때, Decorator 위치 + abort mode + 트리 안의 노드 우선순위를 다 정합해야 한다.

디자이너가 트리를 수정하면 의도치 않은 abort cascade가 발생.

50 노드짜리 BT에서 어느 abort cascade가 어디로 점프할지 예측하는 건 인지적으로 비싸다

---

BT의 transition이 일급 객체가 아니다

BT에서 "X에서 Y로의 transition"이라는 개념이 명시적으로 없다. 
Transition은 abort + 새 traverse의 부산물로 emergent

"이 행동이 끝나면 저 행동으로"라는 자연스러운 표현이 어색하다 — Sequence + condition으로 우회.

"특정 transition에서만 일어나야 할 일"(transition action)을 표현할 자리가 없다

---

BT의 노드 공유 + offset memory가 C++ 부담

노드는 stateless 공유 객체, 인스턴스 상태는 NodeMemory byte buffer + reinterpret_cast


active leaf 외의 다른 subtree의 데이터도 메모리에 항상 allocated.

    └── [Patrol]

4개 노드가 동시에 active

Sequence::Tick이 Decorator를 통해 MoveTo::Tick을 호출, 
MoveTo가 InProgress 반환. 

이 시점에 call stack은 Tick 함수들로 이루어진 stack.


BT의 Active Stack은 multi-tick에 걸쳐 보존된다. 

MoveTo가 InProgress 반환 후 control이 BTComponent로 돌아오지만, 
BTComponent는 active path를 기억해서 다음 tick에 MoveTo를 직접 다시 tick. 

call stack이 한 tick 동안만 존재한다면
함수 호출 stack을 시간 차원으로 늘린 것

Notification에서 RequestExecution까지의 deferred path

Decorator의 callback이 호출되면
  Decorator가 자기 condition 재평가
  결과에 따라 OwnerComp.RequestExecution(this, ...) 호출
  RequestExecution은 BTComponent의 pending request queue에 추가
  BTComponent의 main loop가 안전한 시점에 처리


search lock이라는 추가 메커니즘
BT가 새 active leaf를 search 중이면 lock이 걸리고, 그동안의 abort request는 queue에 쌓임. 
Search 끝난 후 처리.

Decorator는 왼쪽 형제에 대해 abort cascade를 일으킬 수 없다

왼쪽 형제는 priority가 더 높음
Decorator의 LowerPriority arrow는 "낮은 priority를 침범"한다는 정의

반대로, 왼쪽 형제 자신이 (또는 그 안의 decorator가) 우리 decorator의 영역에 침범할 수 있다
왼쪽 형제 안에 LowerPriority decorator가 있으면, 그게 자기 오른쪽인 우리 영역으로 화살표를 발사 가능
(왼쪽 형제가 우리 영역으로 LP arrow를 쏨)

흐름:

[Selector]
  ├── [Both-mode decorator: Cond?]
  │      └── [Subtree A]
  └── [Subtree B]


흐름 1 

Subtree B active 
Cond true가 됨 → decorator의 LP arrow 발사 → Subtree B abort, A로 점프.


흐름 2 — A 진행 중에 Cond true → false

Subtree A active, decorator의 Self arrow 발사 → Subtree A abort
Selector가 B로 fall through.

Active Stack은 Index Sequence

Active path 위의 Composite/Decorator는 매 frame 호출 안 됨
stack에 "있는" 상태로 보존되지만 tick 받지 않음


(1) Abort 시점의 cleanup target 식별
(2) Search 시점의 starting point
(3) Service tick의 활성 여부 판단

---

(1) Reentrance 위험 — Effect가 Trigger의 source를 다시 호출할 가능성.
BB.Set → observer → BT abort → 다른 BB.Set → observer → ... 

(2) Iterator invalidation — Trigger가 collection을 순회 중인데 Effect가 그 collection을 변경

(3) State consistency — Trigger 시점에 시스템이 partial state. 그 partial 상태에서 Effect 실행하면 무의미. Frame이 끝나야 consistent.

---

Defer model

행동이 객체화되면 — 그 객체에 대해 다른 operation이 가능해짐

Inspection — 어떤 의도가 큐에 있는지 살펴볼 수 있음. Debugging.
Prioritization — 여러 의도 중 어느 걸 먼저 처리할지 결정. Scheduling.
Cancellation — 취소 가능. "이 의도를 무효화."
Logging / Replay —  Debugging, save/load.
Optimization — 여러 의도를 합치거나 (batching), 중복 제거.
Distribution — 의도를 다른 thread/machine으로 보낼 수 있음. RPC.

Higher Priority Abort가 없는 이유
이 상황이 BT의 search semantics에서 일어날 수 있는가?

Selector의 search는 left-to-right. 자식 1을 먼저 평가, 가능하면 거기로 진입. 안 되면 자식 2로 fall through
decorator가 자식 2 위치에 있고 자식 1이 active라면, search는 자식 1에서 멈췄으니 자식 2의 decorator는 평가 안 됐을 것.

Abort는 정확히 말하면 BT의 재평가 요청


1
Decorator가 자기 condition 변화를 감지. 
그 변화가 "현재 active path가 여전히 적절한가"에 영향을 미친다고 판단.
BTComponent에 "재평가해달라"고 요청 (RequestExecution)

2
BTComponent가 재평가 수행
새 search의 결과로 active path가 바뀔 수도 있고, 안 바뀔 수도 있음
바뀐다면 — 이전 active path를 cleanup

---

search가 그 자식 위치를 skip했더라도, decorator는 자기 condition observer를 등록한다. 
이유는 "지금 condition이 false라 skip했지만, 나중에 true로 바뀌면 LowerPriority abort 발화해야 하니까."

---

RequestExecution이 "Protected로 점프"라는 명령이 아니라 "재평가해달라"는 요청

그 재평가의 결과는 search algorithm이 결정

Search 흐름

1
현재 active path를 어디까지 unwind할지 결정. decorator의 위치가 reference point
  그 위치 또는 그 위로 unwind

2
Unwind 후 그곳에서 새 search 시작
  Selector가 자기 자식을 다시 left-to-right 평가

3
자식 0 (Left)의 condition 검사. Left의 decorator가 있으면 그 condition 평가. 
Left가 진입 가능하면 → Left가 active. Protected 진입 안 됨
