{

    LaunchWindowsShutdown();

여기서 전역 객체 하나가 태어난다

    int32 PreInit(const TCHAR* CmdLine);

플레이어는 서로 다른 생명주기 층위에 분산된 여러 표현의 집합

생명주기 스택이라는 뼈대

UEngine          ← 프로세스 생명주기
  └── UGameInstance   ← 세션 생명주기  (맵 이동을 살아남는다)
        └── UWorld         ← 레벨 생명주기  (맵 이동 시 교체된다)
              └── AActor        ← 게임플레이 생명주기

이 스택은 단순한 소유 관계가 아니다
각 층위는 아래 층위가 소멸할 때도 살아남아야 하는 상태의 경계다

맵이 바뀌면 UWorld와 그 안의 모든 것이 소멸하지만, 
플레이어의 인벤토리, 누적 점수, 세션 상태는 소멸해서는 안 된다. 
이 "소멸해서는 안 되는 것들"을 담는 그릇이 UGameInstance다.

class UGameInstance : public UObject, public FExec
{
    // 이 게임 세션에 참여 중인 로컬 플레이어들
    TArray<ULocalPlayer*>       LocalPlayers;

    // 현재 활성 World로의 약참조
    TWeakObjectPtr<UWorld>      WorldContext; // FWorldContext를 통해 접근

    // Online Subsystem (Steam, PSN 등) 세션 인터페이스
    TSharedPtr<FOnlineSessionInterface> SessionInterface;

    // GameInstance 수명 내내 살아있는 서브시스템들
    USubsystemCollection<UGameInstanceSubsystem> SubsystemCollection;
};

ULocalPlayer는 UWorld가 아니라 UGameInstance에 소유된다. 
따라서 플레이어는 맵 전환을 살아남는다

플랫폼 신원과 월드 신원 사이의 사영 연산자(projection operator)

플랫폼 신원이란 "이 컴퓨터 앞에 앉아 있는 사람"
이것은 맵과 무관하게 존재

월드 신원이란 APlayerController와 APawn으로 표현되는 
"지금 이 맵에서 특정 위치에 있는 에이전트"
이것은 맵에 종속

UGameViewportClient
플랫폼 I/O와 게임 사이의 막이다

한쪽 면에는 OS의 윈도우, 키보드 이벤트, 마우스 이벤트, 해상도 변경
다른 쪽 면에는 ULocalPlayer들과 게임 렌더링

UGameViewportClient는 이 두 세계가 서로를 알지 못하도록 격리하면서 
번역을 수행한다.

UE는 오랫동안 기능 추가의 전통적 방법이 `UGameInstance`나 `AGameMode`를 상속해서 
가상 함수를 오버라이드하는 것이었다. 

이 방법은 두 가지 병목을 만든다. 
첫째, 모든 것이 한 클래스에 집중된다. 
둘째, 기능의 생명주기가 클래스의 생명주기에 강제로 묶인다.

서브시스템 프레임워크는 이 두 문제를 동시에 푼다. 
서비스를 생명주기 층위별로 독립된 클래스에 분리하고, 
각 층위의 객체가 소멸할 때 그 층위에 등록된 서브시스템도 함께 소멸한다.

생명주기 층위          →    서브시스템 컬렉션 타입
─────────────────────────────────────────────────────
UEngine               →    UEngineSubsystem
UGameInstance         →    UGameInstanceSubsystem
UWorld                →    UWorldSubsystem
ULocalPlayer          →    ULocalPlayerSubsystem

서브시스템은 리플렉션 시스템을 통해 자동 발견된다. 
UWorldSubsystem을 상속한 클래스가 프로젝트에 존재하면, 
UWorld가 초기화될 때 해당 클래스의 인스턴스가 자동으로 컬렉션에 추가된다. 
등록 코드를 따로 쓸 필요가 없다. 
StaticClass() 목록을 순회하면서 UWorldSubsystem의 파생 클래스를 찾아 인스턴스화한다.

---

structurally
these membranes emerged as solutions 
to split monolithic systems into manageable pieces 
while keeping them connected

membrane 
selectively filters what passes through
transforms information crossing it
fundamentally enables the separation it mediates

the membrane and the separation are inseparable
you can't have one without the other
(Insight: Inseparable -> symbebekos, implication, consequence 의 실쳬? )

introducing something like UGameViewportClient
it's actually the generative act that creates the separation itself
(그리고 연결,통합,배치에 대해 설계할 것을 압력)

---

A cell membrane doesn't just sit between inside and outside;
it defines what "inside" means

---

OS input events become UE input events
platform identity becomes PlayerController

it's the membrane asserting the difference between domains

the architectural move is both 
decomposition—splitting what was monolithic into two distinct regions
and then reconnection through these transformative boundaries

---

this connection introduces indirection
what enables substitutability

The game 
doesn't know if it's talking to D3D12 or Vulkan through RHI
doesn't know where assets come from through FLinkerLoad

Each membrane sits between a request and its fulfillment, 
allowing either side to change without the other knowing.

---

prevents direct connections 
that would break the abstraction

The RHI stops game code from calling D3D12 directly
GameViewportClient prevents game logic from depending on OS specifics

A membrane with holes
(Escape Hatch)

---

Every point where a system can grow without exponential coupling
is a membrane absorbing it via contract and insulation

---

natural variation axes—places 
where things change at different rates or for different reasons
lifecycle 
platform I/O changes for different reasons than game logic, 
and session state changes differently than level state

instead of architecture following organizational structure, 
architecture should follow the structure of change itself.

막은 기술적 장치가 아니라 구별 행위가 물질화된 형태다
표시된 것과 표시되지 않은 것. 안과 밖. 이쪽과 저쪽

소프트웨여 시스템은 모델링 시스템이다. 
모델링 시스템은 그것이 모델링하는 세계가 변하기 때문에 변한다
세계의 서로 다른 부분들은 서로 다른 이유로, 서로 다른 속도로 변한다

Conway의 법칙은 시스템이 그것을 설계한 조직의 커뮤니케이션 구조를 반영한다고 말한다
하지만 그것이 놓친 다른 축은 다음과 같다
아키텍처는 변화 구조를 따라야 한다
같은 이유로 변하는 것들은 같은 영역에 있어야 한다. 
다른 이유로 변하는 것들 사이에 막을 놓아야 한다

막을 잘못된 위치에 놓으면
이쪽이 변할 때마다 막이 바뀌고, 저쪽이 변할 때마다 막이 바뀐다
계속해서 재협상되어야 한다
막의 유지 비용이 막의 이득을 초과한다
구별이 세계의 자연스러운 절개 지점을 따라 그어지지 않았다는 신호다

올바른 위치의 막은 안정적이다
ULocalPlayer는 플랫폼 신원과 월드 신원 사이에 놓인다. 
이 둘은 진정으로 다른 이유로 변한다. 
플랫폼 계정 시스템의 변화(Steam에서 다른 플랫폼으로 이전)는 
게임 월드 로직의 변화(레벨 디자인, 캐릭터 밸런스)와 무관하다
