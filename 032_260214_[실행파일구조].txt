The Loader component of the Operating System 
  the entire theater (the RAM)
  the script (the executable file on your disk)
  can the performance (the process) begin?

the loader first reads the file's Header
  Is it compatible with the current CPU architecture?
  Does the user have the necessary permissions to run it?

the loader sets up a Virtual Address Space.
  Code Segment, Data Segment, Stack/Heap

modern loaders often use Demand Paging
  only load parts of the code as they are actually needed

CPU Control Transfer
  OS는 PC 레지스터의 값을 게임 프로그램의 시작 지점인 엔트리 포인트(Entry Point) 주소로
  이때부터 CPU는 OS의 코드가 아닌 게임의 코드를 한 줄씩 읽기 시작

C 런타임 (CRT) 초기화
  실제 엔트리 포인트는 보통 _start 같은 시스템 함수
  CRT 라이브러리가 전역 변수 초기화, 메모리 힙(Heap) 설정 등을 마친 뒤에야 main 함수를 호출

CRT startup 코드 _start / mainCRTStartup
  - 스택 설정
  - 전역 변수 초기화 (static constructors)
  - argc, argv 파싱
  - heap 초기화

main() 
main() return

CRT cleanup (static destructors, atexit 핸들러)

OS에 종료 코드 반환

execve Execute Vector with Environment / Unix 
  성공하면 반환하지 않음
  호출한 프로세스의 코드 자체가 교체되기 때문
  실패할 때만 `-1`을 반환

### 교체되는 것 
가상 주소 공간 전체
├── .text   (코드)
├── .data   (초기화된 전역변수)
├── .bss    (미초기화 전역변수)
├── heap
└── stack   (새로 초기화)

### 유지되는 것 
PID                  ← 프로세스 번호 동일
파일 디스크립터      ← 열린 파일들 (O_CLOEXEC 없으면)
시그널 핸들러 일부
프로세스 그룹 / 세션

**"새 프로그램 실행 = 새 프로세스"가 아니라 "현재 프로세스가 새 프로그램으로 변신"**

fork() + execve() Unix의 프로세스 생성 패턴

pid_t pid = fork();    
if (pid == 0) // 자식 프로세스 
{
    execve("/usr/bin/ls", argv, envp);  // 새 프로그램으로 교체
    exit(1); // 여기 오면 execve 실패
} 
else // 부모 프로세스
{
    wait(&status);     // 자식 종료 대기
}

쉘 `bash`에서 `ls`를 치면
bash
 └─ fork() → bash 복사본 생성
      └─ execve("/bin/ls") → 복사본이 ls로 교체
bash (원본)
 └─ wait() → ls 종료 대기

---

Intel/AMD CISC x86-64 architecture
  variable-length instructions
  Imagine a 3-byte instruction: [AA BB CC].
  If the CPU starts at AA, it works fine.
  If a bug or a hacker forces the CPU to start at BB, the CPU will look at BB and try to find an opcode that matches it.

---

ARM RISC architecture
mobile processors Apple Silicon M1/M2/M3, Snapdragon
  Every single instruction is 4 bytes (FF FF  FF FF)
  it can grab 100 instructions at once and know exactly where each one starts without even looking into
