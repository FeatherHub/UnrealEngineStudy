그 그래프가 실행되려면 결국 어떤 형태의 실행 모델이 필요

첫 번째는 AOT(Ahead-of-Time) 컴파일
그래프를 C++로 변환하고, 그것을 네이티브 코드로 컴파일하는 방식

두 번째는 VM
그래프를 바이트코드로 컴파일하고, 
런타임에 인터프리터가 해석

Blueprint는 C++ UObject 시스템 위에서 동작해야 합니다
이것이 핵심 제약입니다 
VM이 UObject reflection system과 깊게 결합되어야 한다는 것

그리고 Delay, AsyncTask 같은 latent 노드들은 
실행을 중간에 "일시정지"하고 나중에 재개할 수 있어야 합니다
이것은 일반적인 함수 호출 모델로는 자연스럽게 표현되지 않습니다.

---

그래프 → 바이트코드 컴파일
Kismet 컴파일러(FKismetCompilerContext)가 노드 그래프를 분석해 
FKismetCompiledStatement의 시퀀스로 변환합니다

이것이 다시 바이트코드(uint8 배열)로 직렬화되어 
UFunction::Script 필드에 저장됩니다

Blueprint 함수 하나 = UFunction 하나이고, 
그 UFunction의 Script 배열이 바이트코드입니다

실행의 핵심은 FFrame입니다
FFrame은 VM의 스택 프레임으로, 
현재 실행 중인 UObject 인스턴스, 바이트코드 포인터(Code), 로컬 변수 스토리지(Locals), 
그리고 호출 컨텍스트를 담습니다

UObject::ProcessEvent()가 호출되면, 
Blueprint 함수의 경우 ProcessInternal()로 진입합니다. 
여기서 FFrame을 스택에 설치하고, 
바이트코드를 앞에서부터 한 바이트씩 읽으며 opcode를 해석합니다. 
각 opcode(EExprToken enum으로 정의됨)는 
대응하는 exec 함수(예: execLocalVariable, execCallFunction)를 가지며, 
이들은 GNatives라는 전역 함수 포인터 테이블을 통해 dispatch됩니다.

---

opcode 레벨에서 UPROPERTY 접근과 UFUNCTION 호출이 
reflection 시스템을 통해 직접 이루어집니다

협력적 코루틴(cooperative coroutine)
Delay 같은 latent 노드는 실행을 다음 틱까지 "중단"해야 합니다. 
VM은 이것을 FLatentActionManager를 통해 처리합니다. 
Latent 노드는 자신의 재개 지점(resume point)을 등록하고, 
함수 실행이 중단된 상태로 반환됩니다. 
다음 틱에 LatentActionManager가 조건이 충족되면 
해당 FFrame 컨텍스트에서 실행을 재개합니다. 


Blueprint VM은 Game Thread에서만 실행됩니다

VM의 인터프리터 오버헤드가 문제가 될 때, 
UE는 Blueprint Nativization이라는 옵션을 제공합니다. 
Blueprint 바이트코드를 C++ 코드로 변환해 네이티브 컴파일하는 것으로,
VM을 완전히 우회합니다

---

Blueprint가 컴파일되면 
에디터 에셋(UBlueprint)과 런타임 클래스(UBlueprintGeneratedClass)가 분리됩니다

UBlueprint는 에디터 전용 에셋입니다
그래프 데이터, 핀 연결 정보, 변수 메타데이터 등을 소유합니다

UBlueprintGeneratedClass는 UClass를 상속받는 런타임 클래스입니다
UBlueprintGeneratedClass는 Blueprint로 정의된 클래스를 기술합니다

Blueprint에서 정의된 함수 하나하나가 
FuncMap(TMap<FName, UFunction*>)에 UFunction*으로 등록됩니다

UFunction 필드들
Script (TArray<uint8>): 바이트코드 본체. 이것이 VM이 실행하는 대상입니다
PropertyLink: 이 함수의 파라미터와 로컬 변수들을 기술하는 FProperty 연결 리스트
FirstPropertyToInit: 생성자가 필요한 프로퍼티들의 서브셋
ParmsSize: 파라미터 블록의 바이트 크기

---

Blueprint 인스턴스
이 UBlueprintGeneratedClass를 자신의 Class로 가리키는 UObject입니다

바이트코드는 클래스에 귀속되고, 
인스턴스는 실행 컨텍스트를 제공할 뿐입니다.

---

런타임 실행 구성요소

UObject::ProcessEvent(UFunction*, void* Parms)
외부에서 어떤 함수를 호출할 때
UFunction의 플래그를 확인합니다. 
FUNC_Native가 설정되어 있으면 네이티브 함수 포인터를 직접 호출하고 끝납니다. 
Script 배열이 비어있지 않은 Blueprint 함수의 경우 ProcessInternal()로 분기합니다.

ProcessInternal이 하는 일은 하나입니다
FFrame을 스택에 설치하고 인터프리터 루프를 시작하는 것.

FFrame은 Blueprint VM의 핵심 런타임 구조입니다
Node (UFunction*): 현재 실행 중인 함수
Object (UObject*): 이 함수를 실행하는 컨텍스트, self에 해당
Code (uint8*): 현재 바이트코드 포인터(Program Counter). Script 배열 내부를 가리킵니다
Locals (uint8*): 로컬 변수 스토리지. 스택에 ParmsSize만큼 할당된 raw 메모리 블록
PreviousFrame (FFrame*): 이 프레임을 호출한 상위 프레임. 호출 스택 추적에 사용

---

GNatives — opcode dispatch 테이블
GNatives는 전역 함수 포인터 배열로
인덱스가 EExprToken(opcode)에 대응하고 값이 exec 함수 포인터입니다

**주요 EExprToken (opcode) 카테고리**

변수 접근 계열: EX_LocalVariable, EX_InstanceVariable, EX_DefaultVariable
각각 Locals 블록, UObject 인스턴스, Class Default Object를 참조합니다.

함수 호출 계열: EX_CallFunction, EX_CallMathFunction, EX_FinalFunction. EX_FinalFunction
가상 디스패치 없이 직접 호출하는 최적화 경로입니다.

흐름 제어 계열: EX_Jump, EX_JumpIfNot, EX_PushExecutionFlow, EX_PopExecutionFlow. 
브랜치 노드나 Sequence 노드가 이것으로 컴파일됩니다.

대입 계열: EX_Let, EX_LetObj, EX_LetWeakObjPtr. 
타입에 따라 구분됩니다.

종료: EX_Return, EX_EndOfScript.

---

어떤 호출자(C++ 코드, 혹은 다른 Blueprint의 execCallFunction)가 UObject::ProcessEvent(Function, Parms)를 호출합니다.
함수가 에디터/디버거 훅을 거친 뒤, Function->Script.Num() > 0임을 확인하고 ProcessInternal을 호출합니다.
ProcessInternal은 함수의 PropertyLink를 순회해 로컬 변수 크기를 파악하고, alloca로 Locals 메모리를 확보합니다. 
FFrame을 스택에 생성하고, Code를 Script[0]으로 초기화합니다.

while (true) {
    opcode = *Code++;
    GNatives[opcode](Object, Frame, nullptr);
    // Code를 추가로 전진시킴
}

함수 내부에서 다른 함수 호출
execCallFunction이 실행되면, 바이트코드에서 UFunction 포인터를 읽고, 
인자를 Stack에서 평가하여 임시 Parms 블록을 구성한 뒤, 
다시 UObject::ProcessEvent()를 재귀 호출합니다. 
이것이 Blueprint 함수 → Blueprint 함수 호출의 구조입니다. 
FFrame이 하나씩 쌓이며 PreviousFrame으로 연결됩니다.

EX_Return opcode가 실행되면 반환값이 Parms에 기록되고, 
FFrame 소멸자가 로컬 변수들의 소멸자를 호출합니다 
OutParms (FOutParmRec*): out 파라미터들의 연결 리스트.

execDelay(혹은 해당 latent 함수)가 실행되면, 
FLatentActionManager(UWorld가 소유)에 
FPendingLatentAction 파생 객체를 등록합니다. 
이 객체는 재개에 필요한 컨텍스트(링크 ID, 대상 UObject*)를 보존합니다. 
그런 다음 함수는 즉시 반환합니다 — FFrame이 해체됩니다.

조건이 충족되는 다음 틱에,
FLatentActionManager::ProcessLatentActions()가 World Tick 중에 호출됩니다. 
등록된 액션들이 순회되고, 
준비된 액션은 UObject::ProcessEvent()를 통해 
해당 노드의 "재개 지점"에 해당하는 UFunction을 다시 호출합니다.

핵심 관찰
Latent 실행은 FFrame을 일시정지시키는 것이 아닙니다.
FFrame을 버리고 재개 시 새 FFrame으로 다시 진입합니다. 
이것은 coroutine의 "스택 보존" 방식이 아니라, 
"체크포인트 재진입" 방식입니다. 
Latent 노드 이후의 로컬 변수 상태가 보존되지 않는 것은 이 구조의 직접적 결과입니다.

---

Blueprint 그래프의 각 노드는 UK2Node를 상속받습니다
컴파일러가 그래프를 순회하면서 각 노드의 컴파일 메서드를 호출하면
노드 스스로가 자신이 어떤 Statement로 변환되어야 하는지를 기술합니다
변환의 주도권이 컴파일러가 아닌 노드에 있다

FKismetCompiledStatement 
이 IR이 존재하는 이유는 최적화와 변환을 바이트코드 생성과 분리하기 위해서입니다
IR 레벨에서 dead code 제거, 상수 접기 등의 작업을 할 수 있고
백엔드만 교체하면 다른 타겟(Nativization이 바로 이것)으로 내보낼 수 있습니다.
