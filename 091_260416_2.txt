STL의 std::vector, std::string, std::map이 있었지만 성능 특성이 게임에 적합하지 않았다
당시 게임 콘솔(PS2, Xbox, PSP 등)의 컴파일러가 최신 STL을 제대로 지원하지 않았다.
STL 자체의 예외 처리가 문제였다

std::vector<int32>는 UE 리플렉션이 모른다

---

GMalloc
FMemory::Malloc

FMemory 통계 반영

---

서로 다른 컴파일러(MSVC, Clang, GCC, 콘솔별 커스텀 컴파일러) 간의 차이

---

std::shared_ptr은 항상 원자적 참조 카운팅

TSharedPtr<FMyClass>                       // 스레드 안전하지 않음 (기본)
TSharedPtr<FMyClass, ESPMode::ThreadSafe>  // 스레드 안전
TSharedPtr<FMyClass, ESPMode::Fast>        // 별칭, 스레드 안전하지 않음

---

만약 UE가 자체 언어를 만들었다면 

물리 엔진, 이미지 라이브러리, 오디오 코덱, 네트워크 프로토콜 구현, 수학 라이브러리
컴파일러, 디버거, IDE 통합, 프로파일러, 정적 분석기, 린터
C++ 컴파일러(MSVC, Clang, GCC)는 수십 년 최적화 축적
C++는 모든 게임 콘솔, 모바일 OS, 데스크톱에서 지원됨
그래픽 드라이버, 오디오 드라이버는 C ABI로 제공됨

---

엔진 코어와 성능 민감 영역은 C++
게임플레이 로직은 Verse (블루프린트보다 표현력 있는 텍스트 언어)
블루프린트는 비주얼 편집 선호자를 위해 유지
