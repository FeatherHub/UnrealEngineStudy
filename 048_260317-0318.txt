조건부 상태 변환 로직

스턴 상태일 때는 스킬을 쓸 수 없어야 하고, 
버프는 데미지 계산에 영향을 줘야 하며, 
네트워크에서는 클라이언트와 서버가 같은 결과를 봐야 한다.

이 문제를 그냥 두면 각 팀원이 각자의 방식으로 해결하고, 
코드는 스파게티가 된다. 

GAS는 이 복잡성을 처리하기 위한 중앙화된 아키텍처야

UAbilitySystemComponent 
시스템의 허브
Actor에 부착
이하 모든 요소들의 소유자
네트워크 복제의 기준점

UGameplayAbility 
실제로 "무언가를 하는" 로직의 단위
발동 조건, 비용, 쿨다운, 실행 로직을 캡슐화
블루프린트로 서브클래싱 가능

FGameplayAttribute와 UAttributeSet
숫자로 표현되는 게임플레이 상태값들을 담는 구조체
체력, 마나, 이동속도
Base Value와 Current Value의 이중 구조

UGameplayEffect 
Attribute를 변경하는 유일한 공식 경로
즉각적(Instant), 지속적(Duration), 영구적(Infinite) 세 종류
스택, 조건부 적용, Modifier 연산 등을 선언적으로 정의

FGameplayTag와 FGameplayTagContainer
시스템 전체의 "의미 공유 언어"
어떤 어빌리티를 막을지, 어떤 Effect가 활성화되어 있는지, 어떤 상태인지를 
계층적 태그로 표현하고 필터링

AttributeSet의 이중 상태 구조가 왜 필요한가
Base/Current Value

Current Value가 Base + Σ(Applied GE Modifiers)로 계산되는 이유는, 
GE들의 Applied 상태가 언제든 제거될 수 있기 때문
버프 GE가 제거되면 그 Modifier도 계산에서 빠지고,
Base Value는 오염되지 않은 채로 유지돼. 

만약 GE가 Base Value를 직접 수정했다면, GE를 제거해도 Attribute를 원래대로 되돌릴 방법이 없어.

class UGA_Fireball : public UGameplayAbility
{
    // 에디터에서 할당
    TSubclassOf<UGameplayEffect> CostEffectClass;   // GE_FireballCost
    TSubclassOf<UGameplayEffect> DamageEffectClass; // GE_FireballDamage

    virtual void ActivateAbility(...) override
    {
        // 1. 마나 소모 GE 적용 (자기 자신에게)
        ApplyGameplayEffectToOwner(
            Handle, ActorInfo, ActivationInfo,
            CostEffectClass.GetDefaultObject(), 1.0f
        );

        // 2. 파이어볼 발사 (Projectile spawn 등)
        SpawnFireballProjectile();

        // 파이어볼이 적에게 닿으면 OnHit이 불림
    }

    void OnFireballHit(AActor* HitActor)
    {
        auto TargetASC =
            UAbilitySystemLibrary::GetAbilitySystemComponent(HitActor);

        // 3. Damage GE Spec 만들기 — 이 시점에 수치를 주입
        FGameplayEffectSpecHandle SpecHandle =
            MakeOutgoingGameplayEffectSpec(DamageEffectClass, 1.0f);

        SpecHandle.Data->SetByCallerTagMagnitudes.Add(
            FGameplayTag::RequestGameplayTag("Data.Damage"),
            -35.0f   // 런타임에 결정된 데미지
        );

        // 4. 적의 ASC에 적용
        ApplyGameplayEffectSpecToTarget(Handle, ActorInfo, ActivationInfo,
                                        SpecHandle, TargetASC);
    }
};

UGameplayAbility::ActivateAbility
GAS가 어빌리티를 발동시킬 때 호출하는 함수
직접 구현하는 곳 (override point)

UGameplayAbility::ApplyGameplayEffectToOwner
파이어볼을 쏘는 주체는 플레이어이고, 마나는 플레이어에게서 빠져나가야 하니까 ToOwner

1 Handle은 지금 실행 중인 이 어빌리티 인스턴스의 식별자
GAS가 어떤 어빌리티가 이 GE를 만들었는지 추적하는 데 쓰인다

2 ActorInfo는 이 어빌리티와 연결된 Actor 정보 묶음

3 ActivationInfo는 네트워크 컨텍스트
이 발동이 서버에서 일어났는지 클라이언트에서 일어났는지, 예측 키는 무엇인지

4 CostEffectClass.GetDefaultObject()
BP 에서 작성한 GE의 CDO

5 1.0f 는 어빌리티 레밸

State를 직접 건드리지 마라. 모든 변경은 Action → Reducer 파이프라인을 통과해야 한다.

store                          AbilitySystemComponent
store.getState()               AttributeSet 현재값들
store.dispatch(action)         ASC->TryActivateAbility()
Action 객체                    GameplayAbility (의도)
Action payload                 FGameplayEffectSpec (운반 데이터)
Reducer                        GE Modifier + PostGameplayEffectExecute
Middleware                     GameplayTag (파이프라인 게이트)

store.dispatch({ type: "DEAL_DAMAGE", payload: { amount: 35 } })

Redux Reducer는 순간이야
GAS는 여기에 시간 축이 추가돼

// GAS가 허가 판정 후 호출
UGA_Fireball::ActivateAbility(...) {

    // ① 마나 차감 — "나(플레이어)의 ASC에 Cost GE를 dispatch"
    ApplyGameplayEffectToOwner(Handle, ActorInfo, ActivationInfo,
                               CostEffectClass.GetDefaultObject(), 1.0f);

    // 이 시점에 GE_FireballCost가 즉시 실행됨
    // → Mana.CurrentValue -= 20
    // → PostGameplayEffectExecute(Mana) 호출됨

    // ② 투사체 발사. 충돌하면 OnFireballHit 호출
    SpawnFireballProjectile();
}

UGA_Fireball::OnFireballHit(AActor* HitActor) {
    UAbilitySystemComponent* TargetASC = GetASC(HitActor);

    // Action payload 조립 — GE Blueprint + 런타임 수치
    FGameplayEffectSpecHandle Spec =
        MakeOutgoingGameplayEffectSpec(DamageEffectClass, 1.0f);

    // SetByCaller* 슬롯에 수치 주입
    // Redux: { type: "DEAL_DAMAGE", payload: { amount: -35 } }
    Spec.Data->SetByCallerTagMagnitudes.Add(
        FGameplayTag::RequestGameplayTag("Data.Damage"), -35.0f);

    // "적의 ASC에 dispatch"
    ApplyGameplayEffectSpecToTarget(Handle, ActorInfo, ActivationInfo,
                                    Spec, TargetASC);

    // 이 시점에 GE_FireballDamage가 적에게 실행됨
    // → Health.CurrentValue -= 35
    // → 적의 PostGameplayEffectExecute(Health) 호출됨
}

// GAS가 Attribute 변경 후 자동 호출 → Reducer
void UMyAttributeSet::PostGameplayEffectExecute(
        const FGameplayEffectModCallbackData& Data) {

    // switch(action.type) — 어떤 Attribute가 바뀌었나
    if (Data.EvaluatedData.Attribute == GetHealthAttribute()) {

        // 새 값 클램프 (GAS는 이걸 자동으로 안 해줌)
        SetHealth(FMath::Clamp(GetHealth(), 0.f, GetMaxHealth()));

        if (GetHealth() <= 0.f) { /* 사망 처리 */ }
    }

    if (Data.EvaluatedData.Attribute == GetManaAttribute()) {
        SetMana(FMath::Clamp(GetMana(), 0.f, GetMaxMana()));
    }
}

GE는 명령자가 아니라 리간드다. 
AttributeSet은 수신자가 아니라 수용체다.

struct FGameplayAttribute
{
    TFieldPath<FNumericProperty> Attribute;
    // "UMyAttributeSet 클래스의 Health 필드"를
    // 런타임에 참조 가능한 형태로 들고 있는 것
};

`FNumericProperty`는 UE 리플렉션 시스템이 
UPROPERTY로 표시된 숫자 필드에 대해 생성하는 
메타데이터 객체야

GE Modifier가 "Health를 수정해라"고 할 때, 
실행 시점에 이 `FNumericProperty` 포인터를 통해 
AttributeSet 인스턴스 위의 실제 float 메모리 주소를 계산해서 접근해.

리플렉션이 없으면 GE가 Attribute를 선언적으로 지정하는 것 자체가 불가능해. 
이것이 `ATTRIBUTE_ACCESSOR` 매크로가 `GetHealthAttribute()`를 생성하는 이유야
이 함수가 `FNumericProperty*`를 감싼 `FGameplayAttribute`를 반환하고, 
GAS 내부 시스템이 그것을 키로 써.

---

**FName + 전역 레지스트리**

`FGameplayTag`는 내부적으로 `FName`이야. 
그런데 단순한 `FName`이 아니라 `FGameplayTagNode` 트리에 등록된 `FName`이야.

"State.Stunned.Heavy"가 등록될 때
GameplayTagManager는 내부적으로
    "State"
    "State.Stunned"
    "State.Stunned.Heavy"
세 노드를 트리에 등록하고 각각 인덱스를 부여해

FGameplayTag는 계층적 온톨로지 안에 위치한 노드의 핸들이야.

---

FAggregator — CurrentValue 계산의 실제 엔진
AttributeSet 안의 각 `FGameplayAttributeData`는 내부적으로 `FAggregator`를 들고 있어

---

태그의 실제 저장 구조
struct FGameplayTagCountContainer
{
    TMap<FGameplayTag, int32> GameplayTagCountMap;
    // "State.Stunned" → 2  (두 개의 GE가 이 태그를 부여 중)

    TMap<FGameplayTag, int32> ParentTagCountMap;
    // "State" → 2  (부모 태그도 자동으로 카운트됨)
};

`AddTag("State.Stunned")`가 호출될 때 내부적으로 일어나는 일:
"State.Stunned" 카운트 +1
"State" 카운트 +1  ← 부모 태그 자동 처리

이 RefCount 구조 덕분에 같은 태그를 부여하는 GE가 여러 개 중첩되어도 
카운트가 0이 될 때만 태그가 실제로 제거돼
**태그의 존재 여부는 단순한 bool이 아니라 RefCount > 0인지의 여부야.**

---

변하지 않는 것들
GE → AttributeSet 파이프라인 구조
CDO, 태그 레지스트리, FAggregator의 평가 알고리즘, FGameplayAttribute가 가리키는 필드의 위치.

변하는 것들
CurrentValue, ActiveGameplayEffects 컨테이너의 내용, TagCountContainer의 RefCount, GA의 상태

변하지 않는 것들이 전부 "계산의 규칙"이고, 변하는 것들이 전부 "계산의 입력"

---

FGameplayAttribute를 AttributeSet 안의 FGameplayAttributeData 또는 float property를 기술하는 타입
체력을 “35가 깎인 수치”보다 먼저, 어디에 변화가 기록되어야 하는가라는 자리의 문제로 봐요
불변인 것은 값이 아니라 “자리”예요
그 값은 언제나 AttributeSet이라는 구조 안에 놓여 있어요
불변인 것은 체력이라는 변화의 좌석이에요

무언가가 변하려면, 먼저 그 변화가 “이것의 변화”라고 말할 수 있는 동일성의 축이 있어야 해요.
GAS는 그 축을 low-level에서는 reflected property로 붙잡고 있어요.
그래서 GAS의 첫 번째 불변 항은 자리의 존재론, locus ontology예요.

---

The inversion of normal programming flow is probably the real sticking point
instead of writing code that directly does things, 
you're writing declarations and then requesting actions 
that the system either grants or denies

GAS isn't a library you call into
it's an environment you inhabit. 
Your code mostly lives in callbacks
reacting to what the system decides to do,
they orchestrating actions directly

The frustration of working within these constraints is 
actually what makes you feel why they exist

try to work around it and see what breaks
stop studying it from the outside and start living inside it

it's a framework that calls your code, 
a data-driven system where much of the state lives in assets rather than code, 
a network-aware system built around replication, 
and a composition system of small declared pieces

제약을 통한 이해
세 개의 금지

fireball->Execute();           // 이런 거 없어

이 세 금지가 왜 존재하는지를 역방향으로 추적하면 GAS의 전체 구조가 나와.

금지 1을 위반하면 무슨 일이 생기나
코드베이스가 커지면 어떻게 되냐

enemy->Health -= warrior->AttackPower;
enemy->Health -= trapDamage * 0.5f;
enemy->Health -= poisonDPS * deltaTime;

// 버프가 받는 피해를 증폭시킬 때
// ... 어디다 곱해야 하지?
// warrior 공격 코드? 함정 코드? 독 코드? 전부?

버프 하나가 생기는 순간 데미지를 주는 모든 코드를 찾아서 수정해야 해

새로 입사한 팀원이 데미지 코드를 하나 추가하면 
그 팀원은 버프 로직을 끼워야 한다는 걸 어떻게 알아? 몰라

GE → AttributeSet 단일 경로가 존재하는 이유가 이거야. 
버프를 한 군데만 수정하면 모든 데미지 소스에 자동 반영되기 위해서

GE는 "35를 빼라"는 명령이 아니야. 
GE는 "Additive 채널에 -35를 등록해라"야.

변화의 소스를 분산시키지 마라. 집계를 한 곳에서 해라

금지 2를 위반하면 무슨 일이 생기나

        SpawnFireball();

}

금지 3을 위반하면 무슨 일이 생기나

// 애니메이션 시스템

// 입력 시스템
if (character->crowdControlFlags & CCF_STUN) { return; }

네 시스템이 "스턴"을 네 가지 다른 방식으로 표현해. 
"빙결 상태에서는 스턴처럼 행동하되, 불 데미지를 받으면 해제되어야 한다"는 요구사항이 생기면

각자의 bool과 enum과 string과 flag로?

Attribute를 Actor에 직접 두지 마라

{
    float Health;
    float Mana;
    float MoveSpeed;

GE를 코드로 만들지 마라

코드는 절차야. "이 순서로 이렇게 해라"를 명령으로 표현해. 
선언은 GAS가 읽어서 네트워크 복제, 스택 관리, 조건부 적용을 자동으로 처리할 수 있어
절차는 GAS가 읽을 수 없어. 코드는 실행될 뿐이야.

void UGA_Fireball::ApplyDamage(AActor* Target)
{
    auto* TargetChar = Cast<AMyCharacter>(Target);
    TargetChar->Health -= 35.0f;
    TargetChar->Health = FMath::Clamp(TargetChar->Health, 0.f, 100.f);
}

파이어볼 데미지를 35에서 40으로 올려주세요.
그리고 화염 저항 버프가 있으면 데미지가 반감되게 해주세요.
레벨 5 이상이면 3초 화상 DoT도 추가해주세요.

이 요청을 처리하려면 프로그래머를 불러야 해. 
코드를 수정하고, 빌드하고, 테스트해야 해. 
기획이 바뀔 때마다 이 사이클이 반복돼.

GE를 Blueprint 에셋으로 만드는 이유가 이거야
"이 효과는 이런 구조다"를 데이터로 표현해

기획자가 에디터에서 
Magnitude를 35에서 40으로 바꾸고, 
조건 태그에 Buff.FireResist를 추가하고, 
DoT용 GE를 연결해

효과의 구조를 절차가 아니라 선언으로 표현해라

---

후처리 로직을 GE 안에 넣지 마라
"체력이 0이 되면 죽게 하고 싶은데, 이걸 GE 안에 넣으면 안 될까?"

독 데미지 GE도 체력을 깎아. 이 GE 안에도 사망 체크를 넣어야 해. 
환경 데미지 GE도. 반사 데미지 GE도. 
체력을 깎는 GE가 10개라면 사망 체크 코드가 10군데에 있어.

그리고 "갓모드 버프가 있을 때는 체력이 0이 돼도 죽지 않는다"
는 요구사항이 생기면 10군데를 전부 수정해야 해.

PostGameplayEffectExecute가 AttributeSet에 있는 이유가 이거야
체력이 바뀌는 이유가 무엇이든 — 데미지든, 독이든, 환경이든 — 체력이 바뀐 결과에 대한 반응은 한 군데서 일어나야 해.

변화의 원인에 후처리를 붙이지 마라. 변화의 결과에 붙여라

---

이 구조가 없으면 두 번째 데미지 소스가 생기는 순간부터 문제가 시작돼
GAS의 복잡도는 기능 하나를 위한 것이 아니야. 
기능이 열 개, 백 개가 됐을 때 시스템 전체가 무너지지 않도록 하기 위한 거야

---

`PostGameplayEffectExecute`는 **서버에서 실행돼.** 
서버에는 오디오 시스템이 없어. 
사운드를 재생하는 코드를 여기 넣으면 서버에서 조용히 무시되거나, 더 나쁘게는 크래시가 나.

GAS에는 이것을 위한 전용 시스템이 있어. `GameplayCue`야.
GameplayCue는 클라이언트에서만 실행되는 것이 보장돼. 
서버는 "이 Cue를 발동시켜라"는 신호만 복제하고, 실제 재생은 클라이언트가 해.
