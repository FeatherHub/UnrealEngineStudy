엔진 수준의 "tick" 은 다른 리듬을 가진 시간계열들의 synchronization barrier

subsystem 마다 내재적 시간 해상도 (자연 리듬)이 다르다

엔진 Tick 은 "모든 subsystem 이 같은 속도로 뛰어야 하는 지점" 이 아니라 
"모두가 최소 한 번은 접선(tangent) 에서 만나는 지점"이다

Physics Substep

FPhysScene::TickScene
Chaos 의 물리 solver

물리적 수치 적분(특히 PBD 의 constraint solver)은 안정성을 위해 작은 dt 를 요구

프레임 rate 가 떨어져 Δt 가 33ms 가 되어도 
solver 는 이것을 Max Substep Delta Time (예: 16.67ms) 단위로 쪼개 n 번 반복 step 한다
이 경우 한 번의 Game Thread tick 이 물리 세계에서는 두 번의 연속 시뮬레이션 step 이 되는 것이다.


이 substep loop 안에서 또 다른 loop 가 nested 되어 있다
constraint solver 의 iteration

관절, 접촉, 마찰 같은 constraint 들은 한 번의 풀이로는 수렴하지 않으며 
Gauss-Seidel 방식으로 여러 번 반복해야 한다
Position Iterations, Velocity Iterations 같은 설정이 이 루프의 횟수다.


Game Thread Δt ─┬─ Substep 1 (fixed dt) ─┬─ Solver iteration 1
                │                        └─ Solver iteration 2
                │                        
                └─ Substep 2 (fixed dt) ─┬─ Solver iteration 1
                                         └─ Solver iteration 2

"바깥 tick + 고정 dt 분할 + 수렴 iteration"

Skeletal mesh 의 animation 은 기본적으로 매 tick 평가되지만, 
AURO 가 활성화되면 여러 tick 마다 한 번만 본 transform 을 새로 계산하고

내부 루프가 "여러 번" 돌지 않고 "가끔씩" 돈다
바깥 시계( 엔진 tick ) 위에 자기 고유 리듬의 내부 시계가 있다.

Dirty Marking — Deferring

dirty 함수들
MarkRenderStateDirty, MarkRenderTransformDirty, MarkRenderDynamicDataDirty

이벤트를 즉시 처리하지 않고 플래그만 세운 뒤 나중에 일괄 처리하는 구조다

Game Thread 에서 값이 바뀔 때 subsystem 은 bit 하나만 set 한다. 이 set 자체는 거의 공짜다(atomic bit 연산 한 번). 
Tick 의 render-state-update phase 에서 엔진이 dirty bit 이 켜진 Component 들을 순회하며 실제 업데이트 command 를 발행한다

1 deduplication

같은 frame 안에서 transform 을 10번 바꿔도 dirty bit 은 한 번만 set 된다. 
10번의 업데이트가 아니라 마지막 값의 1번만 Render Thread 로 간다.

2 batch
일괄 처리 단계에서 Component 들을 grouping 하여 Render Thread command 를 배치로 보낼 수 있다.

3 consistency
모든 업데이트가 frame 경계의 한 시점에 처리되므로 
Render Thread 가 보는 state 가 일관된 snapshot 이 된다.


시점의 분리를 통해 동기화 지점, 일괄 처리의 기반을 얻는다

  Bloom pass → 여러 해상도의 다운샘플 pyramid texture 들
  Tone mapping → LDR 결과 texture (Low Dynamic Range)

이 모든 texture 들이 VRAM 에 머문다

Game Thread: "이 view, 이 scene, 이 camera 로 그려라" 라는 high-level 요구만 넣음. BeginRenderingViewFamily 호출.

Render Thread (FSceneRenderer): 이 요구를 받아 pass 시나리오를 구성. RDG builder 에 pass 들을 등록.

RDG: 등록된 pass 그래프를 분석하여 실행 순서, resource lifetime, barrier 를 자동 결정

RHI Thread: RDG 가 생산한 RHI command 를 플랫폼별 API 로 번역

GPU: 실제 실행
