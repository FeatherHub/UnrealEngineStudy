순환 참조가 로딩에 일으키는 문제는 재진입이다

A를 채우다 B로, B를 채우다 다시 A로 오는 것은 A의 로드가 안 끝났는데 A에 대한 로드 요청이 또 들어오는 상황이다

재진입이 안전하려면 두 번째 요청이 무한재귀에 빠지지 않고, 
반쯤 된 작업을 중복하지 않고, 그러면서도 유효한 포인터를 받아 가야 한다

캐시에 이미 있으니 새로 만들지 않고(무한재귀 차단)
RF_NeedLoad가 꺼져 있으면 Serialize를 반복하지 않고(중복 차단)
켜져 있으면 그 자리에서 마저 채우면 되며
어느 경우든 포인터 자체는 즉시 유효하다

순환 참조의 해법
생성/직렬화 2단계 분리

주소 교환은 껍데기끼리도 가능

Preload(A) 시작 → A->Serialize 진행 중 B 참조를 만남
→ CreateExport(B)로 B의 껍데기만 생성(이 순간 B의 Serialize는 하지 않는다, 포인터 대입에는 상대의 내용이 필요 없으므로)
→ B 포인터를 A의 필드에 대입 → A->Serialize 끝, A 완성. 

그 후 큐에 들어 있던 B 차례가 오면 → Preload(B) → B->Serialize 중 A 참조를 만남 → A는 이미 캐시에 완성돼 있으니 포인터 대입 → B 완성.

A의 직렬화와 B의 직렬화가 서로의 안에 중첩되지 않고 독립된 시점에 한 번씩 실행된다
큐를 통해 자기 차례에 온다

{

  if (!Obj)
  {
    LoadPackage(PackageOf(Path))                   // 없으면 패키지째 로드

  }

  return Obj

UPackage* LoadPackage(PackagePath)
{
  if (Cooked(PackagePath))                                      // 갈림길은 여기 하나
    return await FAsyncLoadingThread2::LoadPackageAsync(...)    // await = FlushAsyncLoading(RequestID)

  Linker = FLinkerLoad::CreateLinker(PackagePath)
  {
    Loader = OpenFile(PackagePath)
    Read(Summary, NameMap, ImportMap, ExportMap)   // 목차만 적재. 본문은 아직 안 읽음
  }

  for (i : Linker.ExportMap)
  {
    FLinkerLoad::CreateExport(i)                   // 생성
    FLinkerLoad::Preload(ExportMap[i].Object)      // 직렬화
  }

  EndLoad(LoadContext)                             // 잔여 직렬화 + 후처리
  return Linker.LinkerRoot
}

UObject* FLinkerLoad::CreateExport(i)              // 생성 — 존재부터 만든다
{
  Class = Resolve(ExportMap[i].ClassIndex)         // 재귀 지점 ①
  Outer = Resolve(ExportMap[i].OuterIndex)         // 재귀 지점 ②
  ExportMap[i].Object = NewObject(Class, Outer)    // 플래그: NeedLoad | NeedPostLoad
  LoadContext.Queue += ExportMap[i].Object
}

호출이 완료되기 전에 같은 호출이 중첩되는 것

void FLinkerLoad::Preload(Obj)                     // 직렬화 — 내용을 채운다
{
  if (!Obj.NeedLoad) return                        // 플래그 = 재진입 가드
  Obj.NeedLoad = false                             // 순환 대비, 먼저 끈다
  Loader.Seek(Export.SerialOffset)
  Obj.Serialize(Loader)                            // 참조 필드마다 Resolve(idx) → 재귀 지점 ③
}

UObject* FLinkerLoad::Resolve(idx)                 // 모든 재귀의 단일 관문
{
  if (idx > 0) return ExportMap[idx−1].Object  ?? CreateExport(idx−1)                   // 내 패키지
  if (idx < 0) return ImportMap[−idx−1].XObject ?? StaticLoadObject(PathOf(Import))     // 남의 패키지 → 맨 위로 재귀
  return null
}

void EndLoad()
{
  while (Queue.NotEmpty())
  {
    Batch = Queue.TakeAll()
    for (Obj : Batch) Fill(Obj)          // 미완성 소진 (가드 덕에 한 번씩만)
    for (Obj : Batch) Obj.PostLoad()
  }
}

FArchive& FLinkerLoad::operator<<(UObject*& Object)   // 어떤 Serialize 안의 Ar << MyMesh 가 도착하는 곳
{
  FPackageIndex Index;
  *this << Index;                     // ① 파일 커서에서 정수 읽기
  Object = IndexToObject(Index);      // ②③
  return *this;                       // ④ Object가 곧 MyMesh의 참조 — 대입은 이미 끝났다
}

UObject* FLinkerLoad::IndexToObject(FPackageIndex Index)   // ② 부호로 테이블 선택
{
  if (Index.IsExport()) return CreateExport(Index.ToExport());
  if (Index.IsImport()) return CreateImport(Index.ToImport());
  return nullptr;
}

IoStore

    ├─ 카메라에 붙는 이펙트 (1인칭 총구 화염, 레이저 사이트 — 카메라 보정 후 위치여야 안 떨린다)
    └─ 스크린 공간 기준 계산(월드 위젯 앵커, 조준점 투영)

  UWorld::RunTickGroup(TG_LastDemotable)

  ULevelInstanceSubsystem::Tick()              // 레벨 인스턴스 (틱 프레임 밖 — 레벨 삭제 가능성 때문)

  UWorld::FinishAsyncTrace()                   //    비동기 충돌 쿼리 회수

  UWorld::BroadcastTickFlush() -> UNetDriver::TickFlush  // ⑦ 네트워크 송신 — 리플리케이션 발사

  FFXSystemInterface::Tick()                 // ⑧ FX 시스템 — 나이아가라 GPU 시뮬 준비·디스패치 (렌더 연계)

  GEngine->ConditionalCollectGarbage()     // ⑨ GC 판정 — 주기 도달/요청 시 마킹·수거
}

  요청 큐의 패키지들을 생성→직렬화 파이프라인으로 처리, 완료 큐에 적재

      World->UpdateLevelStreaming()  //   레벨 요구 vs 현실 대조: 새 로드 [발사] + 완료 레벨 AddToWorld(시간분할)

      ProcessAsyncLoading(잔여 예산) //   남은 시간 재투입
    }
  }

FlushAsyncLoading은 이 체계의 비상구
아무 지점에서나 완료까지 블로킹으로 완주시키는 버튼
정당한 사용처는 화면이 어차피 멈춰 있는 곳(맵 전환, 로딩 스크린)

프레임 N의 어느 틱에서 발사 → N…N+k 동안 로딩 스레드가 진행(게임 스레드는 무관심) 
→ 프레임 N+k의 HandleUnifiedStreaming에서 소비(콜백 발화, 편입 시작) 
→ 편입이 시간 분할이면 N+k+m에 완료되어 그 다음 틱부터 그 액터들이 틱 그룹에 합류

RHI에 프레임 경계를 알리고, GFrameCounter를 넘긴다

게임 스레드가 프레임 N을 시뮬레이션할 때 렌더 스레드는 N−1을 그리고 GPU는 N−2를 출력한다. 
RedrawViewports는 생산이다: 이번 프레임의 뷰를 계산해 렌더 커맨드를 큐에 넣는 것. 
끝의 동기화는 배관 관리다: 게임 스레드가 렌더 스레드보다 허용치(1~2프레임) 이상 앞서 나갔으면 여기서 기다린다

로딩 스레드는 오브젝트 그래프의 상시 변경자
GUObjectArray에 노드(새 오브젝트)를 추가하고, Serialize로 참조 필드에 간선을 써 넣는다

GC 마킹은 그 그래프의 일관된 스냅샷 순회다.

마킹이 시작된 뒤 태어난 오브젝트는 마킹에서 누락되어 살아있는데 수거되고(디스크에서 막 읽은 오브젝트가 증발 — 크래시)
반쯤 쓰인 참조 필드를 마킹이 읽으면 쓰레기 포인터를 추적한다

스트리밍 게임에서 로딩은 상시 활동이라, 로딩이 조용해질 때까지 GC를 미루면 오픈월드에서는 영원히 미뤄진다

GC 수요를 만드는 주범이 바로 스트리밍이다(셀 인/아웃이 쓰레기를 양산)

즉 로딩이 활발할수록 GC가 더 자주 필요하다

로딩은 임계 구간(오브젝트 생성 등)을 락과 함께 수행하고, GC가 그 락을 잡으면 로딩이 구간 경계에서 멈춘다
