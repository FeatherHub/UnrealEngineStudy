생성·변형·로드/언로드는 그 흐름의 서로 다른 국면일 뿐

Unborn

Hybrid

플레이어 이동과 청크 생성: 동심원 모델

플레이어 월드 좌표 (Wx, Wy, Wz)에서 청크 좌표 (Cx, Cy, Cz)로의 변환은 정수 나눗셈
청크 크기가 16이면 Cx = floor(Wx / 16)
왜냐하면 청크는 16×16×256 블록이므로, 청크 좌표는 월드 좌표를 16으로 나눈 값입니다

이 변환이 프레임마다 수행되어야 하고, 
청크 좌표가 변경된 순간 전체 스트리밍 시스템이 작동

매 Tick마다 플레이어의 현재 청크 좌표를 계산하고, 이전 프레임의 청크 좌표와 비교합니다. 
동일하면 아무 일도 일어나지 않습니다. 
다르면 스트리밍 업데이트가 촉발됩니다

동심원 우선순위 링(concentric priority rings)

Ring 0 (반경 0-2 청크)
즉시 로드.
플레이어가 현재 서 있는 청크와 직접 인접 청크
이 링은 동기적으로, 또는 최소한 프레임 내에 완료되어야 합니다
게임 시작 시 초기 스폰 대기가 필요한 이유

Ring 1 (반경 3-6 청크)
높은 우선순위 비동기 로드
플레이어가 달리면 1-2초 내에 도달할 수 있는 거리
메시까지 완전히 빌드되어 있어야 합니다

Ring 2 (반경 7-12 청크)
중간 우선순위
데이터는 생성하되, LOD를 낮추거나 먼 거리 전용 단순화된 메시만 빌드

Ring 3 (반경 13-16+ 청크)
최저 우선순위
청크 데이터만 생성하고 메시는 빌드하지 않을 수도 있습니다
또는 2D heightmap만으로 "실루엣 메시"를 생성하여 먼 지형의 윤곽만 보여줍니다

Unload Ring (설정 반경 바깥)
언로드 후보
언로드 큐에 넣어 일정 시간 유예를 줍니다.
플레이어가 경계에서 왔다 갔다 할 때 반복 로드/언로드(thrashing)를 방지하기 위해
이 유예 거리를 hysteresis라 부르며, 보통 로드 반경보다 2-3 청크 더 넓게 설정합니다
hysteresis란 시스템이 상태를 변경할 때, 그 상태를 유지하려는 경향입니다.

플레이어가 청크 경계를 넘으면, 새로 로드해야 할 청크들의 목록이 생성

이 목록은 맨해튼 거리 또는 체비셰프 거리 기준으로 정렬된 우선순위 큐에

맨해튼 거리란
taxicab distance
L1 distance
Unlike Euclidean distance, which calculates the straight-line distance, 
Manhattan distance sums the absolute differences of the coordinates along each axis

d = |x1 - x2| + |y1 - y2| + |z1 - z2|

체비셰프 거리
d = max(|x1 - x2|, |y1 - y2|, |z1 - z2|)
3D 공간에서 점 (0, 0, 0)과 (3, 4, 5) 사이의 체비셰프 거리는 5

플레이어의 이동 방향을 고려하는가?
플레이어가 북쪽으로 달리고 있다면, 북쪽의 새 청크가 남쪽의 같은 거리 청크보다 급합니다
이동 벡터의 내적으로 우선순위에 가중치

청크 하나가 Unborn에서 Loaded로 전이되는 과정은 여러 단계로 나뉘며
각 단계의 스레드 친화성이 다릅니다

Stage 1 — Block Data Generation (Worker Thread)
Noise 함수로 블록 배열을 채움
UE의 FAsyncTask 또는 UE5 Tasks::Launch로 실행

Stage 2 — Decoration Pass (Worker Thread)
나무는 청크 경계를 넘을 수 있습니다. 한 청크에서 시작된 나무의 잎이 인접 청크에 걸칠 수 있음

(a) 인접 청크도 동시에 생성되어 있어야 나무를 배치할 수 있다 — 의존성이 생겨서 병렬성이 줄어듦. 
(b) 나무를 배치할 때 인접 청크에 "대기 중인 변형(pending modifications)"을 기록해두고, 그 인접 청크가 나중에 로드될 때 적용한다

Stage 3 — Light Propagation (Worker Thread, but 인접 의존성 있음)
인접 청크가 아직 미생성이면, 경계면 라이트를 잠정값으로 두고, 인접 청크가 로드될 때 경계를 따라 재전파합니다

Stage 4 — Mesh Build (Worker Thread)

경계면의 face를 cull할지 결정하려면 옆 청크의 블록이 불투명한지 알아야 하므로. 
따라서 인접 청크의 블록 데이터가 최소한 Stage 1까지는 완료되어 있어야 합니다

Stage 5 — GPU Upload & Scene Registration (Game Thread)

버텍스/인덱스 버퍼를 RHI를 통해 GPU에 업로드하고, SceneProxy를 등록
이것만은 반드시 GameThread에서 수행
렌더 커맨드 큐는 GameThread에서만 접근 가능하기 때문

인접 청크 의존성
청크 경계에서 눈에 보이는 이음새(seam)
옆 청크와 라이팅이 안 맞거나, 
경계면에 불필요한 면이 렌더링되거나, 
나무가 잘려 나가거나

블록 변형의 데이터 표현: "원본 + 델타" 모델

Full Snapshot
청크의 전체 블록 배열을 저장
월드가 커지면 세이브 파일이 급격히 비대

Seed + Delta
플레이어가 변형한 블록의 좌표와 새 블록 타입의 목록(델타)
로드할 때는 먼저 seed로부터 원본 청크를 재생성하고, 그 위에 델타를 적용

장점: 변형이 없는 청크는 디스크에 아무것도 저장하지 않아도 되므로 세이브 파일이 극적으로 작아짐. 
단점: 로드할 때 noise 재생성 비용이 항상 발생

Procedural generation의 본질은 seed로부터 결정론적으로 재현 가능하다는 것
이건 공짜로 얻는 무한 압축
65,536 블록의 데이터를 seed 하나(8바이트)로 인코딩

---

델타 하나의 단위 구조

struct FBlockDelta {
    FIntVector LocalPosition;  // 청크 내 로컬 좌표 (0-15, 0-15, 0-255)
    uint16 NewBlockID;         // 새 블록 타입 (0 = Air = 파괴)
};

같은 좌표에 파괴 → 배치 → 파괴가 반복되면 델타가 누적됩니다
따라서 저장 시점에 **압축(compaction)**이 필요합니다
같은 좌표의 마지막 델타만 유지
최종 상태가 원본과 동일한 델타는 제거

[ChunkCoord: int32 × 3]
[DeltaCount: uint32]
[Delta0][Delta1]...[DeltaN]

블록 변형이 많은 청크(플레이어가 대규모 건축을 한 경우)
적응적 전략
델타 수가 임계치(예: 전체 블록 수의 30%)를 넘으면 full snapshot으로 전환
RLE(Run-Length Encoding) 또는 palette 압축을 적용

세이브 파일의 전체 구조

청크마다 개별 파일을 만들면 파일 시스템에 수십만 개의 파일이 생깁니다
32×32 청크를 하나의 파일로 묶고, 파일 헤더에 각 청크의 오프셋과 크기를 기록하여 O(1)으로 개별 청크에 접근합니다

FArchive로 region file을 열고, 헤더를 읽어 원하는 청크의 오프셋으로 Seek한 뒤, 해당 청크의 델타 또는 snapshot을 읽습니다
쓸 때는 청크 데이터가 이전보다 커졌으면 파일 끝에 새로 쓰고 헤더의 오프셋을 갱신합니다

append-and-update-header 패턴은 단편화를 만들지만, 주기적 compaction으로 해결합니다

4.1 Unborn → Loaded (최초 방문)
이 전이는 비용이 가장 큽니다
Noise 생성 + decoration + light propagation + mesh build + GPU upload 전체 파이프라인

**Stage 1(블록 데이터 생성)을 실제 로드 반경보다 넓은 범위에서 미리 수행(pre-generation)**

메시는 빌드하지 않되 블록 데이터만 메모리에 올려놓으면, 
나중에 플레이어가 접근할 때 Stage 2부터 시작할 수 있고, 
인접 청크의 블록 데이터 의존성도 이미 해소되어 있습니다

4.2 Dormant → Loaded (재방문)

디스크에서 region file을 읽고, seed로 원본 생성 후 델타 적용, 메시 빌드, GPU 업로드
noise 재생성 비용은 동일하지만, 디스크 I/O가 추가됨
SSD에서는 차이가 미미하고, HDD에서는 유의미합니다

최근 언로드된 청크의 블록 데이터를 메모리에 LRU 캐시로 유지하면, 디스크 I/O와 noise 재생성을 모두 건너뛸 수 있습니다

hysteresis는 언로드 자체를 지연시키고, LRU 캐시는 언로드된 후에도 데이터를 메모리에 유지합니다.

4.3 Loaded → Dormant (언로드)

Step 1 — 델타 compaction
seed로부터 생성한 원본을 비교
최종 델타 목록을 산출

Step 2 — Serialization
region file에 기록. 비동기 I/O로 

Step 3 — Scene Deregistration
SceneProxy 해제, collision body 해제
GameThread에서 수행

Step 4 — Memory Release
블록 배열, 메시 버퍼 해제

비동기 파이프라인에서 완료된 청크가 10개 쌓여 있어도, GameThread는 프레임당 2개만 씬에 등록하고 나머지는 다음 프레임으로 미룹니다
"팝인(pop-in)" — 먼 곳에서 다가오면 청크가 갑자기 나타나는 현상
pre-generation과 LOD로 완화

---

복셀 월드의 청크 관리가 사실상 수제 데이터베이스 엔진을 구현하는 것
캐싱, write-ahead logging, 적응적 압축, 인접 의존성 해소
DBMS의 핵심 개념들이 게임 엔진 내부에서 재발명되는 것

---

연속적인 세계를 불연속적인 단위로 분할했기 때문에, 
분할선을 넘는 모든 연산이 특수 처리를 요구한다

Decoration(나무 등이 경계를 넘음), 
Light Propagation(빛이 경계를 넘어 전파),
Mesh Build(경계면의 face culling에 이웃 데이터 필요),
Block Modification(경계 근처 블록 파괴/설치 시 이웃 청크의 메시/라이트 갱신)

"내 청크의 연산이 이웃 청크의 데이터에 의존한다."

---

이웃 접근의 통합 인터페이스

"좌표 (x, y, z)의 블록을 알고 싶은데, 그 좌표가 내 청크 바깥이면 어떻게 하는가?"

이걸 매 시스템마다 개별적으로 처리하면 코드가 산산조각 나므로, 
먼저 이웃 접근을 추상화하는 단일 인터페이스를 만들어야 합니다

Neighborhood의 조립 시점
청크가 Stage 2(Decoration) 이상으로 진행하기 전에
ChunkManager가 해당 청크의 이웃 6개를 조회하여 Neighborhood를 조립합니다

---

횃불이 청크 모서리에 있으면 빛이 7개 청크에 동시에 퍼져야 합니다

각 청크는 자기 자신의 라이트 데이터만 직접 수정한다
"이웃에게 이만큼의 빛이 유입될 것이다"라는 메시지를 남깁니다
이웃은 자기가 다시 라이트를 계산할 때 그 메시지를 소비합니다
한 청크의 데이터에 대한 쓰기 권한은 오직 그 청크를 처리하는 태스크만 갖습니다

블록 변형 시 이웃 무효화: 연쇄 갱신

---

UE는 이미 다층 좌표계를 가지고 있습니다

World Space
Actor Space (Actor의 Transform)
Component Space (Component의 RelativeTransform)
Local Space (메시 버텍스의 좌표)

이 계층은 USceneComponent::GetComponentTransform()이 반환하는 FTransform에 의해 연결됩니다. 
FTransform은 위치(FVector), 회전(FQuat), 스케일(FVector)을 캡슐화하고, 
TransformPosition(), InverseTransformPosition()으로 계층 간 변환을 수행합니다

복셀 시스템은 이 계층에 두 개의 추가 레이어를 삽입
World → Chunk → Local Block

복셀의 경우 회전과 스케일이 항상 항등(identity)이므로 FTransform이 과잉이고, 
정수 좌표 + 정수 나눗셈으로 충분합니다
FTransform이 존재하는 이유는 일반화된 3D 객체는 임의의 위치·회전·스케일을 가지기 때문
복셀처럼 그 자유도가 제한된 도메인에서는 더 가벼운 표현이 적합

---

FMath vs <cmath>

플랫폼 간 동일한 결과 보장(determinism), 
SIMD 가속 경로 제공, 
NaN/Inf 체크 옵션. 

복셀 좌표 변환에서 floor가 플랫폼마다 다른 결과를 내면 같은 seed에서 다른 월드가 생성되므로, 
이 결정론적 보장이 중요합니다

UE에서 월드의 공간 데이터를 담는 기존 경로는
UStaticMesh → FStaticMeshRenderData → FStaticMeshLODResources → 버텍스/인덱스 버퍼 체인

이 체인은 메시 형태의 데이터를 전제
삼각형 목록, 버텍스 포지션, 노멀, UV. 복셀 블록 배열은 메시가 아닙니다

FChunkData는 UE의 렌더링 파이프라인 바깥에 독자적으로 존재하는 도메인 데이터

UNavigationSystemV1의 NavMesh 데이터
(FRecastNavMeshGenerator가 관리하는 타일 배열)

ALandscapeProxy의 높이맵 데이터
(ULandscapeHeightfieldCollisionComponent의 CollisionHeightData).

이들도 렌더링 메시와는 별개의 원본 데이터를 갖고, 
거기서 메시나 충돌체를 파생시킵니다.

TArray<uint16>
3D 좌표 → 1D 인덱스 변환(x + y*W + z*W*H)
cache-friendly 레이아웃의 대가

AsyncTask([=]() {

UE5에서 비동기 작업을 실행하는 경로는 세 가지입니다

경로 A — FAsyncTask / F AutoDelete AsyncTask + G ThreadPool
UE4 시절부터의 전통적 방법
FNonAbandonable Task를 상속하여 Do Work()를 구현하고,
G ThreadPool->Add QueuedWork()로 제출합니다. 
스레드 풀의 크기는 FPlatformMisc::NumberOfCores Including Hyperthreads() - 2 로 설정

명시적 클래스 정의가 필요해서 보일러플레이트가 많다

경로 B — UE5 Tasks System (UE::Tasks::Launch)
람다 기반으로 훨씬 간결

UE::Tasks::Launch(TEXT("ChunkGen"), [=]() { ... }) 로 워커 스레드에 작업을 던집니다.
내부적으로 FTaskGraph(현재는 LowLevelTasks 백엔드) 위에서 동작하며
태스크 간 의존성을 Prerequisites로 선언할 수 있습니다
복셀 파이프라인의 "Stage 2는 이웃의 Stage 1 완료에 의존한다"는 관계를 자연스럽게 표현할 수 있습니다

---

// stage 1 task for self
UE::Tasks::Launch(TEXT("ChunkGen"), [=]() {
    GenerateBlockData(chunk, WorldSeed);
});

// stage 1 tasks for neighbors
TArray<UE::Tasks::FTask> neighborTasks;
for (auto* neighborChunk : GetNeighborChunks(coord)) {
    if (neighborChunk->Stage1Task.IsValid()) {
        neighborTasks.Add(neighborChunk->Stage1Task);
    }
}

// stage 2 task
UE::Tasks::Launch(
  #1 TEXT("ChunkDecorate"), 
  #2 [=]() { ... },
  #3 Prerequisites(stage1Task, neighborTasks)
)

UProceduralMeshComponent

이 컴포넌트는 내부적으로 FProceduralMeshSceneProxy를 생성
이 프록시가 매 프레임 GetDynamicMeshElements()에서 FMeshBatch를 렌더러에 제출

대비: Static Mesh의 GetStaticMeshElement()
Dynamic 경로는 매 프레임 가시성 판정과 드로콜 제출을 반복
Static 경로는 캐싱된 드로콜 리스트를 재사용
복셀 청크는 변경 빈도가 "가끔"이므로 둘 사이에 위치

FArchive
operator<<가 읽기/쓰기 양방향으로 동작한다

FArchive가 Loading 모드이면 <<가 데이터를 읽어 변수에 채우고
Saving 모드이면 변수에서 데이터를 꺼내 버퍼에 씁니다

FBufferArchive saveAr;          // FArchive의 Saving 모드 구현
SerializeChunk(saveAr, data);   // <<가 변수 → 버퍼 방향

FMemoryReader loadAr(bytes);    // FArchive의 Loading 모드 구현
SerializeChunk(loadAr, data);   // <<가 버퍼 → 변수 방향

최상층 (GameFramework)

복셀은 이 층을 가볍게 사용합니다
이 층의 풍부한 기능(Gameplay Ability, AI Controller, 네트워크 리플리케이션)은 대부분 우회됩니다

중간층 (렌더링 추상화)

SceneProxy, FMeshBatch, VertexFactory
커스텀 SceneProxy를 통해 엔진의 렌더 파이프라인에 직접 메시를 주입합니다

하부층 (시스템 인프라)

FArchive, Task System, FPlatformFile, FMath
이 층은 UE의 "게임 엔진"이라기보다는 "크로스 플랫폼 시스템 라이브러리"에 가깝습니다

최상층을 최소한으로 쓰고, 중간층에서 싸우고, 하부층에 기대는 것입니다.
