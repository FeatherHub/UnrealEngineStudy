Shadow map

광원 시점에서 장면을 한 번 렌더링
깊이(depth)만 저장한 텍스처

이를 이용한 그림자 판정 기법이 shadow mapping

---

1단계 (Shadow Depth Pass)

카메라 대신 광원 위치에 가상의 카메라를 놓고 장면을 렌더링
색은 필요 없고 각 texel에 광원에서 가장 가까운 표면까지의 거리 z_map만 기록
광원이 볼 수 있는 표면들의 거리 지도

파이프라인(VS → clipper → rasterizer → PS → OM)을 그대로 다시 씁니다
카메라 depth prepass와 하드웨어적으로 동일한 종류의 패스
시점과 타깃만 다릅니다

View/Projection 행렬이 카메라가 아니라 광원의 것
클리핑도 광원 절두체 기준으로 동일하게 일어남

렌더 타깃 없이 depth buffer(DSV)만 바인딩
출력은 오직 SV_Depth

광원 N개면 장면을 N + 1번 래스터라이즈

---


2단계 (shadow test)

scene depth로 픽셀의 월드 위치 P를 복원 → 광원 클립 공간으로 변환
shadow map 샘플과 z 비교 → 결과를 shadow mask 텍스처에 기록

입력이 텍스처 두 장(scene depth, shadow map)

---

카메라 대신 광원의 view 행렬과 projection 행렬을 쓴 클립 공간

Directional light면 orthographic projection
spot light면 perspective projection

---

광원 입장에서 보이는 표면만 빛을 받고
광원 시점에서 다른 물체에 가려진 표면은 그림자가 됩니다

bias는 자기 자신의 깊이와 비교할 때 생기는 self-shadowing 노이즈(shadow acne)를 막기 위한 작은 오프셋

정점이 vertex shader를 통과하면 최종적으로 4차원 동차 좌표 (x, y, z, w)를 출력 (HLSL의 SV_Position)

각 성분을 w로 나누면 NDC(Normalized Device Coordinates)가 됩니다 (x/w, y/w, z/w)

−w ≤ x ≤ w, −w ≤ y ≤ w, 0 ≤ z ≤ w

UE는 reversed-Z라 가까운 면이 z′ = 1, 먼 면이 z′ = 0입니다.

패스 5 (Post Process): 역시 풀스크린, 마지막에 backbuffer 기록

모든 오브젝트가 모든 패스에 참여하지는 않습니다. 

패스마다 자기 목록이 따로 있다
(불투명만 base pass, 반투명은 translucency 패스, 그림자 안 드리우는 건 shadow depth 제외, 컬링된 건 아예 빠짐)

---

void RenderFrame()
{
    // ---------- 0. 준비 (CPU) ----------
    // 컬링 후, 패스별로 참여할 드로콜 목록을 미리 만들어 둔다
    VisibleObjects   = FrustumCull(Scene.Objects, Camera);
    PassLists[Depth]       = FilterOpaque(VisibleObjects);
    PassLists[Shadow]      = FilterShadowCasters(Scene.Objects, Lights);   // 광원 기준 컬링
    PassLists[BasePass]    = FilterOpaque(VisibleObjects);
    PassLists[Translucent] = FilterTranslucent(VisibleObjects);

    // ---------- 1. PrePass : 카메라 depth ----------
    SetRenderTargets(depth: SceneDepth, color: none);        // PS 없음
    for (Object in PassLists[Depth])
        Draw(Object, CameraViewProj);                        // depth만 기록

    // ---------- 2. Shadow Depth : 광원마다 ----------
    for (Light in Lights)
    {
        SetRenderTargets(depth: ShadowMap[Light], color: none);
        for (Object in PassLists[Shadow][Light])
            Draw(Object, LightViewProj[Light]);              // 광원 클립 공간
    }

    // ---------- 3. Base Pass : 재질 속성 기록 ----------
    SetRenderTargets(depth: SceneDepth, color: GBufferA..D);
    for (Object in PassLists[BasePass])
        Draw(Object, CameraViewProj);                        // albedo, normal, roughness ...

    // ---------- 4. Lighting : 오브젝트 없음, 화면 픽셀 순회 ----------
    SetRenderTargets(color: SceneColor);
    for (Light in Lights)
    {
        // 4a. shadow projection: 픽셀별 shadow test → 마스크
        ShadowMask = FullscreenPass(SceneDepth, ShadowMap[Light]);
        // 4b. 라이팅 누적 (additive blend)
        FullscreenPass(GBuffer, SceneDepth, ShadowMask) -> SceneColor;
    }

    // ---------- 5. Translucency : SceneColor 위에 blend ----------
    SetRenderTargets(depth: SceneDepth /*test only*/, color: SceneColor);
    for (Object in SortBackToFront(PassLists[Translucent]))
        Draw(Object, CameraViewProj);

    // ---------- 6. Post Process : 텍스처 릴레이 ----------
    Temp0 = FullscreenPass(SceneColor);          // TAA
    Temp1 = FullscreenPass(Temp0);               // bloom 등
    FullscreenPass(Temp1) -> Backbuffer;         // tonemap — 여기서 처음 최종 버퍼에 기록

    Present(Backbuffer);
}

왜 반투명은 deferred가 불가능한가

GBuffer는 픽셀당 표면 1개의 속성만 저장할 수 있습니다. 

Deferred의 전제는 픽셀마다 최종적으로 보이는 표면이 하나(depth test 승자)라는 것인데, 

반투명은 정의상 한 픽셀에 유리 뒤에 연기 뒤에 물처럼 여러 표면이 겹쳐 보이고 
각각의 라이팅 결과가 blend 순서대로 합성되어야 합니다. 

픽셀당 1슬롯인 GBuffer에는 이 레이어들을 담을 수 없으므로, 

각 반투명 표면은 자기 드로콜의 PS 안에서 라이팅까지 끝내고 
완성된 색을 내보내 blend되는 수밖에 없습니다. 
이것이 forward shading의 정의입니다.

한 camera pixel에 투명 표면이 여러 겹 있을 수 있기 때문
  glass front layer
  smoke layer
  water layer
  opaque wall

조명 계산 후 scene color render target에 blend

---

PS_Opaque()
{
    GBufferA = normal;
    GBufferB = metallic, roughness, ShadingModelID;
    GBufferC = albedo;
    // 빛 계산은 4번으로 미룸
}

PS_Translucent()
{
    Color = 0;
    for (Light in LightsAffectingThisPixel)      // 라이트 루프가 PS 안으로 들어옴
        Color += BRDF(Surface, Light) * ShadowFactor(Light);
    return float4(Color, Opacity);               // OM이 SceneColor와 blend
}

---

for pixel in screen_pixels:

    camera_scene_depth = ReadCameraSceneDepth(pixel)

    world_position =  ReconstructWorldPosition(pixel, camera_scene_depth)

    light_space_position = TransformWorldPositionToLightSpace(world_position)

    shadow_map_uv = ProjectLightSpacePositionToShadowMapUV(light_space_position)

    receiver_light_space_depth = light_space_position.depth

    stored_shadow_map_depth = SampleShadowMapDepth(shadow_map_uv)

    if receiver_light_space_depth > stored_shadow_map_depth + shadow_bias:
        shadow_factor = 0
    else:
        shadow_factor = 1

    final_light += EvaluateLighting(pixel_material, light) * shadow_factor

아틀라스

프레임에 spot light 20개가 있으면 shadow map 20장이 필요한데, 
이를 각각 별도 텍스처로 만들면 렌더 타깃 전환 20번(패스 경계의 비용)과 텍스처 20개 관리가 필요

대신 2048 × 2048 텍스처 한 장을 잡고 
라이트마다 512 × 512 구역을 배정하면, 
렌더 타깃을 한 번만 묶고 viewport/scissor만 구역별로 옮기며 연달아 그릴 수 있습니다

Shadow map 한 장 = 가상 카메라 하나의 절두체
원근 투영 하나가 담을 수 있는 시야각은 180° 미만
왜곡 때문에 ~90∼120°가 한계

---

Spot light: 빛이 원뿔 하나로 퍼집니다. 
원뿔은 시야각이 보통 90° 내외라서 원근 카메라 1대로 정확히 덮입니다. 그래서 1장.

Point light: 빛이 전방향 360°로 퍼집니다
카메라 1대로는 물리적으로 불가능하므로, 
정육면체의 6면 방향으로 90° 카메라 6대를 놓아 전 방향을 빈틈없이 분할
cube map

---

Directional light
절두체가 아니라 평행 투영(orthographic)
카메라가 보는 범위 전체(수 km)를 한 장에 담으면 
texel 하나가 수십 cm가 되어 발밑 그림자가 뭉개집니다

카메라 깊이 범위를 구간으로 나눠(예: 0∼10 m, 10∼40 m, 40∼160 m…) 
구간마다 별도의 ortho 카메라와 shadow map을 배정
이것이 캐스케이드(CSM)
구간당 FProjectedShadowInfo 1개씩 

가까운 구간일수록 좁은 영역을 같은 해상도로 덮으니 촘촘해집니다
VSM의 clipmap은 이 아이디어의 연속

Slope-scaled depth bias
Shadow acne(자기 그림자 노이즈)의 해결책
비교 시 여유값(bias)을 주는 것
필요한 여유의 크기가 표면 기울기에 비례
빛에 대해 비스듬한 표면은 texel 하나 안에서 depth가 texelSize × tan θ만큼 변합니다

상수를 키우면 이번엔 그림자가 물체에서 떨어져 나가는 peter-panning이 생깁니다
절충이 bias = constant + slopeScale × max(|∂z/∂x|, |∂z/∂y|),

래스터라이저 고정 기능이 depth 기록 시 자동 적용

불투명 vs 반투명의 그림자

불투명 + masked만 shadow map에 그려지고
반투명은 그림자를 드리우지 않습니다

반투명 그림자(투과율을 누적하는 전용 아틀라스, FSortedShadowMaps::TranslucencyShadowMapAtlases)가 있지만 
일반 depth 비교와는 다른 메커니즘

반투명 forward PS가 shadow map(또는 VSM 페이지)을 직접 샘플링하며, 
보통 탭 수를 줄이거나 volumetric 그림자 근사(translucency lighting volume)를 쓰는 등 
품질을 낮춘 경로를 탑니다

동적 — 장부 형식 자체를 교체 (Fourier Opacity Map)

라이트의 CastTranslucentShadows 옵션
(ULightComponent::CastTranslucentShadows)이 켜지면 

FProjectedShadowInfo::bTranslucentShadow인 per-object shadow가 생성

FProjectedShadowInfo::RenderTranslucencyDepths가 반투명 캐스터들을 
전용 아틀라스(FSceneRenderer::SortedShadowsForShadowDepthPass:FSortedShadowMaps::TranslucencyShadowMapAtlases[i])에 그립니다.

핵심 발상은 저장 내용의 교체

거리 대신 그 표면의 광학적 밀도를 −ln(1−opacity) 형태로 변환해서, 
깊이의 함수인 투과 곡선을 푸리에 급수 계수(cos/sin 몇 개)로 투영해 렌더 타깃 두 장에 씁니다

이 표현의 결정적 장점은 additive blend로 누적된다는 것입니다
유리 뒤 연기 뒤 커튼이 어떤 순서로 그려지든 계수를 그냥 더하면 됩니다

푸리에 항이 몇 개뿐이라 투과 곡선이 뭉툭하고(저주파 근사, ringing)
per-object 단위라 비용이 커서 주연급 오브젝트에만 씁니다
