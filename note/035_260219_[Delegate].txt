      거짓된 독립성
    별로 안 얽혀 있음 -> 코드 상 두 개이고 의미적으로도 두 개임 : 일치

복잡성, Decoupling, Composability (Reusability)

“A를 보면 B가 거의 결정된다” → 두 모듈은 열역학적으로 하나의 거대한 마이크로스테이트로 수렴한다.
B가 바뀌면 A가 자동으로 따라 바뀌게 됨 => 코드상에서 “수동적 동기화” 필요
  변경 한 번이 시스템 전체 엔트로피를 재배분

“형태적 엔트로피(파일 개수)는 2인데, 실재적 엔트로피(독립 상태 공간 크기)는 1”
유지보수 지옥의 열역학적 원인
거짓된 독립성은 엔트로피 계산을 속이는 사기 행위다.

최소한의 “채널 용량”만 사용해 정보를 전달
  인터페이스(중력, 이벤트, 메시지) = 저엔트로피 신호
  발신자와 수신자가 서로를 모른다는 것은 상호 정보 I(A;B) ≈ 0이라는 뜻

각 축이 독립적이므로, 전체 상태 공간의 부피(가능한 조합 수)가 최대곱으로 증가
한 모듈이 다른 모듈의 엔트로피를 오염시키면, 전체 시스템의 가능한 미래 상태 수가 감소

Semantic Elevation
  raw function은 결과를 돌려줄 뿐이지만
  이벤트는 "이 변화가 일어났다"는 맥락과 의도를 얹는다.

다층 전파 스택(Multi-Layered Propagation Stack)
  OSI 7계층처럼 아래층은 속도와 안전을, 위층은 유연성과 의미를 담당한다.
  Layer 0~1 (원시 신호층): Raw 포인터 + 타입 안전. 가장 빠른 “전기 신호” 수준.
  Layer 2 (멀티캐스트층): 한 번의 변화가 여러 곳으로 동시에 퍼지는 “파동”.
  Layer 3~4 (안전·동적층): Weak 기반 자동 정리. 실시간 게임에서 필수인 신뢰성 계층. 객체가 사라지면 자동으로 신호가 끊어지는 “자동 회로 차단기”.
  Layer 5 (Reflection층): Dynamic Delegate + Reflection + BlueprintAssignable. 

C++만으로 충분했던 시대 -> 에디터 플러그인, 멀티플레이어, Niagara, GAS까지 모두 연결해야 하는 시대 -> 디자이너와 프로그래머가 함께 만드는 시대

이 다층 신호 전파 시스템 때문에 UE는 하나의 작은 변화가 수천 개의 예상치 못한 아름다운 결과를 만들어내는 살아있는 세계를 만들 수 있다.
문이 열리는 순간 → 사운드 + 파티클 + AI + UI + 네트워크 + BP 이벤트 + 에디터 프리뷰까지 모두 자동으로 공명한다.
발신자는 그 결과를 미리 알 필요가 없다.
그저 “문이 열렸다”는 의미 있는 신호만 던지면 된다.

Delegate
  Delegate 자체는 “신호”가 아니다. 그냥 연락 도구다.
  기술적 정체성: “함수를 안전하게 담아두는 타입 안전 컨테이너”
  “누구에게 연락할지”를 정해 놓는 기계장치
  단일(Execute) / 멀티(Broadcast) / 동적(Dynamic)으로 세분화(Specialization)

Event
  특정 일이 일어났다는 신호 그 자체
  Event는 Delegate를 사용하지만, 의미 승격이 일어난 상태

Broadcast
  Delegate나 Event의 메서드
  알림을 동시에 퍼뜨리는 동작
  Event는 “편지 내용”, Delegate는 “주소록”

Message System
  채널(태그) + 구조화된 메시지 구조체를 주고받음
  라우팅이 핵심 (특정 채널로만 보내기, 필터링)
  Event = “불이 났다”
  Message = “불이 났고, 위치는 여기고, 크기는 이렇게 크다”

Bus System
  발신자는 버스에 “게시(publish)”만 하고, 수신자는 버스에서 “구독(subscribe)”만 한다.
  Bus = 전파탑 (공공 인프라)
  Delegate/Event = 직접 연결된 전선

세계는 변화하는 주체(Subject)와 그 변화를 관심 있게 지켜보는 관찰자(Observer)로 나뉜다.
주체는 관찰자가 누구인지 몰라도 되고, 관찰자는 주체의 내부를 몰라도 된다.
한 번의 변화가 여러 곳에 동시에 영향을 미쳐도 시스템이 안정적이다.
  Broadcast (또는 Notify) → 모든 등록된 관찰자에게 “이 일이 일어났다”는 신호 전송
  각 관찰자가 자신의 맥락에서 반응

Actor Model
  발신자와 수신자는 완전히 분리되어 있으며, 채널 태그로만 만난다.
  메시지는 payload(데이터)를 필수로 가지며, 단순 “일어났다”가 아니라 “이렇게 일어났다”를 전달한다.
  비동기·지연 처리도 자연스럽게 허용된다.

Bus System
  세계는 중앙 허브 하나가 모든 통신을 중계한다.
  시스템이 커질수록 중앙 허브가 스케일링의 핵심이다.
  발신자가 Bus에 Publish(또는 Post)
  Bus가 내부 토픽/채널 매칭
  모든 구독자에게 비동기/동기 전달 (필터·우선순위 적용 가능)

A가 B의 내부 명세(Specificities)를 알아야만 작동한다면 이 둘은 정보학적으로 하나의 파동함수로 강등된 상태
B를 관측(수정)하는 순간 A의 상태가 결정(붕괴)

A와 B는 국소적 상호작용
A->CallB()->DoSomething();
// 정보는 '접촉'을 통해 전달됨
// "A가 B를 직접 건드린다"
입자는 서로의 위치를 알아야 합니다 (position-dependent).

장 패러다임 (Event System)

// 장 세계관: A는 "장"에 교란을 만듦
OnHealthChanged.Broadcast(NewHealth);

// 장의 요동(excitation)이 공간 전체로 전파
// B, C, D는 각자의 위치에서 장의 변화를 감지

// A는 B, C, D의 위치를 몰라도 됨
// B, C, D는 A의 존재를 몰라도 됨

전자기장 비유:
├─ 전하(charge) → Event Source (Actor, Component)
├─ 장(field) → Delegate/Event infrastructure
├─ 장의 여기(excitation) → Broadcast()
└─ 다른 전하의 반응 → Listeners

J.L. Austin의 화행론(Speech Act Theory)처럼:
// 1. Locutionary Act (발화 행위) - "나는 이것을 말한다"
OnHealthChanged.Broadcast(10);  // 음성학적 발화

// 2. Illocutionary Act (발화수반행위) - "이 말의 의도는..."
DECLARE_EVENT(ACharacter, FOnDeath)  // "죽음이 일어났음"을 선언
FOnDeath OnDeath;

// 3. Perlocutionary Act (발화효과행위) - "듣는 자가 어떻게 반응하는가"
OnDeath.AddUObject(UIManager, &UUIManager::ShowGameOver);
OnDeath.AddUObject(AIController, &AAIController::CelebrateVictory);
OnDeath.AddUObject(SoundSystem, &USoundSystem::PlayDeathSound);

Delegate = 발화 가능성의 문법 (grammar of possible utterances)
Event = 실제 발화된 명제 (uttered proposition)
Listener = 화용론적 해석자 (pragmatic interpreter)

// 발신자의 세계
OnQuestComplete.Broadcast(QuestID);  
// "나는 단지 'Quest X가 완료됨'이라는 사실을 말했을 뿐이다"

// 청자들의 세계
UI: "아, 그럼 나는 축하 메시지를 보여줘야겠구나"
AI: "아, 그럼 나는 새로운 퀘스트를 줘야겠구나"
Sound: "아, 그럼 나는 팡파르를 틀어야겠구나"

// 음성 층위: 그냥 메모리 주소
void* ptr = 0x12345678;

// 형태소 층위: "이것은 함수다"
TDelegate<void(int32)> OnEvent;

// 통사 층위: "이것들을 묶어서"
TMulticastDelegate<void(int32)> OnEvent;

// 의미 층위: "이것은 '체력 변화'를 뜻한다"
DECLARE_DYNAMIC_MULTICAST_DELEGATE_OneParam(FOnHealthChanged, int32, NewHealth);

// 화용 층위: "체력이 바뀌었으니, 너희는 각자 알아서 반응해"
UPROPERTY(BlueprintAssignable)
FOnHealthChanged OnHealthChanged;

// 본래적 존재 (Authentic Being)
class IndependentModule {
    // 이 모듈은 자기 자신의 가능성을 가짐
    // 미래가 열려 있음 (thrown projection)
    // 자기-투사(Selbstentwurf): 자신의 미래를 스스로 기획함
    // 공동-존재(Mitsein): 다른 모듈과 공존하지만 그들에게 종속되지 않음
    void OnEvent(int32 Data) {
        // 나는 이 신호를 "내 방식대로" 해석한다
        // 타인의 구체성(Das Man)에 종속되지 않음
    }
};

// 비본래적 존재 (Inauthentic Being) - tight coupling
class DependentModule {
    OtherModule* other;  // 타인에 의해 규정됨
    
    void DoSomething() {
        // 내가 무엇인지는 other가 결정함
        // 나는 other의 "도구성(Zuhandenheit)"일 뿐
        if (other->specificMethod() == 특정값) {
            // other의 내부 구현에 내 존재가 달림
        }
    }
};

// 세계-내-존재
OnQuestComplete.AddUObject(this, &UMyModule::HandleQuest);

// 내가 "HandleQuest"를 어떻게 구현할지는
// 발신자가 아니라 "나"의 존재론적 구조가 결정함
void UMyModule::HandleQuest(int32 QuestID) {
    // 이것은 "나의 프로젝트(my project)"
    // 타인이 규정하지 않음
}

Event System = 모듈의 본래적 존재 회복

---

Sender → [복잡한 인터페이스] → Receiver
         ↑
      Implementation Details
      (클래스 구조, 메서드 이름, 파라미터 타입...)

Sender → [이벤트] → Receiver
         ↑
      최소 시그널
      (타입 + 파라미터만)

이벤트 = 정보 압축(Information Compression)

---

위상학적 불변량 (Topological Invariants):
1. 연결성 (Connectivity): 누가 누구와 연결되어 있는가
2. 분리성 (Separation): 독립적으로 변형 가능한가
3. 연속성 (Continuity): 변화가 매끄럽게 전파되는가

Event System의 위상학적 성질:
├─ 호모토피 불변(Homotopy invariant): 
│   구현이 바뀌어도 (continuous deformation)
│   "연결 구조"는 보존됨
│
├─ 국소적 작용(Local action):
│   각 모듈은 자기 주변만 봄 (tangent space)
│   전역 구조를 몰라도 됨
│
└─ 층 다발(Fiber bundle):
│   Base space = Event types
│   Fiber = 각 이벤트에 바인딩된 함수들
│   연결(Connection) = Add/Remove 연산

---

**각 층은 다른 종류의 불변성을 보장합니다.**
```
Layer 0: 물리적 불변성 (메모리 레이아웃)
Layer 1: 타입 불변성 (컴파일 타임 체크)
Layer 2: 구조적 불변성 (1:N 관계 유지)
Layer 3: 시간적 불변성 (생명주기 안전)
Layer 4: 의미적 불변성 (직렬화 가능)
Layer 5: 인터페이스 불변성 (BP 호환)
Layer 6: 캡슐화 불변성 (접근 제어)
Layer 7: 전역적 불변성 (시스템 조율)
```

**이것은 물리학의 대칭성과 보존 법칙과 같습니다:**
```
Noether's Theorem:
대칭성 ↔ 보존 법칙

UE Event System:
각 층의 추상화 ↔ 각각의 불변량 보존
```
