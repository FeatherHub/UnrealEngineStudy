FPlatformMisc
Platform Abstraction Layer

FGenericPlatformMisc          (공통 기본 구현, GenericPlatform/)
    └─ FWindowsPlatformMisc   (Windows 특화, WindowsPlatform/)
    └─ FAndroidPlatformMisc   (Android 특화)
    └─ ...

typedef 기반 레이어링
typedef FWindowsPlatformMisc FPlatformMisc;  // 컴파일 타임에 결정

즉 FPlatformMisc는 실제 클래스가 아니라 현재 플랫폼의 구현체에 대한 별칭(alias) 
빌드 시 타깃 플랫폼에 따라 다른 클래스를 가리키게 된다.

모든 메서드는 static이다

시스템 정보: GetCPUInfo(), NumberOfCores(), GetOSVersion()
클립보드: ClipboardCopy(), ClipboardPaste()
크래시 처리: RequestExit(), SubmitErrorReport()
환경 변수: GetEnvironmentVariable()
타이머/성능: GetCycles(), GetTicks()
디버깅: DebugBreak(), IsDebuggerPresent()


Slate
Immediate-Mode GUI 프레임워크
C++ 코드만으로 UI를 선언적으로 구성
UMG가 Slate 위에 올려진 블루프린트 친화적 래퍼

Slate DSL
Slate는 C++ 연산자 오버로딩을 활용해 마치 마크업 언어처럼 읽히는 문법을 만든다

SNew(SVerticalBox)
+ SVerticalBox::Slot()
  .AutoHeight()
  [
      SNew(STextBlock)
      .Text(LOCTEXT("Hello", "Hello World"))
      .Font(FSlateFontInfo(...))
  ]
+ SVerticalBox::Slot()
  .FillHeight(1.0f)
  [
      SNew(SButton)
      .OnClicked(this, &SMyWidget::OnButtonClicked)
  ]

Slate는 일반 3D 렌더링과 별도의 2D 렌더 패스를 가진다.

1. **레이아웃 패스**
2. **페인트 패스** - 드로우 엘리먼트 리스트 생성
3. **배치 처리(Batching)**  - 드로우 콜 병합
4. **RHI 제출** — Slate Renderer가 RHI에 제출

FSlateApplication 
Slate 전체 시스템의 중앙 싱글톤
OS와 UE 위젯 사이의 교환소(dispatcher) 역할

② 창(SWindow) 생명주기 관리

모든 SWindow 인스턴스는 FSlateApplication이 소유

    .ClientSize(FVector2D(800, 600));

FSlateApplication::Get().AddWindow(NewWindow)

AddWindow()를 호출하면 
내부적으로 FWindowsApplication::MakeWindow()가 
CreateWindowEx()를 호출해 HWND를 만들고, 
그 HWND를 FWindowsWindow가 래핑한 뒤, 
FSlateViewport와 연결된다.

③ Slate Tick 주도

매 프레임 엔진 루프에서 FSlateApplication::Tick()이 호출된다. 
입력 이벤트 드레인(큐에 쌓인 이벤트 처리)

Viewport 
Slate 창 안에서 게임 씬이 그려지는 영역은 어떻게 연결되는가?

`SWindow`는 Slate가 관리하는 창이다. 
그런데 게임 씬(3D 렌더링)은 Slate가 그리는 게 아니다. 
별도의 렌더 패스다. 
이 둘을 이어주는 연결고리가 `FSceneViewport`다.

SWindow (Slate 위젯 트리의 루트)
    └─ SViewport (Slate 위젯: "여기 직사각형 영역에 외부 렌더 결과를 끼워넣겠다")
         └─ FSceneViewport (실제 연결 핀)
              └─ UGameViewportClient::Draw() → 씬 렌더러

입력 경로 관점에서:
- 마우스 클릭이 Slate 위젯 위에 오면 → Slate가 처리
- 마우스 클릭이 게임 뷰포트 영역에 오면 → `FSceneViewport`가 받아서 → `UGameViewportClient` → `PlayerController`

FPlatformMemory     "이 기계에서 메모리란 이렇게 동작한다"
FPlatformTime       "이 기계에서 시간은 이렇게 측정된다"
FPlatformProcess    "이 기계에서 프로세스/스레드는 이렇게 다룬다"
FPlatformAtomics    "이 기계에서 원자 연산은 이렇게 보장된다"
FPlatformMisc       "그 외 이 기계의 잡다한 사실들"

FPlatformMemory::Malloc()을 호출한다는 것은
어떤 메모리 관리자 객체에게 요청하는 게 아니라
이 프로세스가 실행되는 기계의 능력을 사용하는 것이다.

HAL의 모든 메서드가 static이라는 사실은 단순한 구현 디테일이 아니다. 
이것은 HAL의 개념적 정체를 드러낸다.
HAL이 묘사하는 것은 상태가 아니다

플랫폼 특화 코드가 코드베이스 전체에 흩어지는 대신, 
HAL이라는 단 하나의 경계선 안에 봉인된다
경계선 밖의 모든 코드는 플랫폼을 모른다.

HAL 경계선 안:    ::VirtualAlloc(), ::QueryPerformanceCounter(), __builtin_expect()
HAL 경계선 밖:    FPlatformMemory::Malloc(), FPlatformTime::Cycles(), 플랫폼 몰라도 됨

HAL은 원소(element) 를 제공한다. 
그 원소로 무엇을 만들지는 위 계층의 일이다.

새로운 플랫폼을 지원하려면 
HAL 구현체 하나만 작성하면 된다.
