{

    TObjectPtr<UMaterial> DefaultMaterial;

    // Tick의 공통 진입점
    virtual void Tick(float DeltaSeconds, bool bIdleMode) = 0;

struct FWorldContext
{
    EWorldType::Type WorldType;
    //  Editor      ← 에디터 뷰포트가 보여주는 World
    //  PIE         ← Play In Editor World
    //  Game        ← 패키징 게임의 단일 World
    //  EditorPreview ← 머티리얼 프리뷰, 썸네일 등

    UWorld* ThisCurrentWorld;
    TArray<ULocalPlayer*> LocalPlayers;
    FSlatePlayInEditorInfo PIELoginInfo;
};

Slate가 아는 것은 "이 영역에 외부 렌더링 결과물을 붙여넣어라"는 것뿐이다
그 인터페이스가 SViewport다:

SWidget (모든 Slate 위젯의 베이스)
    │
    └─ SViewport          ← "여기에 렌더링 결과를 붙여넣는 슬롯"
            │
            └─ FViewportClient* ← 실제 렌더링 담당자에 대한 포인터
                                   Slate는 이것이 누군지 모른다

SViewport는 자신의 영역 크기와 위치만 알고 있다.

FSlateApplication (싱글턴)

    │

층위 3: FViewport — OS 윈도우와 렌더 타겟의 연결

SViewport(Slate)와 FViewportClient(렌더링 요청자) 사이에 FViewport가 있다

SViewport (Slate 위젯)
    │
    └─ FViewport (렌더 타겟 + 입력 라우터)
            │
            ├─ FRenderTarget    ← 렌더링 결과가 써지는 텍스처
            ├─ 크기, 해상도
            └─ FViewportClient* → 실제 렌더링 요청자

FViewport는 "이 크기의 렌더 타겟을 들고 있고, 
렌더링이 필요하면 FViewportClient에게 위임한다"는 중개자다.

"Slate" 라는 이름 아래 있는 것들:

FSlateApplication      ← 싱글턴. 이 시스템 전체의 관제탑.
SWidget과 그 서브클래스들 ← UI 위젯 계층
FSlateRenderer         ← Slate가 렌더링 스레드에 그림을 그리는 방법
FSlateWindowElementList ← 한 프레임에 그릴 드로우 콜 목록

FSlateApplication과 Slate의 관계는 엔진과 FEngineLoop의 관계와 비슷하다
FSlateApplication이 Slate 시스템의 실질적 진입점이자 생명주기 관리자다

FSlateApplication (싱글턴)
    │
    ├─ 모든 SWindow 목록 관리
    ├─ OS 메시지(마우스, 키보드) 수신 → 적절한 위젯으로 라우팅
    ├─ 매 프레임 모든 위젯 Tick
    ├─ 매 프레임 모든 위젯 Paint → 드로우 콜 생성
    └─ FSlateRenderer를 통해 렌더링 스레드에 제출

SWindow와 SViewport는 둘 다 SWidget의 서브클래스다

SWidget                          ← 모든 Slate 위젯의 베이스
    │                               크기, 위치, 이벤트 처리의 공통 인터페이스
    │
    ├─ SWindow                   ← OS 창(HWND)을 감싸는 Slate 위젯
    │       │                       HWND를 소유
    │       │                       Slate 트리의 루트 노드
    │       └─ (소유 위젯들)
    │               └─ SDockTab
    │                       └─ SLevelViewport
    │                               └─ SViewport  ←┐
    │                                              │
    └─ SViewport                 ← "이 영역은 외부 렌더링 결과물을 표시"
                                    렌더링 자체는 모름
                                    FViewport*를 통해 렌더링 결과를 받아 표시

SWindow의 특별한 점은 SWidget이면서 동시에 OS 창과 1:1 대응한다는 것이다
다른 위젯들은 순수 Slate 개념이지만, SWindow는 HWND라는 OS 자원을 들고 있다

SViewport의 특별한 점은 자신의 영역에 직접 그림을 그리지 않는다는 것이다.
"이 사각형 영역에 외부 렌더링 결과를 붙여넣는 창구" 역할만 한다.

FViewport는 Slate와 렌더링의 경계에 있는 중개자다
Slate로부터 "이 크기로 그려줘"를 받고,
FViewportClient에게 실제 렌더링을 위임하고
결과 텍스처를 다시 Slate에 돌려준다

SViewport (Slate 위젯)
    │
    └─ FViewport
            │                          
            └─ FViewportClient*

FEditorViewportClient    → 일반 C++ 객체
    소유자: SEditorViewport (Slate 위젯)
    생명주기: Slate 탭의 생명주기와 동기화
    탭 닫히면 소멸. GC 불필요.
    에디터 전용 → 패키징 게임에 없음 → UObject 인프라 낭비

UGameViewportClient      → UObject
    소유자: UEngine::GameViewport (TObjectPtr)
    생명주기: 엔진의 생명주기와 동기화
    GC 추적 필요한 이유:
        → ULocalPlayer들을 UPROPERTY로 들고 있음
        → UWorld와 연결됨
        → 네트워크 시스템과 연결됨
        → 이 연결들이 GC 추적 대상

UWorld는 게임 스레드에 산다
그런데 렌더링은 별도 렌더링 스레드에서 돌아간다

FScene은 UWorld의 렌더링 측 대리자다
UWorld가 여럿이면 FScene도 여럿이다

UStaticMeshComponent가 에디터에서 이동하면, 
게임 스레드가 렌더링 스레드에 
"이 Proxy의 Transform을 업데이트해라"는 명령을 큐에 넣는다.
