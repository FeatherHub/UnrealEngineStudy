매 Tick의 전수조사

Enhanced Input은 이벤트 구동이 아니다.

"A키가 눌렸다"는 이벤트가 파이프라인을 타고 흘러가는 것이 아니라,
UEnhancedInputLocalPlayerSubsystem이 매 Tick마다 모든 활성 매핑을 전수조사한다

활성 IMC 목록에 등록된 모든 FEnhancedActionKeyMapping 엔트리를 순회
각 엔트리는 (FKey, UInputAction*, Triggers[])의 조합
각 엔트리의 FKey에 대해 "지금 이 키의 값이 무엇인가"를 읽는다

스페이스바를 꾹 누르고 있으면 매 Tick마다 value=1.0이 읽히고, 떼고 있으면 매 Tick마다 value=0.0이 읽힌다
이벤트가 아니라 상태의 반복 샘플링이다.

이 값을 해당 엔트리의 Trigger 체인에 넣는다
Trigger는 내부 상태를 가진 객체이므로, 이전 Tick에서의 축적된 정보(Hold 시간, 이전 프레임의 키 상태 등)와 현재 값을 조합하여
ETriggerState 셋 중 하나를 반환한다: None, Ongoing, Triggered.

---

Trigger의 반환값 자체가 아니라 **이전 Tick의 반환값에서 현재 Tick의 반환값으로의 전이(transition)**가 
최종 ETriggerEvent를 결정한다
  이전 Tick에서 None이었는데 현재 Triggered이면 → ETriggerEvent::Triggered
  이전 Tick에서 None이었는데 현재 Ongoing이면 → ETriggerEvent::Started
  이전에도 Ongoing이고 현재도 Ongoing이면 → ETriggerEvent::Ongoing. 
  이전에 Ongoing이나 Triggered였는데 현재 None이면 → ETriggerEvent::Completed (또는 조건에 따라 Canceled)

즉, Trigger가 보관하는 것은 "발화 여부"가 아니라 "현재 상태"이고, 상태 전이가 이벤트를 생성한다

PlayerController는 BuildInputStack()을 통해 현재 유효한 UInputComponent들의 우선순위 정렬 배열을 구성한다
  PlayerController 자신의 InputComponent
  현재 Possess하고 있는 Pawn의 InputComponent
  PushInputComponent()를 호출하여 삽입된 액터의 InputComponent(UI 위젯이나 차량 등)

InputMapping + Context
입력 매핑들의 맥락

InputMapping(FKey→IA+Trigger 엔트리)을 묶어서 하나의 단위로 활성화/비활성화할 수 있게 하는 껍질

"하나의 단위로" => 일관성, 원자성

개별 행을 하나씩 삭제하고 새 행을 하나씩 삽입 
매핑이 50개면 50번의 원자적 조작이 필요
그 중간 상태에서 일관성이 깨질 수 있다

Context는 "이 번역 규칙들이 유효한 상황의 이름"

---

Context
해석 함수의 암묵적 매개변수

interpretation(symbol, context) → meaning

기호만으로 의미가 결정되면 context-free
기호 외의 정보가 필요하면 context-sensitive

---

CPU Context (Context Switch)

동일한 MOV EAX, [RBX+8] 명령이라도 RBX의 값, 페이지 테이블에 따라 다른 물리 메모리에 접근한다
Context Switch란 이 암묵적 매개변수 세트를 교체하는 것

동일한 명령어가 교체 후에는 완전히 다른 의미(다른 물리 메모리 접근)를 갖게 된다.

---

Graphics Context (D3D Device Context, OpenGL Context)

동일한 DrawIndexed(36, 0, 0) Draw Call 호출이라도, 
어떤 셰이더가 바인딩되어 있는지, 어떤 렌더 타겟에 그리는지, 블렌드 스테이트가 무엇인지에 따라
다른 시각적 결과를 산출한다

이 바인딩된 상태의 총체가 Graphics Context이다
Draw Call 자체에는 "어떤 셰이더로"라는 정보가 없다

UEnhancedPlayerInput::ProcessInputStack 

Step 1: 매핑 재구축
UEnhancedPlayerInput::RebuildControlMappings() → void

MappingContexts라는 멤버(활성 IMC 목록, FEnhancedActionKeyMapping[])를 소유
이것을 순회하여 EnhancedActionMappings(flat 배열)를 재구축한다


Step 2: 전수조사

[for each Mapping in EnhancedActionMappings]
  this->GetKeyState(Mapping.Key) → FKeyState*

  [for each Modifier in Mapping.Modifiers]
     Modifier->ModifyRaw(this, CurrentValue, DeltaTime) → FInputActionValue

  this->ActionInstanceData.FindOrAdd(Mapping.Action)


GetKeyState는 멤버인 TMap<FKey, FKeyState> KeyStateMap을 조회한다. 
Phase 0에서 갱신해둔 저장소

---

Step 3: Trigger 평가

[for each (Action, ActionData) in this->ActionInstanceData]
    [for each Trigger in ActionData.Triggers]
        Trigger->UpdateState(this, ActionData.Value, DeltaTime) → ETriggerState


Step 4: 상태 전이 → ETriggerEvent 추출

[for each (Action, ActionData) in this->ActionInstanceData]

    GetTriggerEvent(ActionData.LastTriggeredState, CurrentState) → ETriggerEvent
    
    ActionData.TriggerEvent = Event
    ActionData.LastTriggeredState = CurrentState


전부 자기 멤버 내의 연산이다
외부 접근 없음

---

Step 5: InputComponent 스택 순회 → 델리게이트 발화

[for i = InputStack.Num()-1 downto 0]
  UEnhancedInputComponent* IC = Cast<UEnhancedInputComponent>(InputStack[i])

  [for each Binding in IC->GetActionEventBindings()]
      if (Binding.Action == Action && Binding.TriggerEvent == ActionData.TriggerEvent)
          Binding.Delegate.Execute(ActionData.Value) // AMyPawn::OnJump(ActionData.Value) 실행

매개변수 전달
"이 관계는 이 호출이 지속되는 동안만 유효하다"는 관계


왜 UEnhancedPlayerInput이 InputStack을 멤버로 소유하지 않는가? 
InputStack의 구성은 매 틱 변할 수 있기 때문이다. 
PushInputComponent/PopInputComponent가 언제든 호출될 수 있고, 
Possess/UnPossess가 Pawn을 바꿀 수 있다. 

스택의 유효 수명은 한 번의 ProcessInputStack 호출 동안이다. 
다음 틱에는 다시 BuildInputStack으로 새로 구축해야 한다.


만약 PlayerInput이 InputStack을 멤버로 보유했다면, 
PlayerInput은 "이 스택이 아직 유효한가"를 매 틱 확인해야 하고, 
스택 변경 시 PlayerInput에게 통지하는 메커니즘이 필요하고,
PlayerInput이 PlayerController의 Possess 상태를 감시해야 한다


즉, PlayerInput이 자기 관심사(입력 평가)가 아닌 것(스택 유효성 관리)까지 책임져야 한다. 
매개변수 전달은 이 책임을 호출자(PlayerController)에게 남겨둔다
"스택을 올바르게 구축하는 것은 네 일이고, 나는 받은 스택을 평가만 한다."

PlayerController BuildInputStack

"이 플레이어에게 관련된 InputComponent들을 수거하라"는 소유 그래프 순회

"자기 InputComponent"는 자기 멤버이고,
"현재 Pawn의 InputComponent"는 GetPawn()(자기 멤버 참조)을 통해 접근 가능하고, 
"Push된 InputComponent"는 자기 멤버 배열이다.


UEnhancedPlayerInput ProcessInputStack

이 함수는 "물리 상태 × 매핑 규칙 × 이전 상태 → 이벤트"라는 연산

물리 상태(KeyStateMap)와 매핑 규칙(EnhancedActionMappings)과 이전 상태(ActionInstanceData)를 전부 멤버로 보유
InputStack만이 외부에서 오고, 그래서 매개변수


AddMappingContext가 서브시스템에 있는 이유

IMC의 활성/비활성은 게임 로직이 결정
게임 로직이 PlayerInput에 직접 접근하는 것은 캡슐화 위반

서브시스템은 PlayerInput의 내부 구조를 감추면서 "매핑 맥락 관리"라는 공개 인터페이스를 제공하는 파사드
