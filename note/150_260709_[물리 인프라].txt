FImplicitObject 

Implicit 은 음함수(암시적) 표면 표현

형상을 φ(x) = 0 을 만족하는 점집합으로 정의한다
S = { x ∈ ℝ³ : φ(x) = 0 }
여기서 φ는 부호거리라 안쪽은 φ < 0, 바깥은 φ > 0

Chaos::GJK/EPA/레이캐스트/오버랩이 형상 종류와 무관하게 동일 인터페이스로 돈다

Solver가 하는 일은 운동방정식·제약의 집합을 받아 그것을 만족하는 상태를 찾는 것

Chaos에서는 운동방정식(뉴턴-오일러)에 제약(비관통 접촉 φ ≥ 0, 조인트 C(x) = 0)을 얹고, 
힘을 적분해 임시 운동을 낸 뒤 
위치·속도를 반복 보정해 
모든 제약이 만족될 때까지 = 수렴할 때까지 돌린다. 

Chaos::FPBDRigidsSolver 는 위치 수준에서 풀고 속도를 역산한다.
형식으로 쓰면 동역학 x_{t+1} = integrate(x_t, F) 아래에서 C(x) = 0(부등식 포함)을 만족하는 x를, 
닫힌형이 아니라 반복법(Gauss-Seidel 계열의 projected iteration)으로 
잔차가 작아질 때까지 찾는 것

제약 해석은 위치와 힘을 평형을 향해 이완시킨다

---

함수 f: ℝ³ → ℝ 의 level set은 f(x) = c 를 만족하는 점집합
φ를 3D 격자에 샘플링해 저장하는 구체 구현

ImplicitObject는 암시적 표면이라는 일반 개념
LevelSet은 φ를 닫힌형 수식이 아니라 격자 샘플로 저장한 특수 구현

구는 해석적 φ = |x − c| − r 을 갖지만 LevelSet은 표로 된 φ를 갖는다
해석적 프리미티브로 표현할 수 없는 임의·유기적 형상(파편, 스키닝된 형상)에 쓴다

void UWorld::Tick()
{
  UWorld::SetupPhysicsTickFunctions()          // Start/End 틱함수를 TG_StartPhysics/TG_EndPhysics 에 장전
  UWorld::RunTickGroup(TG_StartPhysics)        // -> FStartPhysicsTickFunction::ExecuteTick
  UWorld::RunTickGroup(TG_DuringPhysics)       // 게임플레이 틱이 시뮬과 겹쳐 실행
  UWorld::RunTickGroup(TG_EndPhysics)          // -> FEndPhysicsTickFunction::ExecuteTick
}

{
  FPhysScene_Chaos::StartFrame()
}

void FPhysScene_Chaos::StartFrame()
{
  // GT 쪽 dirty 상태를 PT 로 마샬링
  async FGraphEventRef SolverEvent = Chaos::FPBDRigidsSolver::AdvanceAndDispatch_External(Dt)   // PT 시뮬 킥
}

void async Chaos::FPBDRigidsSolver::AdvanceSolverBy()
{
  Chaos::FPBDRigidsEvolutionGBF::AdvanceOneTimeStep()   // 서브스텝마다 1회
  Chaos::FPBDRigidsSolver::BufferPhysicsResults()       // 결과를 GT용 더블버퍼에 기록
}

void Chaos::FPBDRigidsEvolutionGBF::AdvanceOneTimeStepImpl()
{
  Integrate()                                  // 힘/중력 -> 속도 -> 임시 위치
  ComputeIntermediateSpatialAcceleration()     // 브로드페이즈 (Chaos::TAABBTree)
  DetectCollisions()                           // 내로우페이즈 GJK/EPA -> 접촉 제약 생성
  CreateConstraintGraph()                      // 접촉+조인트를 그래프로
  CreateIslands()                              // 상호 제약된 바디를 독립 아일랜드로 분할
  ApplyConstraints()                           // 아일랜드별 반복 solve: 위치 투영 후 속도 해석
}

void FEndPhysicsTickFunction::ExecuteTick()

{

  FChaosScene::SyncBodies()                    // -> FPhysScene_Chaos::OnSyncBodies()
}

{
  // dirty 프록시마다:
  FBodyInstance <- particle handle 트랜스폼      // 해석 결과를 GT 뷰로 복사
  UPrimitiveComponent 트랜스폼 갱신 (컴포넌트 이동)
  // hit/overlap 이벤트, wake/sleep 통지 발행
}

GJK
Gilbert–Johnson–Keerthi

두 볼록 도형이 겹치나를 한 점 문제로 바꾼다
민코프스키 차 A ⊖ B = { a − b : a∈A, b∈B }
이 차집합이 원점 0 을 품는 것

차집합 전체를 짓지 않고 서포트 함수 — 방향 d로 가장 먼 점 — 만 쓴다
support(d) = support_A(d) − support_B(−d)

원점을 향해 방향을 잡아 서포트 점을 하나씩 모아 
심플렉스(3D에선 최대 사면체)를 키우며 그게 원점을 가두는지 본다 

가두면 겹침, 더는 원점에 못 다가가면 분리

원점을 도형-차의 잘 고른 몇 점으로 가둘 수 있나?

EPA
Expanding Polytope Algorithm
얼마나 깊이 파고들었나(관통 깊이·접촉 노멀)

GJK가 원점을 가둔 그 심플렉스를 다면체로 삼아
원점에 가장 가까운 면에 서포트 점을 더해
부풀리길 반복해 차집합 경계에서 원점에 가장 가까운 점으로 수렴한다

그 점까지 거리가 관통 깊이, 방향이 밀어낼 노멀이다

겹친 상태에서 원점에서 풍선을 불어 표면에 가장 얕게 닿는 곳을 찾는 것
가장 적게 밀어내 떼는 방향·깊이

둘 다 서포트 함수 덕에 볼록에서만 성립한다
그래서 Chaos 지오메트리는 볼록 조각이어야 하고, 
오목한 트라이메시는 삼각형 단위(각 삼각형은 볼록)로 처리

게임이 바디를 옮길 때 새 포즈를 파티클에 쓰는 방식이 둘

키네마틱 타깃은 목표 트랜스폼을 세팅하면 솔버가 서브스텝 동안 거기로 옮기며 그 변위에서 속도를 산출한다
플랫폼을 키네마틱 타깃으로 옮기면 솔버가 그 이동에서 속도를 추론한다
그 속도 덕에 플랫폼과 위 물체의 접촉면이 움직이고 있어, 마찰·접촉으로 물체가 함께 실려 간다 
반대로 플랫폼을 텔레포트(속도 없이 직접 기록)하면 접촉면 속도가 0이라 위 물체를 안 나르고, 플랫폼만 밑에서 이동


직접 기록은 텔레포트로, 파티클의 X,R을 곧장 새 포즈로 세팅하되 속도를 산출하지 않는다 
즉시 순간이동, 중간과 상호작용 없음
이동에서 속도를 추론하지 않고 즉시 옮긴다
