CPU의 디버깅을 위한 전용 메커니즘

x86 하드웨어 브레이크포인트 레지스터(DR0~DR7)
특정 메모리 주소에 접근하거나 특정 명령어에 도달하면 CPU가 예외(trap)를 발생

EFLAGS 레지스터의 TF(Trap Flag) 비트를 설정하면 
명령어 하나를 실행할 때마다 인터럽트가 걸려서, 이른바 "single-step" 실행이 가능

더 낮은 레벨에서는 JTAG 같은 온칩 디버그 인터페이스
OS조차 부팅되기 전인 펌웨어/부트로더 단계의 디버깅에 쓰입니다

컴파일러가 -g 같은 옵션으로 빌드하면, 실행 파일 안에 디버그 정보(DWARF 포맷 등)를 생성
기계어 주소와 소스 코드 행 번호의 매핑, 
변수 이름과 그 변수가 저장된 메모리/레지스터 위치의 매핑, 
함수 호출 프레임의 구조(스택 레이아웃) 등

이게 없으면 디버거는 raw 메모리 주소와 레지스터 값만 보여줄 수 있을 뿐, 
"소스 코드의 몇 번째 줄" 같은 정보를 제공할 수 없습니다.

GDB, LLDB, WinDbg, Visual Studio Debugger 같은 도구가 위의 모든 것을 조합

---

gcc -g -o hello hello.c

`-g`를 붙이면 컴파일러가 DWARF 디버그 정보를 
ELF 바이너리의 `.debug_info`, `.debug_line` 등의 섹션에 기록합니다.

gdb ./hello
(gdb) break main
(gdb) run
(gdb) next
(gdb) print x
$1 = 42

break main을 치면, GDB는 디버그 심볼에서 main 함수의 시작 주소를 찾아낸 뒤, 
그 주소의 원래 명령어를 int 3(0xCC, 소프트웨어 브레이크포인트 명령어)으로 교체
이건 ptrace의 메모리 쓰기 기능으로 이루어집니다.

run을 치면, GDB가 fork + exec으로 자식 프로세스를 만들고 ptrace로 추적을 시작합니다
프로그램이 main의 첫 명령어에 도달하면 CPU가 int 3을 만나 트랩을 발생시키고
커널이 SIGTRAP을 생성하며
ptrace를 통해 GDB에 제어가 넘어옵니다

next를 치면, GDB는 디버그 심볼의 행 번호 테이블을 참조해서 
"다음 소스 행"에 해당하는 기계어 주소를 알아내고, 거기까지 실행시킵니다
내부적으로는 CPU의 single-step(TF 플래그)이나 임시 브레이크포인트를 조합해서 구현됩니다.

print x를 치면, GDB는 디버그 심볼에서 변수 x가 현재 스택 프레임의 어느 오프셋(또는 어느 레지스터)에 저장되어 있는지를 찾아내고,
ptrace로 그 값을 읽어와서 42라고 보여줍니다.

소프트웨어 브레이크포인트
특정 주소에 브레이크포인트를 설정하면, 그 주소에 있던 원래 명령어의 첫 바이트를 저장해두고, 그 자리에 0xCC(int 3)를 써넣습니다

CPU가 int 3을 실행하고, 
이는 인터럽트 벡터를 통해 
커널의 트랩 핸들러로 점프합니다.

커널은 해당 프로세스에 SIGTRAP을 보내고
ptrace로 연결된 디버거가 이를 수신합니다

디버거는 원래 바이트를 복원하고, 프로그램 카운터를 한 바이트 뒤로 돌려놓은 뒤, 사용자에게 제어를 넘깁니다

하드웨어 브레이크포인트
소프트웨어 방식은 메모리를 수정해야 하므로, ROM이나 읽기 전용 영역에는 쓸 수 없습니다

CPU의 디버그 레지스터에 감시할 주소를 설정하면, 
CPU가 매 명령어 실행 시 해당 주소인지 비교하고, 일치하면 트랩을 겁니다

DWARF 디버그 정보는 ELF 바이너리 안에 들어갑니다
debug_info, .debug_line 같은 섹션이 실행 파일 자체에 포함됩니다
그래서 -g로 빌드하면 바이너리 크기가 커집니다

PDB는 실행 파일(.exe/.dll)과 별도의 파일로 생성됩니다
hello.exe를 빌드하면 옆에 hello.pdb가 따로 만들어집니다
실행 파일 안에는 PDB 파일의 경로와 GUID가 적혀 있어서, 
디버거가 "이 exe에 맞는 pdb가 어디 있는지" 찾을 수 있습니다.


고객 환경에서 크래시 덤프가 올라왔을 때 그 덤프에 PDB를 매칭시켜 사후 분석이 가능합니다
Visual Studio가 디버깅 중에 자동으로 Microsoft 심볼 서버에서 OS 라이브러리의 PDB를 다운로드받는 것이 이 구조 덕분

---

int main(void) { ... } 
cl /Zi hello.c /link /DEBUG

**A. 함수 심볼 정보**

main이라는 함수가 있고, 
코드 영역에서 주소 0x00401000부터 시작하며, 
길이는 20바이트다." 

디버거가 `break main`을 처리할 때, 바로 이 정보로 "main = 0x00401000"을 알아냅니다. 

크래시 덤프에서 instruction pointer가 0x00401010을 가리키고 있으면, 
디버거는 이 범위 정보를 보고 "main 함수 안, 시작 지점으로부터 +0x10 위치"라고 알려줍니다.

---

int x = 42;
return x;

행 번호 매핑
주소 0x00401000  →  hello.c 3번째 줄  (int x = 42;)
주소 0x00401007  →  hello.c 4번째 줄  (return x;)

지역 변수 정보
main 함수 안에 x라는 변수가 있고, 타입은 int(4바이트)이며, 스택 프레임에서 [rbp - 4] 위치에 저장된다

디버거는 rbp 레지스터 값을 읽고, 거기서 4를 빼서, 
그 메모리 주소의 4바이트를 읽어서, int로 해석해서 
42라고 표시할 수 있습니다.

CPU가 디버그 트랩을 실행
해당 OS 인터럽트 벡터로 점프
OS가 전처리 후 디버거 호출
디버거는 instruction pointer의 현재 값을 얻고
PDB의 행 번호 테이블에서 "이 주소 = 소스 몇 번째 줄"을 조회
"이 함수의 이 시점에서 어떤 변수가 어디에 있는지"를 조회

---

PDB는 프로그램에 등장하는 모든 타입의 구조를 기록합니다

Type 1000: struct Account
  오프셋 0,  필드 "name",    타입: char[32]
  오프셋 32, 필드 "balance", 타입: int
  전체 크기: 36 바이트

디버거에서 `my_acc`를 조사할 때 디버거가 보는 건 메모리의 36바이트 덩어리일 뿐입니다. 
0x41 0x6C 0x69 0x63 0x65 0x00 ... 이런 raw 바이트입니다. 
PDB의 타입 정보가 있어야 "처음 32바이트는 name이라는 char 배열이고 'Alice'라고 읽으면 되고, 
오프셋 32부터 4바이트는 balance라는 int이고 1500이다"라고 해석할 수 있습니다.

함수 시그니처와 파라미터 정보
함수: deposit
  반환 타입: void
  파라미터 1: "acc",    타입: Account* (포인터), 위치: rcx 레지스터
  파라미터 2: "amount", 타입: int,               위치: edx 레지스터
  코드 범위: 0x00401050 ~ 0x00401070

Windows x64 호출 규약에서 첫 번째 인수는 rcx, 두 번째는 rdx에 들어갑니다. 
PDB가 이걸 기록하고 있어야 디버거가 `deposit` 안에서 멈췄을 때 
`acc`의 값(포인터 주소)과 `amount`의 값(500)을 제대로 보여줄 수 있습니다.

`acc`의 값이 예를 들어 `0x00BFFC10`이라면, 
디버거는 PDB에서 `acc`의 타입이 `Account*`임을 알고, 
다시 ``0x00BFFC10`부터 32바이트를 name으로, `0x00BFFC10 + 32`부터 4바이트를 balance로 읽습니다. 
타입 정보가 포인터를 따라가면서 연쇄적으로 적용되는 것입니다.

---

콜 스택 되감기 정보 (Frame Data / Unwind Info)
#0  deposit (acc=0x00BFFC10, amount=500)  at account.c:10
#1  main ()                                at account.c:17

이걸 구성하려면 디버거가 스택을 "되감아야" 합니다. 
현재 함수의 스택 프레임에서 이전 함수의 복귀 주소를 찾고, 
그 주소가 어떤 함수에 속하는지 PDB에서 조회하고, 
그 함수의 스택 프레임 구조를 다시 PDB에서 읽어서 그 이전 함수를 찾고... 
이 과정을 반복합니다. 

PDB에는 각 함수에서 "스택 포인터가 얼마나 이동했는지, rbp가 어디에 저장되었는지, 복귀 주소가 어디에 있는지"가 기록되어 있어서 
이 되감기가 가능합니다.

이 정보가 없으면 콜 스택이 깨집니다. 최적화된 코드에서 "콜 스택을 표시할 수 없습니다" 같은 메시지를 본 적이 있다면, 
바로 이 되감기 정보가 불충분해서입니다.

---

CPU의 실행 흐름이 인터럽트/트랩에 의해 커널 코드로 전환되고
커널이 그 이벤트를 데이터 구조에 기록하고
디버거 프로세스가 시스템 콜을 통해 그 데이터를 읽거나 대상 프로세스의 상태를 수정하는 것입니다

---

x86-64 CPU에서 디버깅과 관련된 하드웨어 요소

int 3 명령어 (opcode 0xCC)
CPU가 명령어 페치 단계에서 
RIP(instruction pointer)가 가리키는 메모리에서 바이트를 읽어올 때 그 값이 0xCC이면 
CPU는 일반적인 명령어 실행 대신 다음 시퀀스를 수행합니다

현재 RIP, CS, RFLAGS를 커널 스택에 push하고
IDT(Interrupt Descriptor Table)의 3번 엔트리에 저장된 주소로 RIP를 설정합니다
이 주소는 리눅스 커널의 트랩 핸들러 함수를 가리킵니다

CPU의 특권 레벨이 Ring 3(유저)에서 Ring 0(커널)으로 전환됩니다.

이 전체 과정은 CPU 회로에 고정 배선된 동작입니다
0xCC라는 바이트 패턴이 디코더에서 특정 마이크로연산 시퀀스를 트리거하는 것이고, 
그 시퀀스가 "스택에 상태 저장 → IDT 참조 → 점프"입니다

---

EFLAGS 레지스터의 TF(Trap Flag) 비트

EFLAGS의 비트 8이 1로 설정된 상태에서 CPU가 명령어를 하나 실행 완료하면
CPU는 int 1과 동일한 시퀀스를 수행합니다
IDT의 1번 엔트리로 점프합니다 커널로 제어가 넘어갑니다
TF는 이 트랩 발생 후 자동으로 0으로 클리어됩니다

---

디버그 레지스터 DR0~DR7
DR0~DR3에 주소를 저장하고,
CPU는 매 명령어 실행 시 해당 주소와 현재 접근 주소를 비교
일치하면 IDT(Interrupt Descriptor Table) 1번 엔트리로 점프합니다

---

리눅스 커널은 부팅 시 IDT의 1번과 3번 엔트리에 자신의 핸들러 함수 주소를 등록합니다
CPU가 0xCC를 만나거나 TF 트랩이 발생하면, 실행 흐름이 이 핸들러로 도달합니다

CPU가 커널 스택에 push해둔 값들(RIP, RFLAGS 등)을 pt_regs라는 구조체로 정리합니다

해당 프로세스의 task_struct(프로세스 메타데이터)에서 ptrace 관련 플래그를 확인합니다. 
이 플래그가 설정되어 있으면, 커널은 대상 프로세스의 상태를 TASK_TRACED로 변경하고
(스케줄러가 이 프로세스를 더 이상 실행하지 않음)
task_struct에 시그널 번호(SIGTRAP = 5)를 기록하고, 
디버거 프로세스가 wait() 시스템 콜에서 블록되어 있다면 그것을 TASK_RUNNING으로 전환

---

ptrace는 하나의 시스템 콜이지만, 첫 번째 인수(request)에 따라 여러 동작을 수행합니다

PTRACE_ATTACH
커널이 대상 프로세스의 task_struct에 "이 프로세스는 디버거 PID N에 의해 추적 중"이라는 필드를 설정
이후 대상 프로세스에서 트랩이 발생하면 위의 핸들러가 이 필드를 참조해서 디버거를 깨웁니다.

PTRACE_PEEKTEXT
대상 프로세스의 페이지 테이블을 참조해서 지정된 가상 주소에 해당하는 물리 메모리의 값을 읽어 반환
디버거 프로세스와 대상 프로세스는 서로 다른 가상 주소 공간을 가지므로, 
디버거가 대상의 메모리를 직접 접근할 수 없습니다. 
ptrace가 커널 권한으로 대상 프로세스의 페이지 테이블을 경유하여 읽는 것입니다.

PTRACE_GETREGS 
대상 프로세스의 task_struct에 저장된 pt_regs 구조체의 내용을 디버거의 버퍼로 복사합니다
이 구조체를 읽으면 정지 시점의 모든 레지스터 값을 얻을 수 있습니다.

---

ptrace 외에, 리눅스는 /proc 파일시스템을 통해 
대상 프로세스의 메모리를 파일처럼 접근할 수 있는 인터페이스도 제공합니다

디버거가 /proc/[대상PID]/mem을 open()하고 특정 오프셋으로 lseek() 한 뒤 read()하면
커널이 해당 오프셋을 대상 프로세스의 가상 주소로 해석하여 그 메모리 내용을 반환합니다

---

1
GDB 프로세스가 fork() 시스템 콜을 호출합니다
자식 프로세스에서 ptrace(PTRACE_TRACEME) 시스템 콜이 호출됩니다
커널이 이 프로세스의 task_struct에 "부모에 의해 추적 중" 플래그를 설정합니다
자식 프로세스가 execve("./hello")를 호출합니다.
커널이 hello 바이너리의 ELF 헤더를 파싱하고, 새 주소 공간을 설정하고, 코드/데이터 세그먼트를 매핑합니다

GDB의 waitpid()가 반환됩니다.

2
GDB 프로세스가 ELF의 .debug_info 섹션를 읽어서
"main" 문자열과 연관된 주소 값(예: 0x401106)을 얻습니다
GDB가 ptrace(PTRACE_PEEKTEXT, child_pid, 0x401106)을 호출합니다
커널이 자식 프로세스의 페이지 테이블에서 0x401106의 물리 주소를 찾고, 그 위치의 8바이트를 반환합니다

3
GDB가 ptrace(PTRACE_CONT, child_pid)를 호출합니다
커널이 자식의 상태를 TASK_RUNNING으로 변경합니다
스케줄러가 자식 프로세스를 선택하면, CPU가 pt_regs에서 레지스터를 복원하고
Ring 0 → Ring 3으로 전환하여 유저 코드 실행을 시작합니다

CRT 초기화 코드를 거쳐 RIP가 0x401106에 도달합니다
CPU 명령어 페치 단계에서 0xCC를 읽습니다

CPU가 RIP/CS/RFLAGS를 커널 스택에 push하고, IDT[3]의 주소로 점프합니다.

커널 트랩 핸들러가 실행됩니다
자식의 task_struct에서 추적 플래그를 확인하고, SIGTRAP을 기록하고, 자식을 TASK_TRACED로 전환하고
GDB의 waitpid()를 깨웁니다
GDB의 waitpid()가 반환값으로 "자식이 시그널 5(SIGTRAP)로 정지됨"이라는 상태 코드를 받습니다

4
GDB가 .debug_info에서 "x" 위치 정보를 읽습니다. "DW_OP_fbreg -4" (프레임 베이스에서 -4 오프셋)라고 기록
GDB가 ptrace(PTRACE_GETREGS, child_pid)를 호출
커널이 자식의 pt_regs 구조체 전체를 GDB의 버퍼로 복사
GDB가 버퍼에서 RBP 값(예: 0x7fffffffde10)을 읽습니다
GDB가 ptrace(PTRACE_PEEKTEXT, child_pid, 0x7fffffffde10 - 4)를 호출
커널이 자식의 가상 주소 0x7fffffffde0c의 물리 메모리에서 값을 읽어 반환

---

프로그램의 어딘가에 오류가 있다
그 오류의 위치와 성격

프로그램이 만 줄이고 오류가 한 곳에 있다면
오류 위치에 대한 엔트로피는 대략 log₂(10000) ≈ 13비트

print문을 하나 찍어서 "이 지점에서는 값이 정상"이라는 결과를 얻으면, 오류 가능 영역이 대략 절반으로 줄어듭니다.
이것은 약 1비트의 정보를 얻은 것입니다

여러 변수의 값, 콜 스택의 구조, 실행 경로의 확인 — 각각이 불확실성을 줄이는 정보입니다.

잘못된 곳에 print를 찍으면 0비트에 가까운 정보를 얻습니다.

리버스 디버거는 "결과에서 원인으로" 직접 역추적할 수 있기 때문에, 
이진 탐색조차 필요 없이 오류 지점에 도달할 수 있습니다.

디버그 심볼은 컴파일이라는 비가역적 변환의 부분적 역함수를 가능하게 하는 보조 데이터
컴파일러가 소스를 기계어로 변환할 때 정보가 소실됩니다. 
변수 이름, 타입 구조, 소스 행 번호 — 이런 것들은 기계어 실행에 불필요하므로 컴파일 과정에서 버려집니다.
디버그 심볼은 이 소실된 정보를 별도로 보존한 것입니다

---

컴퓨터 시스템의 실행 계층을 생각하면 보통 이런 구조입니다
하드웨어 → 커널 → 시스템 라이브러리 → 응용 프로그램.
각 계층은 아래 계층의 서비스를 사용하여 위 계층에 추상화를 제공합니다

디버깅은 어떤 특정 계층에 속하는 것이 아니라, 모든 계층을 수직으로 관통하는 별도의 축입니다

이 구조는 실행(execution)과 평행하게 존재하는 별도의 채널, 즉 메타-실행(meta-execution) 계층이라고 볼 수 있습니다.
프로그램의 "정상적인" 실행 경로(코드 실행 → 시스템 콜 → 하드웨어 연산)와 별도로, 
그 실행을 관찰하고 제어하기 위한 경로(트랩 → 커널 핸들러 → ptrace → 디버거)가 존재합니다

---

디버그 심볼은 이 세 가지 존재 양태 사이의 대응 관계를 명시적으로 기록한 것입니다

int x = 42;는 텍스트 파일 안의 문자열
기계어 mov DWORD PTR [rbp-4], 0x2a는 메모리의 바이트 패턴
실행 중인 프로세스는 레지스터와 메모리 상태의 시간적 시퀀스

이 세 가지는 물리적으로 완전히 다른 객체인데, 
프로그래머는 이것들을 "같은 프로그램"으로 취급합니다.

"소스 3번째 줄 = 주소 0x401106 = 실행 시점 T₃"이라는 삼중 매핑

이 매핑이 없으면 세 가지는 서로 독립적인 객체로 흩어지고, 
"같은 프로그램"이라는 동일성이 기술적으로 수립될 수 없습니다.

최적화 컴파일러는 프로그램의 관찰 가능한 행동(observable behavior)을 보존하면서 내부 구조를 자유롭게 재배치합니다
"프로그램의 정체성은 내부 구조가 아니라 외부 행동에 있다"는 존재론적 입장입니다. 
그런데 디버깅은 정확히 내부 구조를 들여다보는 행위이므로, 최적화와 디버깅은 프로그램의 정체성에 대한 상충하는 관점에 기반합니다.

---

인과적 추론(causal reasoning)이라는 보편적 인지 활동의 한 구현
결과를 관찰하고, 가능한 원인의 공간을 좁혀가며, 실제 인과 경로를 확정하는" 과정
도메인에 관계없이 동일한 논리적 구조

과학적 실험. "어디에 센서를 놓을 것인가"
환자가 증상(결과)을 가지고 오면, 의사는 원인을 찾아야 합니다
법학/수사 현장 보존은 코어 덤프

---

프로그래머가 고려한 실행 경로는 유한하지만, 
프로그램이 실제로 취할 수 있는 실행 경로는 입력과 환경의 조합에 따라 사실상 무한합니다. 
버그는 프로그래머가 고려하지 않은 경로에서 발생

인식론적 기반
프로그램의 행동 공간이 프로그래머의 의도 공간보다 훨씬 크다

---

소설가가 자기 소설의 일관성을 검증하려면 자기 자신이 독자가 되어 전체를 처음부터 읽어야 합니다.
프로그래머가 프로그램을 디버깅하기 위해 CPU의 역할을 머릿속에서 직접 수행하는 것
디버거 없이, 코드를 눈으로 한 줄씩 추적하면서 변수 값을 종이에 적어가는 방식
컴퓨터가 프로그램을 실행해주기 때문에 프로그래머는 의도와 현실의 괴리를 빠르게 발견
전쟁과 평화를 처음부터 끝까지 "실행"하는 데(읽는 데) 수십 시간

프로그래머는 "이 변수가 변경되는 순간 멈춰라"라는 조건을 설정
"안드레이의 위치가 변하는 모든 장면에서 멈춰라"라는 조건을 소설에 걸 수 있는 방법은 없습니다

실질적으로 검색(search)의 문제
워드프로세서에서 "안드레이"를 검색하면 그가 등장하는 모든 위치를 찾을 수 있습니다

---

프로그램에서 변수의 값은 메모리에 명시적으로 존재합니다
소설에서 인물의 "현재 위치"는 명시적으로 저장된 곳이 없습니다

소설에서 "세계 상태"는 독자의 해석으로만 존재하며, 텍스트 자체에 인코딩되어 있지 않습니다
소설의 "상태"가 물리적으로 표현되어 있지 않기 때문

컴퓨터 프로그램이 이 조건을 충족하는 것은 폰 노이만 아키텍처의 근본 설계
상태가 메모리와 레지스터라는 물리적 저장소에 명시적으로 존재한다 덕분

"소설의 임의의 페이지를 펼치면 그 시점의 모든 인물 상태가 자동으로 표시되는 장치"
프로그램을 임의의 시점에서 멈추면 그 시점의 모든 변수 값이 메모리에 존재하기 때문입니다

---

디버깅 인프라는 "형식화의 배당금"이다
복잡한 체계를 구성하는 행위
작성자의 인지 한계로 인해 오류가 불가피

프로그래밍 언어가 형식 언어이기 때문에, 
실행이 기계적으로 가능하고, 상태가 물리적으로 존재하고, 변환 과정이 기록 가능하고, 
따라서 디버깅 인프라가 구축 가능합니다

소설이 자연 언어로 쓰이기 때문에, 
실행이 인간에게 의존하고, 상태가 명시적으로 존재하지 않고, 변환 과정(아이디어→텍스트)이 형식화되어 있지 않고, 
따라서 디버깅 인프라가 원리적으로 제한됩니다.
