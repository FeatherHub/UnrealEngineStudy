UPackage* LoadPackageSync(FName Name)
{
  if (이미 로드됨) return 그것;                     // 메모
  Header = Parse(Read(Name));                      // 지역변수
  for (Import : Header) LoadPackageSync(Import);   // 재귀
  껍데기 생성; 내용 채우기; PostLoad;                 // 본문
  return Root;                                     // 리턴
}

이제 기계가 돈다. 디스크에서 바이트가 도착하면

FPackageImportStore::FindOrGetImportObjectFromLocalIndex()를 거쳐, 
FGlobalImportStore::FindPublicExportObject()나 FindScriptImportObject()가 내준 
살아 있는 포인터를 프로퍼티 자리에 써넣는 순간 — 좌표가 포인터가 되는 순간

그러나 세워졌다고 끝이 아니다. 이 건물들은 남의 세계에 서 있고, 그 세계에는 법이 있다. 

그래서 로더는 만드는 족족 EInternalObjectFlags::Async로 가림막을 치고 ConstructedObjects 명단에 적어두었다. 
자기 관할에서 할 수 있는 마무리는 Event_ExportsDone()과 Event_PostLoadExportBundle()로 끝내지만,

값만 캐시하면 부족하고, 약속(진행 중인 future)도 캐시해야 동시 요청이 합쳐진다

PostLoad가 ALT 몫(Event_PostLoadExportBundle)과 
GT 몫(Event_DeferredPostLoadExportBundle)으로 갈라져 있는 이유

PostLoad는 로더의 코드가 아니라 
각 클래스가 오버라이드한 임의의 유저 코드다. 
임의의 코드는 임의의 것을 만질 수 있다 — 렌더 리소스 등록, 타이머, 다른 오브젝트 검색. 

그런 코드를 ALT에서 돌리는 것은 일반적으로 불법이다(스레드 계약 위반). 
그렇다고 전부 GT로 보내면 GT의 프레임 예산이 로딩에 잡아먹힌다.

그래서 절충이 오브젝트 단위로 일어난다. 
자기 PostLoad가 스레드에 안전하다고 스스로 선언한 클래스(IsPostLoadThreadSafe())의 것들은 ALT에서 미리 처리하고
(Event_PostLoadExportBundle, 설정상 async postload가 켜져 있을 때)

선언하지 않은 나머지만 GT행 컨베이어로 넘긴다(Event_DeferredPostLoadExportBundle)

Event_DependenciesReady

A의 Event_ProcessPackageSummary()가 헤더를 읽고 알게 됐다
나는 B와 C를 import한다

그런데 A가 지금 바로 자기 내용을 채우기 시작하면, 
직렬화 도중 B의 export를 찾는 순간 아직 없을 수 있다.

그래서 A는 자기 Package_DependenciesReady 노드에 barrier를 건다
B 몫 하나, C 몫 하나, 합계 2

그리고 그 barrier의 해제선을 
B와 C의 진행 지점(그들의 export가 세워지고 직렬화되는 지점)에 걸어둔다

B가 거기 도달하면 A의 카운터가 2→1, C가 도달하면 1→0, 
그 순간 Event_DependenciesReady가 fire된다

이 문 앞은 밖을 기다리는 구간이고, 이 문 뒤는 다시는 밖 때문에 멈추지 않는 구간이다.

동기 함수는 원래 다섯 가지 능력을 공짜로 가진다.

call stack        누가 누구를 불렀는가
local variables   중간 값은 어디에 사는가
program counter   지금 어디까지 실행했는가
return            결과는 누구에게 돌아가는가
blocking wait     필요하면 멈춰 기다릴 수 있다

---

비동기 로더는 이 다섯 가지를 잃은 세계다
잃어버린 동기 함수의 기관을 다시 만든 시스템

call stack        → PackageRequestQueue, import dependency
local variables   → FAsyncPackage2
program counter   → AsyncPackageLoadingState, EventQueue
return            → CompletedPackageRequest, completion delegate
blocking wait     → BarrierCount, dependency edge

FZenaphore의 구현

FZenaphoreWaiter::Wait()의 첫 호출은 자기를 대기자 리스트(FZenaphore::HeadWaiter)에 등록만 하고 즉시 리턴한다(SpinCount 0→1).

일감이 끊이지 않으면 두 번째 호출 전에 NotifyOne()이 도착하므로 syscall 없이 계속 도는 것

FAsyncLoadingThread2::EventSpecs:TArray<FAsyncLoadEventSpec>

라우팅표. 공정 k는 기계 Q에서 작업 F로

로딩은 여러 단계로 진행된다
헤더 읽기, 오브젝트 만들기, PostLoad
단계마다 정해야 할 게 두 가지다: 무슨 함수를 실행하나, 어느 스레드에서 실행하나

규칙은 벽에 붙인 안내판 한 장에만 적고, 티켓에는 자기가 몇 번째 단계인지만 적는다

enum EPhase { Wash, Dry, Fold, NumPhases };

struct FPhaseRule                       // FAsyncLoadEventSpec에 해당
{
  void (*Func)(FShirt*);                // 이 단계에서 할 일
  FQueue* Queue;                        // 이 단계를 처리할 작업장(스레드)
};

FPhaseRule Rules[NumPhases] =           // FAsyncLoadingThread2::EventSpecs에 해당
{
  { &DoWash, &BackroomQueue },          // 세탁 — 뒷방(ALT)
  { &DoDry,  &BackroomQueue },          // 건조 — 뒷방(ALT)
  { &DoFold, &CounterQueue  },          // 개기 — 카운터(GT)
};

struct FTicket                          // FEventLoadNode2에 해당
{
  FShirt* Shirt;
  EPhase  Phase;                        // 표의 몇 번째 줄인지만 안다
};

void Fire(FTicket* T)    { Rules[T->Phase].Queue->Push(T); }   // 준비되면 지정 작업장으로
void Execute(FTicket* T) { Rules[T->Phase].Func(T->Shirt); }   // 작업장에서 꺼내면 지정 함수로

FAsyncLoadingThread2::EventQueue:FAsyncLoadEventQueue2

struct FDepletableQueue                  // FAsyncLoadEventQueue2에 해당
{
  std::atomic<FTicket*> Inbox;           // 공용 투입함 — 아무나 꽂는다
  FTicket*             MyTray = nullptr; // 소비자 전용 트레이

  void Push(FTicket* T);                 // 아무 스레드나: Inbox에 원자적으로 건다

  FTicket* Pop()                         // 소유 스레드만 부른다는 계약
  {
    if (!MyTray)
      MyTray = Inbox.exchange(nullptr);  // 꽂힌 것 전부 떼오기 — 경쟁은 이 1회
    return TakeHead(MyTray);             // 이후는 내 트레이에서 경쟁 없이
  }
};

FAsyncLoadingThread2::DeferredDeletePackages:TMpscQueue<FAsyncPackage2*>

워크오더 객체 FAsyncPackage2는 GT와 ALT가 둘 다 만진다
다 쓴 쪽이 그 자리에서 delete하면 문제가 생긴다

해법은 도서관 반납함
이용자는 책을 서가에 직접 꽂지 않고, 반납함에 넣기만 한다
정리는 항상 사서가 한가할 때 몰아서 한다

void GameThread_DoneWith(FAsyncPackage2* P)
{
  if (--P->RefCount == 0)
    DeferredDeletePackages.Enqueue(P);        // 넣고 끝 — GT 비용은 이게 전부
}

void AsyncThread_WhenIdle()                    // ALT 루프의 한가한 틈
{
  FAsyncPackage2* P;
  while (DeferredDeletePackages.Dequeue(P))
  {
    Ledger.ReleaseRef(P->Id);                  // 계좌 잔액 차감 — 원장 주인이 하니 락 불필요
    delete P;                                  // 파괴는 오직 여기서만
  }
}

부수는 코드가 여러 스레드에 흩어져 있으면 그 모든 지점이 서로를 의식해야 하지만, 
부수는 지점을 한 스레드로 몰면 그 걱정이 통째로 사라진다

한 줄 세우기(사슬)는 모든 두 작업이 비교 가능하다는 뜻
임의의 a, b에 대해 a가 먼저거나 b가 먼저거나
제약들은 전순서(total order)가 아니라 부분순서(partial order)를 이룬다
부분순서를 그림으로 그린 것이 DAG

합류(fan-in) — P가 Q와 R을 둘 다 import하면:
    Q:직렬화완료 ──┐
                  ├──▶ P:ExportBundle_Process     (P는 둘 다 기다림 — barrier 2개)
    R:직렬화완료 ──┘

분기(fan-out) — Q를 P와 S가 둘 다 import하면:
                  ┌──▶ P:ExportBundle_Process
    Q:직렬화완료 ──┤
                  └──▶ S:ExportBundle_Process     (Q 하나의 완료가 여럿을 풀어줌)

---

위상 정렬(topological sort)

모든 간선을 존중하는(간선 u→v가 있으면 u가 v보다 앞에 오는) 한 줄 세우기를 찾는 것
부분순서를 위반 없이 전순서로 펴는 작업

Kahn의 알고리즘은 진입차수(들어오는 간선 수) 카운터를 두고, 
0인 정점들을 준비 집합에 넣고, 
하나 꺼내 처리할 때마다 이웃의 카운터를 깎는 식으로 정렬을 소비하면서 만든다

// 동기 세계의 가상 코드
UPackage* LoadPackageSync(FName Name)
{
  FHeader Header   = ParseHeader(ReadFile(Name));    // 지역변수
  TArray<UObject*> Exports;                          // 지역변수
  for (auto& E : Header.ExportMap) Exports.Add(NewObject(...));
  for (auto& E : Header.ExportMap) SerializeInto(Exports[i]);
  for (auto* O : Exports) O->PostLoad();
  return RootPackage;                                // 반환값
}

지역변수들은 스택 프레임에 살고, 함수가 끝나면 사라진다
비동기 세계에서는 이 함수가 수십 개의 이벤트 조각으로 찢겨 여러 스레드에서 띄엄띄엄 실행된다
스택 프레임이 살아남을 수 없다

그래서 지역변수 전부를 힙 객체 하나로 옮긴 것 
FAsyncPackage2

본질은 손으로 만든 코루틴 프레임
멤버는 전부 저 동기 함수의 무언가가 피난 온 것

함수 인자의 피난처

동기 함수라면 인자였을 것들
호출 시점에 확정되고, 실행 중 절대 안 바뀐다

Desc.Priority, 그리고 회신 주소들

FAsyncPackage2::Data:FAsyncPackageData
지역변수들을 한 판에 재단

동기 함수의 지역 배열들(노드, export 슬롯, import 목록)을 피난시키되
조각조각 new하지 않고 한 번의 할당에 몰아넣는다

할당 3번, 해제 3번, 메모리 사방에 흩어짐
Nodes   = new FEventLoadNode2[6];
Exports = new FExportObject[3];
Imports = new FAsyncPackage2*[2];

FAsyncPackageData 방식 — 개수를 전부 미리 아니까 한 판

uint8* Block = Malloc(6*sizeof(노드) + 3*sizeof(슬롯) + 2*sizeof(포인터));
Nodes   = TArrayView<FEventLoadNode2>(Block + 0,   6);   // 전부 view 
Exports = TArrayView<FExportObject>  (Block + ..., 3);   // 실소유자는 블록 하나

---

FEventLoadNode2[...] — 함수 본문을 찢은 조각들의 실행 예약표

동기 함수의 각 구문 블록(파싱하고, 만들고, 채우고, 후처리하고)이 찢겨 나온 조각들

패키지 단계 노드 3(Package_ProcessSummary, Package_DependenciesReady, Package_ExportsSerialized)
번들 1개 × 번들 단계 3(ExportBundle_Process, ExportBundle_PostLoad, ExportBundle_DeferredPostLoad) = 6개

Data 블록에 placement-new로 미리 박힌다

---

동기 세계에선 코드 위치가 곧 진행 상태였지만(이 줄에 왔으면 다 만들어진 것), 조각난 세계에선 슬롯마다 명시적 상태가 필요하다

비었음 / 껍데기 생성됨 / 내용 채워짐 / 실패 / 필터됨(서버 전용이라 클라에선 안 만듦 같은 경우)

TSharedPtr<FStreamableHandle> RockMeshLoadHandle;

void AMyActor::LoadRockMesh()
{
    FStreamableManager& StreamableManager = UAssetManager::GetStreamableManager();

    RockMeshLoadHandle = StreamableManager.RequestAsyncLoad(
        RockMeshAsset.ToSoftObjectPath(),
        FStreamableDelegate::CreateWeakLambda(
            this,
            [this]()
            {
                UStaticMesh* LoadedMesh = RockMeshAsset.Get();

                if (LoadedMesh && MeshComponent)
                {
                    MeshComponent->SetStaticMesh(LoadedMesh);
                }
            }));
}
