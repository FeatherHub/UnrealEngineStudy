오픈 월드가 그 전제를 깬다

모든 것을 동시에 메모리에 올리는 것은 불가능하다
여러 사람이 동시에 다른 구역을 작업
런타임에는 플레이어의 위치에 따라 서로 다른 영역이 동적으로 로드·언로드
빌드 파이프라인은 변경된 영역만 처리

---

World Composition에서는 공간을 "레벨들의 그리드"로 분할
각 서브레벨은 독립적인 .umap 패키지
스트리밍 단위 = 레벨 패키지


World Partition
모든 액터는 퍼시스턴트 레벨의 단일 데이터베이스에 저장
스트리밍 경계는 런타임에 공간 쿼리로 결정

액터는 더 이상 특정 서브레벨에 "속하지" 않는다. 
대신, 액터는 공간 좌표를 가지고, 런타임 시스템이 그 좌표를 기준으로 스트리밍 여부를 결정한다
협업을 위한 잠금 단위는 레벨이 아니라 개별 액터 또는 셀

---

스트리밍의 단위는 Cell
셀은 월드 공간을 일정 크기의 격자로 나눈 것
각 셀은 자신의 공간 영역 안에 있는 액터들의 집합을 대표

런타임에 플레이어(또는 스트리밍 소스)로부터의 거리에 따라 셀 단위로 로드·언로드


하나의 Grid는 지형과 건물 같은 주요 환경 액터를 담당하고 큰 셀 크기(예: 128m)를 쓰는 반면,
다른 Grid는 소도구나 조명처럼 세밀한 액터를 담당하고 작은 셀 크기를 쓸 수 있다. 이것은 "공간 LOD"의 편집 표현이다

---

액터가 어느 셀에 속하는지는 그 액터의 **spatial bounds**를 기준으로 계산된다. 
크기가 매우 큰 액터(예: 지형 전체를 덮는 포스트 프로세스 볼륨)는 
bIsSpatiallyLoaded = false로 설정해서 항상 로드되도록 예외 처리할 수 있다

---

Hierarchical Level of Detail(HLOD) 레이어

런타임 스트리밍이 아니라 오프라인 빌드 타임 프로세스

빌드 시 각 Grid의 셀들이 하나 이상의 HLOD 레이어로 baked

즉, 셀 안의 수천 개 액터들이 하나의 병합된 스태틱 메시와 머티리얼로 합쳐지며, 이것이 HLOD 액터로 저장

런타임에는 어떤 셀이 로드 범위 밖에 있을 때 그 셀의 실제 콘텐츠 대신 HLOD 표현이 보인다
원본 콘텐츠가 메모리에 없는 동안 그것을 대신하는 존재 (Proxy)

---

전통적인 .umap은 레벨 안의 모든 액터를 하나의 파일에 직렬화한다
두 사람이 같은 레벨의 다른 액터를 수정하면 동일한 파일에 충돌이 생긴다. (소스 컨트롤 관점)

One File Per Actor (OFPA)
각 액터가 자신만의 독립적인 .uasset 파일을 갖는다. 
퍼시스턴트 레벨 패키지는 이 개별 파일들에 대한 참조 인덱스 역할만 한다.


OFPA 액터 파일들은 __ExternalActors__ 폴더 아래에 위치하며, 
액터의 GUID를 기반으로 경로가 결정된다.

Data Layer

콘텐츠를 비공간적 기준으로 그룹화하는 메커니즘
공간이 아닌 의미(semantics)에 의한 분류


"낮" 레이어에는 낮에만 보이는 NPC와 오브젝트
"밤" 레이어에는 밤에만 활성화되는 적과 조명
"퀘스트 A 완료 후" 레이어에는 스토리 진행에 따라 달라지는 오브젝트


어떤 액터가 로드되려면 (1) 공간적으로 범위 안에 있어야 하고 (2) 속한 Data Layer가 활성화되어 있어야 한다

World Partition은 “무엇을 기준으로 주변을 로드할 것인가?”가 필요
그 기준점이 Streaming Source


대부분의 경우 플레이어가 기본 Streaming Source처럼 작동
플레이어가 이동하면 주변 셀이 로드되고, 멀어진 셀은 언로드


컷신 카메라, 순간이동 목적지, 미리 로드해야 하는 전투 지역 같은 것도 Streaming Source 개념으로 다룰 수 있어요.

이 관점에서 World Partition은 미래에 필요할 공간 데이터를 예측해서 로드하는 시스템

실무 Tip

Always Loaded Actor가 Spatially Loaded Actor를 강하게 참조하면, 스트리밍 경계가 흐려질 수 있어요
대규모 월드에서는 가능한 한 Soft Reference, ID 기반 조회, Subsystem 중재

큰 Bounds를 가진 Actor, Landscape, Spline, 거대한 Trigger Volume은 예상보다 많은 셀에 영향을 줄 수 있어요.

멀리 있는 도시, 숲, 산맥, 구조물을 계속 보여줘야 한다면 HLOD 설계가 필요

이동속도가 빠른 게임에서는 Loading Range와 예측이 중요
플레이어가 빠르게 이동하면 Streaming Source 주변 셀이 제때 로드되지 않아 hitch, pop-in, collision 부재, 낙하 문제

Unloaded
처음엔 레벨 패키지가 존재조차 하지 않는다 

Loading
스트리밍 요청이 오면 UE의 비동기 패키지 로더(FAsyncLoadingThread 또는 Zen Loader)가 .umap 파일을 디스크에서 읽기 시작한다

LoadedNotVisible
패키지 로드가 완료되면 레벨 안의 액터들이 초기화되지만 아직 UWorld의 실행 컨텍스트에 완전히 합류하지 않은 상태가 된다

LoadedVisible
이 상태에서 레벨을 Visible 상태로 전환하면 액터들이 렌더링, 틱, 물리에 등록되며 실제로 "존재"하기 시작한다


언로드할 때 즉시 패키지를 메모리에서 제거하는 것이 아니라, 먼저 Invisible로 만들고 그 다음 언로드를 진행한다

World Partition은 Level Streaming 위에 구축된 자동화 레이어이며, 
런타임에 World Partition의 모든 셀 로딩은 Level Streaming 메커니즘을 직접 구동한다

---

Level Streaming 설계는 콘텐츠를 패키지 단위로 분리하고, 그 패키지의 실행 컨텍스트 진입을 런타임에 제어한다. 
이 결정은 올바르다. 문제는 이 메커니즘이 완전히 수동이라는 것이다.

무엇이 어떤 패키지에 들어갈지, 어떤 패키지가 언제 로드될지, 그 조건을 누가 판단할지 — 이 모든 결정을 개발자가 직접 내려야 한다
서브레벨을 손으로 나누고, 스트리밍 볼륨을 손으로 배치하고, 어느 볼륨이 어느 레벨을 트리거하는지 수동으로 연결한다
결과적으로 생기는 것은 수십 개의 서브레벨 패키지, 수백 개의 스트리밍 볼륨, 이들 사이의 복잡한 수동 관계망이다

---

World Partition

Level Streaming이 수동으로 처리해야 했던 결정들을 자동화한다

World Partition은 이것을 공간 좌표 기반으로 자동 계산한다. 
액터의 위치를 보고 그것이 속할 셀을 결정하며, 빌드 시에 셀 단위 패키지를 자동으로 생성한다


"언제 어떤 패키지를 로드할 것인가"라는 질문에 Level Streaming은 스트리밍 볼륨이나 수동 코드 호출을 요구한다. 
World Partition은 이것을 스트리밍 소스와의 거리 계산으로 자동화한다.
런타임에 UWorldPartitionStreamingPolicy가 매 틱 스트리밍 소스들의 위치를 평가하고, 어느 셀이 로드 범위 안에 있는지 판단한 다음, 
그 판단 결과를 ULevelStreamingDynamic 요청으로 변환한다

---

World Partition의 결정이 ULevelStreamingDynamic으로 번역되는 순간, 
이후의 처리는 완전히 표준 Level Streaming 파이프라인이 담당한다

비동기 패키지 로더가 셀 패키지를 읽고, 상태 머신이 Unloaded → Loading → LoadedNotVisible → LoadedVisible 전이를 처리하고, 
틱마다 컴포넌트 등록이 분할 처리된다

World Partition은 이 과정을 전혀 모른다 — 요청만 넣고 결과를 기다린다.

---

수동 Level Streaming이 여전히 유효한 경우

공간 좌표가 아닌 게임 상태에 따라 로드 여부가 결정되는 콘텐츠

퀘스트 완료 후에만 나타나는 지역, 특정 이벤트가 발생했을 때만 존재하는 던전, 챕터 전환에 따라 교체되는 환경

ULevelStreamingDynamic::LoadLevelInstance()를 직접 호출

---

World Partition
  = 하나의 큰 World를 자동 Grid Cell로 나누고,
    Streaming Source 주변 Cell을 자동으로 로드·언로드하는 UE5식 월드 스트리밍 시스템

Level Streaming “맵 파일을 메모리에 로드/언로드하고 visibility를 토글하는 기능”

World Partition은 큰 월드를 위한 자동 데이터 관리와 거리 기반 Level Streaming 시스템

---

                         World Streaming 큰 문제
                                  │
                ┌─────────────────┴─────────────────┐
                │                                   │
        Traditional Level Streaming          World Partition
                │                                   │
      사람이 만든 Sublevel 단위            엔진이 만든 Cell 단위
                │                                   │
      .umap을 직접 로드/언로드             Grid / Runtime Cell 자동 스트리밍
                │                                   │
      Trigger / Volume / Blueprint          Streaming Source / Loading Range

---

Traditional Level Streaming의 답:
  "사람이 월드를 여러 Sublevel로 나누고, 직접 로드 규칙을 짠다."

World Partition의 답:
  "하나의 월드를 Actor 단위로 저장하고, 엔진이 공간 Cell로 나눠 자동 스트리밍한다."

---

World Partition에서는 보통 하나의 큰 Persistent World를 유지해요. 
그리고 엔진이 내부적으로 월드를 Grid Cell로 나눠요.

플레이어나 카메라 같은 Streaming Source 주변 Cell만 로드되고, 멀어진 Cell은 언로드돼요

---

ALandscape  (하나, 항상 로드)
├── ALandscapeStreamingProxy_5_4_0
├── ALandscapeStreamingProxy_5_5_0
│   ...


이름의 숫자 _X_Y_0은 이 분할 그리드의 좌표다. _7_7_0은 그리드 (7, 7) 위치의 Proxy이며, 
마지막 _0은 LOD 레벨 또는 섹션 인덱스다.

---

ALandscape는 World Partition 컨텍스트에서 bIsSpatiallyLoaded = false 로 설정된다. 
이것은 "공간 스트리밍 대상에서 제외하라"는 표시다. 
플레이어가 어디에 있든, ALandscape는 항상 로드되어 있다. 

왜냐하면 ALandscape는 지형 전체에 관한 메타데이터 — 섹션 크기, 컴포넌트 수, 머티리얼 레이어 정보 — 를 들고 있으며, 
각 Proxy가 자신이 무엇의 일부인지 알기 위해 이 정보가 필요하기 때문이다


반면 각 ALandscapeStreamingProxy는 bIsSpatiallyLoaded = true다. 
이것들이 World Partition의 공간 스트리밍 대상이다. 플레이어가 접근하면 로드되고, 멀어지면 언로드된다
World Partition은 각 Proxy의 공간 바운드를 보고 어느 셀에 배치할지 결정

로드되지 않은 영역은 시각적으로 어떻게 존재하는가

Level Streaming이 "무엇이 존재하는가"의 경계를 제어한다면, 
HLOD는 존재하지 않는 것의 시각적 유령을 제공한다

원본 콘텐츠가 메모리에 없는 동안, 그것을 대신하는 경량 표현이 항상 렌더링되고 있어야 한다.

AWorldPartitionHLOD

World Partition 셀과 1:1 대응
각 셀마다 하나(또는 레이어 수만큼)의 HLOD 액터가 존재한다
이름의 좌표 X-1_Y-1_Z0은 바로 그 셀의 그리드 좌표

음수 좌표가 존재하는 것(X-1, Y-1)은 그리드의 원점이 월드 중심(0, 0)에 있고 셀들이 양방향으로 펼쳐져 있기 때문이다


원본 셀이 로드 범위 안으로 들어오면 → 그 셀의 콘텐츠가 로드되고 → AWorldPartitionHLOD는 자신을 숨긴다(SetHidden(true))
원본 셀이 로드 범위 밖으로 나가면 → 콘텐츠가 언로드되고 → AWorldPartitionHLOD가 자신을 드러낸다

두 상태 사이에 순간적인 교차 시점이 있는데, 이 전환이 팝인/팝아웃 없이 부드럽게 이어지도록 전환 거리와 페이드 처리가 설계된다

---

실제 셀 콘텐츠의 로드 반경이 예를 들어 500m라면, HLOD 액터의 로드 반경은 수 킬로미터에 달한다
멀리서 보이는 실루엣을 제공해야 하기 때문이다

---

HLOD 폴더는 모든 HLOD 레이어의 컨테이너

HLOD0은 첫 번째(가장 세밀한) HLOD 레이어다

레이어가 여러 개 있다면 HLOD1, HLOD2도 존재할 수 있으며, 각 레이어는 더 큰 셀 크기로 더 넓은 거리를 커버한다.


HLOD0_Instancing은 이 레이어에서 사용된 HLOD 빌더 타입이다
HLOD 빌더 타입은 "원본 셀의 콘텐츠를 어떻게 단순화할 것인가"의 전략이다

Instancing 방식은 셀 안의 동일한 스태틱 메시들을 인스턴스 기반으로 통합한다
원본 메시 에셋을 그대로 쓰되 드로우콜을 GPU 인스턴싱으로 합친다

Merged 방식(모든 메시를 하나의 새 메시로 굽는다)과 대비된다

Instancing은 에셋을 새로 생성하지 않아 빌드 비용이 낮고, 
Merged는 드로우콜이 단 하나가 되어 런타임 비용이 낮다.

ActorTextureStreamingBuildDataComponent_2 

텍스처 스트리밍 메타데이터 컨테이너

HLOD 액터는 빌드 타임에 이미 자신의 표시 거리와 크기가 알려져 있으므로, 이 데이터를 사전 계산해 컴포넌트로 저장해둔다
런타임에 텍스처 스트리머가 이것을 참조해서 적절한 밉맵 레벨을 결정한다.

HLOD / WorldPartitionHLOD는 **“언로드된 세계도 완전히 사라지는 것이 아니라, 먼 거리용 대리 Actor로 다시 나타난다”**

---

LOD는 보통 하나의 Mesh 내부에서 거리별로 더 단순한 모델을 쓰는 방식이에요

StaticMeshActor
└─ StaticMeshComponent
   ├─ LOD0  // 가까운 거리, 고품질
   ├─ LOD1
   ├─ LOD2
   └─ LOD3  // 먼 거리, 저품질


반면 HLOD, 즉 Hierarchical Level of Detail는 하나의 Mesh가 아니라 
여러 Actor / 여러 Component / 한 지역 전체를 묶어서 먼 거리용 대리 표현으로 바꾸는 시스템이에요


가까운 거리
  Tree_001
  Tree_002
  Rock_001
  House_001
  Landscape 조각들
  Fence_001
  Barrel_001

먼 거리
  WorldPartitionHLOD Actor 하나 또는 소수의 Proxy Component

---

LOD
  하나의 Mesh 안에서 디테일을 낮춤

HLOD
  여러 Actor들의 집합을 하나의 더 싼 표현으로 대체함

---

World Partition은 플레이어 주변 Cell만 실제로 로드하려고 해요
그런데 먼 Cell을 그냥 언로드하면 문제가 생겨요
그래서 HLOD가 들어와요.
HLOD는 World Partition의 “시야 연속성 보존 장치”라고 볼 수 있어요

---

UE의 직렬화 시스템은 월드 안의 객체를 두 범주로 나눈다

AActor에서 파생된 것들은 __ExternalActors__에 저장된다. 

AActor가 아닌 UObject — 레벨 데이터 에셋, 일부 외부화된 컴포넌트 — 는 
__ExternalObjects__에 저장된다


---

__ExternalActors__/ActionRoguelike/Map/P_TestMap/0/1L/ZUY6GZP8RVMCAPYCUE2FSY.uasset

소유 맵의 콘텐츠 경로(/Game/ActionRoguelike/Map/P_TestMap)를 그대로 미러링

0/1L/
액터의 GUID(ZUY6GZP8RVMCAPYCUE2FSY에 인코딩된)의 앞 몇 바이트를 가져와 디렉토리 경로를 결정한다
수백 개의 파일을 단일 디렉토리에 두면 대부분의 파일시스템에서 디렉토리 순회 성능이 급격히 저하된다
두 단계 해시 분산이 각 디렉토리의 항목 수를 일정 이하로 유지한다

이것은 콘텐츠-주소 지정(content-addressable) 스토리지의 고전적인 구현 패턴이다
Git의 .git/objects/ 디렉토리 구조와 동일한 원리다

---

ZUY6GZP8RVMCAPYCUE2FSY.uasset

이 GUID는 불변이다. 액터가 맵 안에서 이름이 바뀌거나 이동하거나 수정되어도 이 파일명은 바뀌지 않는다

---

GUID → 파일 경로의 결정론적 매핑

GUID의 첫 바이트들을 가져와 16진수 문자열로 변환하고, 그것으로 디렉토리 경로를 구성한다

World Partition은 "이 셀에 어떤 액터들이 있는가"를 런타임 공간 인덱스(Runtime Hash Grid)에서 GUID 목록으로 얻은 다음, 
각 GUID를 즉시 파일 경로로 변환해서 비동기 로더에 넘길 수 있다

---

World Partition이 런타임에 어떤 셀을 로드해야 한다고 결정하면, 
실제로 무슨 파일들을 로드해야 하는지 알아야 한다. 

이 정보는 빌드 타임에 생성된 Runtime Spatial Hash 데이터 구조에 인코딩되어 있다

이 구조는 그리드 셀 좌표 → 그 셀에 속하는 액터 GUID 목록의 매핑이다

---

1 Runtime Spatial Hash에서 셀 (X, Y)의 GUID 목록을 얻는다
2 각 GUID를 파일 경로로 변환한다 (__ExternalActors__/.../해시/GUID.uasset)
3 비동기 패키지 로더에 이 경로들을 요청한다
4 각 .uasset이 로드되면 그 안의 액터가 역직렬화되고 월드에 등록된다.


디렉토리의 각 파일은 이 흐름에서 하나의 원자적 로딩 단위다
Level Streaming의 "패키지 단위 로드"라는 개념이 OFPA에서는 "액터 단위 패키지"로 세분화된 것이다.
